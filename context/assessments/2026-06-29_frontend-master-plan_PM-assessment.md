# PM Assessment — Frontend Master Plan (2026-06-29)

## 0. Verdict

**The plan is excellent for shipping V1 and factually sound — and it has one systematic blind spot: it optimizes for the mid-Aug gate + Nov-25 launch and under-weights the *future* Nick named** (all platforms, maintainable + scalable, the eventual cloud/B2B/high-end expansion). The near-term plan is strong enough to execute now; the future-proofing layer needs to be added *before* the plan locks the design-system architecture and the website build, because those are the choices most expensive to reverse later.

**Recommendation:** ACCEPT as the foundation and unblock **FE-1 (dashboard live-integration)** immediately — none of the gaps below touch it. In parallel, route the plan back to the frontend lane for a **v2 revision** that folds the P0/P1 gaps (cross-platform, future-scale seams, i18n-readiness, the design-system architecture/governance) **before** the website build starts and before the design-token architecture is set.

## 3. The gaps — flaws, research depth/breadth, and improvements (prioritized)

### P0 — the future Nick explicitly named (fold before the architecture locks)

**G1 — Cross-platform / multi-device strategy is under-developed (the biggest gap).** Nick: "*on all platforms that users and/or customers will want to use both the HomeSynapse system and the website on.*" The plan is web-SPA-centric and treats **responsive as a P2 "confirm."** But smart-home dashboards are used **phone-first** in the real world, and every serious competitor (HA Companion, SmartThings, Hubitat) ships a **native mobile app** with push/geofence/widgets. The plan has no mobile-first design depth, **no PWA/installability strategy**, **no stated native-app position**, and no tablet/wall-panel design (kiosk, always-on, burn-in, large-touch). Why it matters: these are the choices most expensive to retrofit, and "responsive confirm" badly under-scopes a phone-first product.
*Recommendation:* add a **Cross-Platform section** — (a) **mobile-first as a V1 requirement**, not a P2 confirm (the hero must be excellent on a phone); (b) a **PWA strategy** (installable, offline app-shell) — high-leverage, cheap, and *on-brand for local-first* (an installable no-cloud app is a proof point); (c) an explicit **native-app POSITION** for the future (even if "web/PWA for V1, native reconsidered post-launch") plus the **architectural seam** to keep it open; (d) tablet/wall-panel. Research the Capacitor/PWA-vs-native trade for a local-first product.

**G2 — Future-scale seams for cloud/B2B/high-end are not architecturally reserved.** This is the frontend analog of the backend's standing discipline ("build non-precluding of the upscale/cloud/AIoT seams" — Nick's directive from day one). The plan reserves cloud/account *surfaces* to the post-V1 backlog (scope-correct) but does **not reserve the architectural seams**: the design system as a **versioned, publishable package**; the component library reusable across product + marketing + a future **multi-tenant/B2B fleet dashboard**; **role-based / white-label** readiness in the IA + token architecture. "Don't encode hobbyist" (W-3a) is *brand* guidance, not architecture.
*Recommendation:* add a **Future-Proofing Seams subsection** — most importantly, **platform-agnostic design tokens**: today `tokens.css` is CSS-only and cannot feed a native app or a B2B re-theme. Recommend a **single source-of-truth token file in a platform-neutral format (W3C Design Tokens / Style-Dictionary JSON) that *generates* `tokens.css`** (and later iOS/Android/B2B-theme outputs). This keeps V1 identical but makes the multi-platform + white-label future a generation step, not a rewrite. Reserve the multi-tenant/role/white-label seam (don't build it).

**G3 — Internationalization (i18n) is entirely absent (verified: zero i18n in `src/`).** For an EU/prosumer launch audience and a product meant to scale globally, i18n-retrofit is one of the most painful frontend debts. The centralized `format.ts` is *i18n-friendly architecture* — but there is no message-catalog structure, no locale/RTL/number-date-format plan, and the plan never mentions it.
*Recommendation:* state an **i18n-readiness position** — V1 ships English-only, but architect `format.ts` as a **keyed message catalog** (keys, not inline strings) and reserve the locale/RTL/format seams now. Cheap as a convention today; very expensive to retrofit across a grown surface. (This also de-hardcodes copy the same way the `productName` token does — they're the same discipline.)

### P1 — brand consistency, others' mistakes, maintainability/performance

**G4 — "Avoiding others' mistakes" is strong on explainability/trust but misses the frontend-platform mistakes.** The plan nails the trace/attribution + cloud-dependency mistakes. It misses: **(a) breaking-change churn** — Home Assistant is infamous for "every update breaks my dashboard" (Lovelace migrations, deprecations); a trust brand needs a stated **frontend stability / deprecation / design-system-versioning policy** ("we don't break your dashboard"). **(b) Performance-at-scale** — HA crawls with hundreds of entities; the plan has a *bundle* budget but no **runtime render-at-scale** plan (virtualized lists for large run/event tables, a render budget, the poll cost at scale). The contract has cursor pagination (good) but the UI must virtualize + budget. **(c) Config-as-burden** — deferred correctly (V1 is read-only observability), but the eventual authoring UX should explicitly inherit "simplicity over power."
*Recommendation:* add a short **"mistakes we will not repeat" subsection** naming these three with the disciplines (a stability/versioning policy; a render-at-scale budget + list virtualization as a V1 convention even on the curated set; the config-simplicity principle reserved for the future).

**G5 — The typeface decision (D-FE-5) is too quickly dismissed — and type is the most visible brand element.** "Coherence by discipline" means the dashboard renders in **San Francisco on Mac / Segoe on Windows / Roboto on Android** while the website is Inter — i.e., the two surfaces literally differ, and the dashboard's font *changes per user's OS*. For a brand whose thesis is consistency and craft, that is a real compromise the plan waves through ("would blow the budget"). But there's **~75 KB of headroom** (≈22–26 KB used of 100 KB), and a **subsetted Inter-variable woff2 (Latin) is ~15–30 KB.**
*Recommendation:* make D-FE-5 **evidence-based**: actually measure a subsetted Inter-variable woff2 against the live budget. If it fits (~15–25 KB), self-hosting it on the dashboard buys true cross-surface type consistency for a modest, deliberate spend — plausibly worth it for a craft/consistency brand. At minimum, don't decide it by assumption.

**G6 — Design-system maturity/governance + automated quality gates are under-specified (maintainability).** "Seamless and consistently performant" over time needs more than `tokens.css`: there's **no design-system documentation / living component catalog** (a Storybook-class artifact) for maintainability + onboarding the build lane; **no visual-regression testing**; **no automated accessibility gate** (e.g., axe-core in CI) for an AA-committed brand; and **no runtime-performance budget** (Lighthouse-CI / Core Web Vitals for the site; interaction-latency budget for the dashboard) — the `frontend.yml` gate is lint/typecheck/test/build/bundle/contract only.
*Recommendation:* add **design-system governance** (a documented, versioned design system + a component catalog as the single reference for both surfaces and the future B2B lane) and **extend the CI gate** with automated a11y (axe), visual-regression, and a perf budget. These are the machinery that keep the brand consistent and the perf "consistent" as the surface grows — exactly Nick's words.

### P2 — worth naming (lower urgency)

**G7 — Frontend error observability vs the no-telemetry brand (an unsolved, on-brand design problem).** Local-first + no-tracking means: *how do we know when the dashboard breaks in someone's home?* The plan doesn't address it. This is a genuine, brand-aligned design problem, not a contradiction to hide — the on-brand answer is a **local, user-visible/exportable diagnostic** (an error log the user can choose to send), never silent telemetry. Worth naming + a direction.

**G8 — The "live hero embed" needs a demo-data story; plus SEO + WU sizing.** The Astro live-hero embed (elegant) needs a **seeded/replayed demo fixture** to render against — it must *not* phone home to a live backend (that would contradict the no-cloud brand and add a security surface); specify the static demo-data path. Website **SEO/discoverability** (structured data, the follow-the-build funnel measured without tracking) is light. And §2.4's S/M/L sizing is fine for planning but the build lane will want the mid-Aug-critical WUs (FE-1 + the hero AA/state pass + the brand folds) called out as the hard floor (the plan implies this; make it explicit).

## 5. Disposition + next step

1. **Unblock the dashboard now.** FE-1 (live-integration) and the brand folds (FE-2 dark-default, FE-3 name-light) are unaffected by every gap above — they should proceed the moment Nick rules **D-FE-1** (dark-default). The dashboard is the mid-Aug long pole; don't hold it for the revision.
2. **Route the plan back to the frontend lane for a v2** that folds **G1–G3 (P0)** and **G4–G6 (P1)**, *before* the website build starts and before the design-token architecture is set (G2/G8 are the time-sensitive ones — they gate the website + cross-platform foundation).
3. **Hub will reconcile** the plan + this assessment into the spine (and commit, with the v12 beat) once the truncated-tail phantom goes quiescent (currently active; spine writes held).
4. **Counsel gate** the competitive/patent claims before any public copy (residual verification item, §1).

**Net:** a strong plan that ships V1 well — plus a future-proofing layer (cross-platform, neutral tokens, i18n-readiness, design-system governance, the anti-breakage + scale-perf disciplines) so the MVP's frontend choices don't have to be undone when HomeSynapse goes multi-platform, global, and into cloud/B2B. That is the same "non-precluding the future seams" discipline that has governed the backend from day one — applied to the surface a stranger touches.
