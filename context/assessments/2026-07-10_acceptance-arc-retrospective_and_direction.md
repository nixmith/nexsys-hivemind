# The Acceptance-Arc Retrospective — What We Proved, What Broke, and Where That Points

## 0. The one-paragraph version

In a single overnight arc we landed M9.5-DURb (CI-green), certified the moat's end-to-end thesis on silicon (motion → light → honestly-confirmed over measured reporting, on durable identity), survived an unplanned radio-hijacking incident that stress-tested the custody fences harder than any script we would have dared write, closed the 5b durability legs in superior variants, banked a first silicon rep of the restore-from-custody path, found one real defect (the reopen-target locator) and several precision seams — and entered the 72 h soak on final code with ZERO false CONFIRMs across ~25 verdicts and three networks. The system's honesty architecture was the hero of the night; the operator experience and the radio-ops tooling were the pain; the product direction consequence is that observability/explainability and field-reality hardening should move up the stack.

## 1. Accomplished (with the evidence pointers)

1. **M9.5-DURb: authored → audited (zero layer-2 code defects) → landed `04f5f70` → CI green.** The identity-path repair (LINKED requires ≥1 entity + deviceId reuse), the always-print rehydration INFO, the permitJoinDeadline clear, the installed-policy-first fence, the BENCH-VERIFY→SILICON-VERIFIED flips (comment-only, mechanically proven), the apply-caller census. Silicon confirmation same night.
2. **The 5b closing legs, closed better than specified.** L1: real over-the-air announces → `device_relinked` with the SAME deviceIds, zero proposals — held across restart AND across a full network loss + restore AND across a fresh re-formation. L2: the exact 5b entity ULIDs returned by the API across app restarts, a full Pi reboot, and two network swaps. **Identity survived three networks in one night.** This is AMD-99/DUR/DURb doing precisely what it was built for, under fire.
3. **§51 certified live** (event positions in the bench record): the supersession probe PASSED (superseded `command_result` 1 ms after the superseding dispatch; exactly one CT `state_confirmed`; zero CT timeouts) · identify's immediate honest UNCONFIRMED proven at 81 ms with the measured reason verbatim · brightness confirmed 0.33 s (in-corpus) · IAS ×2 twins absorbed (~15 runs, one per motion) · timeout honesty demonstrated in THREE organic classes (rejoin race · absent device ×6 full runs · the no-change class, live-proving the corpus note) · zero false CONFIRMs anywhere.
4. **The restore path's first silicon rep.** `NetworkFormation.resume()`'s re-form-from-custody branch (`zigbee.network_restored_from_parameters`) rebuilt ch20/0x9b65 from `zigbee-network.json` + the secrets-store key — and both devices returned voluntarily within 30 s. A never-exercised code path worked first try in a real incident.
5. **The honesty architecture as incident containment.** Six consecutive boots found a foreign network on the NCP and PIE'd honestly (`network_parameter_mismatch` → PERMANENT_FAILURE) — never adopted it, never clobbered it, kept the core up with the integration isolated. Add the supervisor's 3-strike backoff, the BindException fail-fast on double-launch, and honest `UNKNOWN` availability: every failure mode of the night was contained by design, not luck.
6. **Operator tooling matured mid-session**: `tools/bench.sh` (decisive-verdict launches — kills the $LOG/sleep/grep race class), the Hue-app BT reset recipe, the patched-bellows NCP-clear recipe, ModemManager disabled, journalctl adopted as a bench instrument.

## 2. Failed / pain points — honestly, including the hub's own

### Lens A — the human-software interface

- **The operator interface was raw shell for 90% of the night.** Env-var juggling, racy `sleep`/`grep` verification, multi-line paste hazards (Git Bash mangled a ~40-line heredoc outright), double launches, ^C races that voided observations. `bench.sh` fixed this LATE; it should have existed before the first bench session. **Lesson-as-direction: the operator surface IS a product surface.** The decisive-verdict pattern (launch → poll → print HEALTHY/FAILED with evidence) is the right interaction shape for everything an operator does.
- **Hub command precision failures cost real round-trips** (owned): the nohup/log-glob race; a grep vocabulary that matched `transport_failed` but not `transient_failure`/`integration.failed` (masked the supervisor-retry signature twice); a zigpy-cli subcommand that does not exist; a premature "the leave was probably sent" read. Operator blocks need the same red-first rigor as code: name the expected token AND the failure tokens, and never gate on a race.
- **Physical acts had no feedback contract.** "Did the Hue reset?" (the 6× dance failed silently, 3+ times), "when exactly did the dongle re-seat?" — both unanswerable in the moment; the kernel journal adjudicated both after the fact. **Rule for the playbook: every physical act names its observable success signal in advance, and journalctl is a first-class instrument.**
- **Operator fatigue is a real failure mode.** 3 a.m., "80% sure I did it right," a panic `sudo reboot` that voided the incident's witness window. The system tolerated all of it — that is the architecture's win — but runbooks must be designed for a tired human: fewer steps, hard go/no-go gates, zero judgment calls at night.
- **Verdicts live in SQLite and log greps.** Every §51 result tonight was read as `sqlite3` rows and token greps, interpreted by the hub. A trust product whose evidence only a PM can read has an unshipped core surface: **the explainability dashboard is no longer a nice-to-have lane; tonight is the proof of why it is the product.**

### Lens B — codebase accuracy/precision vs the real world

- **CONFIRMED DEFECT — the reopen-target locator is blind to re-enumeration.** Journal-proven: re-attach at 07:49:08 (same ttyUSB0, same serial, USB devnum 4→5); `reopen_no_target` ×4 AFTER re-attach; detection and backoff textbook; app restart recovers instantly. The primary field scenario (a bumped dongle) is exactly the unmatched one — likely PortIdentity/locator matching too instance-specific. Availability-not-trust. **Post-soak micro-WU (M9.6-RO candidate).**
- **The ch22/0xFC4D foreign-network incident — writer UNATTRIBUTED.** HA exonerated (not installed on the Pi); our app exonerated (custody untouched, no formation logged); the USB layer was demonstrably sick pre-incident (busy → invalid-descriptor → cp210x -110 storms); ModemManager was present and port-claiming. Evidence banked (bellows info dump, six PIE boots). Recurrence during the soak = live-threat signal; a single occurrence in an un-witnessed window is treated as an artifact — but the honest position is: WE DO NOT KNOW, and that is an open investigation row.
- **We had no first-party radio-ops tooling** — clearing the NCP required pip-installing an unmaintained third-party CLI and patching it live. Our own EZSP stack is the most silicon-proven v13 implementation in the room. **Direction: a small `hs-radio` tool (info / leave / custody-restore) built on OUR protocol code** turns the worst incident class of the night into a two-minute, no-internet recovery. Relatedly: the mismatch-PIE is correct, but its sanctioned REMEDY should be a documented first-party flow, not tool archaeology.
- **The restore path does not manage NWK frame counters.** We got away with it (both devices re-associated), but a restore after heavy traffic risks silent replay-drops. Hardening row: persist an outgoing-counter high-water in custody and bump on restore (the zigpy backup/restore precedent).
- **`availability` stays UNKNOWN despite flowing reports.** Conservative-honest (never false-ALIVE) but wrong-shaped for users: a working sensor reads UNKNOWN. Needs the tracker's ALIVE wiring read + likely a small WU. Blocks an honest FE-7 dashboard row.
- **The no-change confirmation class is now a live UX/design question.** Commands to already-set values honestly time out (corpus-documented, proven ×3 tonight). To a user, "turn_on → unconfirmed" on an already-on bulb reads as failure. Candidate design question for Doc 08/AMD: a distinct verdict for already-satisfied commands (pre-dispatch state comparison), or explanation-layer rendering ("already on — nothing to confirm"). One-way-door rules apply: never render it as CONFIRMED (no fresh evidence!) — this is vocabulary, not semantics. RULING NEEDED before any code.
- **Identify-verdict latency variance (81 ms → 10.2 s) exposed a serialization question**: the immediate verdict follows the dispatch's serial exchange, so a degraded/absent device delays its verdict — and if the per-adapter command thread is head-of-line-blocking, ONE deaf device may delay ALL commands' dispatches and verdicts. Needs a code read (the `[integration-cmd-zigbee-0]` threading model); a real multi-device-home precision row if confirmed.
- **One unattributed `key_establishment_failed … device=0xFFFF… VERIFY_KEY_TIMEOUT`** ~45 s post-window (honest verdict on a dangling exchange; likely a duplicate TCLK retry from the Hue rejoin). Watch: recurrence in the soak.
- **Environmental fragility recorded**: cp210x control-request timeouts under kill/relaunch stress; enumeration wedges; the udev-rule RAMP deferral still open; SD-card storage WARN standing.

## 6. What this arc proves about the method

Tests-before-implementation, frozen tokens, evidence-based audits, stop-conditions-before-the-run, and identity-in-the-log survived contact with: a hostile unknown radio event, broken third-party tooling, hardware flake, and an exhausted operator. The certification was supposed to check boxes; instead it adversarially validated the architecture. The direction that falls out is not "more features" — it is: make the honesty VISIBLE (observability), make the recovery OWNED (first-party radio ops), and make the operator surface HUMANE (decisive verdicts everywhere). That is the shape of the next arc.
