**The sequenced roadmap this lane recommends (Nick rules; each row names the decision it interacts with):**

**The three strongest reasons, in order:**

1. **It is the only candidate that pays three strategic bills with one small arc.** A Shelly Gen2 adapter simultaneously (a) makes HomeSynapse **actually multi-protocol** — turning INV-CE-04 protocol-agnosticism, the per-protocol profile-source doctrine (registry dossier §Q6), and battlefield 4's crash-isolation claim from single-example assertions into demonstrated facts ("kill the Zigbee adapter; the WiFi plugs keep working" is the demo, with no simulated-second-adapter asterisk); (b) puts the **first real energy-metering device** in the fleet — the V1 curated set's S31 Lite ZB has *no* power monitoring, so battlefield 5 (Energy Intelligence) currently runs on zero real watts; the Plus Plug US streams apower/voltage/current/aenergy the moment it adopts; and (c) the hardware is **already arriving for the bench anyway** — zero procurement pole, zero new spend, and the B3 stimulus driver work (lane (a)'s appendix) characterizes the same RPC surface the adapter will speak.
2. **It is the cheapest real adapter we will ever build, and it lands inside the runway that matters.** Sized bottom-up against the as-built seams (§6): ~4–6 WUs vs the Zigbee arc's 14, because it subtracts the entire radio substrate (ASH/EZSP codecs, ZCL/ZDO, network formation/custody, interview state machine, sleepy-device machinery — roughly 60–70 of integration-zigbee's 107 types — hub-verified census at audit) and keeps only the protocol-agnostic pattern components (factory/descriptor, adoption slice, classifier, confirmation installer, command handler). Transport is JSON-RPC 2.0 over HTTP/WebSocket — Jackson (LTD-08) and `java.net.http` (already re-exported by integration-api; `IoType.NETWORK` → virtual-thread supervision is **built but unexercised by any real adapter — M14 is its first customer**, a named integration risk (§6.1), not a freebie). On the M9 arc's measured cadence (14 WUs authored+landed 2026-07-02→08, bench-interleaved — repo pointers in §6.3) 4–6 WUs is *days* of authoring; this REC budgets **1–2 calendar weeks** to carry real-silicon bench characterization plus full-bar audits — still comfortably inside the ~Aug 16 runway.
3. **It strengthens, not dilutes, the honesty moat — and proves the moat generalizes.** Gen2 RPC gives device-originated push evidence (`NotifyStatus` over an open WebSocket) tied to command acknowledgment (`Switch.Set` returns the prior state; the status push carries the new state) — mapping cleanly onto AMD-97's characterization vocabulary as `VERIFIED_REPORTS`/`ON_CHANGE`/`CONFIRMABLE` with measured timeouts, and onto the ledger's existing `ExactMatch`/`WithinTolerance` machinery with zero core changes (the confirmation engine is deliberately protocol-agnostic — "any reporting protocol confirms"; PM-assessment §1). A second protocol confirming through the same ledger is the proof that never-false-CONFIRMED is an architecture, not a Zigbee feature. It also gives availability honesty a stronger substrate than mains-Zigbee's (a live WebSocket + ping/pong is seconds-scale bounded-staleness evidence vs the 10-minute mains ping ladder, the like-for-like comparator — the 25 h window is the battery-device case, not this one).

**The strongest counter-arguments, stated honestly (the brief's two, plus a third added at hub audit):**

1. **"WiFi-first reads as commodity — the radio craft is the moat, and the WiFi plug class is the cloud-junk class."** The brand risk is real: every cheap hub does WiFi plugs; our story so far is "we built the radio from the byte layer up." *Mitigation:* scope the class to **local-API WiFi** (Shelly with cloud disabled; ESPHome later) and let the honesty thesis do the differentiating — nobody else renders WiFi-plug commands as evidence-confirmed verdicts with per-device measured windows; the commodity device is precisely where the non-commodity verdict shows. The radio craft continues in parallel regardless (ZNP/Wave-2 is already scheduled; Thread/Z-Wave silicon is the post-launch radio work). *Acceptance:* if Nick judges the narrative cost decisive, the fallback ordering is M14 = Z-Wave-native — but it costs an M9-scale arc on the go/no-go runway and eats the elevated explainability push (retrospective §4 stack), which this lane judges strictly worse.
2. **"Deferring Matter to Q1-2027 risks the 'is it current?' question at launch."** Reviewers and forum threads will ask "does it do Matter?" on day one, and answering "no" costs credibility with exactly the early-adopter audience V1 courts. *Mitigation:* (a) the deferral is the standing, twice-confirmed position (prior-art D4, High confidence; 2026-06-27 currency: "CONFIRM V1 Matter deferral (reinforced)") grounded in verified 2026 Matter reality — fragmentation, platforms 2–4 spec versions behind, bridge-dominated fleets (§3.1); (b) the M15-era commissioning spike converts "no" into "in development, on measured evidence, arriving with the winter arc" — an honest, dated claim in the brand's own voice; (c) the MG24 dedicated-radio path (currency ruling: dedicated Thread radio later, Multi-PAN is dead) means no hardware regret. *Acceptance:* if Matter momentum spikes discontinuously before launch (e.g., a major ecosystem forces controller-side adoption), the winter arc pulls forward — the reserved seams (Doc 05 §3, Doc 18 §3.1/§4) are exactly the insurance that makes that a re-sequencing, not a re-architecture.

3. **"The null alternative: spend nothing on new integrations before the go/no-go."** M14's 4–6+ WUs — and, scarcer, its bench time — compete with the ruled post-soak stack (seed-custody → M9.6-RO → availability-ALIVE → the explainability push → B1 ⊗ M9.6-RO → Wave-2 joins) inside the same five-week window; this REC already wields exactly this argument against its own Z-Wave fallback, so it must answer it aimed at itself. *Mitigation:* M14 is sequenced **behind** the ruled stack by construction (§0: "opens when the post-soak Track-A stack clears"); the go/no-go does **not** depend on it (the Zigbee-only demo already clears the gates — Shelly is demo upside, never gate dependency); and the **slide trigger** makes the priority self-enforcing: if the ruled stack has not cleared by ~Aug 1, or any audit holds an open [REVIEW], M14 slides past the go/no-go without ceremony. *Acceptance:* if the stack runs long, V1 reaches its checkpoint exactly as scoped today and M14 becomes the first post-go/no-go arc — the sequence survives either way; only the demo upside is forfeit.

**Decision this REC asks of Nick (one-turn-rulable):** ratify the sequence above (or reorder); separately rule the three flagged sub-questions: (i) the V1 ship-tag for the Shelly adapter at go/no-go time, (ii) whether the Tier-2 proof-scenario wording ("Zigbee, Z-Wave, energy monitoring" — Six Battlefields §Proof Scenario) is amended to "multi-protocol (Zigbee + local-WiFi) + energy monitoring" for launch, with Z-Wave's proof-scenario slot moving to the professional-channel era, and (iii) the slide trigger — M14 auto-slides past the go/no-go if the ruled Track-A stack has not cleared by ~Aug 1 or any audit holds an open [REVIEW]. §8 lists the remaining open questions only Nick can rule.

## §2 — Local-WiFi devices (the M14 pick) — evidence

### 2.1 Shelly Gen2+ RPC — the protocol, verified against the official docs (all [VERIFIED-current], fetched 2026-07-10)

- **Frame:** Gen2+ devices speak JSON-RPC 2.0; the protocol is symmetric (device can call/notify the client). Requests carry `id`/`src`/`method`/`params`. Source: https://shelly-api-docs.shelly.cloud/gen2/General/RPCProtocol
- **Transports:** HTTP POST/GET (one-shot — "notifications cannot be sent and received through this channel"); a **WebSocket server on the device at `ws://<addr>/rpc`** (persistent, carries notifications); device-initiated outbound WebSocket; MQTT; UDP (no delivery guarantees). Source: https://shelly-api-docs.shelly.cloud/gen2/General/RPCChannels
- **Push semantics:** `NotifyStatus` fires on any component status change (delta form); `NotifyEvent` for non-status events; `NotifyFullStatus` for full state. A client holding an open WS receives unsolicited `NotifyStatus` — prerequisite: send at least one request frame with a valid `src` first. Source: https://shelly-api-docs.shelly.cloud/gen2/General/Notifications
- **The honesty-critical semantic:** `Switch.Set` returns `was_on` — "True if the switch was on **before** the method was executed" (https://shelly-api-docs.shelly.cloud/gen2/ComponentsAndServices/Switch). The RPC response is an *acceptance ack + prior state*, structurally incapable of masquerading as result evidence. The authoritative CONFIRMED evidence is the subsequent device-originated `NotifyStatus` carrying `switch:0.output` (or a fresh `Switch.GetStatus` readback). This is exactly the `state_reported`-then-`state_confirmed` shape the ledger already implements. [inference from the verified doc semantics]
- **Auth:** digest auth "using the SHA-256 hmac algorithm as defined in RFC7616"; username fixed `admin`; protects HTTP + WS. Source: https://shelly-api-docs.shelly.cloud/gen2/General/Authentication
- **Discovery:** mDNS `_shelly._tcp` (Gen2+-specific) + `_http._tcp`; TXT carries `gen`. Source: https://shelly-api-docs.shelly.cloud/gen2/General/mDNS/
- **Cloud-off, local intact:** `Cloud.SetConfig {enable:false}`; Shelly support (2025-05-19) confirms local control without internet — caveat scoped to battery/sleeping sensors (not mains plugs). Sources: https://shelly-api-docs.shelly.cloud/gen2/ComponentsAndServices/Cloud · https://support.shelly.cloud/en/support/solutions/articles/103000354007
- **Generations:** Gen1 = HTTP + CoIoT (different protocol, out of pilot scope); **Gen2 (Plus/Pro), Gen3, Gen4 all speak the same Gen2+ RPC** — one adapter covers the current and future Shelly line. Source: https://shelly-api-docs.shelly.cloud/gen1/ + Gen2+ docs family.

### 3.2 Honesty-thesis fit — strong for native devices, structurally diluted at bridges

- Matter's read/subscribe interaction model is **device-originated, subscription-based, versioned, and liveness-checked**: subscriptions negotiate min/max report intervals, carry per-path `DataVersion` filters, and terminate if no report arrives within the max interval — verified to SDK source (`ReadHandler.h`, `AttributePathParams.h`) [VERIFIED-current: https://handbook.buildwithmatter.com/how-it-works/interaction-model/ · https://raw.githubusercontent.com/project-chip/connectedhomeip/master/src/app/ReadHandler.h]. Invoke responses give a CASE-sessioned per-command status distinct from the subsequent state report. **For native devices this is arguably stronger confirmation evidence than Zigbee's attribute reports** (versioned + liveness-bound) [inference]. The AMD-97 vocabulary absorbs it without change: `VERIFIED_REPORTS`/`ON_CHANGE`/`CONFIRMABLE`, with invoke-status as the dispatch-level ack.
- **The bridge caveat:** a Matter bridge answers for its bridged endpoints; attestation attaches to the bridge, not the leaf device — report provenance is bridge-sourced, one hop weaker than native [inference; bridge mechanics VERIFIED-current via the openHAB binding docs]. Bridged share of the installed base is **unquantified anywhere** [genuinely unverifiable], but directionally large (Hue, Aqara, SwitchBot, Yale ride bridges; CSA's own marketing chair publicly warned against Matter-logo-on-bridged-device mislabeling [community-reported: https://www.matteralpha.com/explainer/what-the-heck-is-wrong-with-advertising-for-bridged-matter-devices]). **Consequence for the moat:** a HomeSynapse Matter integration must characterize bridged endpoints honestly (provenance recorded as bridge-attested; the never-false-CONFIRMED bound stated per-endpoint) — this is a design row for the winter arc, and a differentiator opportunity: nobody else surfaces the native-vs-bridged evidence distinction to users [inference].

### 3.3 Local-first fit — the 2026 development that changes the winter arc's cost

## §5 — MQTT as an integration class — evidence (fetched 2026-07-10)

### 5.1 The honesty trap, spec-precise

- **Retained delivery is unconditional on device liveness:** MQTT 3.1.1 §3.3.1.3 [MQTT-3.3.1-6]: "When a new subscription is established, the last retained message, if any, on each matching topic name MUST be sent to the subscriber" — identical behavior in MQTT 5.0 §3.3.1.3 [VERIFIED-current: https://docs.oasis-open.org/mqtt/mqtt/v3.1.1/os/mqtt-v3.1.1-os.html · https://docs.oasis-open.org/mqtt/mqtt/v5.0/os/mqtt-v5.0-os.html]. A retained payload is **broker memory** — the device may have been dead for months. It can seed a REPORTED/stale cache; it can never ground CONFIRMED [inference from spec].
- **QoS is per-hop (client↔broker), not end-to-end** (§4.3 both specs): a PUBACK proves broker receipt, never device action [VERIFIED-current + inference].
- **LWT** (§3.1.2.5 [MQTT-3.1.2-8]) is broker-inferred offline detection, delayed up to the keep-alive window — a useful *negative* gate (LWT-offline ⇒ suppress CONFIRMED), never positive freshness [VERIFIED-current].
- **MQTT 5 Message Expiry** (§3.3.2.3.3) expires retained copies — a partial mitigation that is opt-in per publisher and rare in practice [VERIFIED-current + inference].
- **The honest ingestion rule that falls out [inference]:** MQTT-sourced evidence counts only when received on a **live subscription** (never the retained replay burst), gated on availability topics, with per-message arrival-time freshness bounds; command outcomes cap at honest UNCONFIRMED unless a fresh post-command state message arrives inside the window. This is implementable in the AMD-97 vocabulary (`reportsAuthoritative` per convention, `BEST_EFFORT` ceilings, `CONFIRM_FROM_CACHE_OR_READBACK` degrade rules) — but it is per-convention modeling work, not one adapter.

### 5.2 Schema reality

No payload schema standard exists — topics carry opaque bytes. De-facto schemas: **Home Assistant MQTT discovery** (config topics under `homeassistant/…`; HA's own docs recommend re-publishing discovery on birth rather than trusting retained discovery — HA itself treats retained topics as fragile) [VERIFIED-current: https://raw.githubusercontent.com/home-assistant/home-assistant.io/current/source/_integrations/mqtt.markdown]; **Homie** is alive-but-marginal (spec site shows v5.0; the convention repo's last tagged release is v4.0.0, 2019-08-15; main consumer openHAB) [VERIFIED-current: https://homieiot.github.io/ · https://github.com/homieiot/convention; adoption characterization community-reported]. Tasmota has moved OFF generic HA discovery to its own `tasmota/discovery/#` topics [VERIFIED-current: https://raw.githubusercontent.com/tasmota/docs/master/docs/Home-Assistant.md]. An MQTT integration class is therefore really **N convention dialects** (HA-discovery, Tasmota, Homie, z2m payloads), each needing its own freshness/confirmability characterization [inference].

### 5.3 zigbee2mqtt-bridge implications, specifically

- z2m app = **GPL-3.0** (its stack layers zigbee-herdsman + converters = MIT) [VERIFIED-current LICENSE files]; corpus currently **5,521 devices / 583 vendors** [VERIFIED-current: https://www.zigbee2mqtt.io/supported-devices/].
- **Architectural fact:** a z2m bridge is a **second Zigbee network on its own coordinator** — it cannot share HomeSynapse's native EZSP radio [inference, architectural]. So "z2m-bridge" is not a shortcut to device breadth on our radio; it is a coexistence/migration feature for households that already run z2m. Evidence arrives broker-mediated with retained defaults; the §5.1 rules apply in full.
- **Consequence [inference]:** never build z2m-bridge as the moat-carrying Zigbee path (our native stack IS the moat — identity custody, measured confirmations, §51 certification); if ever built, it is an adoption/migration on-ramp inside the MQTT class, explicitly evidence-bounded, and the D5 adapt-the-data ruling (converter corpus → our IR, licenses pinned at ingest) remains the sanctioned way z2m *knowledge* reaches us [banked: license-feasibility assessment §4-A; D5].
- zwave-js-ui's MQTT gateway mode exists (MIT, active — v11.19.1, 2026-06-02) but its WebSocket server path is the better-evidence channel — same lesson at protocol scale [VERIFIED-current: https://github.com/zwave-js/zwave-js-ui/releases].

### 5.4 Implementation cost + Java stack

Client-side is the easy part: **HiveMQ MQTT Client (Apache-2.0, active — 1.3.14, 2026-05-05)** is the recommended JVM client; Eclipse Paho Java is EPL-2.0/EDL-1.0 dual but **maintenance-dormant** (last release 1.2.5, 2020-07-15); embedded-broker option Moquette (Apache-2.0, semi-active, 0.18 2024-12-27); external Mosquitto (EPL-2.0/BSD-3) [all VERIFIED-current: https://github.com/hivemq/hivemq-mqtt-client · https://github.com/eclipse-paho/paho.mqtt.java · https://github.com/moquette-io/moquette · https://github.com/eclipse-mosquitto/mosquitto]. The real cost is semantic (§5.1–5.2) plus a known runtime gap: one-instance-per-type [banked: PM-assessment §2 names "no second MQTT broker" as the V1 example]. **Timing verdict [inference]:** MQTT's value concentrates exactly where the Doc 18 community/data-channel wave lands (months 12–24) — long-tail DIY devices, existing z2m/Tasmota households, community-contributed convention profiles. Build it there, with the freshness rules above as design law from day one.

## §6 — The Shelly pilot, CONCRETE: a minimal Gen2 adapter mapped to the as-built seams, in WU terms

### 6.1 What the seams already give us (zero new core work)

- **SPI:** `integration-shelly` implements the frozen two-type contract — `IntegrationFactory.descriptor()/create(IntegrationContext)` → `IntegrationAdapter` (8 methods; `initialize()` no-external-I/O per INV-RF-03, blocking `run()`, idempotent `close()`, `commandHandler()`).
