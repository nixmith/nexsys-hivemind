# M2.5 Arch-Debt Retrospective — Deferred Build Gates and the PM-Side Closeout Gap

**Date:** 2026-04-11
**Auditor:** Hivemind (governance meta-session)
**Subject:** The `NO_DIRECT_TIME_ACCESS` violations shipped through M2.2 and M2.4, caught by M2.5, fixed in `d6a6065`
**Subject 2:** The 3-week gap in BCP Phase 2 (PM-side closeout) execution
**Verdict:** Two independent governance failures, compounding. Neither was a single person's mistake. Both are now fixed structurally.

---

## 1. What Happened

### 1.1 The arch-rule violations

**M2.2** (`696ac37`, 2026-04-10) introduced `Instant.now()` inside `MigrationRunner.java`, located in `com.homesynapse.persistence`.

**M2.4** (`4b20786`, 2026-04-10) introduced `System.nanoTime()` inside `JacksonWarmup.java`, also located in `com.homesynapse.persistence`.

Both are direct time-source accesses. Both live in a package that is not on the `NO_DIRECT_TIME_ACCESS` whitelist (which covers only `com.homesynapse.app..`, `com.homesynapse.platform..`, and `com.homesynapse.test..`). The ArchUnit rule `HomeSynapseArchRulesTest > noDirectTimeAccess` would have failed on `./gradlew check` for either milestone.

Neither milestone failed. The violations persisted through two commits and ~24 hours of additional work, silently, until M2.5's test run on 2026-04-11 caught them. Commit `d6a6065` resolved the debt:
- `MigrationRunner` and `DatabaseExecutor` now take an injected `Clock`
- `JacksonWarmup` was simplified to avoid the timing call
- Test-side `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` propagation was added across the persistence test module

After `d6a6065`, M2.5 (`5279e7a`) landed clean.

### 1.2 The PM-side closeout gap

The Block Completion Protocol (BCP) was authored during Phase 2. It had two phases: Phase 1 (Coder-side closeout — update MODULE_CONTEXT, coder-handoff, coder-lessons, cross-agent-notes) and Phase 2 (PM-side closeout — update PROJECT_SNAPSHOT, pm-handoff, strategic-context-map §6, weekly plan progress, drift check).

Between 2026-03-20 (Phase 2 interface-specification closeout) and 2026-04-11, BCP Phase 1 fired correctly after every M1.x and M2.x milestone — `coder-handoff.md`, `coder-lessons.md`, MODULE_CONTEXT.md files, and `cross-agent-notes.md` (fresh 2026-04-10 entry) were all updated in real time.

BCP Phase 2 did not fire after any of those milestones. Every PM-side artifact — PROJECT_SNAPSHOT.md, pm-handoff.md, strategic-context-map.md §6, master-release-plan.md, the weekly plan files — remained frozen at its 2026-03-20/24 state while code advanced through M1.a, M1.b, M1.c, M1.d, M1.e, M1.f, M1.g, M1.h, M1.i, AMD-33, M2.1, M2.i, M2.2, M2.3, M2.4, M2.5, and the arch-debt fix.

The staleness compounded across three weeks until visible drift prompted this audit.

## 2. Root Cause Analysis

### 2.1 Why the arch-rule violations shipped

The proximate cause is that neither the M2.2 nor the M2.4 coder session ran `./gradlew check`. The project's established policy is to **defer** `./gradlew check` to Nick's sandbox-external environment, because the coder's sandbox cannot reliably execute Gradle (earlier experiments showed disk-space exhaustion and classpath issues). This is a rational policy — it is not the root cause of the violations.

The root cause is that **deferred build gates were not tracked as open risks.** When a coder-handoff artifact said "build gate deferred to Nick," there was no mechanism to surface that deferral to future sessions, to track whether Nick had actually run the gate, or to block downstream milestones until a deferred gate was resolved. A deferred gate was effectively forgotten the moment the handoff was filed. Two weeks and two milestones later, nobody had to answer "did M2.2's build gate ever actually run?" because nobody was asking.

Compounding the root cause: the violations would have been caught by **any** persistence test run, not just the ArchUnit test. But the coder's own unit tests for `MigrationRunner` and `JacksonWarmup` passed in isolation, because the `NO_DIRECT_TIME_ACCESS` ArchUnit check runs in the app module's test task, not in persistence's own `:core:persistence:test` task. The coder had signal that "my tests pass" without having signal that "`./gradlew check` would pass."

### 2.2 Why the PM-side closeout gap opened

The BCP was written with "Block" vocabulary exclusively. When Phase 3 began, the Coder continued to run Phase 1 closeout — Phase 1 responsibilities are straightforwardly transferable from "Block X complete" to "Milestone M2.y complete." But Phase 2 never ran, for three independent reasons:

1. **No automatic trigger.** There was no "PM session fires when a milestone lands" automation. PM sessions fire when Nick starts one. After each coder session, Nick moved directly to the next coding task without starting a PM session, so the PM closeout gate simply did not fire.

2. **No session-start freshness check.** When a PM session eventually did fire, there was nothing in the agent's normal workflow that said "check whether the hivemind is stale relative to the codebase before doing anything else." The PM assumed the state in PROJECT_SNAPSHOT.md was current, used it as ground truth, and never discovered the drift.

3. **Vocabulary drift.** The BCP explicitly used "Block" vocabulary. When the codebase moved to "Milestone" vocabulary for Phase 3, the BCP's applicability became ambiguous. A PM session that read the BCP might reasonably conclude "this protocol is for Phase 2 blocks — Phase 3 milestones must have their own process" and not run Phase 2 closeout at all.

---

## 3. The Two Failures Compound

These two failures interlock. If either had been fixed independently, the other would have been caught.

- If **PM-side closeout had been running**, a PM session on 2026-04-10 (after M2.2 landed) would have read the coder-handoff, seen "build gate deferred," asked the question "is there a mechanism tracking this?", noticed there wasn't, and escalated. The M2.4 violation would never have been committed because M2.3 would have been blocked on the M2.2 deferral.

- If **deferred build gates had been tracked as open risks**, the M2.5 coder session would have seen "M2.2 and M2.4 have unresolved build gates" in its session preamble and either run the gates before starting or refused to proceed. M2.5's test run would have caught the violations via intent, not via accident.

Neither fix alone would be sufficient going forward — both are needed. Both are landing in the 2026-04-11 governance overhaul.

## 4. Structural Fixes (2026-04-11 Governance Overhaul)

### 4.1 BCP → WUCP generalization

The Block Completion Protocol is renamed the **Work Unit Completion Protocol** and generalized to cover both "Block X" (Phase 2) and "Milestone M{x}.{y}" (Phase 3) work units. The protocol now opens with a new rule:

> **No work unit is "done" until both WUCP phases have been executed. Completion of a work unit is a prerequisite for starting the next.**

A thin redirect is left at `block-completion-protocol.md` so existing cross-references continue to resolve.

### 4.2 Freshness preflight

A new reference file `project-manager/references/freshness-preflight.md` defines a 10-step procedure that runs at the start of every PM session (and mirrors into the Coder session). The preflight compares `PROJECT_SNAPSHOT.md`'s last-updated date against the most recent commit that touched `core/`, `platform/`, `api/`, or `app/`. If the snapshot is stale by more than one work unit, the only allowed task for the session is running WUCP Phase 2 retroactively for the last completed work unit.

The PM SKILL.md Mode 3 (Director) pre-flight now **asserts** this — the session cannot proceed to normal Director work without passing the freshness check.

### 4.3 Deferred build gate tracking

The WUCP Phase 2 checklist now includes:

> **Deferred build gate audit.** For each coder-handoff completed since the last PM-side closeout, check the "Deviations" / "Deferred" sections. Every deferred `./gradlew check` run must be logged on `pm-handoff.md` under "Open Risks" with a specific closure condition ("resolved when Nick reports a successful `./gradlew check` on or after commit X"). The risk remains open until that condition is met.

### 4.4 Test code is in scope for arch rules

The PM coding-instruction template now includes, in the "What to Watch Out For" section, an explicit reminder for Phase 3 briefs targeting modules outside `com.homesynapse.{app,platform,test}..`:

> Tests must inject `Clock` (typically `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)`). Do not use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in test classes. ArchUnit scans test code in non-whitelisted packages.

This is also codified in `coder/references/testing-standards.md`.

### 4.5 Dual skill-location sync

WUCP Phase 2 now ends with a `diff -rq` check confirming that `ClaudeFolder/nexsys-hivemind/{coder,project-manager}/` and `.claude/skills/nexsys-{coder,project-manager}/` are byte-identical. A non-empty diff fails the closeout.

## 5. Lessons for the Future

1. **Deferred ≠ resolved.** Any policy that defers a step (build gate, integration test, manual review) must be paired with a tracking mechanism that keeps the deferral visible until it is resolved. Absent tracking, deferrals become omissions.

2. **Protocols are not self-enforcing.** BCP Phase 2 was documented in the protocol file — it still did not execute, for three weeks, because nothing triggered it. A protocol that requires a human to remember to trigger it will eventually fail silently. Session-start pre-flight checks are the mechanism that converts a documented protocol into an enforced protocol.

3. **Vocabulary drift hides scope drift.** When Phase 2 ended and Phase 3 began, the project's work-unit vocabulary changed from "Block" to "Milestone" without the governance artifacts being updated. This created ambiguity about whether Phase 2 artifacts (like the BCP) applied to Phase 3 work. That ambiguity was enough to suppress the Phase 2 closeout. Rename early.

4. **Tests catch bugs your tests don't.** `MigrationRunner` and `JacksonWarmup` had passing unit tests. The bug — an ArchUnit rule violation — was caught by a test in a different module. The lesson is not "write better tests" but "the full build is a test, and anything short of the full build is an approximation." Policies that routinely skip the full build must be paired with compensating controls (tracking, reviews, or deferred-but-eventual execution).

5. **State staleness is exponentially more expensive than state freshness.** The arch-debt fix took a few hours once identified. The retrospective audit (this file) took many more hours. The governance overhaul takes an entire session. All of that work would have been zero if BCP Phase 2 had run once, ~24 hours after M2.2 landed. The lesson: invest in freshness, not in reconstruction.
