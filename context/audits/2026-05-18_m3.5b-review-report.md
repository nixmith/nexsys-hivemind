## Layer 1: Correctness Verification

### Item 1: DeadLetter record fields match V002 columns — PASS

The V002 migration (`V002__subscriber_dead_letter_queue.sql`) declares 11 columns: `dlq_id` (INTEGER PK AUTOINCREMENT), `subscriber_id` (TEXT NOT NULL), `sequence_key` (TEXT NOT NULL), `event_position` (INTEGER NOT NULL), `event_id` (BLOB(16) NOT NULL), `cause_class` (TEXT NOT NULL), `cause_message` (TEXT NOT NULL), `attempt_count` (INTEGER NOT NULL DEFAULT 1), `first_seen_at` (INTEGER NOT NULL), `last_attempt_at` (INTEGER NOT NULL), `diagnostics` (TEXT nullable). The `DeadLetter` record declares exactly 11 fields with a clean camelCase mapping to each column. Types are correct: `long` for `dlqId`/`eventPosition` (INTEGER columns), `int` for `attemptCount`, `Ulid` for `eventId` (BLOB(16)), `Instant` for `firstSeenAt`/`lastAttemptAt` (converted via `TimeConversion.toMicros`/`fromMicros` in `SqliteDeadLetterStore`), `String` for text columns, and `String` (nullable) for `diagnostics`. The compact constructor enforces non-null on all required fields and validates `eventPosition >= 0`, `attemptCount >= 1`. The Javadoc table at lines 20–34 explicitly documents the field-to-column mapping. No discrepancies.

### Item 2: CheckpointSerializer null handling — PASS

The serializer correctly handles null `staleAfter`: in `toSerializable` (line 229–257), the `staleAfter` field from `EntityState` is passed through directly to `SerializableEntityState` with no filtering. In `fromSerializable` (line 259–281), it is passed back to the `EntityState` constructor as-is. Null attribute values are preserved: `toSerializable` iterates the attributes map and explicitly handles `null` `AttributeValue` entries by writing `null` string values (lines 236–237). `fromSerializable` rebuilds attributes via `HashMap.put` (line 266), which accepts null values — the code explicitly comments "DO NOT use Map.copyOf (throws on null values)" at line 261. `Map.copyOf` does not appear anywhere in the serializer. The `CheckpointSerializerTest` has dedicated tests: `nullStaleAfterPreserved` (line 107) and `nullAttributeValuesPreserved` (line 119) both pass. The Javadoc at lines 72–78 explicitly documents the null-preservation requirement and the `HashMap.put` rationale.

### Item 3: AtomicCheckpointWriter pattern consistency — PASS

The new `writeAtomicCheckpointWithDlqPark` method (lines 213–284) follows the exact same pattern as `writeAtomicCheckpoint` (lines 128–184): (1) parameter validation on caller's thread, (2) `writeCoordinator.submit(STATE_PROJECTION, ...)`, (3) `conn.getAutoCommit()` saved, (4) `conn.setAutoCommit(false)`, (5) try block with prepared statements, (6) `conn.commit()`, (7) catch with `conn.rollback()` + LOG.error on rollback failure + rethrow, (8) finally with `conn.setAutoCommit(originalAutoCommit)`. The error handling is identical in structure — both catch `Exception`, both attempt rollback with a nested try-catch, both log at ERROR level on rollback failure with context-specific messages, and both rethrow the original exception. The only structural addition is the third prepared statement block for the DLQ upsert (step 3, lines 249–265) and the null-diagnostics handling (`setNull` for null, `setString` for non-null) which mirrors `SqliteDeadLetterStore.park` exactly. No subtle differences in the transaction management pattern.

### Item 4: SqliteDeadLetterStore upsert SQL — PASS

The `UPSERT_SQL` constant (lines 67–79) correctly uses `ON CONFLICT(subscriber_id, event_position) DO UPDATE SET` targeting the V002 `UNIQUE(subscriber_id, event_position)` constraint. The UPDATE SET clause updates exactly the right columns: `attempt_count = excluded.attempt_count`, `last_attempt_at = excluded.last_attempt_at`, `cause_class = excluded.cause_class`, `cause_message = excluded.cause_message`, `diagnostics = excluded.diagnostics`. Critically, `first_seen_at` is NOT in the UPDATE SET list — it stays frozen at the original INSERT value, which is the correct behavior (first failure time should never change on re-park). The `dlq_id` (AUTOINCREMENT PK) is also not in the INSERT column list — SQLite assigns it automatically, and it stays frozen across upserts. The INSERT binds 10 parameters matching the 10 non-PK columns in the correct order. The `AtomicCheckpointWriter.DLQ_SQL` at lines 69–81 is identical SQL — they are consistent.

### Item 5: SubscriberDlq backward compatibility — PASS

## Layer 2: Coherence Check

### Item 7: SqliteStateStore vs InMemoryStateStore behavioral parity — PASS (with minor note)

Both implement `StateStore` with identical method signatures: `get(EntityId)`, `put(EntityId, EntityState)`, `getAll()`, `clear()`. Both use `ConcurrentHashMap<EntityId, EntityState>` as the backing store. Both return `Optional.ofNullable(backing.get(entityId))` for `get()` — same behavior for unknown entities (Optional.empty). Both use `Collections.unmodifiableMap(new LinkedHashMap<>(backing))` for `getAll()` — identical defensive-copy strategy. Both validate non-null on `get`/`put` parameters. The only behavioral differences beyond the interface contract are: (1) `SqliteStateStore` eagerly loads state from a `ViewCheckpointStore` checkpoint on construction (lines 120–121), adding checkpoint-based initialization; (2) `SqliteStateStore.clear()` also resets `loadedProjectionVersion` to 0; (3) `SqliteStateStore` adds `serialize(int)`, `loadedProjectionVersion()`, and `viewName()` as package-private methods for checkpoint integration. These additions are strictly additive — the `StateStore` interface contract is satisfied identically by both implementations.

## Layer 3: Engineering Quality

### Item 11: Memory footprint of SqliteStateStore — PASS (acceptable risk)

Estimating per the Doc 03 §8 target of 10,000 entities: each `ConcurrentHashMap` entry is approximately 48 bytes (Node object header + hash + key ref + value ref + next ref). Each `EntityState` record has 9 fields: `EntityId` (~40 bytes, wraps a `Ulid` which is two longs), attributes `Map` (a `HashMap` with typically 3–10 entries, each ~100 bytes for key String + `StringValue` wrapping a String value = ~500–1000 bytes per entity), `Availability` enum ref (8 bytes), `stateVersion` long (8 bytes), three `Instant` fields (~24 bytes each = 72 bytes), nullable `staleAfter` `Instant` (24 bytes or null), `stale` boolean (padded to 8 bytes). Conservative estimate: ~1.5 KB per entity including the attributes map with 5 entries. At 10,000 entities: ~15 MB for the state map. The `ConcurrentHashMap` overhead itself (segments, bins) adds maybe 1–2 MB. Total: ~17 MB. On a Pi with `MemoryMax=2G`, this is under 1% of the heap budget — well within safe limits. Even at 100,000 entities (10x the design target), this would be ~170 MB, still under 10% of the 2 GB budget. Checkpoint serialization temporarily doubles memory (snapshot copy in `serialize()` at line 175), but only briefly. This is safe.

### Item 12: CheckpointSerializer robustness — CONCERN (non-blocking)

The serializer buffers the entire `byte[]` in memory for both serialization (`objectMapper.writeValueAsBytes` at line 173) and deserialization (`objectMapper.readValue(data, ...)` at line 201). Jackson's `writeValueAsBytes` builds the full byte array in memory; `readValue` builds the full object tree in memory. At 10,000 entities with ~1.5 KB per serialized entity, the checkpoint payload would be approximately 15–20 MB. This is acceptable for the current design target but has no guardrail: there is no size limit, no warning log, and no streaming alternative. If entities grow to have large attribute maps (e.g., 50+ attributes with long string values), the checkpoint could reach 50–100 MB, which would create GC pressure on the Pi. This is not a blocking issue because the Doc 03 §8 target is 10,000 entities and the current design explicitly chose checkpoint-based durability over per-entity SQLite writes. However, a future milestone should consider adding a WARN log when the serialized payload exceeds a configurable threshold (e.g., 10 MB).

### Item 13: AtomicCheckpointWriter code duplication — CONCERN (non-blocking)

The writer has two methods (`writeAtomicCheckpoint` and `writeAtomicCheckpointWithDlqPark`) with nearly identical transaction management logic. The autoCommit-disable / commit / rollback / autoCommit-restore-in-finally pattern is copy-pasted. The subscriber checkpoint and view checkpoint prepared statement blocks are duplicated verbatim — same SQL, same bind order, same parameter mapping. There is no shared private method for the transaction wrapper or for the two common prepared statement blocks. The drift risk is real: if a future change modifies the subscriber checkpoint SQL or bind order in one method but not the other, the inconsistency would be subtle and hard to catch in review. A shared `executeInTransaction(Connection, ThrowingRunnable)` helper and factored `bindSubscriberCheckpoint`/`bindViewCheckpoint` methods would eliminate this. Not blocking because (a) both methods have identical SQL constants (`SUBSCRIBER_SQL`, `VIEW_SQL`), (b) the DLQ SQL constant (`DLQ_SQL`) is only used by the three-way method, and (c) the existing tests cover both paths. Track as technical debt for the next cleanup pass.

This is infrastructure-grade testing for the M3.5b scope.

### Item 15: SqliteDeadLetterStore connection handling — CONCERN (non-blocking)

The store does not defensively handle the case where `DatabaseExecutor` has been shut down. If `park()` is called after `dbExecutor.shutdown()`:

- `dbExecutor.writeCoordinator().submit(...)` would attempt to submit a task to the write coordinator's executor, which is presumably shut down. The behavior depends on `WriteCoordinator`'s implementation — it may throw a `RejectedExecutionException`, which would propagate as an uncaught exception to the subscriber's virtual thread.

- Similarly, `findBySubscriber()` / `findByPosition()` / `countBySubscriber()` would call `dbExecutor.readExecutor().execute(...)` on a shut-down read executor.

There is no `isShutdown()` check or try-catch around the executor submission in any of the store's methods. The `SqliteDeadLetterStore` follows the same pattern as `SqliteCheckpointStore` and `SqliteViewCheckpointStore` — none of the existing stores have defensive shutdown handling. This is consistent but means all stores share the same vulnerability. In production, the shutdown sequence (orchestrated by `SqlitePersistenceLifecycle.stop()`) should ensure no callers are active when the executor is shut down. The risk is during ungraceful shutdown or test tear-down races. Not blocking because it's a pre-existing pattern across all stores, but worth noting for a future hardening pass.

## Summary

### Blocking issues

None.

### Non-blocking concerns

1. **CheckpointSerializer has no size guardrail** (Item 12): The entire checkpoint is buffered in memory with no size limit or warning. At the 10,000-entity design target this is ~15–20 MB and safe, but there's no alert if payloads grow unexpectedly. Recommend adding a WARN log above a configurable threshold.

2. **AtomicCheckpointWriter code duplication** (Item 13): The two-way and three-way methods duplicate the transaction wrapper and both common prepared statement blocks. Factoring shared infrastructure would reduce drift risk. Track as tech debt.

3. **No concurrent access tests for SqliteStateStore** (Item 14): The `ConcurrentHashMap` backing provides correctness by construction, but no test proves it under concurrent load. Low priority given the backing data structure's guarantees.

4. **No post-shutdown defensive handling in stores** (Item 15): `SqliteDeadLetterStore` (and all sibling stores) will throw uncaught exceptions if methods are called after `DatabaseExecutor.shutdown()`. This is a pre-existing pattern, not an M3.5b regression. Track for a future hardening pass.

5. **event-bus MODULE_CONTEXT type count may be off by one** (Item 10): Claims 32 types, actual count appears to be 33. Minor documentation discrepancy.

### Quality observations

1. **Javadoc quality is exceptional.** Every public type and method has comprehensive documentation including field-to-column mapping tables, null-handling contracts, idempotency semantics, and cross-references to design docs and amendments. The `DeadLetter` record's Javadoc is a model of what persistence-boundary records should look like.

2. **Contract test architecture is well-designed.** The `DeadLetterStoreContractTest` abstract class in event-bus testFixtures, subclassed by `SqliteDeadLetterStoreContractTest` in persistence, mirrors the established pattern from `CheckpointStoreContractTest` and `ViewCheckpointStoreContractTest`. This ensures behavioral parity between any future DLQ store implementation and the SQLite production one.

3. **Null-safety discipline is rigorous.** The explicit avoidance of `Map.copyOf` (which throws on null values) in `CheckpointSerializer.fromSerializable`, `SqliteStateStore.getAll()`, and `CheckpointData`'s compact constructor shows awareness of the `EntityState.attributes()` nullable-value contract. The code comments calling this out as a "gotcha" are a good defensive practice.

4. **The ObjectMapper divergence is correctly handled.** Two ObjectMapper configurations (`NON_NULL` for events, `ALWAYS` for checkpoints) is the right architectural call, and it's thoroughly documented in Javadoc, test comments, and MODULE_CONTEXT gotchas. The composition-root follow-up is correctly scoped out of M3.5b.

5. **V004 migration is appropriately minimal.** Adding operational indices without table or column changes is the right approach for a non-breaking schema evolution. The `CREATE INDEX IF NOT EXISTS` pattern makes it safe to apply on existing databases.

### Overall assessment

M3.5b is clean, well-documented, and correct — the code reads as production-grade infrastructure, not "works in tests" scaffolding.
