## Findings (severity-ranked)

### F-1 · HIGH · v14 is ACCEPTED by negotiation but the interview/ingestion wire dialect is v13-only — on v14 silicon the radio goes silently deaf

- **Defect class:** protocol-semantics divergence (v13/v14 dialect assumption — priority surface 4).
- **Claim:** `EzspCoordinatorProtocol.startSession()` accepts a negotiated version up to 14 (`MAX_SUPPORTED_PROTOCOL_VERSION = 14`, lines 234–242 accept 13–14) and pins the v14 codec for STATUS widths (D-M92-4), but the two M9.3 wire bindings hardcode v13-only layouts: `sendUnicastLocked(...)` writes `messageTag` as ONE byte at offset [14] + length at [15] (EzspCoordinatorProtocol.java:718–738), and `EzspIncomingMessage.parse(...)` reads sender at [14..15] and the message length at [18] (EzspIncomingMessage.java:70–90).
- **Re-derived evidence (bellows, fetched this session):** `v14/commands.py` OVERRIDES both commands — `sendUnicast: (message_type, nwk, aps_frame, message_tag: uint16_t, LVBytes)` and `incomingMessageHandler: (message_type, aps_frame, nwk, eui64, binding_index, address_index, lqi, rssi, timestamp: uint32_t, LVBytes)` — while `v13/commands.py` inherits the v4-lineage shapes the code implements (messageTag u8; lqi/rssi/sender/bindingIdx/addrIdx flat, length at [18]). On v14 the length byte lives at offset [30] (1+11+2+8+1+1+1+1+4), not [18].
- **Concrete failing scenario:** a Dongle Plus MG24 from the EmberZNet 8.0.2 ship-batch (the AMD-96/E1 documented reality: "other units of the same model are reported shipping EZSP v14") negotiates 14 → accepted, WARN-free. Every subsequent `incomingMessageHandler` parse reads parameters[18] (= byte 5 of the sender EUI64) as the length; for a typical frame that value ≠ remaining bytes → `Optional.empty()` → the frame is DROPPED. Result: zero announces, zero reports, zero interview responses — a permanently deaf adapter that passed negotiation, with no PIE and no restart-loop signal (the failure is pure silence). Outbound: the interview's `sendUnicast` request carries the length byte where v14 expects messageTag's high byte — the NCP misparses the APS payload boundary; UNICAST corruption on every interview step.
- **Adjacent known items do NOT cover this:** the MODULE_CONTEXT pins only the `lookupNodeIdByEui64` v14 reply and the `networkInit` arity as bench-verify items; H4 covered the wide-STATUS path only. The 0x0034/0x0045 layout divergence is unrecorded anywhere in the repo (grep: no v14 branch, no TODO, no MODULE_CONTEXT line).
- **Fix direction:** either narrow acceptance to `== 13` with v14 → PIE-until-dialect-complete (the R4 band-edge spirit: never best-effort an uncharacterized dialect), or version the 0x0034/0x0045 codecs behind the same seam as `decodeStatus` — and add both to the M9.4 v14 bench-verify list either way.

### F-3 · HIGH (M9.4-authoring input) · TOLERANCE confirmation is structurally unreachable end-to-end — no path can ever construct a commanded-target WithinTolerance

Net: the two capabilities the bench measured CONFIRMABLE with tight latency bands (brightness 353–686 ms, CT 6.7–8.4 s — the AMD-97 reference values) can never render CONFIRMED or UNCONFIRMED through any current code path; they are silently optimistic.
- **Why this is a finding and not a restatement of the known gap:** beat-4's "engine-unwired" framing covers the `Confirmability`/`ConfirmationCharacterization` consumption and the per-device tuning path. - **Concrete scenario:** hero-adjacent demo — automation sets CT 2700 K; the bulb lands at 2702 K and reports it (measured drift class); the system records NOTHING (no track, no confirm, no timeout) while the corpus says CONFIRMABLE/15000 ms. The claim→truth table for "honest confirmation" fails on the first non-boolean command.
- **Fix direction:** M9.4 charter item — ledger (or executor) derives the expectation from `command_issued.parameters` + the capability's ConfirmationPolicy (+ AMD-97 per-device override); populate `expectedOutcomes` semantics for parameterized commands or replace the static-outcome source with the factory seam the ledger javadoc already anticipates (`ExpectationFactory`).

### F-4 · MEDIUM · dedup has no time window — periodic identical reports at consecutive TSNs chain-drop indefinitely, and solicited read-backs share the twin-suppression scope

- **Defect class:** edge-case logic defect in the measured dedup contract (priority surfaces 3/5).
- **Claim:** `ReportDeduplicator.isDuplicate(...)` (full read) classifies duplicate iff payload-equal AND TSN same-or-successor (mod 256) — no time bound, though the measured twin phenomenon is 8–21 ms. Two consequences: (a) a quiet battery device whose periodic max-interval report repeats an unchanged payload at a consecutive TSN (no interleaving traffic on that device advanced its TSN) is classified a twin — and because `isDuplicate` re-records the dropped frame's TSN as `last`, the NEXT periodic report is again a successor → the drop CHAINS until the value changes or the device re-announces. Scenario: SNZB overnight, no motion; battery reports (unchanged 200) at TSN 50/51 (real twins, dropped ✓), then hourly TSN 52, 53, 54… — every subsequent periodic battery report suppressed for the whole quiet period. `availability` survives (listener.onFrame fires pre-dedup ✓) but the event log records no battery observations. (b) Solicited `ReadAttributesResponse` frames flow through the same scope (ZclIngestionUnit.handleZcl dedups BEFORE the command-type branch, and the 0x01 branch publishes state_reported): two deliberate read-backs of an unchanged attribute with consecutive OUR-side TSNs → the second response is swallowed → the M9.4 `CONFIRM_FROM_CACHE_OR_READBACK` degrade path (and the idempotent-command readback confirm, AMD-97 caveat 1) loses its confirming observation → a CONFIRMABLE command renders UNCONFIRMED (honest-direction miss, but a brand-eroding false negative the corpus explicitly prices).
- **Settled-ground note:** the dedup payload-scope [INFO] ruled payload-vs-whole-frame; it did not rule the missing time bound or the solicited-response scoping — this refines, not re-litigates.
- **Fix direction:** bound the duplicate predicate with a short window (e.g. ≤ 1 s — 50× the measured twin gap), and/or scope dedup to unsolicited classes only (Report_Attributes 0x0A + IAS notifications), exempting 0x01 responses.

### F-6 · MEDIUM (M9.4 dispatch-facing) · device-cache NWK reuse leaves the collision victim's record pointing at another device — the dispatch identity join would misdirect commands

### F-9 · MEDIUM · ZCL invalid-value markers decode as real values — battery 0xFF publishes battery_pct 127, boolean 0xFF publishes on=true

- **Defect class:** codec edge defect (signed/unsigned + invalid-marker semantics, priority surface 4).
- **Claim:** (a) `PowerConfigurationHandler.normalize` guards only `raw >= 0` (line read) — the ZCL uint8 invalid marker 0xFF (a real-world "battery unknown" report, common on budget sensors) publishes `battery_pct = 127`; 201–254 likewise publish > 100 %. (b) `ZclCodec.decodeValue` case 0x10 decodes ANY non-zero byte as `true` — the ZCL boolean invalid marker 0xFF becomes a genuine `on=true` state observation (grep confirms no invalid-marker handling anywhere in ZclCodec beyond the string-length 0xFF).
- **Concrete scenario:** a sensor reports battery 0xFF while its cell is actually at 40 % → dashboard shows 127 %, and a "battery below 20 %" automation is structurally unable to fire later ONLY if the device resumes real reports; meanwhile the recorded state is nonsense. For (b): a flaky bulb reports OnOff 0xFF → `on=true` recorded → an M9.4 EXACT_MATCH turn_on confirmation could CONFIRM against an invalid-marker report (wrong-verdict adjacency).
- **Fix direction:** skip 0xFF for uint8 battery (and clamp/skip > 200) with a DEBUG log; treat bool values other than 0x00/0x01 as non-observations (skip the record, keep the rest of the frame).

### F-10 · MEDIUM · commands issued during the ledger's TRANSITION window are misclassified as crash-recovery in-flight — expired_on_restart on live commands, re-issue offers for commands still executing

- **Defect class:** concurrency/lifecycle boundary defect (the restart-honesty class the M9.4 prove-list names).
- **Claim:** `setMode(mode != LIVE)` makes TRANSITION deliveries accumulate into `replayInFlight` (StandardPendingCommandLedger.java:251–253, 256–259, 426–441); `onCaughtUp()` classifies EVERYTHING accumulated (476–492) with no freshness test, though each candidate's event-derived `deadline` is right there on the record. A `command_issued` published while the ledger drains its catch-up window is therefore treated as pre-crash residue: NOT_IDEMPOTENT → `command_result(expired_on_restart)` PUBLISHED for a command the adapter is actively executing; IDEMPOTENT → added to `reissueOffered` (a double-actuation signal once M8.2 drains it — the ghost-command class); and in all cases the command is NOT live-tracked, so its genuine confirmation window never runs.
- **Concrete scenario:** restart with a long log; the ledger is in TRANSITION for N seconds of catch-up; an automation fires `activate_scene` (NOT_IDEMPOTENT) in that window → the log permanently records `expired_on_restart` for a command that executed normally — a false lifecycle verdict in the audit trail, plus a suppressed confirmation.
- **Fix direction:** in `classifyRestart()`, candidates whose deadline is in the FUTURE (relative to the injected clock) are live — index them as tracked commands instead of classifying; only past-deadline candidates are crash residue.

### F-12 · LOW/MEDIUM (M9.4 wiring-facing) · a malformed lazy profile body detonates at findProfile time — one broken user profile can restart-loop the adapter
