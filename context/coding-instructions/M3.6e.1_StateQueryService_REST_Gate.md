## What This Implements

M3.6e.1 is the M3.6 capstone — the moment the composition root becomes externally queryable. It delivers three things: (1) `MaterializedStateQueryService`, the production `StateQueryService` implementation that reads materialized entity state from the `StateProjection`'s live `StateStore`; (2) `ReadinessFilter`, a Javalin `before` handler that returns 503 with diagnostic headers when the projection is not yet LIVE; (3) Javalin HTTP server bootstrap in `HomeSynapseCore` with per-profile thread pool sizing. After this WU, `HomeSynapseCore.stateQueryService()` returns the real service instead of `ThrowingStateQueryService`, and an HTTP endpoint exists to gate downstream consumers.

## Technical Specification

### `MaterializedStateQueryService` (package-private final class)
**Package:** `com.homesynapse.state`
**Responsibility:** Production read-only facade over the StateProjection's live StateStore. Lock-free reads via ConcurrentHashMap internals.
**Implements:** `StateQueryService`

**Constructor:**
```java
MaterializedStateQueryService(StateStore stateStore, ReadinessSource readinessSource, Clock clock)
```
All three injected by the composition root. `Clock` is required for staleness recomputation at read time.

**Methods:**

```java
@Override
public Optional<EntityState> getState(EntityId entityId) {
    // Delegate to stateStore.get(entityId)
    // If present, recompute EntityState.stale using clock.instant() and staleAfter
    // Return Optional.of(recomputed) or Optional.empty()
}

@Override
public Map<EntityId, EntityState> getStates(Set<EntityId> entityIds) {
    // For each entityId, call stateStore.get(entityId)
    // Omit unknown entityIds from result (do NOT insert null or Optional.empty)
    // Recompute stale for each present entity
    // Return unmodifiable map
}

@Override
public StateSnapshot getSnapshot() {
    // Delegate to stateStore.getAll() (or equivalent — verify against StateStore interface)
    // Recompute stale for each entity
    // Return StateSnapshot with stateStore.getViewPosition() and the recomputed map
}

@Override
public long getViewPosition() {
    // Delegate to stateStore.getViewPosition()
    return stateStore.getViewPosition();
}

@Override
public boolean isReady() {
    // Delegate to readinessSource.mode() == SubscriberMode.LIVE
    return readinessSource.mode() == SubscriberMode.LIVE;
}
```

**Behavioral contracts:**
- All reads are lock-free. ConcurrentHashMap provides per-entity consistency; cross-entity reads are weakly consistent.
- `stale` field MUST be recomputed at read time using `clock.instant()` and `EntityState.staleAfter()`. The projection writes `stale=false` unconditionally — the real staleness answer depends on when you ask. This is HomeSynapse's #1 architectural differentiator.
- `getStates` omits unknown entityIds silently — no exception, no null entry.
- `getSnapshot` returns ALL entities with a single consistent `viewPosition`.
- Thread safety: fully thread-safe (all state comes from ConcurrentHashMap + atomic long in StateStore).

**Stale recomputation logic:**
```java
// EntityState has: Optional<Instant> staleAfter()
// If staleAfter is empty → stale = false (no staleness contract)
// If staleAfter is present and clock.instant().isAfter(staleAfter.get()) → stale = true
// Otherwise → stale = false
//
// To produce the recomputed EntityState, use EntityState's copy/withStale pattern
// (check what EntityState provides — it may be a record with a wither, or you may
// need to construct a new instance with the corrected stale flag)
```

### `ReadinessFilter` (public class)
**Package:** `com.homesynapse.api.rest`
**Responsibility:** Javalin `before` handler that returns 503 when the StateProjection is not yet LIVE.
**Implements:** `io.javalin.http.Handler` (Javalin's handler functional interface)

**Constructor:**
```java
public ReadinessFilter(ReadinessSource readinessSource)
```

**Behavior:**
```java
@Override
public void handle(Context ctx) throws Exception {
    SubscriberMode mode = readinessSource.mode();
    if (mode != SubscriberMode.LIVE) {
        ctx.status(503);
        ctx.header("X-HomeSynapse-Projection-State", mode.name());
        ctx.header("Retry-After", "5");
        ctx.json(Map.of(
            "type", ProblemType.STATE_STORE_REPLAYING.typeUri(),
            "status", 503,
            "title", ProblemType.STATE_STORE_REPLAYING.title(),
            "detail", "State projection is in " + mode.name() + " mode; query results are not yet available."
        ));
        return; // do NOT call ctx.next() — request is complete
    }
    // LIVE → do nothing, request continues to the matched endpoint
}
```

**Notes:**
- Registered via `app.before("/api/*", readinessFilter)` in the Javalin server setup.
- Must NOT swallow exceptions — let Javalin's error handling deal with unexpected failures.
- The `Retry-After: 5` header is a fixed advisory value (not computed from projection progress).
- Response body follows RFC 9457 problem detail format using `ProblemType.STATE_STORE_REPLAYING`.

## Locked Decisions That Apply

- **DEC-M3-09 (Clock injection):** Single `Clock` per JPMS module. MaterializedStateQueryService receives `Clock` via constructor. The `NO_DIRECT_TIME_ACCESS` ArchUnit rule bans `Instant.now()`, `System.nanoTime()`, `System.currentTimeMillis()`, `Clock.systemUTC()`, `Clock.systemDefaultZone()`.
- **DEC-M3-16 (Composition-root visibility):** Package-private implementations behind public interfaces. `MaterializedStateQueryService` is package-private; consumers see `StateQueryService`.
- **LTD-11 (No Broker):** Virtual threads — use `ReentrantLock`, never `synchronized`.
- **AMD-26 (Virtual thread carrier pinning):** All SQLite ops on platform threads. MaterializedStateQueryService reads from ConcurrentHashMap (no SQLite), so no pinning concern here — but be aware that `StateStore` reads are NOT SQLite reads; they're in-memory map lookups.

## Invariants That Must Hold

- **INV-QS-01 (Read-only query service):** `StateQueryService` NEVER mutates state. All methods are pure reads. The `MaterializedStateQueryService` must not call any write method on `StateStore`.
- **INV-SUB-MODE-01 (Mode lifecycle):** `SubscriberMode` transitions are unidirectional: COLD → REPLAY → TRANSITION → LIVE. `isReady()` must delegate to `readinessSource.mode() == LIVE`, not maintain its own boolean.

## What to Watch Out For

1. **`EntityState.stale` recomputation is NOT optional.** The projection writes `stale=false` unconditionally. If you return the `EntityState` from the store without recomputing, every entity will report as not-stale, which is wrong. This is the single most important behavioral contract in the entire query service. Check how `EntityState` handles the `stale` field — it may be a record (immutable), in which case you need to construct a new `EntityState` with the corrected value.

2. **`Map.copyOf` and null attribute values.** `EntityState.attributes()` may contain null values per the contract. `Map.copyOf` throws on nulls. If you need to create an unmodifiable map in `getStates`, use `Collections.unmodifiableMap(new LinkedHashMap<>(...))`, NOT `Map.copyOf`. (See state-store MODULE_CONTEXT gotcha.)

3. **`module-info.java` for rest-api is empty.** Phase 2 left it with zero `requires` directives. You need to add the production dependencies. The `requires` list should include at minimum: `com.homesynapse.state` (for ReadinessSource, SubscriberMode), `com.homesynapse.event.bus` (for SubscriberMode enum — verify which module exports it), and `io.javalin`. Check whether Javalin's module name is `io.javalin` or something else.

4. **Javalin on virtual threads.** Javalin 6.x supports virtual threads natively. However, `QueuedThreadPool` uses platform threads by default. For MVP this is fine — the HTTP thread pool is deliberately small and platform-threaded. Do NOT configure Javalin to use virtual threads for request handling without checking that all downstream code avoids `synchronized` blocks.

5. **`ReadinessFilter` must NOT call `ctx.next()` when rejecting.** Javalin `before` handlers that set a response status and body must return without calling continuation. If you set status 503 and also allow the request to continue, Javalin will overwrite your 503 with the endpoint's response.

6. **Lifecycle module already has `requires com.homesynapse.state` (transitive).** Adding rest-api as a dependency is the new edge. Verify that `com.homesynapse.api.rest` exports the package containing `ReadinessFilter` so lifecycle can see it.

7. **`NO_DIRECT_TIME_ACCESS` ArchUnit rule.** MaterializedStateQueryService uses `Clock.instant()` for staleness — this is the correct pattern. Do NOT use `Instant.now()`. The rule applies in `core/state-store`.

8. **`ThrowingStateQueryService` stays.** Do not delete it. It's still useful as a stub for tests that don't need the real query service. Just change `HomeSynapseCore.stateQueryService()` to return the real implementation.

9. **Javalin `showJavalinBanner = false`.** We're a headless embedded system, not a web app. Suppress the ASCII art banner.

10. **`stop()` ordering in HomeSynapseCore.** Javalin stop should come FIRST in the shutdown sequence (before scheduler, before unsubscribe) — stop accepting new queries before tearing down the state they would query.
