# Coding Task: M6.2 — Secret Store + Per-Scope Key-Management Infrastructure

**Subsystem:** Configuration System (`com.homesynapse.config`) + the `PayloadCipher` seam in Persistence (`com.homesynapse.persistence`) + composition-root wiring in `com.homesynapse.app`
**Design Doc:** Doc 06 (LOCKED) §3.1 stage 3/§3.4/§4.8/§12.1–§12.3 + Doc 15 (LOCKED) §3.5/§3.8/§4.2/§7.3/§8 fences + ratified AMD-68 · M6 charter CARRY 1

## What This Implements

The encrypted secrets plane and the key-management infrastructure every later crypto consumer builds on. Three deliverables: (1) **`ScopeKeyManager`** in config — generates/loads the 256-bit machine-local root key (`.root-key`, `0400`, `SecureRandom`, zero-config per Doc 15 §3.5), derives per-scope KEKs via HKDF-SHA256 (`HKDF(root, "scope:"+scopeId)` — Doc 15 §4.2), generates and wraps per-scope DEKs (AES-256-GCM), owns the `scope_keys` store, and exposes its own `encrypt`/`decrypt` surface; (2) **`SecretStore`** implementation — `secrets.enc` as AES-256-GCM-encrypted JSON under the `config_secrets` scope DEK (Doc 15 §7.3 shared-root unification — the secret store is one scope in the hierarchy, NOT a second key system), with the AMD-68 `setAll(Map)` all-or-nothing durable write and AMD-16 per-operation backup rotation, plus `!secret`/`!env` stage-3 tag resolution in `YamlLoader` (the tags are FATAL-unknown today — fail-closed staging ends here); (3) the **E2 composition-root bridge** — the `PayloadCipher` interface consumer-defined in persistence, implemented by NOBODY in config (see CARRY 1), bridged by a thin adapter constructed in `com.homesynapse.app.Main` and threaded through `HomeSynapseCore` to persistence, where it is HELD for M6.3's write-path consumption. Zero new module edges, verified in both directions.

## CARRY 1 (verbatim from the M6 charter / Locked Doc 15 §3.8 — do not paraphrase)

> **Key-manager placement — JPMS-cycle resolution (m3).** The key manager (root key, scope KEK/DEK derivation, wrap/unwrap, `scope_keys`) is key-management and therefore belongs with **configuration** (`com.homesynapse.config`, M6) — but the *encryptor of event payloads* is **persistence**, which must not statically depend on config (verified at HEAD: persistence does **not** `requires com.homesynapse.config`; config `requires transitive com.homesynapse.event` only — a new persistence→config edge plus any future config→persistence need would risk the **same cycle class as the AMD-52 `event↔device` cycle that forced the value-model relocation**). The resolution is the **AMD-45 `AtomicCheckpointSink` seam pattern**: a narrow **`PayloadCipher` interface is *consumer-defined in persistence*** (`encrypt(scopeId, plaintext) → (ciphertext, iv)` / `decrypt(scopeId, keyVersion, ciphertext, iv) → plaintext`), the config-resident key manager (`ScopeKeyManager`) **exposes its own `encrypt`/`decrypt` surface but does *not* implement the persistence-exported `PayloadCipher` type directly** — implementing it in `config` would force a `config → persistence` edge. Instead the **top-level composition root `com.homesynapse.app`** (`Main`) constructs `ScopeKeyManager`, wraps it in a thin `PayloadCipher` adapter (a lambda over `encrypt`/`decrypt`), and **injects that adapter down through `HomeSynapseCore` (lifecycle) into the persistence write path**. `app` is the correct host because it is the **only** module that already `requires` **both** `com.homesynapse.config` **and** `com.homesynapse.persistence` (verified at HEAD: `app/homesynapse-app/src/main/java/module-info.java` `requires com.homesynapse.config` *and* `requires com.homesynapse.persistence`; `lifecycle` requires `persistence` but **not** `config`), so the bridge adds **no** new module edge; hosting it in `lifecycle`/`HomeSynapseCore` would force a new `lifecycle → config` edge. The `scope_keys` store is owned on the **config/key-manager** side (mirroring the secret store's own `secrets.enc` file). **Net JPMS result: `persistence` does not `requires` `config`, `config` does not `requires` `persistence`, and only `app` `requires` both — which it already does at HEAD — so the cycle is closed in *both* directions with zero new edges** (the AMD-45 `AtomicCheckpointSink` injection-at-the-composition-root discipline, applied to a config-supplied implementation). The verbatim `module-info.java` for persistence and config **must be embedded in the M6 coding instruction** (the standing Research-6 rule) before any code is written, and the no-cycle property re-verified at issue.

**Zero-new-edge re-verification (PM, at authoring against `9035110`):** persistence `module-info` carries NO `requires com.homesynapse.config` ✓ · config carries NO `requires com.homesynapse.persistence` ✓ · app carries BOTH ✓ (embeds below). **The Coder re-verifies all three at execution.** Note the known-stale Doc 15 §8.1 table row reading "implement `PayloadCipher`" for `ScopeKeyManager` — the §3.8 text above GOVERNS (parked currency nit); do not implement the persistence type in config.

## Settled Decisions (NOT open questions)

- **DP-1 (root key — Doc 15 §3.5):** 256-bit from `SecureRandom`, file `.root-key` under the injected config-dir `Path`, POSIX `0400` (best-effort + WARN on non-POSIX filesystems; `Assumptions`-guard the permission test like `ConfigLayoutTest`'s symlink row). Generated lazily on FIRST need (first secret write or first scope-DEK creation) — zero-config installs with no secrets never create it (INV-CE-02). Never logged, never in events.
- **DP-2 (key hierarchy — Doc 15 §4.2):** scope KEK = HKDF-SHA256(root, info=`"scope:"+scopeId`), re-derived on demand, cached per active scope, never stored. Scope DEK = 256-bit `SecureRandom`, wrapped by the KEK with AES-256-GCM (fresh random 96-bit IV per wrap), stored in `scope_keys` with `keyVersion` starting at 1. **HKDF is hand-rolled** (RFC 5869 extract+expand over `javax.crypto.Mac`/HmacSHA256, package-private helper, RFC 5869 Appendix-A test vectors) — JDK 21 has no HKDF API and the catalog gains NO new dependency.
- **DP-3 ([R6] `scope_keys` store):** a config-side JSON file `scope_keys.json` under the config-dir, mirroring `secrets.enc` ownership (CARRY 1 sentence). Content: array of `ScopeKey(scopeId, keyVersion, encryptedDek, iv, createdAt, destroyedAt?)` rows (Doc 15 §8.2 shape; byte arrays Base64 in JSON via jackson-databind — already a `requires`). Written with the same atomic temp→fsync→rename→fsync-dir discipline. PLAINTEXT file is correct — DEKs in it are wrapped (ciphertext); it contains no secret material readable without the root key.
- **DP-4 (`ScopeKeyManager` public surface — config):** `public interface ScopeKeyManager` with `ScopeCipherResult encrypt(String scopeId, byte[] plaintext)` / `byte[] decrypt(String scopeId, int keyVersion, byte[] ciphertext, byte[] iv)` / `static ScopeKeyManager create(Path configDir, Clock clock)` (DEC-M3-16 public-gateway precedent; impl class `StandardScopeKeyManager` package-private). `public record ScopeCipherResult(byte[] ciphertext, byte[] iv, int keyVersion)` — config-module + `java.base` components only. `encrypt` lazily creates the scope's DEK (and the root key) on first use; per-call fresh random 96-bit IV. Array components: NO defensive copies (hot path, trusted callers) — document in Javadoc. `ReentrantLock` for store mutations (LTD-11). Decrypt of a `destroyedAt`-marked or absent (scope,version) → throw (the unreadability property; the shred *operation* that sets `destroyedAt` is post-MVP — only the test exercises destruction by direct store manipulation).
- **DP-5 (`PayloadCipher` seam — persistence):** `public interface PayloadCipher` in `com.homesynapse.persistence` (exported package): `EncryptedPayload encrypt(String scopeId, byte[] plaintext);` / `byte[] decrypt(String scopeId, int keyVersion, byte[] ciphertext, byte[] iv);` + `public record EncryptedPayload(byte[] ciphertext, byte[] iv, int keyVersion)`. `java.base`-only signatures → NO new `requires`, no `-Xlint:exports` exposure. Javadoc cites Doc 15 §3.8 + the M6.3 consumption point. **No persistence code calls it in M6.2.**
- **DP-6 (the bridge — app):** `Main` constructs `ScopeKeyManager.create(configDir, clock)` and adapts: `PayloadCipher cipher = new PayloadCipher() { … delegate both methods, mapping ScopeCipherResult → EncryptedPayload … }` (or lambda-per-method holder — Coder freedom; keep it in `Main` or a private static method, NOT a new public type). Thread it: `HomeSynapseCore` (lifecycle) gains a `PayloadCipher` constructor parameter (lifecycle already reads persistence types — no new edge) and passes it to the persistence factory/lifecycle component, which HOLDS it in a field Javadoc-marked "consumed by the M6.3 at-rest write path." Test harnesses pass a no-op/null-object cipher (add `PayloadCipher.noOp()`? NO — keep the surface minimal: harnesses pass the real adapter over a temp-dir ScopeKeyManager, or `null` if the field is nullable-documented. PIN: nullable field, `@Nullable`-documented, M6.3 makes it required — smallest honest seam).
- **DP-7 (secrets.enc under the shared root — Doc 15 §7.3):** `secrets.enc` = AES-256-GCM ciphertext of the §4.8 JSON form `{ "entries": [...], "version": 1 }`, encrypted under the **`config_secrets` scope DEK** via the same ScopeKeyManager machinery (scope_id literal `"config_secrets"`). The legacy single-static-`.secret-key` design is superseded BEFORE first release — no migration path needed (no production secrets exist). The file's GCM IV is stored alongside the ciphertext (small header or sibling JSON envelope `{keyVersion, iv, ciphertext}` — Coder freedom, document the on-disk form). Fresh random IV per write; the OR-M6-NONCE counter-nonce discipline is the M6.3 per-event path, NOT this full-file-rewrite path (volume = occasional CLI/rotator writes; random-IV collision risk negligible — state this in Javadoc so M6.3 doesn't cargo-cult it).
- **DP-8 (AMD-68 `setAll` — §2.1/§2.2 verbatim contract):** `SecretStore` grows 4→5 methods with the AMD-68 §2.1 Javadoc (all-or-nothing, durable-before-return, creates store on first use, empty map → `IllegalArgumentException`). Implementation: decrypt current store → merge all entries (create/update, `createdAt`/`updatedAt` from injected `Clock`) → encrypt → write-temp → fsync → atomic rename → fsync dir. `set(k,v)` delegates to `setAll(Map.of(k,v))` (uniform atomicity); `remove` uses the same write discipline. Writes serialize on a `ReentrantLock`.
- **DP-9 (AMD-16 per-operation backups — Doc 06 §3.4):** before every mutating operation that rewrites `secrets.enc` (`setAll`/`set`/`remove`), copy the existing file to `secrets.enc.bak.{N}` (N monotonic, 1-indexed; max 5 retained, oldest deleted). First-ever write (no existing file) takes no backup. The restore CLI and `secrets_restored` event are OUT of scope (M13).
- **DP-10 (stage-3 tag resolution — Doc 06 §3.1/§3.4):** `YamlLoader` registers two new tag constructors beside `!include`: `!secret <key>` → `SecretStore.resolve(key)`; missing key → **FATAL `ConfigIssue`** naming the key (collected, not thrown mid-parse — match the `!include` failure pattern); `!env <VAR>` / `!env <VAR>:<default>` → injected env lookup (`Function<String,String>`, composition root passes `System::getenv`; tests pass a map) with the default form preventing the FATAL. Both LoadSettings keep the explicit **CoreSchema** (the gotcha; G2). Resolved values exist only in the parse tree; the decrypted store is held for the load duration only and discarded (§3.4). Resolution applies to root AND included documents. `StandardConfigurationService` threading: ctor gains `SecretStore` + env-lookup params (adapt to the current arity found at G7); the declared-pair-wired-twice gotcha still holds.
- **DP-11 (SecretStore impl + construction):** package-private `StandardSecretStore implements SecretStore`, ctor `(Path configDir, ScopeKeyManager keyManager, Clock clock)`; public gateway `SecretStore.create(Path, ScopeKeyManager, Clock)` static factory (mirrors DP-4). `resolve` of an absent key throws `IllegalArgumentException` (MODULE_CONTEXT decision #7 — pin: REUSE `IllegalArgumentException`, no new exception type; the §3.4 parse-time FATAL comes from the tag constructor catching it). `list()` decrypt-on-demand, names only.

## What to Watch Out For

- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this module's test code — the `NO_DIRECT_TIME_ACCESS` ArchUnit rule scans non-whitelisted test classes and fails `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructor/`@BeforeEach`. (config, persistence, lifecycle: all non-whitelisted. `SecureRandom` is fine — the rule is time-scoped.)
- **CoreSchema lockstep (MODULE_CONTEXT gotcha):** your `YamlLoader` edits touch LoadSettings construction — both root and included settings MUST keep `.setSchema(new CoreSchema())`. The pinned regression test exists; do not weaken it.
- **GCM discipline:** 96-bit IV, 128-bit tag, fresh `SecureRandom` IV per encryption, IV stored beside ciphertext, NEVER reused. The M6.3 counter-nonce/OR-M6-NONCE machinery is explicitly NOT this WU — full-file random-IV rewrites are the correct model here (DP-7).
- **Don't leak `javax.crypto` types on any public API** — all crypto behind the package-private impls; public records carry `byte[]`/`int`/`String` only.
- **`-Xlint:exports` / `-Werror`:** `PayloadCipher`/`EncryptedPayload` are `java.base`-only by design (DP-5) — if you find yourself wanting a cross-module type on that surface, STOP (it breaks the zero-new-edge property). `HomeSynapseCore`'s new param is a persistence type — lifecycle's module-info already reads persistence; verify whether lifecycle `requires transitive` persistence covers the ctor-param exposure (ctor params of a public class in an exported package DO count for `[exports]` lint — check lifecycle's existing posture before assuming; if lifecycle already exposes `PersistenceFactory`-built types in its API, the posture is already transitive).
- **ArchUnit Rule 5:** config is the exempt filesystem consumer — your file IO under the injected `Path` is legal. Do NOT add file IO in persistence or lifecycle for this WU.
- **POSIX permissions on non-POSIX filesystems:** `Files.setPosixFilePermissions` throws `UnsupportedOperationException` on Windows — wrap, WARN, continue (the `ConfigLayoutTest` `Assumptions` precedent for the test).
- **Decrypted-store lifetime (§3.4):** decrypt → resolve tags → discard; do not cache plaintext entries on the service or loader between loads. Zero out intermediate `byte[]`s where practical (best-effort hygiene, not a contract).
- **Sequencing with M6.4:** if M6.4 landed first, `StandardConfigurationService` has reload/write implemented — your tag-resolution threading must flow through BOTH load and reload paths (the reload re-parse resolves tags too; §3.3 step 1 = full §3.1 stages). If M6.4 has NOT landed, reload is still staged — touch only the load path and leave the staging intact.
- **The Doc 15 §8.1 stale row:** "ScopeKeyManager … implement PayloadCipher" — do NOT. §3.8/CARRY 1 governs (the adapter lives in app). Known parked nit.
