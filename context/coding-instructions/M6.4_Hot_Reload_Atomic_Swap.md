## What This Implements

M6.1 shipped the config **load** path and staged `reload()`/`write()` as `UnsupportedOperationException`. M6.4 un-stages them: an explicit `reload()` re-runs the §3.1 pipeline against disk, diffs the candidate against the active model into a `ConfigChangeSet`, classifies every changed section through the AMD-66 listener seam (falling back to per-property `x-reload`, unannotated → `PROCESS_RESTART`), rejects the whole candidate on any FATAL/ERROR (the active model is never degraded — §3.3/§3.6/C5), and otherwise **atomically swaps** the active `ConfigModel` so every in-flight reader sees wholly-old or wholly-new, never a torn mix. `write()` is the §3.5 UI/API path over the same lock: optimistic-concurrency check on `fileModifiedAt`, validate-the-mutated-copy, atomic write-to-temp-fsync-rename, then reload. Two deferred load-path obligations also land here because M6.4 owns atomic file writes: the §3.7 step-7 **migration write-back + pre-migration backup** (R2 ruling) and the **per-ERROR `config_error` startup publication** (R1 ruling). The reload publishes the new AMD-70 `config.section_reloaded` event — which requires the full P2 manifest fan-out in event-model/persistence.

## Settled Decisions (NOT open questions — implement as stated)

- **DP-1 (atomic swap mechanism):** the active model stays a `volatile ConfigModel` field (the existing `getCurrentModel()` read point); the swap is a single volatile reference assignment performed while holding the reload/write lock. `ConfigModel` is immutable (records + `Map.copyOf`), so reference-swap = torn-read-free. No `synchronized` (LTD-11).
- **DP-2 (one lock):** ONE `ReentrantLock` serializes `reload()` AND `write()` (§3.5 step 3 — write acquires it; write's step-7 reload runs under the already-held lock; make the internal reload path re-entrant-safe or structure as private `reloadLocked()`). `getCurrentModel()` never blocks (volatile read only).
- **DP-3 (reload pipeline order — §3.3 verbatim):** re-parse (full §3.1 stages incl. AMD-67 migration + default-merge + validation) → diff → classify → atomicity check (any FATAL or ERROR in candidate → throw `ConfigurationReloadException`, active model untouched, NO events published, NO listener classifications applied) → swap → notify (`config.section_reloaded` per changed section) → return `ReloadResult(newModel, changeSet, warningOnlyIssues)`.
- **DP-4 (classification source):** for each changed section: registered `ConfigurationChangeListener.onSectionChanged(previous, candidate)` (synchronous, BEFORE publish — AMD-66-INV-02); no listener → most-restrictive per-property `x-reload` among the section's changed keys; unannotated property → `PROCESS_RESTART` (AMD-66 §2.3). A **throwing listener rejects the candidate** (active model preserved, surfaced as a reload failure — AMD-66 §4). Listener return drives the section's `appliedClassification`; M6.4 **reports** classifications (`ReloadResult` + event) — it does NOT restart integrations or the process (§3.3 step 7; the supervisor consumes classifications in M9).
- **DP-5 (diff semantics):** diff active-vs-candidate over section `values` key-by-key (`Objects.equals`); removed key → `newValue` null, added key → `oldValue` null (both nullable on `ConfigChange`). Sections present only in one model count as changed. `ConfigChange.reload` carries the per-property `x-reload` classification (the schema annotation), independent of the listener's section-level override. Add the §4.3 filter methods `hot()`/`integrationRestart()`/`processRestart()` to `ConfigChangeSet` now (Phase-3 per MODULE_CONTEXT decision #10).
- **DP-6 (write path — §3.5 steps 1–8 verbatim):** caller supplies `List<ConfigMutation>` + the `fileModifiedAt` token. Under the lock: stat file; mtime ≠ token → `ConcurrentModificationException` (§6.7). Apply mutations to a deep copy of the raw map (null `newValue` = remove key → revert-to-default semantics); serialize to YAML via snakeyaml-engine emitter; validate the mutated copy — FATAL/ERROR → `ConfigurationValidationException`, file untouched; **before the FIRST UI/API write ever performed on this file, create `config.yaml.bak.{ISO8601}` and log a WARNING** (§3.5 mitigation — REC-131; "first" = backup file absence check, timestamp from the injected `Clock`); atomic flush = write `config.yaml.tmp` same-directory → fsync file → rename → fsync directory; update the token; run the reload pipeline; release lock. Comment loss on write is the **documented Locked-doc limitation** — do NOT attempt comment preservation (Out of Scope).
- **DP-7 (⚠ PM ruling [R4] — reload/write publish `config.section_reloaded` ONLY; `ConfigChangedEvent` stays production-unpublished):** Doc 06 §12.4 (Locked, security) forbids config **values** in event payloads, but the frozen `ConfigChangedEvent(configPath, previousValue, newValue)` is value-bearing by shape — publishing it would violate §12.4; changing it is an AMD. Resolution: the §3.3 step-6 notify obligation is discharged by AMD-70's `config.section_reloaded` (names + counts + classification — §12.4-safe). `ConfigChangedEvent` remains registered in all manifests, gains no publish site; its reconciliation is a queued FUTURE-AMD (Research 13 REC-138 family). If you find this untenable in implementation, STOP and escalate — do not publish values.
- **DP-8 (new event record):** `ConfigSectionReloadedEvent(String sectionPath, int changeCount, int issueCount, String appliedClassification)` in `com.homesynapse.event` (flat package), `@EventType(EventTypes.CONFIG_SECTION_RELOADED)`, `CONFIG_SECTION_RELOADED = "config.section_reloaded"`. AMD-70 §2.1 payload verbatim; all components `java.base` (type-residency rule — `appliedClassification` = `ReloadClassification.name()`, the enum is CONSUMED never referenced). Guards: non-null/non-blank `sectionPath`, non-null/non-blank `appliedClassification`, `changeCount >= 1`, `issueCount >= 0`. One event per CHANGED section, published after listener classification, same virtual thread (AMD-70 §4).
- **DP-9 (⚠ PM ruling [R5] — publish metadata):** `config.section_reloaded` publishes exactly like the ruled `config.validation_completed`: DIAGNOSTIC priority, `EventOrigin.SYSTEM`, **null `eventTime`**, `publishRoot`, well-known system subject (`SubjectRef.system(SystemId)`), null actor. Publish failure is logged and NEVER fails the reload (AMD-70-INV-01 precedent).
- **DP-10 (R1 — per-ERROR `config_error` startup publish):** in `load()` (NOT `reload()` — Doc 06 §4.5: "at startup"), after each **completed** validation pass, publish one `ConfigErrorEvent` per **ERROR**-severity issue (not WARNING, not FATAL rows), with the DP-9 metadata. Fires on FATAL passes too, before the throw (the R3/`validation_completed` consistency rule). Mapping: `path` = issue path; `severity` = `Severity.name()`; `message` = issue message; `appliedDefault` = `String.valueOf(appliedDefault)` or the literal `"(none)"` when null (the record is non-null-guarded). **§12.4 fence:** if the issue's schema path carries `x-sensitive: true`, publish `message` as `"[REDACTED]"` and `appliedDefault` as `"(none)"` — never the invalid value. Implement the sensitivity lookup as a small package-private helper over the composed schema (walk for `x-sensitive` annotations; dormant until sensitive properties exist — correctness over coverage).
- **DP-11 (R2 — §3.7 step-7 migration write-back + backup):** when `load()` actually ran a migration chain AND the post-migration document passed validation without FATAL, persist the migrated document: first copy the original file to `{config_path}.pre-migration-v{fromMajor}.{fromMinor}` (never auto-deleted — §3.7), then atomically write the migrated YAML (same temp-fsync-rename discipline) with `schema_version: { major, minor }` stamped to the declared pair, then refresh the `fileModifiedAt` token from the NEW file mtime. A crash anywhere in the window leaves either the intact original or backup+original (recoverable) — never a torn file. Zero-config (absent file) and no-migration loads write nothing back.
- **DP-12 (reload re-parse CoreSchema — pinned obligation #3):** every `LoadSettings` built for the reload path carries the explicit YAML 1.2 `CoreSchema` — in practice: **reuse `YamlLoader`**, do not construct new `LoadSettings` anywhere else. The write path's emitter `DumpSettings` needs no schema but keep emission in one place. (The M6.1a gate-fix lesson; MODULE_CONTEXT gotcha.)

## What to Watch Out For

- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this module's test code — the `NO_DIRECT_TIME_ACCESS` ArchUnit rule scans non-whitelisted test classes and fails `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructor/`@BeforeEach`. (config is NOT whitelisted.)
- **CoreSchema lockstep (MODULE_CONTEXT gotcha, verbatim):** snakeyaml-engine's DEFAULT LoadSettings schema is the JSON schema — `~` parses as the string `"~"`. Any new `LoadSettings` construction site must `.setSchema(new CoreSchema())`. The reload re-parse is exactly such a site — route through `YamlLoader` (DP-12).
- **`fileModifiedAt` is real file mtime** (M6.1a DP-5 resolution): refresh from the actual file after every write/write-back; clock-sourced only for the absent-file zero-config case. Mtime granularity on some filesystems is coarse — tests that touch-then-write should force distinct mtimes explicitly (set `Files.setLastModifiedTime`), not sleep.
- **Event-time rule:** derived/observability events carry **null `eventTime`** — never `Instant.now()` in the publish path.
- **`-Werror` / `[exports]`:** no module-info changes here, but new public methods on `ConfigChangeSet` reference only same-module types — keep it that way. Run the P5 shift-left check before handoff: targeted `./gradlew :config:configuration:compileJava :core:event-model:compileJava :core:persistence:compileJava` if the sandbox permits (NOT the full check — that stays Nick's gate).
- **The diff must compare post-pipeline models** (post-migration, post-default-merge) — diffing raw text would classify cosmetic edits as changes.
- **Don't double-publish:** `write()` triggers the reload pipeline → `config.section_reloaded` fires from the reload leg only; no write-specific event (DP-7).
- **ArchUnit Rule 5:** `com.homesynapse.config..` is exempt from `NO_DIRECT_FILESYSTEM_IN_CORE` (ratified AMD-71 ruling, banner in `HomeSynapseArchRules`) — your file IO is legal; do NOT widen the exemption or touch the rule.
- **`ConfigurationReloadException` vs `ConfigurationValidationException`:** reload rejection throws the former (`config.reload_failed`, 422); write-validation rejection throws the latter (event-model, checked). `ConcurrentModificationException` is unchecked (MODULE_CONTEXT gotcha).
- **Stream.of aggregation sites:** `HomeSynapseCore`/`IntegrationTestHarness` consume the manifest — verify-no-edit, don't assume (the M4.C lesson).
