## What This Implements (and the exact boundary of "amendment-free")

In production today the canonical attribute map is **never populated**: `StateProjection.applyToState` writes an entity's `attributes` only on inbound `state_changed`, and the sole production producer of `state_changed` is the `DerivationRule` — currently the no-op constant lambda `MINIMAL_DERIVATION_RULE = context -> List.of()` in `HomeSynapseCore`. Every state-reading path is therefore dark.

**M4.0b-1 makes derivation real for *new* events.** It promotes a production `DerivationRule` that, on an inbound `state_reported` whose value differs from the prior canonical value, emits a `state_changed` draft; the projection (existing machinery) publishes it on LIVE, records it in the self-produced filter, and applies it — populating `attributes`. It also swaps the placeholder `MinimalProjectionAdvancer` for the REC-28 `DispatchingProjectionAdvancer`.

**What it deliberately does NOT do (the amendment-free boundary):** it does **not** bump `projectionVersion`, so no reconciliation/replay-from-zero fires and **historical** attributes (events logged before this WU) are **not** backfilled. Historical backfill is M4.0b-2, which needs the AMD-41 §3.2.4 refinement (P2-blocked). Concretely: after this WU, a fresh boot reads the existing version-1 checkpoint, does not reconcile, and the real rule populates attributes only for `state_reported` arriving **after** the rule ships. That is the intended, correct boundary for a pre-launch system with no persistent deployments to upgrade.

## Behavioral Contracts (the deliverables)

### 1. Production `DerivationRule` — lift `EchoStateRule` into production
Create a production `DerivationRule` implementation in `com.homesynapse.state` (package-private), exposed to the composition root through a **public static factory** (DEC-M3-16 gateway — e.g. a `DerivationRule.production()` / `ProductionDerivationRule.create()`-style factory on a public type in `com.homesynapse.state`; exact name/shape is yours, see Pushback). Lift the change-detect logic verbatim-in-spirit from the testFixtures `EchoStateRule`:

- On inbound `state_reported` (`payload instanceof StateReportedEvent sr`): `key = sr.attributeKey()`, `newValue = sr.value()`, `oldValue =` the prior canonical value for `key` (String form: `StringValue.value()` else `rawValue().toString()`).
- If `Objects.equals(oldValue, newValue)` → return `List.of()` (no change → no derived event).
- Else emit one `StateChangedEvent(key, oldOrEmpty, newValue, env.eventId())` wrapped in an `EventDraft(EventTypes.STATE_CHANGED, 1, env.eventTime(), env.subjectRef(), EventPriority.NORMAL, EventOrigin.SYSTEM, payload, env.actorRef(), null)`. (Confirm the `EventDraft` arg list against source — gate G8.)
- Non-`state_reported` payloads → `List.of()`.

**String semantics only.** Comparison and `StateChangedEvent.{oldValue,newValue}` remain `String` (the existing serialized form). **Do not** introduce `QuantityValue`/`ArrayValue` or a typed comparator — those are M4.B3 / M4.0b-2 and the types do not exist in source (gate G9). The rule **MUST** be deterministic (INV-PROJ-01) and **MUST NOT** publish or mutate the store (the projection owns both phases).

### 2. `DispatchingProjectionAdvancer` (REC-28) replaces `MinimalProjectionAdvancer`
Provide a `ProjectionAdvancer` implementation in `com.homesynapse.state` that dispatches by event type via a **constructor-injected, package-private per-event-type handler set** (no `ServiceLoader`, no `.handlers` subpackage — REC-28 mods A/B/C). It replaces `MinimalProjectionAdvancer` (today in `com.homesynapse.lifecycle`) at the composition root. It honours the existing `ProjectionAdvancer` contract exactly: open a short-lived bounded read on `EventStore.readFrom(fromPosition, min(maxRows, DEFAULT_MAX_ROWS))`, invoke the caller's `processor` per envelope in `globalPosition` order, close the read before returning, and return an `AdvanceResult(lastProcessedPosition, eventsProcessed, hasMore)`. **The advancer MUST NOT publish** and MUST NOT spawn threads (AMD-41 §3.2.1, `ProjectionAdvancer` Javadoc).

**⚠ Key design decision — read before you build the dispatch (Pushback expected).** There is a genuine tension between the REC-28 assessment language ("the dispatch table IS the derivation mechanism; no new events") and **plan §4.2, which overrides it**: the production `DerivationRule` *publishes* real `state_changed` so automation triggers wake. In current source these are **two separate collaborators** — `StateProjection` calls `rule.evaluate(...)` directly in the LIVE `onEvent` path (which does **not** use the advancer at all), and uses the advancer only in `processBatch`. Therefore:

- The **`DerivationRule` remains the publishing seam** (Contract 1). Do **not** move derivation/publication into the advancer's handlers — that would bypass derivation on the LIVE `onEvent` path and contradict plan §4.2.
- `DispatchingProjectionAdvancer`'s "per-event-type handlers" dispatch the **read/forward** concern (which event types the projection cares to forward to its processor), **not** derivation. For unknown/uninteresting types the handler is a no-op forward (or a skip), but it **must not** change the `globalPosition`/cursor accounting the existing contract guarantees.
- **`AdvanceResult` has no `skipped()` factory** (gate G7). The assessment's "`AdvanceResult.skipped()`" does not exist in source. If your dispatch needs a skip signal, either (a) keep `eventsProcessed` accurate and forward all types (recommended — simplest, preserves the M3.7 cursor semantics), or (b) propose adding `AdvanceResult.skipped()` as an explicit deviation and justify it. **Do not silently invent it.**

If, after reading source, you conclude the cleanest REC-28-faithful design differs from the above (e.g. the dispatch belongs in the rule rather than the advancer, or the advancer should stay minimal and only be renamed), **say so before implementing** — this is the highest-judgment call in the WU.

## What to Watch Out For

- **Gateway visibility (the recurring trap).** Both new types are package-private in `com.homesynapse.state`; `HomeSynapseCore` (in `com.homesynapse.lifecycle`) cannot construct them directly. Expose each via a **public static factory on a public same-package type** (DEC-M3-16; the `StateQueryService.materialized(...)` and M3.6e.1 `RestFilters` precedents). Do not promote the impl classes to `public` and do not add a `requires` to reach them.
- **Don't move derivation into the advancer.** Plan §4.2 is explicit: the rule publishes `state_changed`. The LIVE `onEvent` path doesn't touch the advancer, so advancer-side derivation would silently never run on LIVE. (See Contract 2's design-decision box.)
- **`AdvanceResult.skipped()` is a phantom** (G7) — same failure mode as the `MinimalDerivationRule` phantom. Verify against source; forward all types and keep `eventsProcessed` accurate unless you propose `skipped()` as an explicit deviation.
- **No `projectionVersion` bump, no backfill.** If you find yourself wanting to bump to `2` (so historical attributes reconstruct), STOP — that is M4.0b-2 and needs the P2-blocked amendment. This WU populates attributes for **new** events only; that is correct and intended.
- **String change-detect only** (G9). The `EchoStateRule` `lookupAttribute` `StringValue.value()` / `rawValue().toString()` form is the whole comparison surface. Typed comparison (REC-90) and `QuantityValue`/`ArrayValue` (M4.B3) are out of scope and their types don't exist yet.
- **Determinism = no clock branching** (INV-PROJ-01). The injected `Clock` is for derived-event timestamps only; never branch on its value.
- **Clock injection in tests — no exception.** `NO_DIRECT_TIME_ACCESS` fails the build on `Instant.now()`/`System.nanoTime()`/`System.currentTimeMillis()` in non-whitelisted test code. Inject `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)`.
- **M4.0a lesson — gate/prove the full path, not the happy case.** M4.0a's D-1 slipped because a REPLAY-side path was left unexercised. Make Contract 4 genuinely exercise REPLAY (a differing `state_reported` under REPLAY that *would* publish on LIVE), not an empty-log no-op.
- **Backward-compatible change shape (M4.0a lesson).** Prefer additive seams (new package-private types + a factory) over changing the `StateProjection.create(...)` signature. The signature already accepts `DerivationRule` and `ProjectionAdvancer` — you are swapping the *instances* passed at the composition root, not the API.
