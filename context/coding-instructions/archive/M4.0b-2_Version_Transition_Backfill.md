## Constraints to Cite by ID (look up full text before relying on them)

- **AMD-50 §2.2 supersession** — while the gate is active, re-derived current-rule drafts are authoritative for `attributes`; a logged prior-version `state_changed` advances `stateVersion` (cursor) but its attribute/`lastChanged` write is **suppressed**.

- **AMD-26/27** — sqlite-jdbc through bounded **platform-thread** executors; no virtual threads for DB I/O. **LTD-11** — `ReentrantLock`, never `synchronized`.
- **DEC-M3-05** — snapshot format (Jackson JSON + version headers); the reconstructed attributes persist through the existing checkpoint path (no new serializer).

## Behavioral Contracts (the deliverables)

### 1. Bump `projectionVersion` 1→2
In `HomeSynapseCore` step 6, change the literal `1` passed as `StateProjection.create(...)` arg #2 to `2`. This is the trigger — first boot on a version-1 checkpoint now mismatches and reconciles.

### 2. The reconciliation provenance gate (AMD-50 §2.2)
Add a projection-internal `backfillActive` flag (default `false`):
- **Set `true`** in `StateProjection.initialize()`'s reconciliation branch — the branch that already detects `persistedVersion != projectionVersion` (escape hatch off), calls `stateStore.clear()`, sets `cursorPosition = 0L`, and populates `reconciledFromVersion`/`reconciledToVersion`/`reconciledAt`. (Do **not** set it on the escape-hatch path or when versions match.)
- **Clear `false`** when the projection exits REPLAY — in `onCaughtUp()` (the REPLAY→LIVE signal), alongside the existing `caughtUpFired` CAS + REC-80 metric emission.
- The flag is the §2.2 gate: backfill apply (Contract 3) and supersession suppression (Contract 4) act **only** while it is `true`.

### 3. Backfill apply of re-derived drafts during REPLAY (AMD-50 §2.1/§2.3) — gate BOTH paths
While `backfillActive`, the re-derived `state_changed` drafts the rule produces during the reconciliation replay must be **applied to in-memory state** (so historical attributes reconstruct) — **without publishing** and **without a second `stateVersion` increment**.

- This must be added to **both** REPLAY derivation paths — `onEvent` **and** `processBatch` (AMD-50 §7; the **D-1 lesson** — gate every path that produces the guarded behavior, not just one). `onEvent` is the active production REPLAY path (`ReplayDriver`→`supervisor.deliver`→`onEvent`); `processBatch` carries the same derivation logic. Both currently apply derived drafts to state **only** inside their `mode == SubscriberMode.LIVE` block. Add a parallel `else if (backfillActive)` (REPLAY) apply that calls a **narrow attribute-write path** — not `publisher.publish`, not the full `applyToState` `state_changed` branch.
- **Narrow write (AMD-50 §2.3):** introduce a private helper (e.g. `applyBackfillAttribute(EntityId, StateChangedEvent sc, Instant causeEventTime)`) that updates only the entity's `attributes` map (`put(attributeKey, new StringValue(newValue))`) and `lastChanged`, and **preserves `stateVersion`** (no increment), `lastReported`, and `lastUpdated`. It must NOT route through `applyToState` (which increments the cursor and re-stamps timestamps). The triggering `state_reported`, applied normally, owns the single `stateVersion +1` and `lastReported`.
- **Timestamp source — `lastChanged` MUST be the causing event's time, NOT the projection wall-clock (`clock.instant()`).** The re-derived draft already inherits `eventTime` from its causing `state_reported` envelope (the derived-event-time rule); use that log-fixed value. Wall-clock would be both **semantically wrong** (it stamps every reconstructed historical attribute as "changed at rebuild time" rather than when the report occurred, and re-stamps them on every later transition) and a **latent rebuild non-determinism that a fixed test clock silently masks** — which is exactly the class of bug AMD-50 exists to prevent. Confirm which envelope field carries the deterministic, log-fixed time (`eventTime`); if a production `state_reported` can carry a null `eventTime`, confirm a deterministic fallback (e.g. the envelope's recorded/ordering time) — **never** the projection wall-clock.
- The backfill apply respects the rule's output: no draft (unchanged value) → no attribute write.
- **Resulting invariant (state it; build to it):** under an active gate, the narrow backfill helper is the **sole writer of `attributes`**, and every inbound logged `state_changed` is suppressed (cursor-only, Contract 4). Attribute values are therefore fully determined by re-derivation over the `state_reported` history, **independent of logged-event interleaving** — there is no apply-order ambiguity between re-derived and logged values.

### 4. Supersession suppression in `applyToState` (AMD-50 §2.2)
While `backfillActive`, when a **logged prior-version `state_changed`** replays as inbound, it must advance `stateVersion` (it is a log event — cursor, INV-01) but **must not** write `attributes` or `lastChanged`. Make the `applyToState` `state_changed` branch gate-conditional: under `backfillActive`, build the updated `EntityState` like the "other payload" branch (`stateVersion + 1`, `lastUpdated = now`, everything else from `prior`) — i.e. skip the `newAttrs.put(...)` and the `lastChanged = now`. Outside the gate, the branch is unchanged (writes attributes as today). *(For the production 1→2 case this path is never hit — the pre-2 log has no `state_changed` — but it is required for generality and is exercised by the §5 supersession test.)*

### 5. Remove `Clock` from `DerivationContext` (AMD-50 §2.4)
Change the `DerivationContext` record from `(EntityState priorState, EventEnvelope envelope, Clock clock)` to `(EntityState priorState, EventEnvelope envelope)`. Update: the record's compact constructor/Javadoc; the **two** `StateProjection` construction sites (`onEvent` and `processBatch` — drop the `clock` arg); `ProductionDerivationRule` (it already ignores the clock — remove the now-absent reference if any); and any test constructing a `DerivationContext`. This makes INV-03 airtight (no clock to branch on). The projection keeps its own `Clock` for timestamps; only the rule's context loses it.

### 6. AMD-50-INV-04 — gate–checkpoint coherence (verify, don't add a path)
Confirm by inspection that the **only** path that rebuilds in-memory state from `position = 0` is the `initialize()` reconciliation branch (version mismatch, escape hatch off). The escape-hatch path resumes from the stale snapshot (`cursorPosition = record.position()`), not from zero. Add a STOP-gate note in your report; if you find any other from-zero path that could run with `loadedProjectionVersion() == projectionVersion` and the gate off, STOP and report (it would leave historical attributes empty/stale).

### 7. Tests (AMD-50 §5 — all six; extend, do not greenfield)
Extend `ReconciliationTest` and `StateProjectionContractTest`:
1. **Rebuild idempotency** — complete M4.0b-1's `rebuildIdempotency_...` by adding the backfill assertions it deferred. **Assert equality on attribute-map *values* + `stateVersion`** (and `lastChanged`, which is event-time-sourced per Contract 3 and therefore deterministic). **Do NOT assert byte-identical full `EntityState`**: `lastUpdated`/`lastReported` are stamped with the projection wall-clock by `applyToState` (all branches, pre-existing) and are not reproducible across real rebuilds — under a fixed test clock they would coincidentally match, masking exactly the non-determinism this WU must avoid claiming away.
2. **Backfill ≡ native (attribute *values*)** — after a 1→2 reconciliation, attribute **values** equal a native log's; assert `stateVersion` equals the **reconciliation log's** event count (the `state_reported` count for 1→2), **not** the native log's, and **no double-increment** (INV-01).
3. **One-shot** — a second restart at version 2 finds persisted-version 2 → no reconciliation → backfill does not run (gate inactive, INV-02).
4. **Steady-state safety** — a post-transition (gate-inactive) replay applies logged `state_changed` once as inbound; re-derived drafts discarded for state.
5. **Supersession / generality (the regression guard for the AMD-50 review fix)** — a synthetic N→M reconciliation over a log containing a **spurious prior-version `state_changed`** (e.g. seed a `state_reported` "20.0" then "20.0000001" and a logged `state_changed`→"20.0000001", with a test rule that treats them as equal) reconstructs the **current rule's** value ("20.0"), the spurious change suppressed; `stateVersion` matches the prior snapshot's. *(Use a test `DerivationRule` that suppresses the within-tolerance change — you do NOT need the real typed comparator; a string/equality test rule that returns no draft for the second report suffices to exercise supersession.)*
6. **Determinism** — the production rule yields identical drafts for identical `(priorState, envelope)`; with the clock removed there is no clock-shift case.

## What to Watch Out For

- **Timestamp model — `[REVIEW]`, surfaced for Nick (conscious interim).** `applyToState` stamps `lastUpdated`/`lastReported` (and the LIVE `state_changed` branch's `lastChanged`) with the **projection wall-clock** across all branches and modes — pre-existing, so the full `EntityState` timestamp set is not rebuild-deterministic. M4.0b-2 fixes only the **new** write it introduces (the backfill `lastChanged` → event-time) and scopes its determinism tests to attribute values + `stateVersion`. **It does not change the existing `applyToState` timestamp behaviour.**
  - **Document the resulting interim explicitly** (in your completion report **and** `core/state-store/MODULE_CONTEXT.md`): after this WU, **`lastChanged` has mixed semantics** — **event-time** when set by the reconciliation backfill, **wall-clock** when set by the LIVE `applyToState` `state_changed` branch. This is a **conscious, named interim**, not a bug: the new backfill code is correct, the pre-existing wall-clock wart is deliberately untouched. Name it so a future reader does not flag it as a defect.
  - The unifier is a **separate, scheduled follow-up WU** (NOT this one): move the projection's whole timestamp model (`lastChanged`/`lastReported`, possibly `lastUpdated`) to event-time. Per Nick's 2026-05-29 call it needs a **design beat first** — it is a behavioral-contract change to `EntityState` semantics (Doc 03, likely AMD-11 staleness) that may be **amendment-worthy**, and it has read-path blast radius (`MaterializedStateQueryService` stale-recompute, any query/observability surface exposing those timestamps). Do **not** silently expand into it here.
- **Gate BOTH REPLAY paths (the D-1 lesson).** The backfill apply belongs in `onEvent` **and** `processBatch`. M4.0a's D-1 was exactly a guard added to one path and missed on another; do not repeat it. A test must exercise the production path (`onEvent` under REPLAY).
- **Narrow attribute write — do not reuse the `state_changed` branch.** Backfill drafts carry **no** `stateVersion` increment (INV-01) and touch only `attributes`/`lastChanged`. Routing them through `applyToState` would double-increment the cursor and re-stamp timestamps. Use the dedicated helper.
- **Supersession is gate-conditional, and `false` outside the gate.** The `applyToState` `state_changed` suppression must apply **only** while `backfillActive`. A bug that suppresses attribute writes in LIVE would make derivation a no-op in production. Test both regimes.
- **Cursor determinism is the invariant under test.** `stateVersion` = count of processed log events, identical across rebuilds (INV-01). Backfill drafts don't count; logged events (incl. suppressed `state_changed`) do.
- **`Clock` removal ripple.** `DerivationContext` loses its clock; the projection keeps its own. Grep every `new DerivationContext(...)` (two in `StateProjection`, plus tests) and update arity. Don't accidentally remove the projection's `clock` field.
- **Idempotency (INV-ES-05).** The 1→2 reconciliation replays the whole log; `applyToState` and the backfill helper must be idempotent under re-application. A second boot at version 2 must NOT reconcile (Contract 7 test 3).
- **Clock injection in tests.** `NO_DIRECT_TIME_ACCESS` — inject `Clock.fixed(...)`. The reconciliation/backfill tests need a fixed clock for the projection.
