# Spike: Does relocating `AttributeValue` to a `com.homesynapse.value` leaf break the event↔device cycle and compile clean?

**Question:** If we create a new JPMS module `com.homesynapse.value` (leaf, `requires` nothing but `java.base`), move the 10 self-contained value types into it, add `requires com.homesynapse.value` to every module that names those types, and update all imports — does the full module graph become **acyclic** and **compile clean** under `-Xlint:all -Werror`, with `com.homesynapse.event` **not** requiring `com.homesynapse.device`?

**Success criteria (all must hold):**
1. `./gradlew compileJava compileTestJava` (or per-module `:check` minus tests, see Methodology) succeeds with **zero** `cyclic dependence` errors and zero `-Xlint`/`-Werror` failures across the whole graph.
2. `com.homesynapse.event`'s `module-info.java` does **NOT** contain `requires com.homesynapse.device` (the cycle is genuinely broken, not papered over).
3. `AttributeValue` / the 8 variants / `AttributeType` resolve from `event`, `device`, `state`, `persistence`, **and every other importer the grep finds** (see Methodology step 5).
4. No type beyond the planned 10 had to move into `com.homesynapse.value` to make it compile (i.e., `AttributeSchema` and `AttributeValueUpcaster` stayed in `com.homesynapse.device`).

**Failure criteria (report and stop):**
- Any residual `cyclic dependence` after the relocation.
- A forced move of `AttributeSchema`, `AttributeValueUpcaster`, or any other type into `value` to compile (→ the minimal-move scope in design-note §3 is wrong; report what forced it).
- A module that names the value types but cannot reach them even with `requires com.homesynapse.value` (→ a transitive-export issue; report it).

**Time-box:** small — this is a package-rename + a new module + `module-info` edits + import updates + a compile. No tests are written or modified for the spike beyond what is needed to compile `compileTestJava`.

---

## Methodology

Read `homesynapse-core-docs/design/2026-05-31_AttributeValue_Module_Relocation_Design_Note.md` (esp. §3 what moves, §4 the `requires` graph) first. Then, in a **throwaway git worktree/branch off `98f705b`**:

1. **Create the module.** Add `include("core:value-model")` to `settings.gradle.kts` (next to the existing `core:event-model`/`core:device-model`/`core:state-store` includes). Create `core/value-model/build.gradle.kts` mirroring the **simplest leaf** core module's conventions plugin (e.g. the `homesynapse.*-conventions` plugin id pattern device-model uses) but with **no** `api(project(...))`/`implementation(project(...))` dependencies — `com.homesynapse.value` depends on nothing but `java.base`. Create `core/value-model/src/main/java/module-info.java`:
   ```java
   module com.homesynapse.value {
       exports com.homesynapse.value;
   }
   ```
2. **Move the 10 types** (`git mv`, then rewrite the `package` line `com.homesynapse.device` → `com.homesynapse.value` in each):
   `AttributeValue.java`, `BooleanValue.java`, `IntValue.java`, `FloatValue.java`, `StringValue.java`, `EnumValue.java`, `QuantityValue.java`, `ArrayValue.java`, `DegradedAttributeValue.java`, `AttributeType.java`
   from `core/device-model/src/main/java/com/homesynapse/device/` → `core/value-model/src/main/java/com/homesynapse/value/`. Public contracts unchanged — only the `package` declaration changes.
3. **Add the `requires com.homesynapse.value` edges** (design-note §4): `requires transitive com.homesynapse.value` on `com.homesynapse.event` and `com.homesynapse.device` (both re-export the types on their public API); `requires com.homesynapse.value` (transitive where the module re-exports value types on its own API, plain otherwise) on **every other module the grep in step 5 finds** — known so far: `com.homesynapse.state`, `com.homesynapse.persistence`, **`rest-api`** (4 files import the value types — `api/rest-api`). Add to any others the grep reveals (automation, websocket-api, integration-runtime, observability — do not assume; grep). Add the matching `api`/`implementation` Gradle dependency on `:core:value-model` to each such module's `build.gradle.kts`.
4. **Add in-package imports inside device-model.** Every device-model file that previously used `AttributeValue`/`AttributeType`/a variant **in-package** (no import, because they lived in `com.homesynapse.device`) now needs `import com.homesynapse.value.…;`. This is the easy-to-miss part — it is NOT just the one file that already had an explicit import. `AttributeSchema` and `AttributeValueUpcaster` are the obvious ones; grep device-model for bare uses.
5. **Sweep all imports repo-wide.** Run (and act on):
   ```
   grep -rlE "com\.homesynapse\.device\.(AttributeValue|BooleanValue|IntValue|FloatValue|StringValue|EnumValue|QuantityValue|ArrayValue|DegradedAttributeValue|AttributeType)" --include=*.java . | grep -v /build/
   ```
   Update every `import com.homesynapse.device.{thatType}` → `import com.homesynapse.value.{thatType}`. The known module hits at `98f705b`: `core/state-store` (~15 files), `api/rest-api` (~4), `core/persistence` (~3), `core/device-model` (~1 explicit + the in-package users from step 4). The grep is authoritative — edge and import-update **every** module it returns.
6. **Compile the graph.** `./gradlew compileJava compileTestJava` (or, if faster, `:core:value-model:compileJava` then `:core:device-model:compileJava :core:state-store:compileJava :core:persistence:compileJava :api:rest-api:compileJava` + their `compileTestJava`). The build conventions already enforce `-Xlint:all -Werror`. A clean compile across all modules with no `cyclic dependence` is the answer.

## Where to put it
- Code: a **throwaway worktree/branch** — NOT a commit to the production tree until the production M4.0b-4a instruction runs. (If the worktree is clean and complete, M4.0b-4a may promote it rather than redo the diff — note that in the spike report.)
- Findings: report back (a) PASS/FAIL against the success criteria, (b) the **complete** set of modules that needed `requires com.homesynapse.value` (especially any beyond event/device/state/persistence/rest-api — this corrects design-note §4), (c) whether any type beyond the 10 was forced to move, (d) whether the vestigial `device → event` edge could be dropped (does anything in device-model main still import `com.homesynapse.event`? — informational, do NOT change it in the spike).

## What this spike must NOT become
Production code. It is the empirical gate. If it passes, the PM issues the production **M4.0b-4a** relocation instruction (full format: MODULE_CONTEXT updates, green `./gradlew check`, WUCP, commit). If it fails, the findings revise design-note §3/§4 before any production relocation.
