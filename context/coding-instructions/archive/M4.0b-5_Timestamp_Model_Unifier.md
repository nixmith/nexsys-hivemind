## What This Implements

`StateProjection` currently stamps `EntityState`'s three activity timestamps — `lastChanged`, `lastUpdated`, `lastReported` — from the **projection wall-clock** (`clock.instant()`) in every LIVE `applyToState` branch (and in entity adoption seeding), while the AMD-50 reconciliation backfill stamps `lastChanged` from **event-time** (`eventTime ?? ingestTime`). The same observable fields are therefore wall-clock in LIVE and event-time on a reconciliation replay, so they **diverge across every `projectionVersion` bump** — a replay-determinism gap (AMD-52 made the 3→4 reconciliation a live path), and a latent contradiction of Doc 03 §4.1, which already specifies `event_time ?? ingest_time` for all three.

This WU sources all three activity timestamps from `eventTime ?? ingestTime` in **every** projection path (LIVE `applyToState` all branches, plus entity-adoption seeding), making the materialized `EntityState` a pure function of the event log for those fields (AMD-53-INV-01, which extends AMD-50-INV-03 from the `DerivationRule` to the projection's materialization). `staleAfter`/`stale` are explicitly **excluded** and stay wall-clock (AMD-53-INV-02 — they are real-time freshness, compared against `Instant.now()`). The materialized-output change bumps `projectionVersion` **4 → 5**, riding AMD-50's frozen reconciliation backfill unchanged; the upgrade-boot reconciliation heals legacy wall-clock timestamps so checkpoint == replay output for every entity.

This is small and bounded: the source of three timestamp fields. **No** attribute-value, `availability`, `stateVersion`, event-shape, checkpoint-shape, codec, or `module-info` change.

## Settled Decisions (from AMD-53 — NOT open questions)

- **DP-1 (the rule, §2.1).** `lastChanged`, `lastUpdated`, `lastReported` ← `eventTime ?? ingestTime` of the **causing envelope**, in every `applyToState` branch and in adoption seeding. `applyToState` computes the stamp **once** from its inbound `envelope` (the envelope is in hand — no new plumbing) and uses it for every activity-timestamp write in that call.
- **DP-2 (which field per event).** Each field follows its existing ownership — you are changing only the *source*, not which branch writes which field: `state_reported` writes `lastUpdated`+`lastReported`; LIVE `state_changed` writes `lastChanged`+`lastUpdated`; supersession/`availability_changed`/other write `lastUpdated`; `lastChanged` stays suppressed under the active backfill gate.
- **DP-3 (adoption seeding, §1.5).** `initialEntityState` seeds the three from the triggering event-time, not wall-clock — so a never-changed/never-reported field is still a pure function of the log. Thread the seed instant in from both call sites (`applyToState` → its envelope; `applyBackfillAttribute` → its `causeEventTime`).
- **DP-4 (carve-out, §2.2 / AMD-53-INV-02).** `staleAfter`/`stale` are **untouched** and stay wall-clock (real-time freshness). Do not event-time them. Do not wire staleness-threshold resolution (still deferred — `applyToState` carries `staleAfter` forward as `prior.staleAfter()`; leave it).
- **DP-5 (the bump, §2.4).** `projectionVersion` 4 → 5 at the composition root; rides AMD-50's frozen backfill unchanged (AMD-50 §2.5) — author **no** new backfill or §3.2.4 refinement.
- **DP-6 (`clock` stays injected).** `StateProjection` keeps its injected `clock` for its legitimate real-time uses — `reconciledAt` (L675), `lastCheckpointAt` (L689), `replayStartedAt`/replay-duration metric (L693/L512), and the staleness machinery. It is simply no longer the source of any `EntityState` activity timestamp. `applyToState` itself should no longer read the clock at all once the change lands.

## Technical Specification

1. **`applyToState`:** compute the event-time stamp once at the top from the inbound envelope (reuse `backfillTimestamp(envelope)` — consider renaming it to a neutral name like `eventTimestamp` since it is no longer backfill-only; that is a HOW detail). Replace `Instant now = clock.instant();` as the activity-timestamp source. In each branch's `new EntityState(...)`, the activity-timestamp positions that were `now` become the event-time stamp; positions that were `prior.xxx()` (e.g. suppressed `lastChanged`) stay `prior.xxx()`. After the change, `applyToState` does not call `clock.instant()`.
2. **`initialEntityState`:** change to `initialEntityState(EntityId entityId, Instant seed)` (or equivalent) and seed `lastChanged`/`lastUpdated`/`lastReported` from `seed`. Call sites: `applyToState` L783 passes its computed event-time stamp; `applyBackfillAttribute` L930–931 passes its `causeEventTime`. `staleAfter` stays `null`, `stale` stays `false`.
3. **`applyBackfillAttribute` / `backfillTimestamp`:** behaviorally unchanged (already event-time). Update the `[REVIEW]` note (L918–921) to state the split is resolved (LIVE and backfill now agree by construction, AMD-53).
4. **`HomeSynapseCore`:** literal `4 → 5`; comment block → "projectionVersion 5 (M4.0b-5, AMD-53): event-time activity-timestamp materialization; the 4→5 reconciliation heals legacy wall-clock `lastChanged`/`lastUpdated`/`lastReported` (AMD-50 §2.5 backfill, unchanged)."
5. **`applyToState` Javadoc** (L778-781 region and the method header): update any prose describing wall-clock timestamping to event-time.

## Test Requirements (the verification gate — AMD-53 §5)

Extend `StateProjectionContractTest` (testFixtures) and `ReconciliationTest`. **These tests define "correct."**

1. **(GATE) Headline determinism — LIVE ≡ replay-from-zero, all three fields.** Process a multi-entity log LIVE; capture each entity's `lastChanged`/`lastUpdated`/`lastReported`. Rebuild the same log from `position = 0`. Assert the three timestamps are **identical per entity**. **The test `Clock` must be `Clock.fixed(...)` at an instant that differs from every envelope `eventTime`/`ingestTime` in the corpus** — so a wall-clock regression cannot pass. This is the test that makes the §2.4 caveat a gate: it fails if `lastUpdated`/`lastReported` are not event-time post-replay, not only `lastChanged`.
2. **Event-time sourcing + null fallback.** A `state_changed` and a `state_reported` whose `eventTime` ≠ processing wall-clock → the field equals `eventTime`. An envelope with `eventTime == null` → the field equals `ingestTime`. Cover all three fields plus the `availability_changed` `lastUpdated`.
3. **Carve-out unchanged.** `stale` still flips on real `Instant.now()` vs `staleAfter` (read-time + scanner), independent of the activity-timestamp change.
4. **Adoption-seeding determinism (§1.5).** A never-changed entity (only `state_reported` in its history) has a **deterministic** `lastChanged` across two independent rebuilds — the seed event-time, not a wall-clock value.
5. **4→5 reconciliation heal (§2.4 / §3.4).** Materialize an entity under a wall-clock-stamped checkpoint (the legacy regime), run the 4→5 reconciliation, assert the three activity timestamps now equal a fresh-from-zero replay's, and that `stateVersion`/attribute values are unchanged (AMD-50-INV-01 — no double-increment).
6. **No-op report keeps `lastChanged`.** A `state_reported` whose value matches canonical state advances `lastUpdated`/`lastReported` (to event-time) and `stateVersion`, but leaves `lastChanged` unchanged (Doc 03 §3.2 LIVE contract).

## What to Watch Out For

- **The §2.4 caveat is the whole point — verify it, don't assume it.** The 4→5 heal re-derives all three fields **only because** `applyToState`'s `state_reported` branch and the adoption seeding are fixed; the backfill helper alone touches only `lastChanged`. Test #1 and #5 must prove `lastUpdated`/`lastReported` come out event-time after a replay, or the bump half-corrects.
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` anywhere in test code. The ArchUnit rule `NO_DIRECT_TIME_ACCESS` scans test classes in non-whitelisted packages (`com.homesynapse.state` is **not** whitelisted) and will fail `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` and pass it via constructor / `@BeforeEach`. (Per §5 #1 the fixed clock must also differ from the corpus event/ingest times.)
- **Keep `lastChanged` suppressed under the active backfill gate** (supersession branch) — change only its `lastUpdated`. Do not "fix" the suppression; it is AMD-50 §2.2 and is exercised by an existing test.
- **Do not touch** the typed-value pipeline, `AttributeValue` codec, event/checkpoint shapes, attribute values, `availability`, or the `stateVersion` cursor (AMD-53 §6). Only the *source* of three timestamp fields changes.
- **`eventTime ?? ingestTime` is always well-defined:** `ingestTime()` is never-null; `eventTime()` is nullable. Reuse `backfillTimestamp`'s exact logic — do not invent a different fallback.
- Do **not** alter the AMD-50 backfill mechanism, and do **not** add a new `projectionVersion` reconciliation refinement — the 4→5 transition rides AMD-50 §2.5 unchanged.
