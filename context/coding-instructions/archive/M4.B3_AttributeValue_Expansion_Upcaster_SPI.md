## What This Implements

The device model's canonical typed-value contract is the sealed `AttributeValue` hierarchy in `com.homesynapse.device`. Today it admits exactly five primitive shapes (`BooleanValue`, `IntValue`, `FloatValue`, `StringValue`, `EnumValue`). AMD-47 expands it by three public records so the MVP device set can be represented faithfully: `QuantityValue` (carries the (value, unit) "moat decision" at the value layer, normalized to canonical units at construction), `ArrayValue` (ordered, full-replacement list), and `DegradedAttributeValue` (the subtype-level upcast-failure fallback, mirroring `DegradedEvent`). It adds the `AttributeValueUpcaster` SPI ‚ÄĒ the migration seam for evolving stored values across type changes ‚ÄĒ and three `AttributeType` constants (`QUANTITY`, `ARRAY`, `DEGRADED`) so each value variant keeps a 1:1 type classifier and exhaustive switches stay total.

This WU delivers the **types + SPI + the schema-level INV-04 guard + the AMD-47 ¬ß5 contract tests**. It does **not** wire the upcaster into the projection path and does **not** touch the typed-store / `projectionVersion` machinery ‚ÄĒ those are AMD-52 / M4.0b-3 (see Out of Scope and the two carried items below).

## Technical Specification

### `QuantityValue` (record) ‚ÄĒ `com.homesynapse.device`
**Responsibility:** Carry a physical quantity as a canonical-normalized magnitude + canonical unit symbol.
**Implements:** `AttributeValue`

```java
public record QuantityValue(double value, String unit) implements AttributeValue {
    public QuantityValue { /* normalize + validate ‚ÄĒ see contract */ }
    @Override public Object rawValue();          // boxed Double (canonical magnitude), never null
    @Override public AttributeType attributeType(); // QUANTITY
}
```

**Behavioral contract (AMD-47-INV-03):**
- The **stored components are canonical**: the compact constructor takes the supplied `(value, unit)`, looks up the dimension for `unit`, converts the magnitude to that dimension's **canonical unit**, and **reassigns** `value` = canonical magnitude and `unit` = canonical unit symbol. Construction with the canonical unit is the identity conversion.
- `rawValue()` returns the canonical magnitude boxed as `Double` (never null) ‚ÄĒ keeps a `QuantityValue` magnitude-comparable with a `FloatValue` for generic processing.
- **Fail-closed at construction** (deterministic, never silently coerced, never degraded):
  - `unit == null` ‚Üí `NullPointerException`.
  - `unit` blank ‚Üí `IllegalArgumentException`.
  - `value` non-finite (`Double.isNaN` / `isInfinite`) ‚Üí `IllegalArgumentException`.
  - `unit` **unrecognised** (not in the catalogue) ‚Üí `IllegalArgumentException`.
- **Pure & deterministic:** no external units library, no I/O, no locale/clock dependence. Unit matching is **exact string equality** against the catalogue keys (no locale-folding; do not lower-case). Two same-dimension `QuantityValue`s are magnitude-comparable on the canonical `value`.

**Canonical-unit catalogue + conversion table (PM-specified, aligned to Doc 02 ¬ß3.7 + the MVP capability set).** Implement these conversions; the data-structure representation is your call. You MAY extend within the same fail-closed/deterministic contract, but do not add a units library.

| Dimension | Canonical unit | Accepted input unit ‚Üí conversion to canonical |
|---|---|---|
| Temperature | `"¬įC"` | `"¬įC"` ‚Üí identity; `"K"` ‚Üí `v - 273.15`; `"¬įF"` ‚Üí `(v - 32) * 5.0/9.0` |
| Power | `"W"` | `"W"` ‚Üí identity; `"kW"` ‚Üí `v * 1000.0`; `"mW"` ‚Üí `v * 0.001` |
| Energy | `"Wh"` | `"Wh"` ‚Üí identity; `"kWh"` ‚Üí `v * 1000.0`; `"J"` ‚Üí `v / 3600.0`; `"kJ"` ‚Üí `v / 3.6` |
| Illuminance | `"lux"` | `"lux"` ‚Üí identity; `"klx"` ‚Üí `v * 1000.0` |
| Ratio / percent | `"%"` | `"%"` ‚Üí identity |

> **fp determinism note (put in test design):** "same-dimension different-unit inputs land on **identical** canonical `value`" (¬ß5 test #2) is bit-exact only when the conversion factor is exactly IEEE-754-representable. Use **multiplicative** dimensions for the cross-unit equality assertion ‚ÄĒ e.g. `new QuantityValue(2, "kWh").value()` **==** `new QuantityValue(2000, "Wh").value()` (both exactly `2000.0`); `1 kW == 1000 W`; `1 klx == 1000 lux`. **Temperature is affine** (`K`/`¬įF` involve an offset) and is generally **not** bit-exact across units ‚ÄĒ assert it within a tolerance (or with `isCloseTo`), and use it for the "repeated construction is byte-identical" check (same input ‚Üí same bits) rather than the cross-unit equality check.

### `ArrayValue` (record) ‚ÄĒ `com.homesynapse.device`
**Responsibility:** Ordered, unmodifiable, full-replacement list of element values.
**Implements:** `AttributeValue`

```java
public record ArrayValue(List<AttributeValue> elements) implements AttributeValue {
    public ArrayValue { elements = List.copyOf(elements); } // rejects null + null elements; empty OK; unmodifiable
    @Override public Object rawValue();             // the unmodifiable List<AttributeValue>, never null, possibly empty
    @Override public AttributeType attributeType(); // ARRAY
}
```

**Behavioral contract (AMD-47-INV-05):**
- Compact constructor does `elements = List.copyOf(elements)` ‚ÄĒ `List.copyOf` throws `NullPointerException` on a null list **and** on any null element; an empty list is permitted; the result is unmodifiable.
- `rawValue()` returns the unmodifiable `List<AttributeValue>` (never null).
- **Full-replacement semantics ‚ÄĒ no delta/patch.** This is a record property, not behavior: a new `ArrayValue` wholly replaces a prior value for the attribute (relied on by the bounded-window advancer). Element homogeneity is a **schema-level** concern for the future validator, **not** enforced by this record. Nesting (`ArrayValue` of `ArrayValue`) is permitted by the type but discouraged by schema ‚ÄĒ do not add a nesting guard.

### `AttributeValueUpcaster` (interface) ‚ÄĒ `com.homesynapse.device`
**Responsibility:** Migration seam for evolving stored attribute values across `AttributeValue` type changes. No `ServiceLoader` (DECIDE-04 ‚ÄĒ constructor injection downstream).

```java
public interface AttributeValueUpcaster {

    /** Whether this upcaster can transform a stored value of the given subtype-name / schema version. */
    boolean canUpcast(String storedTypeName, int fromSchemaVersion);

    /**
     * STRICT mode: returns the upcast value, or throws if the stored value cannot be upcast
     * (unknown/unsupported type or version, or malformed rawForm). Core projections use this ‚ÄĒ
     * a failed upcast halts; a DegradedAttributeValue is NEVER produced here.
     */
    AttributeValue upcast(String storedTypeName, String rawForm, int fromSchemaVersion);

    /**
     * LENIENT mode (default): returns the upcast value, or a DegradedAttributeValue preserving
     * the raw form and failure reason. Never throws for an un-upcastable value. Diagnostic/forensic
     * tools use this. Assumes a non-blank storedTypeName (a real stored subtype name).
     */
    default AttributeValue upcastLenient(String storedTypeName, String rawForm, int fromSchemaVersion) {
        try {
            if (!canUpcast(storedTypeName, fromSchemaVersion)) {
                return new DegradedAttributeValue(storedTypeName, rawForm,
                        "no upcaster for type " + storedTypeName + " at version " + fromSchemaVersion);
            }
            return upcast(storedTypeName, rawForm, fromSchemaVersion);
        } catch (RuntimeException e) {
            String reason = (e.getMessage() == null) ? e.toString() : e.getMessage();
            return new DegradedAttributeValue(storedTypeName, rawForm, reason);
        }
    }
}
```

**Behavioral contract (AMD-47-INV-02 / -04):**
- `upcast` (strict) is the failure-throwing primitive. The thrown type is your call (Coder freedom) ‚ÄĒ recommend an unchecked exception; `IllegalArgumentException`/`IllegalStateException` is acceptable, or a small dedicated `AttributeValueUpcastException extends RuntimeException` if you prefer precise catching. The lenient default catches `RuntimeException`, so either works.
- `upcastLenient` (lenient) never throws for an un-upcastable value ‚ÄĒ it yields a `DegradedAttributeValue` with `rawForm` preserved and a non-blank reason.
- **No implementation is authored** ‚ÄĒ only the SPI + this contract. The two-mode doctrine parallels the event upcaster / `DegradedEvent` (Doc 01 ¬ß3.10).
- **Ordering (INV-02) ‚ÄĒ production both-paths wiring is M4.0b-3 (DP-1).** Do NOT wire this into `StateProjection`. The M4.B3 test exercises the SPI contract directly via a test upcaster (below).

## What to Watch Out For

- **`CheckpointSerializer:238` and `EnumTransition:23` are NOT exhaustive switches** ‚ÄĒ they are `instanceof` patterns with fallbacks. The new variants flow through their generic branches (e.g. `CheckpointSerializer` will `.toString()` a `QuantityValue`/`ArrayValue`/`DegradedAttributeValue` via the `else { rawValue().toString() }` path). That lossy flatten is **by design** in the AMD-47 window (typed-store is AMD-52). **Do not "fix" it, do not add `default`/new branches, do not change it.**
- **`QuantityValue` is a record but its components are canonical.** Normalize **inside the compact constructor** by reassigning `value` and `unit` (records permit assigning the component parameters in a compact constructor). Do not add a static factory that bypasses the canonical form, and do not store the raw input unit.
- **Affine vs multiplicative conversions.** Temperature (`K`,`¬įF`) carries an offset ‚ÄĒ do not model it as a single scale factor, and do not assert bit-exact cross-unit equality for it (see the fp determinism note). Energy/power/illuminance are multiplicative and safe for the exact-equality test.
- **Unit matching is exact string equality** against the catalogue keys ‚ÄĒ including the `¬į` glyph. Use ASCII-safe source but the symbols are literal (`"¬įC"`, `"¬įF"`). Do not lower-case, trim beyond the blank check, or locale-fold. `@DisplayName`s must be ASCII (M3.7 lesson) ‚ÄĒ keep the degree glyph out of display names; it's fine in assertion data.
- **`DegradedAttributeValue` mirrors `DegradedEvent` precisely** ‚ÄĒ `rawForm` is non-null **only** (blank allowed), while `originalTypeName`/`failureReason` are non-blank. Do not over-validate `rawForm`.
- **Do not propagate the stale JSR-385 notes** from MODULE_CONTEXT (Key-Decision #3, GOTCHA ~L249, Phase-3-Notes "JSR 385 integration"). They are superseded by REC-93 and are being retired at this milestone's closeout. New Javadoc must say hand-rolled `String` units, never "Phase 3 adds JSR 385."
- **`module-info` must stay byte-identical.** If you believe a new `requires`/`exports` is needed, STOP and flag ‚ÄĒ it should not be (only `java.util.List`/`java.lang`/in-package types are used).
- **Tests must inject `Clock` if they touch time.** device-model is **not** on the `NO_DIRECT_TIME_ACCESS` whitelist; the ArchUnit rule scans test classes. Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in any test. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` if needed. AMD-47's types are clock-free by design (INV-03 forbids clock dependence in normalization), so this should be a non-issue ‚ÄĒ but the rule still scans, so keep `Instant.now()` etc. out of the new test classes.
