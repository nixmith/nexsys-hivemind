## What This Implements

HomeSynapse writes its event log append-only and immutable. Doc 15's D2 decision is that the **sensitive-PII categories are encrypted at rest *from launch*** — because encrypt-on-write is **now-or-never on an immutable log** (you cannot retrofit encryption onto already-written plaintext) and because encrypting-per-scope-now is what preserves future crypto-shreddability (INV-PD-07). M6.2 already shipped the key-management infrastructure (`ScopeKeyManager`: machine-local root key, HKDF scope KEKs, wrapped per-scope DEKs in `scope_keys.json`) and the **consumer-defined `PayloadCipher` seam** in persistence — but persistence does **not yet call it**: the adapter is constructed, threaded through `HomeSynapseCore`, and held (the 5-arg ctor's nullable `PayloadCipher`). 

**M6.3 is the consumption step.** It (1) gives the config-side key manager a **counter-nonce payload-encryption path** distinct from M6.2's random-IV secrets path; (2) adds the `payload_iv` + `dek_ref` columns to the events table (migration V005); (3) resolves each event to an encryption scope and, for the confirmed encrypted set `[identity, presence_personal]`, **encrypts the serialized payload before the single-writer INSERT** and decrypts on read-back; and (4) wires the held cipher through the frozen `PersistenceFactory` gateway into the write path (the R-2 step the M6.2 closeout named). The cryptographic correctness gate is **OR-M6-NONCE**: the per-scope GCM nonce is a **counter** (never random — Doc 15 §3.4), and it must be **durable and strictly monotonic across crash and restore**, or (key, nonce) reuse catastrophically breaks AES-256-GCM for that scope.

### 1 — Counter-nonce payload cipher (config side)

#### `ScopeKeyManager.encryptPayload` (new method on the existing interface)
**Package:** `com.homesynapse.config`
**Responsibility:** Encrypt an event payload under the scope's current DEK with AES-256-GCM and a **deterministic counter nonce** (Doc 15 §3.4), distinct from the random-IV `encrypt` used for secrets.

```java
/**
 * Encrypts an event payload under the scope's current DEK with
 * AES-256-GCM and a per-scope COUNTER nonce (Doc 15 §3.4) — never a
 * random IV (the GCM birthday bound is reached at event volume).
 *
 * <p>The per-scope 96-bit nonce is derived from a strictly monotonic
 * counter that is DURABLE and re-initialized from the persisted
 * high-water mark on boot, never from memory (OR-M6-NONCE). The
 * counter high-water mark is persisted atomically with — or ahead of —
 * returning the nonce, so a crash after this call but before the
 * persistence INSERT can only leave a gap, never a reuse.</p>
 *
 * @param scopeId   an encrypted scope (e.g. {@code "identity"},
 *                  {@code "presence_personal"}); never null/blank
 * @param plaintext the serialized payload bytes; never null
 * @return ciphertext (GCM tag appended), the counter nonce as the iv,
 *         and the DEK version; never null
 * @throws IllegalStateException if a key file is corrupt or the
 *         cryptographic operation fails
 */
ScopeCipherResult encryptPayload(String scopeId, byte[] plaintext);
```

**Behavioral contracts:**
- **Counter monotonicity (per `(scopeId, keyVersion)`):** every successful `encryptPayload` returns a nonce strictly greater than every nonce ever *persisted to durable storage* for that `(scopeId, keyVersion)`. Two stored ciphertexts under one DEK can never share a nonce.
- **Durable-ahead-of-return:** the counter high-water mark is fsync-durable on the config side **before** the nonce is handed back (so an INSERT that never happens leaves a harmless gap, not a reuse).
- **Boot re-init from persisted max, never memory:** on construction/first-use after a restart, the counter resumes from the persisted high-water mark + 1. (If you instead choose to re-derive from persistence's `MAX(payload_iv)` per scope — see Settled Decisions — that handshake is brokered at `app`, never a config→persistence edge.)
- **Thread safety:** counter increments serialize on the existing `ReentrantLock` (LTD-11 — no `synchronized`; virtual threads call this on the publish path). `javax.crypto.Cipher` stays per-call.
- **Decrypt is unchanged:** `decrypt(scopeId, keyVersion, ciphertext, iv)` already round-trips any GCM ciphertext regardless of how the nonce was generated — reuse it as-is.

> **Fence (carry verbatim into the code comments):** random-IV **here** is M6.2 secrets (`encrypt`); counter-nonce **there** is M6.3 payloads (`encryptPayload`). Keep the two paths visibly separate. Do not delete or alter `encrypt`.

### 4 — Write-path consumption (`SqliteEventStore`)

At the existing serialize point (`byte[] payloadBytes = codec.encode(draft.payload());`, before the `payload` bind):
1. Resolve the event's scope-id (§2). If it is in the encrypted set **and** a cipher is present: `EncryptedPayload ep = cipher.encrypt(scopeId, payloadBytes);` → bind `payload = ep.ciphertext()`, `payload_iv = ep.iv()`, `dek_ref = scopeId + ":" + ep.keyVersion()`.
2. Else: bind `payload = payloadBytes`, `payload_iv = NULL`, `dek_ref = NULL` (today's behavior).
3. **Read path:** when hydrating a row with non-null `dek_ref`, parse `scope_id:key_version`, call `cipher.decrypt(scopeId, keyVersion, payloadBytes, payload_iv)` → feed the plaintext to the codec. Null `dek_ref` ⇒ decode directly (today's path).

**Null-cipher fail-safe:** if `payloadCipher == null` (the M6.2 production state / tests that don't wire crypto), the store writes plaintext for ALL scopes — *but* if `at_rest_enabled` is true AND a sensitive scope would have been encrypted, that is a **fail-closed misconfiguration**, not a silent plaintext write. Decide the contract: with encryption enabled and an encrypted-scope event but no cipher, **throw** (`IllegalStateException`, Doc 15 §6 "no silent plaintext fallback") rather than persist sensitive-PII plaintext. (Confirm against the disposition; flag if it conflicts.)

> **Encryption thread placement (Doc 15 §3.2/§5):** the design intends crypto to run on the **publishing virtual thread, before the single-writer INSERT** (keeps the single writer free of crypto cost; JDK-intrinsic, no carrier pinning, AMD-26). If the cleanest seam is inside `append()` on the writer for the low-volume sensitive scopes, that is acceptable for MVP volume — but call out the choice and the reasoning. This is perf, not correctness; the counter-nonce durability is the correctness gate.

## Settled Decisions (resolve these as stated; they are NOT open questions)

- **DP-A (counter ownership):** The per-scope nonce counter is owned by `StandardScopeKeyManager` (config side), persisted durably **alongside the DEK** (Doc 15 §3.4 "stored alongside the DEK"), ahead-of-return, re-init from persisted max on boot. This keeps the `PayloadCipher` seam **unchanged** (the cipher generates the nonce internally and returns it in `EncryptedPayload.iv`). *Pushback welcome on the mechanism* (e.g. re-deriving the boot seed from persistence's `MAX(payload_iv)` brokered at `app` — zero extra fsync, but couples to a persistence read): if you can show it is cleaner AND preserves the boot-from-durable-max property AND adds no config↔persistence edge, raise it. The **safety property is non-negotiable**; the storage mechanism has freedom.
- **DP-B (seam unchanged):** Do NOT modify the `PayloadCipher` interface or `EncryptedPayload` record (frozen M6.2). M6.3 is consumption + a new config-side method + wiring. (R-2: extending `PersistenceFactory.start` IS the wiring step, not a Phase-2 change.)
- **DP-C (nonce construction):** 96-bit nonce from the counter, strictly increasing per `(scope, keyVersion)`, collision-free within a DEK. A big-endian counter in the 96-bit field is acceptable; document the construction. (Embed any RFC/test vectors you pin verbatim — standing practice: re-derive independently.)
- **DP-D (restore co-design — design-not-preclude only):** M6.3 must NOT preclude **DEK-rotation-on-restore**. The `dek_ref = scope_id:key_version` schema already supports multiple versions; keep version handling correct even though rotation *automation* is post-MVP. The full crash-safe backup/restore co-design is **side-research R-α (DOCS Project), explicitly OUT of scope this WU** — do not build it. M6.3 closes the *write-path* half of OR-M6-NONCE; the restore half is co-designed when R-α returns.

## Open Risk Carried — OR-M6-NONCE (embed verbatim; [BLOCKING-for-M6-impl])

> #### OR-M6-NONCE — Counter-nonce crash/restore durability — [BLOCKING-for-M6-impl] (NEW 2026-06-07; carried from Doc 15 §6 at Lock)
> - **Severity:** HIGH for the M6 at-rest-encryption implementation (a cryptographic-correctness gate — catastrophic if missed). NOT a blocker for Doc 15's Lock: the review (J6/J7) correctly classified it as blocking the M6 *implementation*, not the *design*. Logged here so it is not lost now that Doc 15 is Locked (the explicit M6 carry the brief required).
> - **Detail:** Doc 15 §3.4 mandates counter-based deterministic 96-bit GCM nonces per scope (never random — to avoid the GCM birthday bound). If the per-scope nonce counter repeats after a crash, **or** a restored backup reintroduces a counter value already used under the same DEK, then (key, nonce) reuse **breaks AES-256-GCM confidentiality AND authentication** for that scope (the catastrophic GCM nonce-reuse failure). Documented in Doc 15 §6 (Failure Modes — "Counter-nonce reuse across crash/restore"), §13.4 (Testing Strategy), and the §16 summary; it is the AMD-86-INV-01 corollary registered in `Architecture_Invariants_v1.md` §35.
> - **Owner:** PM (embed verbatim in the M6 coding instruction) / Coder (implement at M6). **Resolution condition:** the M6 at-rest-encryption WU (a) persists the per-scope counter high-water mark atomically with (or ahead of) the encrypted write and re-inits from the persisted max on boot — never from memory; and (b) co-designs the deferred backup/restore feature (foundation-readiness F3) so a restore can never resume a scope at a counter ≤ any value already used under that DEK (rotate the DEK on restore, or carry the high-water mark in the backup). Closed when the M6 impl ships with the §13.4 kill-mid-encrypt nonce-monotonicity test GREEN.

**M6.3 discharges (a) fully** (the write-path counter durability). **(b) is design-not-preclude only** this WU (DP-D) — the full co-design is R-α.

- **LTD-11 (No Broker / virtual threads):** crypto + counter increment run on the publish virtual thread. Use `ReentrantLock`, never `synchronized` (vthread pinning). The existing `ScopeKeyManager` lock model already complies — extend it, don't add `synchronized`.
- **LTD-14 / fail-closed:** no silent plaintext fallback for a sensitive scope when encryption is enabled (Doc 15 §6).
- **LTD-15 (logging):** never log key material, plaintext payloads of encrypted scopes, or the root key. Structured SLF4J only; redact.
- **LTD-04 (ULID):** unchanged — identifiers stay typed ULID BLOB(16); encryption is on the `payload` column only, not on ids/metadata (the canonical-metadata chain input stays plaintext, Doc 15 §3.3).
- **AMD-71-A (config injection):** the config dir is an injected `Path` — no config→platform edge. The counter store path resolves under the injected `configDir`, like `scope_keys.json`.

## Invariants That Must Hold

- **INV-PD-03 (Encrypted Storage / at-rest posture, AMD-86):** sensitive-PII scopes written encrypted-at-rest under per-scope DEKs. *Test:* an `identity`/`presence_personal` event persists with non-null `payload_iv`/`dek_ref` and ciphertext `payload`; a non-sensitive event persists plaintext with NULL `payload_iv`/`dek_ref`.
- **INV-PD-07 (Crypto-shred lifecycle — infra MVP):** the per-scope key model + `scope_keys` + `dek_ref` versioning seat the post-MVP shred operation. *Test (design-time):* simulate destroying a scope key → that scope's rows fail to decrypt (manager throws), all other scopes + the chain still verify. - **AMD-86-INV-01 (§35):** encrypt-on-write is irreversible — the encrypted-set membership is written at write time and cannot be retrofitted. *Test:* membership is decided at INSERT, not at read.

## What to Watch Out For

- **OR-M6-NONCE is the whole milestone.** The catastrophic failure is (key, nonce) reuse. The only safe design re-inits the counter from **durable** state on boot, never from memory. A nonce used for an encrypt that *crashed before the INSERT* is harmless (it produced no stored ciphertext) — what must never happen is two **stored** ciphertexts sharing a nonce under one DEK. Re-init from the persisted high-water (or from `MAX(payload_iv)` per scope) guarantees this.
- **Random-IV / counter-nonce fence.** `ScopeKeyManager.encrypt` (random IV) is for M6.2 secrets and must stay exactly as-is. The new `encryptPayload` (counter) is for M6.3 payloads. The current `Main.payloadCipher` adapter wires the *random-IV* `encrypt` — re-pointing it is a required M6.3 change, easy to miss.
- **Cold-start cipher warmup.** (M6.3 scoping input from the Pi-evidence session.) First op ~2–3 ms, first ~100 encrypts 80–700 µs — C2 compiling the software-AES path on the Pi (no ARM crypto extensions on the Pi-4 floor). Warm it.
- **Thread placement.** Doc 15 wants crypto on the VT before the single-writer INSERT. If you encrypt inside `append()` on the writer, justify it; either way the counter must be thread-safe under concurrent publishers (the `ReentrantLock` model covers it).
- **Null-cipher production state.** The M6.2 production state is `payloadCipher == null` (held-not-consumed). Tests and the no-crypto path must still work; but with encryption *enabled*, a sensitive event + null cipher fails closed — do not regress to plaintext sensitive-PII.
- **The chain covers stored bytes.** Encrypting changes `payload` to ciphertext; the chain hash is computed over whatever `payload` holds (Doc 15 §5). Do not double-handle — encryption happens before the chain input is taken from the stored bytes. (Chain activation itself is OUT of scope, but keep the byte ordering correct so a later activation sees consistent stored bytes.)
- **`dek_ref` parse discipline.** `scope_id:key_version` — `scope_id` could in principle contain a colon; use a last-colon split or a fixed format and pin it with a test.
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` anywhere in `persistence` or `config` test code. **Enforcement reach (corrected 2026-06-13):** the `NO_DIRECT_TIME_ACCESS` ArchUnit rule runs only from `com.homesynapse.app`'s test classpath, so `./gradlew check` ENFORCES this for app-visible classes only — production code in any module IS caught, but the gate will NOT catch a direct-time-access violation in `persistence`/`config` **test** code. Treat Clock-injection as a **self-enforced convention** here; do not assert the rule reaches these test source sets. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` via constructor/`@BeforeEach`. (M2.4/M2.5 both tripped the production-code form of this rule — see `coder-lessons.md` 2026-04-10.)
