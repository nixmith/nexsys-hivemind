# M7.2b Action-Model Decision — Retry vs. Confirmation (D2/REC-162), evidence-based

**The deliberation Nick asked for, decided on evidence.** The 2026-06-21 competitive research (§3, "the decisive M7.2b input") makes this near-decided rather than an open fork. Below: the framed decisions for co-sign, the one genuine scope fork that's Nick's to rule, and the mid-August go/no-go criteria.

## The evidence in one paragraph
Across HA, Hubitat, SmartThings, openHAB, and Node-RED, the "says on, but it's off" problem is real and universal, and every trace records that a command was **sent**, not that the device **acted** — the HA core team calls the missing in-flight/confirmation UX "really bad UX" and hasn't fixed it (architecture discussion #740). Retry is the wrong default: it is **unsafe without a confirmation channel** and **dangerous on momentary actuators** — Hubitat is the only platform with built-in retry, and they gate it to idempotent device classes (switches/dimmers/locks/shades/thermostats), keep it **off by default**, and explicitly warn against it for garage doors/gates/valves (re-sending a toggle reverses the outcome). The unoccupied territory is **making the command outcome visible**, which HomeSynapse's architecture already produces.

## Framed decisions (Nick co-signs)

**D-M7.2b-1 — Keep no-engine-retry as the V1 default (REC-162 / AMD-90-INV-01 holds).**
The field evidence backs "don't auto-retry": engine-default retry is unsafe without confirmation and dangerous on actuators. The automation layer issues a command and records its outcome; it does not autonomously re-dispatch. This is already the contract — the research **confirms** it rather than changing it. *Confidence: high.*

**D-M7.2b-2 — The differentiator is command-outcome *visibility*, not retry.**
Make every action's outcome first-class and legible: **`dispatched → confirmed | unconfirmed(timeout) | failed(reason)`**. This is the run-coupled-reliability terminal contract (Doc 16 §3.4: deterministic terminal + recorded reason) plus the `PendingCommandLedger`'s command→`state_confirmed` correlation (the `Expectation` model + AMD-90 confirmation). **No competitor shows this; users beg for it; the architecture already produces the data** — a near-free differentiator. The honest "unconfirmed" state (we sent it, the device never reported back) is the trust win — it's the state every other platform hides behind an optimistic lie.

**D-M7.2b-3 — Retry is a post-MVP opt-in, gated and excluded by device class.**
When/if retry ships post-V1: opt-in (off by default), **only for idempotent target classes** (adopt Hubitat's list verbatim: switches, dimmers, locks, shades, thermostats), **explicitly excluded for momentary actuators** (garage doors, gates, valves), and **only behind a confirmation channel** (retry without confirmation is the unsafe case). Do not pull any of this into V1. *This forecloses nothing and is reversible.*

## ⚠ The one real scope fork — Nick's call

**The "did it actually DO it?" hero half depends on M7.3 — is M7.3 in V1?**
The research elevates **"did the device actually do it?"** to a co-equal differentiator (with "why didn't it fire?"). But the `confirmed | unconfirmed` distinction requires the **`PendingCommandLedger` (M7.3)** to correlate the command to a `state_confirmed` event. M7.2b alone yields `dispatched`/`failed(reason)` (the terminal contract); the **confirmed/unconfirmed** half needs M7.3. M7.3 was **not** explicitly in the V1 IN-list (it was "depends on M7.2a, sequence later"). So:

- **Option A (RECOMMENDED): pull M7.3 into the V1 critical path.** The "did it actually confirm?" half is, per the research, half the hero and a near-free differentiator the architecture already produces. Without it, the demo shows "command sent" — the exact thing every competitor shows and users distrust. M7.3 is Doc-16-confirmed UNAFFECTED/small (a focused FSM, no new event types beyond the ledger slice, builds against injected deps, no hardware). Sequencing: M7.2a-2 → M7.2b → **M7.3** → causal-read-API → AB-4 → M9. The causal-read-API then surfaces *both* hero halves (the causal chain + the command outcome).
- **Option B: V1 ships `dispatched`/`failed` only; confirmed/unconfirmed is post-MVP.** Keeps M7.3 out of V1 (smaller scope), but the hero demo can't answer "did the light actually confirm?" — it shows intent, not outcome. Weakens the differentiator to roughly parity with HA's trace on this axis.

**PM recommendation: Option A.** It's the higher-leverage half of the differentiator, the architecture already produces the data, and M7.3 is small and hardware-independent. The cost is one more serial Core milestone before validation — affordable against the ~16-week window, and it's exactly the kind of differentiated-thin-slice depth the V1 thesis is built on. **This is a V1-scope change → Nick rules.**

## Other residual items (recommendations; not blocking the co-sign)
- **`unconfirmed` timeout policy:** a per-device-class default timeout (a fast switch vs a slow thermostat differ), configurable, with a sane global default. Pin at M7.3 scoping; not a V1-blocker.
- **Device-class idempotency taxonomy:** adopt Hubitat's supported-class list as the seed for the (post-MVP) retry gate AND, usefully, as the V1 metadata that marks which targets are momentary (so the UI can label a momentary actuator's outcome honestly even in V1). Small, worth carrying.
- **No new anti-requirement needed:** D-M7.2b-1/3 are entirely consistent with the standing "no engine retry unless Nick reopens REC-162" anti-requirement — this record *is* the M7.2b ruling that anti-requirement deferred to, not a reopen.
