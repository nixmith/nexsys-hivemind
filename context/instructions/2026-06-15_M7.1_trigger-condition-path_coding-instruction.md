## What This Implements

The Automation Engine turns HomeSynapse from passive monitoring into active automation via the Trigger-Condition-Action model. This WU builds the **first half** — the path from an incoming event to a *decision to run*: load automation definitions through the shipped config substrate, index their triggers for O(1) event-type lookup, evaluate the matched triggers (including AMD-25 `forDuration` sustained-state timers), resolve selectors to entity sets, and evaluate conditions against a single trigger-time state snapshot (AMD-03). It publishes the **run-initiation slice** of the M7 event vocabulary (the trigger-side diagnostics + the reshaped `automation_triggered` record). It does NOT execute actions, manage Run lifecycle/concurrency, dispatch commands, or track the pending-command ledger — those are M7.2/M7.3. The module is Phase-2 interface-only today (53 types, 0 tests, 0 impl); this WU writes the first production code and tests against the AMD-88/89-reshaped contracts.

## What to Watch Out For

- **Type-residency is a hard compile cycle, not a lint** (SD-1): a single automation-resident type in an event record forces `event→automation` and the build fails with a JPMS cycle. Do the one-line direction check on every record component.
- **`EntityRole`, NOT `EntityCategory`** (AMD-89 §1.2): `EntityCategory` does not exist; `EntityRole {PRIMARY, DIAGNOSTIC, CONFIG}` is the M4 type in `com.homesynapse.device`. (Distinct from `EventCategory` in event-model — don't conflate.)
- **`EntityRef` in Doc 07 maps to `EntityId`** (MODULE_CONTEXT gotcha); `CommandIdempotency` is in event-model not device-model; `Availability` is from state-store; `AutomationId` is in platform-api (do not duplicate); `RunId` IS automation-resident.
- **`EventTrigger` has NO `forDuration`** (AMD-25) — do not add it.
- **The C1-interim pin (SD-3):** resist the natural urge to publish `automation_triggered` from the evaluator — it must wait for M7.2's completing side. Publishing it here re-introduces exactly the orphan-run risk the pin prevents and would break the converge-verified invariant (zero production publish sites today).
- **Reshape regret-window:** `AutomationTriggeredEvent` reshape is only safe because zero producers + zero persisted instances exist. Re-confirm at issue (STOP-gate); if a producer/persisted instance appeared, this becomes schema-versioning, not a reshape.
- **Published field sets are a one-way wire commitment (red-team F7/C4 — `context/audits/2026-06-15_redteam_reversibility-audit.md`):** every flattened component list in this slice that actually *publishes* — the M7.1 diagnostic events (rows 3, 11–16, 19) — freezes permanently on the immutable, (soon-)hash-chained log at its first publish; there is no later reshape without a log migration (FLATTEN itself is near-zero-migration, but the *field set* is not). Treat each diagnostic record's components as final-on-first-publish: complete and correct *before* the event is published, not after. The reshaped `automation_triggered`/`automation_completed` records (rows 1/2) are held by SD-3 until M7.2, so their wire locks at M7.2's first publish — but settle their *complete* field set at M7.1 scoping now (converge C4: the placeholders cannot yet express `INTERRUPTED`/`CONDITION_NOT_MET` or the F5 fields), so M7.2 inherits a final shape, not a second reshape.
- **VT pinning (LTD-11):** the trigger index, timer map, and registry are concurrent-access — `ReentrantLock`/lock-free only. Check transitive library calls on the publish path too.
- **REPLAY re-derive-never-re-execute (§3.10):** duration timers are not persisted; rebuilding them on REPLAY must not fire side effects — it reconstructs timer state from the event log only.
- **Composition-root ordering:** automation must subscribe AFTER state-store catch-up, or trigger evaluation runs against an incomplete snapshot. This is the converge-flagged latent-defect site — pin it.
