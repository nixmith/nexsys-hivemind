## What This Implements

The automation engine today can decide *which* automations match an incoming event (M7.1 landed the trigger/condition evaluators), but it cannot *run* one: there is no `RunManager` implementation, no Run lifecycle, no cascade governance, and the run-lifecycle event vocabulary is only partially registered. This milestone builds the **run lifecycle** — the finite-state machine that takes a matched automation and a triggering event, enforces concurrency and cascade rules, drives the run through `EVALUATING → RUNNING → terminal`, and publishes the run-lifecycle events that make a run *observable* (and, downstream, *explainable*).

It also lands the **`RunCausalChain`** — the per-run lineage type (AMD-91) that replaces the bare `int cascadeDepth`. This is the data structure the explainability hero view ("why did this fire? / why didn't it?") is built on: a run's causal chain is what lets the dashboard render the chain of automations that led to a firing, and the deterministic cycle/depth suppression is what lets it explain a *non*-firing. Getting this type and its events right is the foundation of the product differentiator.

**This is the run-lifecycle HALF of M7.2a.** The `ActionExecutor`, `CommandDispatchService`, `CommandValidator`, `ConflictDetector`, run-trace, and the action/condition/conflict events (AMD-92 rows 4/5/6/9) are **M7.2a-2** and are explicitly out of scope here. The `RunManager` FSM you build drives an **injected `ActionExecutor`** (the existing Phase-2 interface) — in this milestone you build and test the FSM against a **stub/fake `ActionExecutor`**; the real one arrives in 2a-2 with zero FSM rework.

## Settled Decisions (NOT open questions — implement as stated)

- **DP-A (RunCausalChain shape):** AMD-91 §2.1 — `record RunCausalChain(List<ChainLink> ancestors)`, nested `record ChainLink(RunId runId, AutomationId automationId)`. `depth()` is DERIVED (`ancestors.size()`), not stored. `root()` = empty ancestors. `extend(ChainLink)` returns a NEW chain (defensive copy). `containsAutomation(AutomationId)` = any `ChainLink.automationId().equals(id)`. Compact ctor: `List.copyOf(ancestors)` + null-check. Automation-resident; **never** appears in an event payload (AMD-92-INV-01) — it crosses the event boundary only flattened (`int cascadeDepth` from `depth()`; `List<AutomationId> chain` from the ancestors' automationIds).
- **DP-B (RunContext swap):** field 7 `int cascadeDepth` → `RunCausalChain causalChain` (non-null). Single-component swap; the other 7 components untouched. Add `Objects.requireNonNull(causalChain, "causalChain must not be null")`. The old depth value is now `causalChain.depth()`. **Zero downstream construction sites exist** (survey-confirmed) — the only edits are the record declaration + its Javadoc.
- **DP-C (initiateRun swap):** `initiateRun(...)` last param `int cascadeDepth` → `RunCausalChain parentChain`. `parentChain` is the **ancestor chain the new Run inherits** — `RunCausalChain.root()` for user/device/time-initiated (root) Runs; for a cascade Run triggered by parent Run P, the caller passes `P.causalChain().extend(new ChainLink(P.runId(), P.automationId()))`. The FSM checks `parentChain.containsAutomation(automation.automationId())` (cycle) and `parentChain.depth()` vs the configured max (depth) BEFORE admitting the Run, and stores `parentChain` as the new `RunContext.causalChain`.
- **DP-D (cascade governance — AMD-91 §2.3):** depth-limiting semantics UNCHANGED (config key `automation.max_cascade_depth`, default 8, range 1–32). Depth-exceeded → suppress + publish `cascade_depth_exceeded` (row 17). Cycle (chain membership) → suppress + publish `cascade_loop_detected` (row 18). Suppression is a pure function of `parentChain` + config — **no windowed/evictable suppression set** (AMD-91-INV-01). The old AMD-04 in-memory `(correlation_id, automation_id)` set is **not built**.
- **DP-E (C1-interim hold CLOSES here):** M7.1 registered `automation_triggered` (row 1) but deliberately did NOT publish it (no completing side existed). This milestone gives it its completing partner: the `RunManager` publishes `automation_triggered` at Run **initiation** and `automation_completed` at the **terminal** transition. Both publish sites land here.
- **DP-F (FLATTEN — AMD-92 §2.1):** the new event records live in `com.homesynapse.event` (flat) and reference only event-resident-or-below types. `RunId`/`RunStatus`/`RunCausalChain` NEVER appear in a payload — `runId` is a bare `Ulid`, `finalStatus` is a `String` (`RunStatus.name()`), `cascadeDepth` is an `int`, the cycle path is `List<AutomationId>` (AutomationId is platform-resident → allowed). Violating this inverts the JPMS edge to `event→automation` = a hard compile cycle.
- **DP-G (event-time + causality):** run/diagnostic events publish with inherited or null `eventTime` — **never** `Instant.now()` in the publish path. All Run events publish via `EventPublisher.publish()` carrying the triggering event's `CausalContext` (§3.7) — this is what makes traces assemble and chains event-derivable.
- **DP-H (C8 stamping):** every automation-originated publish carries `actorRef = AutomationId` on the ENVELOPE (B2-C8, ratified). No payload-level actor field. Re-pin the stamping test against the ratified four-kind convention.

## Technical Specification

### Phase A — `RunCausalChain` (new) + the `RunContext` / `RunManager` swaps

#### `RunCausalChain` (record) — CREATE
**Package:** `com.homesynapse.automation`
**Responsibility:** the immutable per-Run causal lineage; the authority for cascade depth + cycle suppression; the type the explainability projection (Doc 16 §3.3) reads.

```java
/** Automation-resident; NEVER appears in event payloads (AMD-92-INV-01 / AMD-91-INV-02). */
public record RunCausalChain(List<ChainLink> ancestors) {

    public record ChainLink(RunId runId, AutomationId automationId) {
        public ChainLink {
            Objects.requireNonNull(runId, "runId must not be null");
            Objects.requireNonNull(automationId, "automationId must not be null");
        }
    }

    public RunCausalChain {                       // compact ctor
        Objects.requireNonNull(ancestors, "ancestors must not be null");
        ancestors = List.copyOf(ancestors);       // defensive, immutable
    }

    /** A root Run's chain: no ancestors, depth 0 (preserves the old cascadeDepth==0 root semantics). */
    public static RunCausalChain root() { return new RunCausalChain(List.of()); }

    /** Returns a NEW chain with {@code parent} appended (this chain is unchanged). */
    public RunCausalChain extend(ChainLink parent) {
        Objects.requireNonNull(parent, "parent must not be null");
        var next = new ArrayList<ChainLink>(ancestors);
        next.add(parent);
        return new RunCausalChain(next);
    }

    /** Derived — equals the old int cascadeDepth by construction. */
    public int depth() { return ancestors.size(); }

    /** True if {@code id} already appears in the lineage (the deterministic cycle test). */
    public boolean containsAutomation(AutomationId id) {
        Objects.requireNonNull(id, "id must not be null");
        return ancestors.stream().anyMatch(link -> link.automationId().equals(id));
    }
}
```

**Behavioral contracts:** value-based equality (records); `extend` allocates (chains ≤ 32 by the depth ceiling — trivial); immutable after construction.

### Phase C — `StandardRunManager` (the FSM) + `RunManagerAssembly`

**Responsibility:** own the Run lifecycle from initiation to terminal; enforce dedup (C2), concurrency modes (§3.6), and cascade governance (§3.7.1); publish the run-lifecycle events; finalize zombie Runs on REPLAY→LIVE. Package-private final class; exposed only through `RunManagerAssembly.runManager(...) → RunManager`.

**Injected dependencies (constructor — the FSM builds against seams, NOT concrete subsystems):**
- `EventPublisher` (event-bus) — the publish surface for all run-lifecycle events.
- `ActionExecutor` (the existing Phase-2 interface) — drives the RUNNING state. **Stub it in 2a-1 tests; the real impl is 2a-2.**
- `RunConditionGate` (CREATE — a small functional seam: `boolean conditionsHold(AutomationDefinition automation, RunContext context)`) — drives the EVALUATING decision. Stub it in 2a-1; 2a-2 wires the real `ConditionEvaluator` + trigger-time `StateSnapshot` + the `automation_condition_evaluated` (row 4) diagnostic behind it. This seam is what keeps the EVALUATING state in the FSM while deferring condition logic + row 4 to 2a-2.
- `Clock` (java.time) — injected (§4c); all timing (`durationMs`, deadlines, the auto-disable window) derives from it. **Never `Instant.now()`/`System.*`.**
- `RunManagerConfig` (CREATE — automation-resident plain record, NO config-module dep — `core→config` is BANNED): `int maxCascadeDepth` (default 8, range 1–32), `int autoDisableThreshold` (default 5), `Duration autoDisableWindow` (default per Doc 07). The composition root (lifecycle — allowed to depend on config) reads the YAML and constructs this; the FSM receives plain values.

**State machine (per Run, on its own virtual thread — LTD-01):**
```
initiateRun → [admission checks] → EVALUATING → RUNNING → COMPLETED
                                        │            └─(executor throws)→ FAILED
                                        └─(conditions false)→ CONDITION_NOT_MET   (does NOT consume a mode slot)
   RESTART-mode cancel of an active Run → ABORTED
   REPLAY→LIVE zombie (triggered, never completed) → INTERRUPTED
```

**Admission sequence inside `initiateRun` (order matters):**
1. **C2 dedup** — key `(automationId, triggeringEventId)`. If a Run for this pair is active or already terminal in the current LIVE window, return `Optional.empty()` — no Run, no event. (Dedup map is separate from cascade suppression — keep them distinct.)
2. **Cascade cycle** — if `parentChain.containsAutomation(automation.automationId())`: suppress, publish `cascade_loop_detected` (row 18; `chain` = the parentChain ancestors' automationIds + this automationId path per AMD-91), return empty. Terminal-absent (no Run object).
3. **Cascade depth** — if `parentChain.depth()` ≥ `config.maxCascadeDepth()`: suppress, publish `cascade_depth_exceeded` (row 17), return empty.
4. **Auto-disable check** — if the automation is currently disabled (failure count reached threshold within the window): suppress (publish nothing new; it was disabled when `automation_disabled` fired), return empty.
5. **Concurrency-mode admission (§3.6):** see the table. On rejection, publish `automation_run_skipped` (row 7) and return empty; on RESTART-cancel, publish `automation_run_cancelled` (row 8) for the cancelled Run.
6. **Admit:** build `RunContext` (`causalChain = parentChain`, `stateSnapshotPosition` from the captured snapshot position, `definitionHash`), publish **`automation_triggered`** (row 1 — its publish site, closing the M7.1 C1-interim hold), register the Run active (status `EVALUATING`), spawn its VT, return the `RunId`.

**Concurrency modes (§3.6) — admission/cancellation behavior:**
| Mode | A trigger fires while a Run is active | maxConcurrent |
|---|---|---|
| `SINGLE` (default) | Drop the new trigger → `automation_run_skipped` reason `mode_busy`, severity from `maxExceededSeverity`. | 1 |
| `RESTART` | Cancel the active Run via `Thread.interrupt()` → that Run terminates `ABORTED` + `automation_run_cancelled`; start the new Run. | 1 |
| `QUEUED` | Enqueue sequentially; if active count ≥ `maxConcurrent`, drop → `automation_run_skipped` reason `queue_full`. | bounded |
| `PARALLEL` | Run concurrently; if active count ≥ `maxConcurrent`, drop → `automation_run_skipped` reason `queue_full`. | bounded |

**Run body (on the VT):** EVALUATING → call `conditionGate.conditionsHold(automation, ctx)`; false → terminal `CONDITION_NOT_MET` (does NOT consume a mode slot — critical-review/MODULE_CONTEXT contract), publish `automation_completed` with `finalStatus="CONDITION_NOT_MET"`. True → RUNNING → call `actionExecutor.execute(automation.actions(), ctx)`; normal return → `COMPLETED`; thrown exception → `FAILED` (capture `failureReason`); interrupt (RESTART cancel/shutdown) → `ABORTED` (capture `abortReason`). On every terminal: publish `automation_completed` (row 2) with `actionCount`/`commandCount` (from the executor's reported tallies — 0 in 2a-1's stub), free the mode slot, and update the auto-disable failure window.

**Auto-disable (critical-review 7.1/7.2):** track per-automation failure timestamps; when failures within `autoDisableWindow` reach `autoDisableThreshold`, mark the automation disabled and publish `automation_disabled` (row 10) once (`failureCount`, `windowMinutes`, `lastError`, `lastRunId`). All failure types count equally. Re-enable is operator-driven (out of scope here).

**REPLAY zombie finalization (Doc 07 §3.10 — DP re-derive-never-re-execute):** expose a method (e.g. `finalizeZombieRuns(...)`) the composition root calls on REPLAY→LIVE: for each Run whose `automation_triggered` exists with no matching `automation_completed`, publish `automation_completed` with `finalStatus="INTERRUPTED"`, `abortReason="interrupted_by_crash"`. This frees mode slots and closes the lifecycle. **Reconstruct the zombie set from the immutable event log only** (E91-1 reconstruction-source pin — never a windowed/in-memory map). The FSM does NOT re-execute the Run.

**Concurrency/thread-safety:** active-Run state and the dedup/auto-disable maps use `ConcurrentHashMap` + `ReentrantLock` where compound updates need atomicity — **never `synchronized`** (LTD-11; VTs pin on `synchronized`). Each Run executes on its own VT (LTD-01). All public methods are safe for concurrent calls.

**C3 ordering (deterministic execution order):** when one event matches multiple automations, the caller (the subscriber — M7.2a-2/wiring) initiates Runs in **priority descending, then `automationId` ascending** order. Provide a `public static Comparator<AutomationDefinition> byExecutionOrder()` for the caller and unit-test it; the `RunManager` itself processes `initiateRun` calls in the order received (it does not re-sort a batch).

## What to Watch Out For
- **JPMS payload residency = a HARD COMPILE CYCLE, not a lint (AMD-92-INV-01 / the M7.1 SD-1 gotcha).** No `com.homesynapse.automation` type (`RunId`/`RunStatus`/`RunCausalChain`/`ChainLink`) may appear in any `com.homesynapse.event` record — it inverts the edge to `event→automation` and the build fails with a cycle. Flatten: `runId`=bare `Ulid`, `finalStatus`/statuses=`String`, `cascadeDepth`=`int`, cycle path=`List<AutomationId>` (AutomationId is platform-resident, allowed). `Ulid`/`AutomationId`/`EntityId`/`EventId` come from `com.homesynapse.platform`/`com.homesynapse.event` — already on event-model's edges.
- **The checked-`SequenceConflictException` test-declaration trap (recurs every milestone — coder-lessons 2026-04-10; bit M6.3 and AB-2).** Any test that drives a publish/write path must declare `throws SequenceConflictException` on the method (sibling-test `throws` pattern). Sweep every new/edited test that publishes. Declarations only — never touch assertions to dodge it.
- **`core → config` is BANNED at every scope** (`assertAllowedModuleDependencies`, stricter than the `-Xlint:exports` check — the FIX-07 instruction-defect lesson). The `RunManager` takes a plain `RunManagerConfig` record (automation-resident), NOT a `ConfigurationService`. The composition root (lifecycle) reads YAML and injects values.
- **`§4c` Clock injection (automation is non-whitelisted).** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this module's test OR production code — the `NO_DIRECT_TIME_ACCESS` ArchUnit rule scans non-whitelisted classes and fails `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructor/`@BeforeEach`.
- **`CONDITION_NOT_MET` must NOT consume a concurrency-mode slot** (MODULE_CONTEXT cross-module contract) — otherwise a SINGLE-mode automation whose conditions keep failing would deadlock itself. Test this explicitly.
- **Cascade suppression is terminal-ABSENT, never retried** (REC-162 anti-requirement). A suppressed Run produces only its diagnostic — no Run object, no re-issue, ever.
- **`Thread.interrupt()` is the RESTART/shutdown cancellation mechanism** — the run VT must observe interruption and terminate ABORTED. Don't swallow `InterruptedException`.
- **REPLAY reconstructs from the event log only** (E91-1) — never a windowed/in-memory map; otherwise determinism (AMD-91-INV-01) breaks in fact while honored in letter.
- **`AutomationId` ≠ `EntityId`; `EntityRole` ≠ `EntityCategory` ≠ `EventCategory`** (M7.1 gotchas) — don't conflate.
- **The reshaped `AutomationCompletedEvent` has zero production producers today** (survey-attested) — the reshape is regret-proof now; but you MUST update its dedicated test (`hasSize(3)`→7 + 10 ctor calls) and the 2 `TestEventSamples` builders, or the gate bounces.
