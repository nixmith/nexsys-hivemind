## What This Implements

M7.2a-1 landed the **run-lifecycle FSM** (`StandardRunManager`): a Run is admitted, its conditions are gated (condition-before-mode), it transitions `EVALUATING → RUNNING → terminal`, and it publishes `automation_triggered`/`automation_completed`. But the FSM today runs against **stub seams**: `ActionExecutor` does nothing real, `RunConditionGate` is a fake, no command is dispatched, no conflict is detected, and `stateSnapshotPosition`/`actionCount`/`commandCount` are placeholders. **M7.2a-2 makes the engine actually act.** It implements the execution half — the action executor that runs the five Tier-1 action types on the Run's virtual thread, the command-dispatch path that routes commands to integration adapters, the validator that gates commands against capabilities, the post-execution conflict detector, and the real condition gate that evaluates conditions over the trigger-time state snapshot. It mints the four AMD-92 diagnostic events these surfaces emit (rows 4/5/6/9) — the events the explainability hero view's causal chain will read. **This is the half that turns "a Run happened" into "a light turned on, and here is exactly why."**

## Settled Decisions (NOT open questions — implement as stated)

- **DP-A — The four M7.2a-1 carries are REQUIRED here (none is optional):**
  1. **QUEUED is a true sequential single-flight drain.** 2a-1 left QUEUED admission-equivalent to PARALLEL (it must NOT ship as PARALLEL). Implement strict sequential draining: at `maxConcurrent`, a QUEUED Run enqueues and drains one-at-a-time as slots free, in C3-comparator order. Test it (a sequencing test, absent in 2a-1).
  2. **Row 4 carries the correct causal chain.** The `automation_condition_evaluated` (row 4) publish must sit on the triggering event's `CausalContext` (derived/chained), `actorRef = AutomationId`, inherited/null `eventTime` (never `Instant.now()`). Publish row 4 from the gate impl (or from the FSM at the fixed gate call site) — the call site is already correct (condition-before-mode); you add the impl + the publish.
  3. **Dedup claim precedes the condition gate.** Move the C2 dedup claim *before* the gate so a raced duplicate cannot double-publish row 4. (2a-1 left dedup after the gate.)
  4. **Real `stateSnapshotPosition` + tallies.** Capture the trigger-time `StateSnapshot` `viewPosition` into `RunContext.stateSnapshotPosition` (no more 0L). Populate `automation_completed`'s `actionCount`/`commandCount` with real values (no more 0).
- **DP-B — Condition-before-mode is the LOCKED contract (already honored).** The 2a-1 Coder placed the gate before mode admission (a CONDITION_NOT_MET Run never enters `activeRuns` / never consumes a mode slot). KEEP this. Do not reintroduce the transient-slot ordering.
- **DP-C — Tier-1 actions ONLY.** `StandardActionExecutor` handles the 5 Tier-1 permits (`CommandAction`, `DelayAction`, `WaitForAction`, `ConditionBranchAction`, `EmitEventAction`). The 3 Tier-2 permits (`ActivateSceneAction`, `InvokeIntegrationAction`, `ParallelAction`) are **reserved** — the exhaustive switch must include them, throwing a clear `UnsupportedOperationException("Tier 2 action reserved: <type>")` (NOT silent no-op). Use a no-`default` exhaustive `switch` so the compiler enforces completeness.
- **DP-D — `ActionExecutor.execute` returns a tally, not `void`.** Carry #4 needs real `actionCount`/`commandCount` on `automation_completed`; `void` cannot report them. Widen to `ActionExecutionResult execute(...)` where `ActionExecutionResult(int actionCount, int commandCount)` is a new package-private/public record in `com.homesynapse.automation`. `StandardRunManager` uses the returned tally to populate the terminal event. (Coder pushback welcome on the exact channel — but the contract "automation_completed carries the real action/command counts" is firm.)
- **DP-E — Command events are PRODUCED, not minted.** `command_issued`/`command_dispatched`/`command_result`/`command_confirmation_timed_out` already exist (EventTypes + records). The dispatch service publishes `command_dispatched` (DIAGNOSTIC) on successful handoff and `command_result` (status `invalid`/`unroutable`) on failure. Do **not** create new command event types/records.
- **DP-F — Tier-1 conflict detection is report-only (D6).** `StandardConflictDetector` scans post-execution; both contradictory commands have already executed; it only emits `automation_conflict_detected` (row 9). No suppression/priority resolution (Tier 2).
- **DP-G — §6.2 action fail-fast + `UnavailablePolicy`.** On an action failure, the sequence stops (fail-fast) and the Run terminates with the recorded reason; an unavailable target is handled per the action's `UnavailablePolicy` (Doc 07 §3.9). Emit `automation_action_completed` with `outcome="error"` + `errorDetail`, and let the FSM record the terminal reason. (The deterministic-terminal-state *contract freeze* is M7.2b — here, implement the §6.2 behavior; do not invent new terminal semantics.)

### `StandardActionExecutor` (impl)
- Exhaustive no-`default` switch over the 5 Tier-1 permits; the 3 Tier-2 throw `UnsupportedOperationException` (DP-C).
- Sequential on the Run VT (the caller's thread — `StandardRunManager` runs the RUNNING phase on the Run VT). Emit `automation_action_started` (row 5) before each, `automation_action_completed` (row 6) after.
- `CommandAction` → `CommandDispatchService.dispatch(...)` (increment commandCount). `DelayAction` → suspend the VT (a virtual-thread-friendly sleep — `Thread.sleep(Duration)` is fine on a VT, it does not pin). `WaitForAction` → block until condition true or timeout. `ConditionBranchAction` → evaluate inline, execute the matching branch. `EmitEventAction` → publish the custom event.
- §6.2 fail-fast (DP-G). Return `ActionExecutionResult(actionCount, commandCount)` (DP-D).

### `StandardCommandDispatchService` (impl)
- Resolve `targetRef` → integration via `DeviceRegistry.getIntegrationForEntity()`. Validate via `CommandValidator`. On success: hand off to the adapter + emit `command_dispatched` (DIAGNOSTIC). On invalid/unroutable: emit `command_result` (status `invalid`/`unroutable`). Thread-safe.

### `StandardConflictDetector` (impl)
- Scan `triggeredRuns` for contradictory commands to the same `entityRef`; emit `automation_conflict_detected` (row 9). Report-only (DP-F).

### `StandardRunConditionGate` (impl)

Return the boolean; publish `automation_condition_evaluated` (row 4) with the `EvaluatedEntityState` list (the observed entity states the conditions read). Pure w.r.t. FSM admission (returns a value; no FSM mutation). Row 4 publish carries the correct CausalContext (DP-A.2).
