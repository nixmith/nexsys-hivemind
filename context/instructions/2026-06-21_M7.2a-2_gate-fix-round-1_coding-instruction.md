# M7.2a-2 ‚Äî Gate-Fix Round 1 (build RED ‚Üí GREEN)

## Situation

`./gradlew check` fails at `:core:automation:compileJava` with **8 errors / 4 warnings**, all from **two Java record component/static-factory name collisions**. A record auto-generates a public accessor for each component, so a record may not also declare a method (static factory or otherwise) whose name equals a component name. The in-session "adversarial verification" was an LLM read pass and did not catch this ‚Äî the compile gate did. Nothing is committed.

## PM pre-review scope (so you don't re-litigate the rest)

The PM read **all** M7.2a-2 production/main source host-side and verified type-consistency of: the widened `ActionExecutor`/`RunConditionGate` interfaces + their impls + the FSM call sites (`conditionsHold(...)` 4-arg, `execute(...)` returning `ActionExecutionResult`, the trigger-time `stateQuery.getSnapshot()` capture); every event-construction site against the real signatures ‚Äî `EventDraft` (9 components, `Ulid actorRef`, nullable `Instant eventTime`/`idempotencyKey`), `CausalContext.chain(Ulid, Ulid)`, `SubjectRef.entity/automation`, the existing `CommandDispatchedEvent(Ulid,Ulid,String)` + `CommandResultEvent(Ulid,String,String,String)`, and the four new event records incl. nested `EvaluatedEntityState`/`ConflictEntry`; `AttributeValues` (exists, automation-package, no import needed); `.value()` ‚Üí `Ulid` on every typed-ULID wrapper; `RunManagerAssembly`'s 6-arg wiring; `AutomationRegistry.get(...)`. **`ActionExecutionResult` and `CommandRef` are NOT collisions** (`failed()` instance + `failed(int,int,String)` static is legal overloading; no component named `failed`/`skip`). **The two collisions below are the only main-compile blockers found.** Apply ONLY these fixes (minimal diff) ‚Äî do not restructure anything else, do not change any contract, counts, or `module-info`.

## Defect 1 ‚Äî `ValidationResult` (new code): `valid` component vs `valid()` factory

**File:** `core/automation/src/main/java/com/homesynapse/automation/CommandValidator.java`
**Error:** `invalid accessor method in record ValidationResult` ‚Äî `record ValidationResult(boolean valid, String reason)` also declares `public static ValidationResult valid()`, which collides with the implicit boolean accessor `valid()`. Consequence at `StandardCommandDispatchService.java:110` (`if (!validation.valid())`): `valid()` binds to the static factory (returns `ValidationResult`) ‚Üí `bad operand type ValidationResult for unary operator '!'`.

**Fix (rename the no-arg factory; keep the boolean accessor):**
- In `CommandValidator.ValidationResult`, rename `public static ValidationResult valid()` ‚Üí `public static ValidationResult accepted()`. Keep the `VALID` constant and `invalid(String)` exactly as-is.
- Update the factory call site `StandardCommandValidator.java:54`: `return ValidationResult.valid();` ‚Üí `return ValidationResult.accepted();`
- **Do NOT change** the boolean-accessor reads (`StandardCommandDispatchService.java:110` `if (!validation.valid())`) ‚Äî once the factory is renamed, `validation.valid()` correctly binds to the boolean accessor.
- Grep test sources for the factory and rename there too: `grep -rn "ValidationResult.valid()" core/automation/src/test` ‚Üí rename each to `ValidationResult.accepted()`. (Boolean-accessor reads `.valid()` on an instance stay.)

## Defect 2 ‚Äî `ModeDecision` (regression of working M7.2a-1 code): `admit`/`restartCancel`/`enqueue` components vs same-named factories

**File:** `core/automation/src/main/java/com/homesynapse/automation/StandardRunManager.java`
**Error:** `record ModeDecision(boolean admit, String skipReason, boolean restartCancel, boolean enqueue)` declares `static ModeDecision admit()`, `restartCancel()`, `enqueue()` ‚Äî all three collide with the implicit boolean accessors (decl errors ~L722/726/730 "accessor method must be public"; call-site errors ~L266/267/275 "ModeDecision cannot be converted to boolean"). At M7.2a-1 this record had no factories and was built with `new ModeDecision(...)`; the 2a-2 rewire introduced the collision.

**Fix (recommended ‚Äî drop the 3 colliding factories, build via the canonical constructor; this matches the M7.2a-1 style and is the minimal change):**
- Delete the three factory methods `admit()`, `restartCancel()`, `enqueue()` (~L722‚Äì732). **Keep `skip(String)`** ‚Äî it does not collide (no `skip` component).
- In `decide(...)` (~L526‚Äì536), replace the factory calls with the canonical constructor:
  - `ModeDecision.admit()` ‚Üí `new ModeDecision(true, null, false, false)`
  - `ModeDecision.restartCancel()` ‚Üí `new ModeDecision(true, null, true, false)`
  - `ModeDecision.enqueue()` ‚Üí `new ModeDecision(false, null, false, true)`
  - leave `ModeDecision.skip("mode_busy")` / `ModeDecision.skip("queue_full")` unchanged.
- **Do NOT change** the admission-site reads `decision.admit()` / `decision.restartCancel()` / `decision.enqueue()` (~L266/267/275) ‚Äî they bind to the boolean accessors once the colliding factories are gone.

Acceptable alternatives (Coder's choice; pick one collision-breaking approach): rename the three factories to non-colliding names and update the `decide()` call sites; or rename the three boolean components (e.g. `shouldAdmit`/`shouldRestartCancel`/`shouldEnqueue`) and update the three admission-site reads. The recommended canonical-constructor fix is the least churn and reverts the gratuitous refactor.

## STOP-check to internalize (new recurring trap)

**A record component named `X` already defines a public accessor `X()`. Never declare a `static` factory (or any method) named `X()` in that record ‚Äî it is a compile error, and call sites silently bind to the wrong one.** Before handoff, scan every new/edited record: for each component name, confirm no same-named method exists. (Factories that don't match a component name ‚Äî `invalid(...)`, `skip(...)`, `succeeded(...)`, `failed(int,int,String)` ‚Äî are fine.)
