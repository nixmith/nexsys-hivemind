## What This Implements

When an automation fires a command today (post-M7.2a-2), the engine dispatches it and publishes `command_dispatched` ‚Äî but **nothing closes the loop**. The Pending Command Ledger closes it: it correlates each issued command to the device's actual reported state, so the product can say, honestly, **`dispatched -> confirmed | unconfirmed | failed`** instead of every competitor's optimistic "command sent." This is the differentiator gap **no competitor fills** (the 2026-06-21 explainability research), and it is the half of the demo a stranger feels immediately: *"did the light actually confirm it turned on?"*

**This milestone is a Phase-3 IMPLEMENTATION of an already-specified seam ‚Äî not a greenfield design.** The Phase-2 interface and its data types ALREADY EXIST in `core/automation` (source-verified at `1b0b6c9`):
- `PendingCommandLedger` (interface): `void trackCommand(PendingCommand)`, `Optional<PendingCommand> getCommand(EventId)`, `List<PendingCommand> getPendingForEntity(EntityId)`, `int pendingCount()`.
- `PendingStatus` (enum, **5 states**): `DISPATCHED, ACKNOWLEDGED, CONFIRMED, TIMED_OUT, EXPIRED`.
- `PendingCommand` (record, 8 components): `(EventId commandEventId, EntityId targetRef, String commandName, String targetAttribute, Expectation expectation, Instant deadline, CommandIdempotency idempotency, PendingStatus status)`.

What is **missing and that you build:** the production `StandardPendingCommandLedger` implementation, its subscriber wiring, its projection-handler registrations (DQ-3), and its test suite. There is **no** `StandardPendingCommandLedger` and **no** `PendingCommandLedgerTest` in the tree today (verified).

## Technical Specification

### The subscriber (Doc 07 ¬ß3.11.2)
Register `pending_command_ledger` with:
```
filter: SubscriptionFilter(
          Set.of("command_issued", "command_result", "state_reported", "state_confirmed"),
          EventPriority.DIAGNOSTIC,   // state_reported is DIAGNOSTIC
          "entity")
coalescing: DISABLED                  // correctness-critical (Doc 01 ¬ß3.6) ‚Äî every event matters
```
Its own virtual thread + checkpoint, like the other two automation subscribers.

### In-memory state ‚Äî a DERIVED, rebuildable projection of the log (INV-SA-03)
Maintain a **dual index** (the interface demands both lookups, and Doc 07 ¬ß3.11.2 vs the MODULE_CONTEXT Phase-3 note disagree on the single map ‚Äî so carry both, explicitly): an `EntityId -> List<PendingCommand>` map (for `getPendingForEntity`, the ¬ß3.11.2 shape) **and** an `EventId -> PendingCommand` index (for `getCommand(EventId)`). This in-memory state is **never the source of truth**: it is rebuilt during REPLAY by reprocessing the command-lifecycle events from the log (INV-SA-03 ‚Äî explanation/state is a pure projection of the immutable log; there is no parallel store). Use `ConcurrentHashMap` + lock-free or `ReentrantLock` updates ‚Äî **never `synchronized`** (LTD-11), and **publish outside any lock**.

### The FSM (Doc 07 ¬ß3.11.2; the 5 `PendingStatus` states are canonical over the prose‚Äôs lowercase words)
1. `command_issued` for the target -> record a `PendingCommand` keyed `(targetRef, commandName)` with `expectation` (from `ExpectationFactory`), `deadline` (= injected-clock now + `confirmationTimeoutMs`), `idempotency`, status `DISPATCHED`. **If the command‚Äôs confirmation policy is OPTIMISTIC, do NOT track it** (AMD-90 ‚Äî `ConfirmationPolicyGateTest`).
2. `command_result` -> if `acknowledged`, status `ACKNOWLEDGED`; if `rejected`/`timed_out`, remove the entry and emit `command_confirmation_timed_out` **only if none was already emitted for it** (idempotent emission).
3. `state_reported` for the target -> `expectation.evaluate(reportedValue)`: `CONFIRMED` -> status `CONFIRMED`, **emit `state_confirmed`** referencing **both** the `command_issued` and the `state_reported` event ids (the existing `StateConfirmedEvent(commandEventId, reportEventId, ...)`), then drop the entry; `NOT_YET` -> keep waiting; `FAILED` -> per Doc 07 (no autonomous retry ‚Äî record/age out).
4. **Timeout:** a `pollExpirations()` step compares the **injected `Clock`** to each entry‚Äôs `deadline`; expired -> status `TIMED_OUT`, **emit `command_confirmation_timed_out`** (DIAGNOSTIC, referencing `command_issued` + any `command_result`) once. **No wall-clock `sleep`; no `Instant.now()`** ‚Äî expiry is a clock comparison so tests are deterministic (REC-156/167; step a `MutableClock`).
5. **Crash recovery (REPLAY -> LIVE), by idempotency class** (Doc 01 ¬ß3.7): `command_issued` without a matching terminal -> `IDEMPOTENT` = re-offer for re-issue **after** the subscriber reaches LIVE; `NOT_IDEMPOTENT` = mark status `EXPIRED` (the `expired_on_restart` case ‚Äî never silently re-fire a momentary actuator); `CONDITIONAL` = offer to the integration adapter for evaluation. **The ledger itself NEVER autonomously re-issues a command** (AMD-90-INV-01 / D2/REC-162) ‚Äî re-issue is a signal it raises, executed (if ever) above the engine at M8.2, never inside the ledger or `ActionExecutor`.

### Events produced/consumed ‚Äî ALL PRE-EXISTING (no mints)
Consumes: `command_issued`, `command_result`, `state_reported`, `state_confirmed`. Produces: `state_confirmed` (on confirmation), `command_confirmation_timed_out` (on timeout/rejection). Causal threading (the M7.3-scoped piece the dispatch service deferred): publish on the `CausalContext` that links `command_issued` -> outcome (`correlation_id` carried from the triggering automation; `causation_id` = the `state_reported` id for `state_confirmed`). **`PendingStatus` is automation-resident and MUST NOT appear in any event payload** (AMD-92-INV-01 ‚Äî it is a hard JPMS cycle, not a lint; the outcome crosses the event boundary only via the already-flattened existing events).

### Error handling
A degraded substrate (snapshot/store read throws) fails the read closed and is recorded, consistent with the M7.2b run-coupled terminal contract ‚Äî it does not crash the subscriber. A missing/absent attribute is a valid input (typed-absent), not an exception (the M7.2b `DegradedAttributeValue` precedent). SQLite/checkpoint/publish go through the Persistence executor (platform thread), not the subscriber‚Äôs VT (LTD-03).

## Invariants That Must Hold
- **AMD-92-INV-01 (residency):** no automation-resident type (`PendingStatus`, `RunStatus`, ‚Ä¶) in any `com.homesynapse.event` payload. Flatten to `Ulid`/`String` if ever needed (it is not, here).
- **AMD-92-INV-02 (manifest-before-publish):** you add no event type, so nothing to register ‚Äî but confirm every event you publish (`state_confirmed`, `command_confirmation_timed_out`) is already in the manifest (it is).
- **AMD-90-INV-01 (no retry):** the ledger correlates and reports; it never re-issues a command autonomously, at any confirmation policy.
- **INV-SA-02 (federation non-preclusion):** key/identify pending commands by the **globally-unique typed ULIDs** already on the events ‚Äî **no site-local-sequential counter, no new persisted identity**. Scope stays an absent-defaults-to-local discriminator reserved at the envelope level; do not bake a single-site assumption into the ledger‚Äôs keys. (This is what lets the same ledger feed a multi-site / enterprise-audit projection later ‚Äî Doc 16 ¬ß3.3 ‚Äî with no event-log migration.)
- **INV-SA-03 (explanation is a pure projection):** the in-memory ledger is a derived view, fully rebuildable from the log; introduce **no parallel trace store**. The outcome (`confirmed`/`unconfirmed`/`timed_out`) must be reconstructable from persisted events alone.
- **LTD-11:** no `synchronized`; `ReentrantLock`/lock-free only; publish outside locks.
- **Clock injection (LTD + REC-156/167):** all time via an injected `Clock`; no `Instant.now()`/`System.nanoTime()`/`System.currentTimeMillis()` in production or test (see the arch-rule note below).

## Test Requirements (tests FIRST)
Write tests before the impl. Inject `Clock.fixed(...)` / a `MutableClock`; pass it into the ledger ctor.
- **FSM transitions:** each of the 5 `PendingStatus` transitions (DISPATCHED -> ACKNOWLEDGED -> CONFIRMED; DISPATCHED -> TIMED_OUT; restart -> EXPIRED).
- **Confirmation:** a `state_reported` matching the `Expectation` -> exactly one `state_confirmed` referencing the right `commandEventId` + `reportEventId`; a non-matching report -> no emission, entry retained.
- **Timeout:** step the clock past `deadline` -> exactly one `command_confirmation_timed_out`; stepping further emits no duplicate.
- **Idempotency on restart (REPLAY):** rebuild from a log with an in-flight `command_issued`; assert IDEMPOTENT re-offers post-LIVE, NOT_IDEMPOTENT -> `EXPIRED`, CONDITIONAL -> adapter offer. Assert the rebuilt in-memory state equals the pre-crash state (pure-projection).
- **`ConfirmationPolicyGateTest`:** an OPTIMISTIC command is never tracked (`pendingCount()` unchanged; no timeout ever emitted).
- **Dual-index lookups:** `getCommand(EventId)` and `getPendingForEntity(EntityId)` both resolve a tracked command.
- **Publish-count pins:** assert the emission counts of `state_confirmed` / `command_confirmation_timed_out` across the FSM tests. **Do not modify the 71/41/53 registry/category count assertions** (they must stay green unchanged ‚Äî a moved count is a red flag you minted a type).
