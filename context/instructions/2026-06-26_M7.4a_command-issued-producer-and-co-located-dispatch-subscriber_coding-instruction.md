# Coding Task: M7.4a — `command_issued` Producer + Co-Located `command_dispatch_service` Subscriber

**Subsystem:** Automation Engine (`core/automation`) + composition root (`lifecycle`)
**Design Doc:** Doc 07 Automation Engine §3.11.1 (Locked; **AMD-95-reconciled 2026-06-26** — event-driven/co-located dispatch); the §1 deeper-M7 architecture decision record (D1–D4, RATIFIED 2026-06-25)
**Phase:** 3-Implementation
**Task Brief Reference:** v7 mission-control §4.C (close governance → M7.4a); the corrected Core sequence (governance close-out → **M7.4a** → M7.4b → thin causal-read-API → AB-4 → M9 → validation)

## What This Implements

The engine ACTS as units through M7.3, but the command pipeline is **not wired** (OR-M7-WIRING): `StandardActionExecutor` calls `CommandDispatchService.dispatch(...)` **in-process**, nothing ever emits `command_issued`, and the Pending Command Ledger has no live input. This WU makes the command hop **substrate-native** per §1 D1: the executor **emits `command_issued` to the log**, and `command_dispatch_service` becomes a **real bus subscriber** that consumes `command_issued` and performs the dispatch. The log becomes the single source of truth for dispatch (not merely its record). For MVP the subscriber runs **co-located / in-process** with the engine (a single local emit→consume hop; V2-A measured ≈1.9 ns over a direct call). This is M7.4a — the producer + the dispatch subscriber + a paired teardown. **M7.4b** (separate) adds the live `pending_command_ledger` subscriber, the `pollExpirations()` tick, the E2E composition-root gate, and the D2 replay CI test.

## Technical Specification

### The producer (in `StandardActionExecutor`)
At the current in-process dispatch site, **emit** `command_issued` rather than calling `dispatchService.dispatch(...)`:

```java
// BEFORE (M7.3 / b296e76):  dispatchService.dispatch(commandEventId, target, action.commandName(), params);
// AFTER  (M7.4a):  publisher.publish(EventDraft for CommandIssuedEvent(
//                      target /*Ulid targetEntityRef*/, action.commandName() /*commandType*/,
//                      serializedParams /*String parameters*/, resolvedTimeoutMs, idempotencyClass),
//                      causalContext);  // correlation = Run's; causation = the triggering/action event
```

**Behavioral contracts:**
- The executor **no longer dispatches in-process**; it publishes exactly one `command_issued` per command action (the AMD-31 issue-order is preserved — one publish per action, in action order; the subscriber then dispatches in `global_position` order).
- `confirmationTimeoutMs` precedence: the target capability's `CommandDefinition.default_timeout` → the `automation.command_pipeline.default_confirmation_timeout_ms` config fallback (Doc 07 §9). `CommandIdempotency` resolved as today.
- `command_issued` carries **no** expectation/confirmation/policy (the frozen 5-comp record — AMD-95 §2.B/§2.C). The ledger (M7.4b) resolves the expectation from the capability.

### The co-located dispatch subscriber (`StandardCommandDispatchService implements Subscriber`)
- `onEvent(EventEnvelope)`: if the event is `command_issued` **and the subscriber is in LIVE mode**, run the existing resolve→validate→handoff→emit path (`command_dispatched` on success, `command_result{invalid|unroutable}` on failure). The two-hop resolution (AMD-95): `EntityRegistry.findEntity(targetRef).deviceId()` → `DeviceRegistry.findDevice(...).integrationId()` → the adapter's `CommandHandler.handle(CommandEnvelope)`.
- **D2 (pure-function-replay) — the load-bearing constraint:** the subscriber performs **no external side-effect on REPLAY**. The bus FSM only delivers to a runtime subscriber in LIVE, but pin it explicitly: guard the dispatch on `mode == LIVE` and prove it with a replay test (the full CI replay-gate is M7.4b; M7.4a ships the unit-level proof).
- **Teardown:** a `stop()`/`close()` that releases any held resource. Even if the dispatch service holds none today, the composition root MUST call it (the M7.3 leak was a subscriber registered with no paired teardown → held SQLite read connection → `@TempDir` failure). Establish the pattern now.

### Composition-root wiring (`HomeSynapseCore`)
- Inject the `EventPublisher` into `StandardActionExecutor` (it already publishes its own diagnostics; confirm the publisher is in scope).
- Register `command_dispatch_service` via `eventBus.subscribeRuntime(new SubscriberInfo("command_dispatch_service", ...), commandDispatchSubscriber)` — **after** the state projection reaches LIVE (mirror the `automation_engine` Step-3.4 ordering; a dispatch subscriber must not act on the replay catch-up).
- **Paired teardown (MANDATORY):** in BOTH shutdown branches (~L735 and ~L777), call the subscriber's `stop()`/`close()` alongside `triggerEvaluator.close()`. This is the explicit fix for the reverted-M7.3 wiring.

### D3 (additive event versioning) — forward constraint, not a mint here
M7.4a reuses the **existing frozen** `command_issued` (zero event-type mint — AMD-95). No new/changed event shape ships in M7.4a, so no upcaster is authored now; **carry D3 as a constraint**: if any event shape is added/changed later in the pipeline it must carry the additive `version` field + an upcaster (the §1 D3 framework). Note this in the MODULE_CONTEXT banner so M7.4b/M9 honor it.

## Locked Decisions / Invariants That Apply
- **LTD-11 (No Broker / VT):** `ReentrantLock`, never `synchronized`; publish outside any held lock.
- **LTD-04 (ULID):** typed wrappers; `command_issued`'s `targetEntityRef` is a `Ulid`.
- **AMD-31 (dispatch ordering):** commands route in `global_position` order — free from the log once dispatch is a subscriber; preserve the per-Run issue order at the producer.
- **AMD-90-INV-01 (no engine retry):** the subscriber dispatches once; never autonomously re-issues.
- **§1 D2 (pure-function-replay):** no external side-effect on replay — the central correctness property of this WU.
- **INV-ES-04 (write-ahead):** `command_issued` is durable before the dispatch subscriber is notified (the bus guarantees this; do not bypass the publisher).

## What to Watch Out For
- **The reverted-M7.3 trap (the #1 risk).** A runtime subscriber registered with **no paired `stop()`** holds its resources (a SQLite read connection in the ledger's case) → JUnit `@TempDir` cleanup fails on Windows → the composition-root tests fail at teardown. Register the dispatch subscriber **only with** a matching teardown in BOTH shutdown branches. This WU exists partly to establish that teardown pattern before M7.4b adds the ledger subscriber.
- **D2 is the whole point — do not let a side-effect run on replay.** Guard dispatch on LIVE; the replay unit test is non-negotiable. The bus FSM makes this structural, but pin it.
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this module's test code — the `NO_DIRECT_TIME_ACCESS` ArchUnit rule scans non-whitelisted test classes and fails `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructor/`@BeforeEach`.
- **Record component / static-factory name collision STOP-check** (the M7.2a-2 gate-fix): for every new/modified record (e.g. `CommandDispatchAssembly.Components`), confirm no static factory / instance method / nested helper shares a component-accessor name. Prefer distinct factory names.
- **No new module edge.** The executor/dispatch already ride `event.bus` (requires transitive) + `event`/`device`/`platform`. Verify `module-info` is UNCHANGED before assuming; if a new exported-API type names another module's type, apply the `requires transitive`↔Gradle `api` lockstep — but this WU should need none.
- **`getIntegrationForEntity` does not exist** — use the two-hop `EntityRegistry.findEntity → deviceId → DeviceRegistry.findDevice → integrationId` (AMD-95; the M7.2a-2 note).
- **AMD-41 / publish-from-subscriber:** the dispatch subscriber publishes (`command_dispatched`/`command_result`) — this is a bus subscriber, NOT a projection advancer (AMD-41 bars publishing inside a read-tx advancer handler). Confirm it registers as a runtime subscriber, not a projection handler.
