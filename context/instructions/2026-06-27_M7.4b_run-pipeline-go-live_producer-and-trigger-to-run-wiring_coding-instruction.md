# Coding Task: M7.4b — The Run Pipeline Goes Live (producer wiring + the trigger→run handoff)

**Subsystem:** Composition root (`lifecycle`) + a small public seam addition in `core/persistence` and `core/automation`
**Design Doc:** Doc 07 Automation Engine §3.2 / §3.6 / §3.9 / §3.11.1 (Locked; AMD-95-reconciled); the §1 deeper-M7 architecture decision record (D1, RATIFIED 2026-06-25)
**Phase:** 3-Implementation
**Task Brief Reference:** v8 mission-control §4.C (drive M7.4b after M7.4a closes GREEN); the corrected Core sequence (M7.4a → **M7.4b** → M7.4c → M7.4d → thin causal-read-API → AB-4 → M9)

## What This Implements

The automation engine has been **built and unit-tested as units** through M7.4a, but the run lifecycle has **never been wired into the live composition root**. Today `HomeSynapseCore` registers the `automation_engine` subscriber, whose `onEvent` calls `evaluator.evaluate(event)` and **ignores the returned matches** (the literal comment in `AutomationEngineSubscriber` is "Run initiation is M7.2"). There is **no `RunManager` in `HomeSynapseCore`** — so a fired trigger evaluates but never initiates a run, no action executes, and nothing emits `command_issued`. The M7.4a `command_dispatch_service` subscriber is wired and waiting, with **no live producer**.

This milestone assembles the run pipeline **for the first time** and connects it: a matched trigger drives `RunManager.initiateRun(...)` (a **direct, co-located, in-process call** — the run lifecycle is the engine's internal orchestration; D1's event-driven rule governs the *command dispatch hop*, which is already a subscriber). The RunManager FSM runs the condition gate and the M7.4a executor; the executor emits `command_issued`; the M7.4a `command_dispatch_service` consumes it and routes it (emitting `command_dispatched`). After this WU the **producer half of the live pipeline is real**: trigger → run → condition → action → `command_issued` → dispatch.

**Out of scope — these are M7.4c / M7.4d (do NOT build them here):** the live `pending_command_ledger` subscriber, the `pollExpirations()` `SharedScheduler` tick, the E2E composition-root **confirmation** gate (issue→confirm), and the D2 pure-function-replay CI test + its INV-id registration. M7.4b proves the producer side with a focused composition-root test (trigger → `command_issued` → `command_dispatched`); the full issue→confirm E2E lands in M7.4c.

## Technical Specification

### 1. The persistence param-serializer seam (`PersistenceFactory`)
Add a public method returning a `Function<Map<String,Object>,String>` that serializes a parameter map to a JSON object string using the persistence `ObjectMapper`. The mapper already round-trips value-model `AttributeValue` (via `AttributeValueSerializer`/`PersistenceJacksonModule`, AMD-52-pinned), which is exactly what the executor's resolved parameters may contain (M7.2b `ComputedValue` resolution yields `AttributeValue`/literals). **Contract:** total over any `Map<String,Object>` whose values are JSON-serializable by the persistence mapper (literals + `AttributeValue`); the executor already floors blank/null output to `"{}"`. Do not expose the `ObjectMapper` itself.

### 2. The run pipeline construction (`HomeSynapseCore`)
Construct, in dependency order, from the registries/services already assembled in Steps 3.1–3.3 (`entityRegistry`, `eventPublisher`, `stateQueryService`, `clock`):
- `StandardSelectorResolver` (over `entityRegistry`);
- `StandardConditionEvaluator(selectorResolver, clock)`;
- `StandardActionExecutor(entityRegistry, selectorResolver, conditionEvaluator, stateQueryService, eventPublisher, clock, defaultConfirmationTimeoutMs, persistenceFactory.commandParameterSerializer())` — use `PendingCommandLedgerAssembly.DEFAULT_CONFIRMATION_TIMEOUT_MS` (30000) as the executor's `defaultConfirmationTimeoutMs` for a single consistent V1 default (the launch-scope single-global-default; confirm there is not already a config key for it);
- `StandardRunConditionGate(conditionEvaluator, …)` — verify the full ctor;
- `RunManagerConfig` from the loaded automation config or `RunManagerConfig.defaults()`;
- `RunManagerAssembly.runManager(eventPublisher, actionExecutor, conditionGate, stateQueryService, clock, config)` → `RunManager`.

All of `StandardSelectorResolver`/`StandardConditionEvaluator`/`StandardActionExecutor`/`StandardRunConditionGate` are **public** (verified) — the composition root constructs them directly; no new assembly seam is required for them.

### 3. The trigger→run handoff (`AutomationEngineSubscriber` + `AutomationEngineAssembly`) — the central design point

**Behavioral contract (the WHAT — fixed):**
- In **LIVE** mode only, when `evaluator.evaluate(event)` returns a non-empty `List<AutomationId>`, the subscriber initiates **exactly one root Run per matched automation**, by calling `runManager.initiateRun(automation, triggeringEvent, matchedTriggers, resolvedTargets, RunCausalChain.root())`.
- `automation` is the `AutomationDefinition` looked up from the registry by id.
- `RunCausalChain.root()` — these are trigger-initiated (root) Runs, not cascades.
- REPLAY is unchanged: **no run initiation on replay** (D2 — the run pipeline must not re-fire on recovery). The existing replay-timer accumulation path stays exactly as-is.
- Idempotency/dedup, concurrency-mode rejection, cascade suppression, and auto-disable are all the **RunManager's** job (it returns `Optional.empty()` for those) — the subscriber does not second-guess them; it initiates and lets the FSM decide.

**The derivation (the HOW — your design call; pushback welcome):** `evaluate()` returns only `List<AutomationId>`, but `initiateRun` needs `matchedTriggers` (`List<Integer>`) and `resolvedTargets` (`Map<String,Set<EntityId>>`). Resolve this cleanly. Recommended approach: a thin package-private run-initiator that, per returned `AutomationId`, (a) looks up the `AutomationDefinition` via the registry, (b) derives the matched trigger indices, and (c) resolves the automation's selector targets via the `SelectorResolver`, then calls `initiateRun`. **If deriving `matchedTriggers` requires re-checking triggers (duplicating `evaluate`'s internal work), raise it** — it may be cleaner to widen `evaluator.evaluate(...)` to return a richer match result (e.g. `record TriggerMatch(AutomationId, List<Integer> matchedTriggerIndices)`) than to re-derive. The contract above is fixed; the shape of the derivation (and whether `evaluate`'s return type widens) is yours to argue with evidence. Whatever you choose, **the matched-trigger indices and resolved targets passed to `initiateRun` must be correct** — a wrong `matchedTriggers` or `resolvedTargets` corrupts the run trace and the causal chain.

### 4. Registration ordering + teardown
- The `automation_engine` subscriber registration (Step 3.4) is **unchanged in shape** — it still registers after the state projection reaches LIVE (the catch-up invariant). What changes is that the subscriber now carries the `RunManager`. Construct the pipeline + RunManager **before** Step 3.4.
- The M7.4a `command_dispatch_service` (Step 3.4b) is already wired — **do not touch it**; M7.4b simply gives it a live producer.
- **Teardown:** the run pipeline objects (selector/condition/executor/gate) are stateless collaborators and need no teardown. The **RunManager** holds in-flight Run state and runs actions on per-Run virtual threads. **Verify whether `StandardRunManager` exposes a `close()`/`shutdown()` and whether in-flight Run VTs need to be interrupted at shutdown** — if so, add the paired teardown in BOTH shutdown branches alongside `triggerEvaluator.close()` (the M7.3 teardown discipline generalizes: anything that starts a thread/holds a resource in `start()` gets a paired stop in `shutdown(...)`). If `StandardRunManager` has no such resource, state that explicitly (justified no-op). Raise it either way.

### D2 / D3 (carried constraints — not built here)
- **D2 (pure-function-replay):** the run pipeline must not initiate runs on replay (the subscriber's LIVE-only guard). The CI replay-gate is **M7.4d**; M7.4b must not regress the property — confirm the subscriber initiates runs only in LIVE.
- **D3 (additive event versioning):** M7.4b mints **no** new event shape (it wires existing producers). No upcaster here. Carry D3 forward.

## Locked Decisions / Invariants That Apply
- **§1 D1 (event-driven/co-located dispatch):** the COMMAND hop is the subscriber (M7.4a, already live after this WU feeds it). The trigger→run step is the engine's internal orchestration — a direct in-process `initiateRun` call. Do not make trigger→run event-driven.
- **§1 D2 (pure-function-replay):** no run initiation / no external side-effect on replay.
- **AMD-91 (causal chain):** root Runs pass `RunCausalChain.root()`; the run-lifecycle events thread the chain (the RunManager does this).
- **AMD-31 (dispatch ordering):** preserved by the existing producer (one `command_issued` per action in order) + the dispatch subscriber (global-position order).
- **AMD-90-INV-01 (no engine retry):** unchanged — the pipeline dispatches once.
- **LTD-11 (No Broker / VT):** `ReentrantLock`, never `synchronized`; publish outside any held lock. The RunManager + executor already honor this; the new wiring adds no locks.
- **core → config ban:** the FSM never sees a `ConfigurationService`; the composition root reads `homesynapse.yaml` and passes a `RunManagerConfig` value (the `RunManagerAssembly` contract).

## What to Watch Out For
- **The trigger→run derivation is the #1 risk.** `evaluate()` returns only `List<AutomationId>`; getting `matchedTriggers` and `resolvedTargets` correct (without duplicating `evaluate`'s work or corrupting the run trace) is the heart of this WU. If re-derivation is ugly, widen `evaluate`'s return — and flag it. A wrong `matchedTriggers`/`resolvedTargets` is a silent causal-chain corruption no compile catches.
- **In-flight Run VTs at shutdown (the M7.3 teardown class, generalized).** The RunManager runs actions on per-Run virtual threads. Confirm whether shutdown must interrupt in-flight runs / whether `StandardRunManager` exposes a stop. A leaked VT or file handle surfaces as a wall of same-signature `@TempDir` `IOException`s in the lifecycle composition-root tests on Windows — the fingerprint of a leak, not N logic bugs. Mirror `triggerEvaluator.close()` in BOTH shutdown branches if a stop is needed; justify a no-op if not.
- **Persistence serializer must NOT expose `ObjectMapper`.** Return `Function<Map<String,Object>,String>`. Exposing `ObjectMapper` forces `requires transitive com.fasterxml.jackson.databind` onto persistence's exported API → `-Xlint:exports` → `-Werror` hard failure. Keep Jackson internal.
- **No new module edge.** `lifecycle` already requires automation/persistence/state/event (it wires them today). The new serializer accessor returns `java.util.function.Function` (java.base). Verify `module-info` needs **no** change in lifecycle, automation, or persistence; if you believe one does, STOP and report with the `requires transitive`↔`api` analysis (the recurring lockstep trap).
- **Record component / static-factory name collision STOP-check** (the M7.2a-2 gate-fix): for any new record (e.g. a `TriggerMatch` if you widen `evaluate`), confirm no static factory / instance method / nested helper shares a component-accessor name.
- **Replay safety:** the subscriber must initiate runs **only in LIVE**. The bus FSM + the evaluator's replay-suppression already gate it; do not initiate runs from the REPLAY path. This is the M7.4d D2 property in embryo — do not regress it.
- **`com.homesynapse.automation` test code must inject `Clock`** (the `NO_DIRECT_TIME_ACCESS` convention; the rule scans non-whitelisted test classes). Use `AutomationTestSupport.FIXED_CLOCK`/`mutableClock()`. (Lifecycle's own tests are whitelisted, but inject Clock for determinism anyway.)
