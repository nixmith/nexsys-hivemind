## What This Implements

M7.4b made the engine ACT live producer-side: a trigger now drives a Run that emits `command_issued`, which the co-located `command_dispatch_service` routes to `command_dispatched`. But the **`pending_command_ledger` is still not registered** — built and unit-tested in M7.3, its LIVE wiring was deferred because its only live input (`command_issued`) did not exist until M7.4b. So today nothing tracks a dispatched command to confirmation, and the "did it actually confirm?" hero half cannot render on a live pipeline.

This milestone registers the ledger LIVE (a runtime bus subscriber, **after** the state projection reaches LIVE, with a **paired `eventBus.unsubscribe(...)` teardown** — the reverted-M7.3 lesson), drives its `pollExpirations()` deadline sweep from the `SharedScheduler`, and lands the **E2E composition-root CONFIRM gate**: boot the real core, fire a seeded automation, and assert the full causal chain `command_issued -> command_dispatched -> state_confirmed` — with the confirming `state_reported` injected synthetically (there is no real device/adapter until M9). After this WU, **OR-M7-WIRING is CLOSED**: the live end-to-end demo — fire, dispatch, confirm — works against the real composition root.

**Out of scope — M7.4d (do NOT build here):** the D2 pure-function-replay **CI** test (a seeded-log replay asserting zero dispatch/run/confirmation side-effects, the V2-A harness template) and registering D2's canonical `INV-…` id. M7.4c's E2E test is a happy-path *confirmation* proof, not the replay-safety gate.

## Technical Specification

### 1. Expose the tick (`PendingCommandLedgerAssembly`)
Add `Runnable expirationTick` to `Components` (the 3rd component), constructed as `ledger::pollExpirations`. The composition root schedules it; nothing else changes. (Record-component STOP-check: `Components(ledger, subscriber, expirationTick)` — confirm no static factory / method shares an accessor name.)

### 2. Extend the scheduler (`SharedScheduler`)
The ledger expiry sweep is a 3rd periodic task. The scheduler is constructed BEFORE the ledger and starts its tasks at construction, so a 3rd-constructor-arg approach would force reordering the scheduler after the ledger (and the ledger registration must follow projection-LIVE). The cleaner path: a post-construction `schedulePeriodic(...)` registered at Step 3.4c. `shutdown()`/`shutdownNow()` cancels all tasks including the new one — confirm, and keep the `safelyInvoke` wrapper (a throwing tick must not cancel the cadence). **If you prefer a 3rd ctor arg + reordering the scheduler construction after the ledger, argue it with evidence — but a leaked tick or a broken refill/health cadence is the failure to avoid.**

### 3. Register the ledger LIVE (`HomeSynapseCore` Step 3.4c)
Mirror Step 3.4b exactly (the dispatch subscriber): construct via the assembly, `subscribeRuntime` with `coalesceExempt=true` AFTER projection LIVE, store the subscriber-id, and **pair the teardown** (`eventBus.unsubscribe(PendingCommandLedgerAssembly.SUBSCRIBER_ID)` in BOTH shutdown branches, alongside the dispatch unsubscribe). The ledger holds a per-subscriber SQLite read connection (the M7.3 leak) — `eventBus.unsubscribe(...)` is what releases it; there is no separate `close()`.

### 4. The E2E confirm gate (the deliverable that proves the hero)
Extend the M7.4b `RunPipelineWiringTest` pattern:
1. Boot the real core to LIVE with a seeded automation (a command action on a seeded entity), a seeded entity+device+integration, **and a capability whose command declares an `expectedOutcome`** (mirror `PendingCommandLedgerTest`'s seeding — `onOff()` may or may not carry one; if not, seed a capability that does, so the path CONFIRMS rather than OPTIMISTIC-bypasses).
2. Fire the triggering event; await `command_issued` + `command_dispatched` (the M7.4b chain).
3. **Inject the synthetic `state_reported`** for the target entity with the value matching the expected outcome (there is no real device until M9 — publish it to the bus, mirroring how `PendingCommandLedgerTest` drives confirmation).
4. Assert: a `state_confirmed` event is published; its causal chain threads back to the `command_issued` (correlation = the Run's; the queryable causal chain `command_issued -> command_dispatched -> state_confirmed`); and the ledger drains (`pendingCount()` returns to 0 / `getCommand(...)` empty).
5. Boot + fire + confirm + `shutdown(...)` cleanly — no `@TempDir` leak (the ledger's read connection released by the paired unsubscribe; the tick cancelled by `scheduler.shutdown()`).

Also add a focused timeout/tick test if practical (a command with no `state_reported` before the deadline -> `pollExpirations()` -> `command_confirmation_timed_out`), using an injected/steppable clock — but the E2E **confirm** path is the required gate; the timeout path is already unit-covered in M7.3.

## What to Watch Out For
- **The reverted-M7.3 `@TempDir` trap is the #1 risk — again.** The ledger is the exact subscriber whose missing teardown got M7.3 reverted. Register it ONLY with `eventBus.unsubscribe(PendingCommandLedgerAssembly.SUBSCRIBER_ID)` in BOTH shutdown branches. Grep the teardown path for the `command_dispatch_service` unsubscribe and add the ledger's beside it — symmetry is the check.
- **Scheduler construction ordering.** `SharedScheduler` is built at ~L426 (before the ledger) and starts tasks at construction. Do not introduce a null tick or a reorder that breaks the refill/health cadence. The post-construction `schedulePeriodic` keeps the ordering intact; if you reorder instead, verify the rate-limiter/health-check timing is unaffected.
- **The confirmation seeding is the subtle part of the E2E.** A command only CONFIRMS if the target capability declares an `expectedOutcome` AND a matching `state_reported` arrives (the M7.3 reality: `CommandDefinition.expectedOutcomes()`, NOT `ExpectationFactory`; `ConfirmationMode.DISABLED` would OPTIMISTIC-bypass and never confirm). Mirror `PendingCommandLedgerTest`'s seeding so the E2E exercises a real CONFIRM, not a bypass. If you can only get an OPTIMISTIC bypass, STOP and report — the gate must prove a true confirmation.
- **Replay safety:** the ledger must register after projection LIVE and not act on replay (M7.3 already guards publishes during REPLAY; the composition-root ordering is the other half).
- **`com.homesynapse.automation` test code must inject `Clock`** (the `NO_DIRECT_TIME_ACCESS` convention). The timeout/tick test in particular must use a steppable clock, never wall-sleep.
- **Record component / static-factory name collision STOP-check** on the widened `Components(ledger, subscriber, expirationTick)`.
