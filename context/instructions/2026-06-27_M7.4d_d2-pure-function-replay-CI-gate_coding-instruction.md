## What This Implements

D2 (the **pure-function-replay invariant**, RATIFIED 2026-06-25) requires that device dispatch and **all external side-effects** (adapter calls, network I/O, notifications — anything observable outside the log) **never run on log replay**; they run **only** on new-command handling in **LIVE** mode. D1's event-driven shape already makes this *largely structural*: every runtime subscriber acts only in LIVE, and the event bus's `REPLAY -> TRANSITION -> LIVE` FSM stops subscribers from publishing during REPLAY. This work unit does **not** add that property — it **names, guarantees, and CI-pins** it with a composition-root **seeded-log replay test** that boots the real pipeline, replays a representative command-lifecycle log in REPLAY mode, and asserts **zero** new dispatch / run-initiation / confirmation side-effects. Because `ci.yml` runs `./gradlew check`, a JUnit test in the tree is automatically a CI gate — so this WU is the production analog of the throwaway V2-A spike (`spike/dispatch-latency-and-log-growth/` B1, which proved 1e6 replayed `command_issued` -> zero dispatch). It is the **last item in the M7.4 arc**; it is safety/governance hardening, **not new functionality**.

There is an **existing UNIT-level D2 proof** — `CommandDispatchSubscriberTest.onCommandIssued_inReplay_doesNotDispatch` (asserts an in-REPLAY `command_issued` yields an empty publisher). M7.4d adds the **integration-level** proof: the *whole assembled pipeline* (trigger->run->dispatch->ledger) is replay-pure end-to-end, on the real composition root.

## Technical Specification

### `RunPipelineReplaySafetyTest` (JUnit test class)
**Package:** `com.homesynapse.lifecycle`
**Responsibility:** Prove, at the composition-root level, that a full REPLAY pass over a seeded command-lifecycle log produces ZERO external side-effects (D2).

**Behavioral contract the test asserts:**

1. **Seed** the persisted event store (before boot, into the `@TempDir` store the same way `RunPipelineConfirmWiringTest` seeds) with a **representative command-lifecycle log** — a sequence that, *if replayed unsafely, would re-fire a real command*. At minimum: a seeded automation definition + a trigger-matching event + a `command_issued` + a `command_dispatched` + a `state_reported` + a `state_confirmed` (a complete prior run, as it would sit in a recovered store). The seed MUST contain at least one `command_issued` whose replay through the dispatch subscriber would, absent the LIVE guard, emit a `command_dispatched` — otherwise the test is vacuous (see Watch-Out).
2. **Boot** the real composition root so the bus performs its `REPLAY` pass over the seeded store (`REPLAY -> TRANSITION -> LIVE`), exactly as a crash-recovery boot would.
3. **Assert ZERO new side-effects produced during the REPLAY pass:**
   - **No new `command_dispatched`** beyond those already in the seeded log (the dispatch subscriber did not act on replayed `command_issued`).
   - **No new run initiated** (RunInitiator / `automation_engine` did not initiate a Run from a replayed trigger).
   - **No new `state_confirmed`** and no ledger confirmation emitted from replay (the ledger rebuilt its state as a pure fold; it did not confirm).
   - **No dispatch-seam invocation** — instrument the actual outbound dispatch seam (the device-adapter call point the `command_dispatch_service` drives; today that is the `command_dispatched` publish) with a recorder; assert the recorder count is **0** across the replay window.
4. **Positive control (regression-meaningfulness):** after the system reaches **LIVE**, drive ONE new trigger and assert the seam DOES fire (a new `command_issued -> command_dispatched`). This proves the test would actually CATCH a regression where replay acted — a green assertion of "zero during replay" is only meaningful if the same harness sees non-zero in LIVE.

**Implementation freedom:** how you observe the seam (a recording publisher/sink, a counting `EventBus` spy, a test double injected at the composition root) is yours — mirror `RunPipelineConfirmWiringTest`'s mechanism. Specify the CONTRACT (zero side-effects during replay; non-zero in LIVE), not a data structure.

### Event Types Consumed (seeded / asserted)
Seeded into the replay log: `command_issued`, `command_dispatched`, `state_reported`, `state_confirmed`, plus the trigger/automation-definition events the existing E2E test uses. Asserted-absent-during-replay: any NEW `command_dispatched` / `command_result` / `state_confirmed`. (Re-derive exact `EventTypes` constants from `EventTypes.java` — counts hold 71/41/53; this WU mints none.)

### Error Handling
None new. The test fails (red) if any side-effect is observed during replay, or if the positive control does not fire in LIVE. Error messages in any new test helper use Register C voice.

## Test Requirements

**Test class:** `RunPipelineReplaySafetyTest` (the deliverable itself).

| Test Method | Scenario | Assertion |
|---|---|---|
| `seededCommandLog_onReplay_producesZeroDispatchSideEffects` | Seed a full prior command-lifecycle log; boot; let the bus REPLAY it. | Zero new `command_dispatched`; dispatch-seam recorder count == 0 across the replay. |
| `seededTrigger_onReplay_initiatesNoRun` | The seeded log contains a trigger-matching event. | Zero Runs initiated during replay (no new run-lifecycle events). |
| `seededConfirmableCommand_onReplay_confirmsNothing` | The seeded log contains a `command_issued` with an open expectation. | Zero `state_confirmed` emitted during replay; the ledger rebuilds state as a pure fold only. |
| `afterLive_newTrigger_doesDispatch` (positive control) | After reaching LIVE, fire one new trigger. | The seam DOES fire (>=1 `command_dispatched`) — proving the gate catches regressions. |

Keep the seeded log **small and deterministic** — this is an INVARIANT gate, not a perf run (the V2-A spike covers scale at 1e6). It must be fast under `./gradlew check`.

## What to Watch Out For

- **Vacuous-test trap (the most important one).** If the seeded log contains nothing whose replay *would* dispatch/run/confirm under an unsafe implementation, "zero side-effects" is trivially true and pins nothing. The seed MUST include at least one `command_issued` (open expectation) and one trigger-matching event so that a *broken* (acts-in-REPLAY) implementation would visibly fail this test. The positive control closes the same gap from the other side.
- **Mode-transition timing.** The assertion window is the REPLAY phase. Mirror `RunPipelineConfirmWiringTest`'s handling of the boot REPLAY->LIVE transition (the M7.4c `awaitLedgerTracked()` precedent shows the timing is real). Capture the during-replay side-effect set deterministically — do not race the LIVE transition. If the bus exposes a mode/phase signal, gate your capture on it; if not, STOP and report (you need a deterministic boundary, not a sleep).
- **Instrument the SEAM, not just the event.** Today the observable dispatch side-effect is the `command_dispatched` publish (no real adapter until M9). Assert on that, but structure the recorder so that when M9 lands a real adapter call, the same gate covers it (a forward note in MODULE_CONTEXT). Do not hard-code an M9 assumption.
- **`@TempDir` + read-connection teardown** (the M7.3 leak class, closed at M7.4a/b/c). If you assemble the real core, ensure clean teardown (the paired close/unsubscribe the composition root already does) so the test passes on Windows `@TempDir`.
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this test — use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via the composition root's seam (as `RunPipelineConfirmWiringTest` does). `com.homesynapse.lifecycle` is **not** on the arch-rule whitelist; treat Clock-injection as a self-enforced convention here (the rule's enforcement reach is app-test-only, but follow the convention).
- **Reuse, don't fork, the boot harness.** Mirror `RunPipelineConfirmWiringTest`; if a shared test-support helper for booting+seeding exists, use it rather than duplicating boot mechanics.
