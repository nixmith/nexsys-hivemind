## What This Implements

The M7.4 arc made the engine *act and confirm* end-to-end; M7.5a makes that **visible on real data** — the gate-2 enabler. It delivers the two hero reads: `GET /api/v1/runs` (the "why did this fire?" entry list) and `GET /api/v1/runs/{runId}/causal-chain` (the "why did it fire? / did it actually confirm?" tree). Both are **pure projections over the immutable event log** (Doc 16 §3.3; INV-SA-03 — "explanation is a projection of the log; no parallel trace store"). There is **no new write path and no new event type**: M7.5a reads the run-lifecycle events (`automation_triggered`/`automation_condition_evaluated`/`automation_action_started`/`automation_action_completed`/`automation_completed`) and the command-confirmation events (`command_issued`/`command_dispatched`/`command_result`/`state_confirmed`/`command_confirmation_timed_out`) that the live pipeline already produces, and assembles them into the frozen v1.1 response shapes.

Per Doc 16 §3.3 (LOCKED) the projection is an **`ExplanationService`** in the **automation read-side** that assembles a **`RunExplanation`** from the log. The rest-api layer consumes that service across a query-service boundary (the `StateQueryService` precedent; the `QUERY_SERVICE_READ_ONLY` ArchUnit rule forbids rest-api touching persistence directly) and maps the projection records to the frozen wire JSON.

## Settled Decisions

### §A. Nick's three rulings (2026-06-27 — co-signed; NOT open questions)

- **DP-A1 — Status vocabulary is mapped at the read boundary.** The frozen `/runs` `status` vocabulary `{COMPLETED, FAILED, SKIPPED, CANCELLED, INTERRUPTED}` is the public contract; the internal `RunStatus` enum stays out of the wire. Map exactly:
  - `COMPLETED → COMPLETED`, `FAILED → FAILED`, `INTERRUPTED → INTERRUPTED` (identity);
  - `CONDITION_NOT_MET → SKIPPED` (the run did not execute actions because conditions were not met);
  - `ABORTED → CANCELLED` (the run was stopped by concurrency mode / shutdown);
  - `EVALUATING` / `RUNNING` are **non-terminal** — the `/runs` list is **terminal runs only**, so these never appear in the list (in-flight runs are out of V1 scope; a future `inProgress` flag would be a low-friction additive contract change like the `name` field C8, never a breaking rename).
  - **No information is lost:** the precise internal reason rides the frozen `terminalReason` string field. `status` is the coarse public term; `terminalReason` carries the exact engine detail (e.g. `status:"SKIPPED", terminalReason:"condition_not_met: …"`).
  - **Where the mapping physically sits** (the rest-api handler vs the automation-side DTO) is **your design call** — the only rule is that the internal `RunStatus` enum must not appear in the public response, and the mapping must not require inverting a JPMS edge.

- **DP-A2 — M7.5a ships the FULL `actions[].outcome` enum now.** Because M7.3 + M7.4c made confirmation live end-to-end, the contract's "ship `DISPATCHED`/`FAILED`/`SKIPPED` only if M7.3 isn't ready" fallback is moot. Ship `{DISPATCHED, CONFIRMED, UNCONFIRMED, FAILED, SKIPPED}`. Derivation per action's command (pure projection by causation over the log):
  - `CONFIRMED` = a `state_confirmed` event correlates the command to its expected outcome (the M7.4c `EXACT_MATCH` path).
  - `UNCONFIRMED` = dispatched, but no `state_confirmed` by the deadline — i.e. a `command_confirmation_timed_out` (or ledger expiry) — the honest "we don't know" state.
  - `FAILED` = a `command_result` failure / recorded reason.
  - `DISPATCHED` = sent and still in-flight at read time, **or** terminal-by-design for a command whose target capability `ConfirmationMode == DISABLED` (no confirmation expected — label honestly as "this device class doesn't report confirmation"; **never** render a false `CONFIRMED`).
  - `SKIPPED` = the action was not executed (e.g. `UnavailablePolicy.SKIP` on an offline target, or fail-fast earlier in the sequence).
  - **The confirmation-MODE nuance is load-bearing.** Confirming modes — `{EXACT_MATCH, ENUM_MATCH, TOLERANCE, ANY_CHANGE}` — can resolve to `CONFIRMED`/`UNCONFIRMED`; `DISABLED` stays `DISPATCHED`. (M7.4c established `StandardCapabilities.onOff().turn_on` carries an `ExpectedOutcome` + `ConfirmationMode.EXACT_MATCH`.)

- **DP-A3 — The projection lives automation read-side, per LOCKED Doc 16 §3.3.** Build the run-list + causal-chain slice of the `ExplanationService` + the `RunExplanation` record in `com.homesynapse.automation`, derived from the log, **never persisted**. rest-api consumes it across the query-service boundary. **Not** a new module (that would be a Doc-16 supersession — OUT for V1's thin slice). **Critical correctness point (confirmed against source):** the hero tree is **NOT** a serialization of `RunCausalChain`. `RunCausalChain` is only the *cascade ancestry* (`ancestors: List<ChainLink{runId, automationId}>`) → it feeds **only** the contract's `cascade:{parentRunId, depth}` field. The `trigger`/`conditions`/`actions`/`outcome` tree is projected from the AMD-92 run-lifecycle **events**. The frozen contract's "maps 1:1 onto `RunCausalChain`" is loose shorthand; the authoritative spec is Doc 16 §3.3's `RunExplanation` assembly. The frozen **response shape is unchanged** — only the internal derivation differs — so this is **not** a cross-lane contract change.

### §B. Scope guard (from the breakdown — binds this WU)

**IN:** the run list + the causal-chain detail (the `ExplanationService` *run slice* + `RunExplanation`). **OUT (do not build):** `NonFiringExplanation` / `GET /automations/{id}/non-firing` (→ M7.5b); `GET /automations` (→ M7.5b); the events feed `GET /events` + `GET /health` (→ M7.5c); `AuditRecord` / the enterprise audit projection (post-V1; `chain_hash` inert); component-authoring / `ComponentRef` attribution (the component model is OUT for V1 — `RunExplanation` may carry a null/omitted component field but builds no attribution); any query language, cross-run analytics, per-non-match trigger recording, or WebSocket push (M12 OUT list, D-OPEN-3 firm).

## Technical Specification

### 1. Automation read-side — `ExplanationService` + records

#### `ExplanationService` (interface) — `com.homesynapse.automation`
**Responsibility:** the read projection that answers "list recent runs" and "explain one run", from the log.
**Methods (contract — finalize exact signatures against source; these are the shape):**
```java
/** Recent TERMINAL runs, newest-first, optionally filtered by automation; a bounded page. Pure projection — no writes. */
RunPage listRuns(java.util.Optional<AutomationId> automationId, long afterPosition, int limit);

/** The assembled causal-chain for one run, or empty if no such run is found in the log. Pure projection — no writes. */
java.util.Optional<RunExplanation> explainRun(RunId runId);
```
- `RunPage` = a small record `(List<RunSummary> runs, long nextAfterPosition, boolean hasMore)` (the projection's own pagination carrier; the rest-api layer maps it to the wire `pagination{nextCursor,hasMore,limit}` via the existing `CursorToken`/`PaginationCodec`). Returning a typed carrier (not the rest-api `PagedResponse`) keeps the automation module free of any rest-api edge.
- **Thread-safety:** stateless over an injected thread-safe `EventStore` + registries; safe for concurrent reads (each request on its own virtual thread, LTD-01).

#### `RunSummary` (record) — the `/runs` list element
Components (map 1:1 to the wire, except `status` which the rest-api layer maps per DP-A1): `runId` (RunId), `automationId` (AutomationId), `automationName` (String, from `AutomationRegistry.get(...)`; null/slug-fallback if the definition is gone), `triggeredAt` (Instant), `status` (**RunStatus** — the internal enum; the rest-api boundary maps it), `terminalReason` (String, nullable).

#### `RunExplanation` (record) — the causal-chain tree (Doc 16 §3.3)
Components (V1 subset; nested records in the same flat package):
- `runId` (RunId), `automationId` (AutomationId), `automationName` (String, nullable).
- `trigger` — `TriggerView(String type, SubjectRefView subjectRef, Instant matchedAt, String firingValue)`.
- `conditions` — `List<ConditionView>` where `ConditionView(String expression, boolean evaluated, boolean result, List<ObservedStateEntry> observedState)`; `ObservedStateEntry(String entityId, String attribute, String value)` projects the row-4 `EvaluatedEntityState`. **Field-mapping note (verified against source):** the event carries `conditionType` (a type name like `"StateCondition"`), `result` (boolean), and `evaluatedState`; it has **no** `expression` and **no** `evaluated` field. Derive `expression` from `conditionType` (a human rendering; the YAML expression text is not on the event), set `evaluated = true` (the event exists only for an evaluated condition — short-circuited ones emit none), and map `EvaluatedEntityState.entityRef` (`EntityId`) → String `entityId`, with nullable `value` tolerated.
- `actions` — `List<ActionView>` where `ActionView(String type, SubjectRefView targetRef, String command, Map<String,Object> params, ActionOutcome outcome, String reason)`; `ActionOutcome` = enum `{DISPATCHED, CONFIRMED, UNCONFIRMED, FAILED, SKIPPED}` (DP-A2).
- `outcome` — `OutcomeView(RunStatus status, String reason, long durationMs, int actionCount, int commandCount)` (rest-api maps `status` per DP-A1).
- `cascade` — `CascadeView(RunId parentRunId /*nullable*/, int depth)` — from the run's flattened cascade data (see the cascade note below).
- `component` — nullable `ComponentRefView` placeholder, **always null in V1** (component model OUT).
- `SubjectRefView(String type, String id)` — the flattened `{type,id}` the wire uses.

> All `RunExplanation`/`RunSummary`/nested component types name only automation-own types + platform identity (`AutomationId`/`RunId`/`EntityId`, all on existing `requires transitive` edges) + `java.base` (`Instant`/`String`/`List`/`Map`/enum). **No new automation module edge** → `core/automation/module-info.java` is **UNCHANGED**. Confirm via the JPMS bullet in the P2 survey.

#### The projection algorithm (`StandardExplanationService`)
- **`listRuns`** — read terminal-run markers via `eventStore.readByType("<automation_completed type constant>", afterPosition, limit)` (confirm the exact `EventTypes` constant). Each terminal event → one `RunSummary` (runId, automationId, status+reason from the event, `triggeredAt` + `automationName` resolved — see the design point). Filter by `automationId` when present. Newest-first: the wire default is `DESC`; `readByType` returns ascending `globalPosition` — reconcile ordering in the projection (e.g. read a window and reverse, or page from `latestPosition()` downward). **[Design point — pushback welcome]:** the precise paging/order strategy (the EventStore exposes ascending `afterPosition` paging only) is yours; keep it O(page), not O(whole-log), and make the cursor opaque (encode `globalPosition` via `CursorToken`).
- **`explainRun(runId)`** —
  1. **Resolve `runId → correlationId`.** The endpoint receives a `runId`, but `readByCorrelation` takes a `correlationId`. The run-lifecycle events carry both the run id and the correlation id, so resolve by locating one of this run's events (e.g. its `automation_triggered`/`automation_completed`) and reading its `correlationId`. **[Design point — pushback welcome]:** the resolution mechanism (a bounded `readByType` locate-by-runId scan vs. a small in-projection `runId→correlationId` index vs. another seam) is yours; the rule is correctness for any historical run + bounded cost. Do not add a new persisted index/event (that would breach zero-mint / no-new-write-path).
  2. `eventStore.readByCorrelation(correlationId)` → the full (bounded) chain.
  3. **Filter to THIS run** — a single correlation can contain multiple runs (multi-automation fan-out / cascade). Keep only the `automation_*` events whose run id == `runId`, plus the `command_*`/`state_confirmed`/`command_confirmation_timed_out` events causally linked to this run's actions (follow `causationId`).
  4. **Assemble** `RunExplanation`: `trigger` from `automation_triggered`; `conditions[]` from `automation_condition_evaluated` (+ `EvaluatedEntityState` → `observedState`); `actions[]` from `automation_action_started`/`automation_action_completed`; `outcome` from `automation_completed`; per-action `outcome` per DP-A2 (join the action's command to its `command_issued → command_dispatched → state_confirmed | command_confirmation_timed_out | command_result` by causation, gated by the target capability `ConfirmationMode`).
  5. **`cascade`** — `depth` from the run's flattened `cascadeDepth` (int) on the lifecycle event (the boundary carries the flattened int + `List<AutomationId>` chain, per the AMD-91-INV-02 gotcha — `RunCausalChain` itself never crosses the event boundary). `parentRunId` is **best-effort/nullable** in V1: the flattened chain carries `AutomationId`s, not `RunId`s, so a parent *run* id may not be directly recoverable — set it when resolvable (e.g. via the causing event's run), else null. **[Design point — pushback welcome.]**
- **Empty/absent:** `explainRun` returns `Optional.empty()` for an unknown runId (→ 404 at the boundary). `listRuns` returns an empty page (NOT 404) when no runs match.

### 2. rest-api — endpoints + gateway + the wire mapping

Mirror the M3.6e.2 pattern exactly: package-private `Handler` impls registered via a **public `RestFilters` gateway method**, driven by the package-private `EndpointContext` SPI, RFC-9457 bodies via `EndpointResponses`.

- **`RestFilters.installRunQueryEndpoints(Object javalinApp, Object explanationService, LongSupplier viewPositionSupplier, Clock clock)`** — registers `GET /api/v1/runs` and `GET /api/v1/runs/{runId}/causal-chain`. **`explanationService` is `Object`-erased and cast internally** (the `installAdminEndpoints(... Object bus ...)` precedent) — this keeps `com.homesynapse.automation` OFF rest-api's exported API, so the new `requires` stays **plain** (non-transitive) and `build.gradle.kts` stays `implementation(...)`. Do NOT type the param as `ExplanationService` on the public signature (that would force `requires transitive com.homesynapse.automation` + `api(...)` — a larger blast radius into websocket-api/app; avoid it).
- **`ListRunsEndpoint`** (mirror `ListEntitiesEndpoint`): query params `automationId` (optional ULID; 400 `invalid-parameters` on malformed), `limit` (default 50, clamp [1,100], malformed→default), `cursor`/`since` (opaque via `PaginationCodec`; malformed→400), `sort` (default DESC). Calls `explanationService.listRuns(...)`, maps each `RunSummary` to the wire object **applying the DP-A1 status mapping**, returns `200 {data:[...], pagination:{nextCursor,hasMore,limit}, meta:{viewPosition,timestamp}}` + the `X-HomeSynapse-View-Position` header. Empty `data` (not 404) when none.
- **`GetRunCausalChainEndpoint`** (mirror `GetEntityEndpoint`): path param `runId` (parse `RunId.of(Ulid.parse(...))`; `IllegalArgumentException`→400 `invalid-parameters`). Calls `explanationService.explainRun(...)`; `Optional.empty()`→404 `not-found`. On success maps `RunExplanation`→the wire tree (**status mapping per DP-A1; outcome enum per DP-A2**), returns `200 {data:{...}, meta:{...}}`. The causal-chain of a TERMINAL run is immutable → a **strong ETag** is appropriate (`ETagProvider.fromEventId(...)`-style keyed on the run's terminal event id); the list is a weak ETag from `viewPosition`.
- **Cross-cutting (inherited — do not re-implement):** AB-1 bearer auth + the `state-store-replaying`(503) readiness gate are already installed on `/api/*` by `installAuth` + `installReadinessGate`; registering these routes under `/api/*` inherits both. `meta:{viewPosition,timestamp}` from `viewPositionSupplier` + `clock`. RFC-9457 via `EndpointResponses`/`ProblemType`.

## What to Watch Out For

- **`RunCausalChain` is NOT the tree.** It is the cascade ancestry only (`ancestors: List<ChainLink{runId, automationId}>`); it never crosses the event boundary (AMD-91-INV-02 — events carry the flattened `int cascadeDepth` + `List<AutomationId>` chain). Build `trigger`/`conditions`/`actions`/`outcome` from the AMD-92 **events**, and use the flattened cascade data for `cascade`. Do not try to read `RunCausalChain` from an event payload — it is not there.
- **No historical-run query exists.** `RunManager` exposes only ACTIVE runs (`getActiveRun`/`getStatus`); completed runs live only in the log. The projection MUST read the `EventStore`, not the `RunManager`.
- **One correlation can hold many runs.** Always filter `readByCorrelation`'s result to the requested `runId` (and to that run's causally-linked command events). A naive "all events in the correlation" assembly will bleed sibling/cascade runs into the tree.
- **Event payloads flatten types (the M7.1 residency gotcha).** `runId` is a bare `Ulid`, statuses are `String`, no automation-resident type appears in any event record (it would invert the JPMS edge). Wrap/parse at the projection (`RunId.of(...)`, `RunStatus.valueOf(...)`) — and handle an unknown status string defensively (don't throw on a future value).
- **Cursor/order mismatch.** `EventStore` pages ascending by `globalPosition` via `afterPosition`; the wire default is `DESC`. Reconcile in the projection without an O(whole-log) scan. Keep cursors opaque (`PaginationCodec`).
- **The `Object`-erasure gateway is deliberate** — it keeps `com.homesynapse.automation` off rest-api's exported API so the new `requires` stays plain. Do not "clean it up" to a typed param (that forces `requires transitive` + `api(...)` — a wider blast radius).
- **Record component / static-factory collision STOP-check** — for EVERY new record (`RunSummary`, `RunExplanation`, `TriggerView`, `ConditionView`, `ActionView`, `OutcomeView`, `CascadeView`, `ObservedStateEntry`, `SubjectRefView`, `RunPage`): confirm no static factory / instance method / nested helper shares a component-accessor name. (This shipped twice past LLM self-review in M7.2a-2; only `javac` caught it.)
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in automation or rest-api test code — the `NO_DIRECT_TIME_ACCESS` ArchUnit rule scans non-whitelisted test classes and fails `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructor/`@BeforeEach`. (automation + rest-api are both outside the `{app,platform,test}` whitelist.)
- **`ApiRequest.body`/JSON-leak discipline** — keep Jackson off any exported signature; the handlers serialize via the established rest-api response path (mirror `ListEntitiesEndpoint`'s `ctx.json(...)`), not a hand-rolled writer.
