**Subsystem:** automation (read-side projection — extends the M7.5a `ExplanationService`) + rest-api (HTTP surface). Composition-root wiring in lifecycle.
**Design Doc:** Doc 16 (Superior Automation Layer) §3.3 + §4 — the explainability read projection (`NonFiringExplanation`) — **LOCKED**; Doc 09 (REST API) — **LOCKED**; Doc 13 (Web-UI Observability MVP) — **LOCKED**. ## What This Implements

M7.5a shipped the hero "why did this fire? / did it confirm?" read on the live log. **M7.5b completes the differentiator** — the *co-equal* "why *didn't* it fire?" read, the single most-differentiated surface in the product (no competitor distinguishes these reasons at all), plus the supporting automation list. Both are **pure projections** — `/non-firing` over the immutable event log + the in-memory `AutomationRegistry`, `/automations` over the registry (+ a best-effort last-run lookup). There is **no new write path and no new event type**: M7.5b reads the same run-lifecycle + confirmation events M7.5a already projects, plus the registry that already holds every `AutomationDefinition`.

Per Doc 16 §3.3/§4 (LOCKED) the projection is the **automation read-side** `ExplanationService` assembling a **`NonFiringExplanation`** from existing diagnostics. **M7.5b extends the M7.5a `ExplanationService`** (it does not create a new service): two new methods on the existing interface, two new public records, new derivation in the existing `StandardExplanationService`. The rest-api layer consumes the service across the same query-service boundary M7.5a established (the `QUERY_SERVICE_READ_ONLY` ArchUnit rule forbids rest-api touching persistence directly) and maps to the frozen v1.1 wire JSON.

## Settled Decisions

### §A. Hub rulings (NOT open questions)

- **DP-B1 — V1 ships the frozen contract's 4-value `verdict`; the Doc-16 §4 `SuppressionReason` (7-value) depth is POST-V1.** This is the M7.5a precedent applied again: the binding V1 wire surface is the FROZEN v1.1 contract (§B3) + the Nick-delegated scope-freeze breakdown, **both co-signed**; Doc 16 §4 is the *eventual/richer* model. Concretely:
  - The wire `verdict` is the contract's **`{CONDITION_NOT_MET, NEVER_TRIGGERED, ACTED_BUT_UNCONFIRMED, DISABLED}`** — computed from **existing run records + absence + config** (the breakdown's explicit V1 data sources), NOT from the full suppression-diagnostic aggregation.
  - Doc 16 §4's **`SuppressionReason`** (`MODE_SUPPRESSED, CASCADE_LOOP, CASCADE_DEPTH_EXCEEDED, CONDITION_NOT_MET, DEFINITION_NOT_LOADED, TARGET_UNAVAILABLE, READ_FAILED_CLOSED`) and the per-`(automation, triggering-event)` keying are the **deep "why did the trigger not match" diagnosis** the breakdown explicitly defers ("no per-non-match trigger-evaluation recording … post-V1"). **Do NOT build `SuppressionReason` in M7.5b.**
  - **Type-name alignment:** name the V1 record **`NonFiringExplanation`** (the Doc-16 §4 type) carrying the V1 field subset (verdict + explanation + triggerSummary + lastEvaluation + lastRelevantRunId), with a Javadoc forward-note that the `SuppressionReason`-keyed enrichment is a post-V1 extension of this same record. The V1 `verdict` is a **distinct contract enum** (`NonFiringVerdict`), NOT a subset of `SuppressionReason` (the contract's `NEVER_TRIGGERED`/`ACTED_BUT_UNCONFIRMED`/`DISABLED` are not suppression reasons — they are the V1 dashboard vocabulary). This is the loose-shorthand reconciliation the M7.5a instruction §A/§DP-A3 already established for `RunExplanation` vs the contract — **the frozen response shape is the contract; Doc 16 §4 is the internal-derivation/eventual spec.** Not a cross-lane change (the frozen contract shape is unchanged); flagged to Nick as the one place V1 ships a coarser shape than Doc 16 §4's eventual model.

- **DP-B3 — The automation list reads the registry, not the log.** `GET /automations` is a projection over `AutomationRegistry.getAll()` (the in-memory single-source-of-truth for definitions). The only log touch is the best-effort `lastRunId` per automation (nullable; reuse the `listRuns` newest-first scan, limit 1). Component summaries derive from the definition's `triggers()`/`conditions()`/`actions()` (see §1).

### §B. Scope guard (from the breakdown — binds this WU)

**IN:** `GET /api/v1/automations/{id}/non-firing` (the `NonFiringExplanation` V1 slice) + `GET /api/v1/automations` (the `AutomationSummary` list). **OUT (do not build):** `SuppressionReason` / the deep per-(automation,event) suppression diagnosis (post-V1 — DP-B1); the events feed `GET /events` + `GET /health` (→ M7.5c); `AuditRecord` / the enterprise audit projection (post-V1; `chain_hash` inert); `AutomationComponent`/`ComponentRef`/`ComputedValue` authoring + real component attribution (the component model is OUT for V1 — the list's `components[]` is a thin derived summary off the existing trigger/condition/action definitions, not the Doc-16 §4 component model); any query language, cross-run analytics, per-non-match trigger recording, or WebSocket push (M12 OUT list, D-OPEN-3 firm). **No change to M7.5a's `/runs*` endpoints or to `RunExplanation`/`RunSummary`/`RunPage`** beyond the additive interface methods.

## Technical Specification

### 1. Automation read-side — extend `ExplanationService` + the two new records

#### `ExplanationService` (interface) — two additive methods, `com.homesynapse.automation`
```java
/**
 * The "why did this NOT fire?" projection for one automation (Doc 16 §3.3; the frozen v1.1
 * §B3 verdict surface). Pure projection — no writes. Returns empty if no automation with this
 * id is known to the registry (→ 404 at the boundary). The verdict is derived from existing
 * run records + config + absence within the expected-since window (DP-B2).
 *
 * @param automationId         the automation to diagnose; never {@code null}
 * @param expectedSincePosition the inclusive lower-bound global log position defining the
 *                              "expected since" window; {@code <= 0} means the default window
 * @return the assembled non-firing explanation, or empty if the automation is unknown
 */
Optional<NonFiringExplanation> explainNonFiring(AutomationId automationId, long expectedSincePosition);

/**
 * Lists all loaded automations as component-based summaries (the frozen v1.1 §B3 list). Pure
 * projection over the {@link AutomationRegistry} (+ a best-effort last-run lookup per
 * automation). Registry order; never {@code null} (empty when none loaded).
 */
List<AutomationSummary> listAutomations();
```
- **The `over(EventStore, AutomationRegistry)` factory is UNCHANGED** — both deps are already injected; the new methods use the same two collaborators.
- **Thread-safety:** unchanged — stateless over the injected thread-safe `EventStore` + `AutomationRegistry`.

#### `NonFiringExplanation` (record) — Doc-16 §4 type name, V1 field subset (DP-B1)
Components (all wire-1:1 except the enums, which the rest-api layer renders to the wire strings):
- `automationId` (AutomationId), `automationName` (String, from the definition), `enabled` (boolean).
- `verdict` (**`NonFiringVerdict`** — the V1 enum, see below).
- `lastRelevantRunId` (RunId, **nullable**).
- `explanation` (String — a plain-language sentence; Register C voice, see §Code Quality).
- `triggerSummary` (String — "what would fire this, in plain words", derived from the definition's triggers).
- `lastEvaluation` (`LastEvaluationView`, **nullable** when there is no relevant run).
- Nested `LastEvaluationView(Instant at /*nullable*/, String conditionsResult /*nullable*/)`.
- Nested enum **`NonFiringVerdict { CONDITION_NOT_MET, NEVER_TRIGGERED, ACTED_BUT_UNCONFIRMED, DISABLED }`** — the frozen v1.1 verdict vocabulary. **Javadoc note:** this is the V1 dashboard verdict; the Doc-16 §4 `SuppressionReason` (7-value, per-triggering-event) suppression diagnosis is a **post-V1** enrichment of this same surface (DP-B1) — do not conflate.

#### `AutomationSummary` (record) — the `/automations` list element
- `automationId` (AutomationId), `name` (String), `enabled` (boolean), `components` (`List<ComponentView>`), `lastRunId` (RunId, **nullable**).
- Nested `ComponentView(String type, String summary)` — a thin derived view: one entry per trigger / condition / action of the definition, `type` = a stable kind label (e.g. the definition's class simple name or a `"trigger"`/`"condition"`/`"action"` kind), `summary` = a short human rendering. **This is NOT the Doc-16 §4 `AutomationComponent` model** (that is OUT — §B); it is a presentation summary off the existing `triggers()`/`conditions()`/`actions()`. **Source-verified shape note:** `TriggerDefinition`/`ConditionDefinition`/`ActionDefinition` are **sealed interfaces with empty bodies** (no common `.summary()`/`.type()` accessor) — `TriggerDefinition` permits 12 records (`StateChangeTrigger`/`EventTrigger`/`TimeTrigger`/…), `ConditionDefinition` 7, `ActionDefinition` 8. So derive `type` from `getClass().getSimpleName()` (the proven M7.5a `buildTrigger` precedent — `StandardExplanationService` already does `triggers().get(0).getClass().getSimpleName()`), and `summary` either as the simple name or via a small per-permit pattern-match over the concrete records where a richer rendering is cheap. Do NOT expect a uniform accessor on the sealed parents.

> All new records name only automation-own types + platform identity (`AutomationId`/`RunId`, already `requires transitive`) + `java.base` (`Instant`/`String`/`List`/`boolean`/enum). **No new automation module edge → `core/automation/module-info.java` is UNCHANGED.** Confirm via the JPMS bullet in the P2 survey.

#### The derivation algorithm (`StandardExplanationService`)

**`explainNonFiring(automationId, expectedSincePosition)` — DP-B2 (verdict priority; window = `expectedSincePosition` or a default 24h-equivalent lookback):**
1. `definition = automationRegistry.get(automationId)`. **Absent → `Optional.empty()`** (→ 404 at the boundary; the automation is unknown).
2. Resolve `automationName`, `triggerSummary` (from `definition.triggers()` — a plain rendering, e.g. "front-door opened"), `enabled`.
3. **If `!definition.enabled()` → verdict `DISABLED`** (`lastRelevantRunId` = the most recent run if cheaply available else null; `lastEvaluation` = null; explanation e.g. *"Automation 'Night Arrival' is currently disabled."*). DISABLED short-circuits — a disabled automation's non-firing reason is that it is off, regardless of history.
4. Else find the **most-recent terminal run for this automation within the window** (reuse the `listRuns` newest-first `readByType(AUTOMATION_COMPLETED, …)` scan, filtered by `automationOf(...)==automationId`, bounded to runs at/after `expectedSincePosition`, take the newest 1):
   - **none in window → `NEVER_TRIGGERED`** (`lastRelevantRunId` = null; `lastEvaluation` = null; explanation states it has not been triggered + the `triggerSummary`).
   - status `CONDITION_NOT_MET` → **`CONDITION_NOT_MET`** (`lastRelevantRunId` = that run; `lastEvaluation` = `{at: run time, conditionsResult: "false"}`; explanation e.g. *"Triggered, but condition 'after sunset' was not met."*).
   - the run derives an action outcome ∈ {`UNCONFIRMED`, `FAILED`} (reuse the M7.5a per-action `deriveOutcome` over the run's command chain) → **`ACTED_BUT_UNCONFIRMED`** (`lastRelevantRunId` = that run; `lastEvaluation` = `{at, conditionsResult: "true"}`; explanation e.g. *"Fired, but the device did not confirm the change."*).
   - **[DP-B2 — the clean-success edge; Nick ruling welcome, see §Design Points]:** otherwise the most-recent in-window run is a clean confirmed success → **recommended V1 default:** return `NonFiringVerdict.NEVER_TRIGGERED` with `lastRelevantRunId` = that successful run (non-null) and an explanation that distinguishes it from a true never-trigger (*"Last fired and confirmed at <time>; no non-firing detected in the requested window."*). The non-null `lastRelevantRunId` + the explanation let the UI tell "never ran" (null) from "ran fine" (non-null) apart. (The frozen enum has no success value; the additive low-friction fix — a post-V1 `FIRED_CONFIRMED` verdict, the `inProgress`/`name` C8 precedent — is the recommended growth path. Pushback + Nick ruling welcome.)
5. `explanation`/`triggerSummary` are plain-language, Register C (no "we"/"sorry"/"please"); keep them short and deterministic (no wall-clock-relative phrasing that breaks replay determinism — render absolute times or omit).

**`listAutomations()` — DP-B3 (registry projection):**
- `automationRegistry.getAll()` → for each `AutomationDefinition`: `automationId`, `name`, `enabled`; `components[]` from `triggers()`+`conditions()`+`actions()` (one `ComponentView` each, kind + short summary); `lastRunId` = the most-recent terminal run for that automation (bounded `readByType` newest-first, limit 1; **nullable** — null when the automation has never run).
- **Cost note:** a naive per-automation full-log scan for `lastRunId` is O(automations × log). Keep it bounded: either (a) a single newest-first pass over `AUTOMATION_COMPLETED` collecting the first-seen run per `automationId` until all known automations are covered or a page budget is hit, or (b) accept `lastRunId=null` past a bounded scan budget (the field is explicitly nullable/best-effort). **[Design point — pushback welcome.]** Registry order; the dataset is small (a home has tens of automations).

### 2. rest-api — endpoints + gateway + the wire mapping

Mirror the M7.5a `ListRunsEndpoint`/`GetRunCausalChainEndpoint` pattern exactly: package-private `Handler`s registered via a **public `RestFilters` gateway**, the package-private `EndpointContext` SPI, RFC-9457 bodies via `EndpointResponses`.

- **`RestFilters.installAutomationQueryEndpoints(Object javalinApp, Object explanationService, LongSupplier viewPositionSupplier, Clock clock)`** — registers `GET /api/v1/automations` and `GET /api/v1/automations/{id}/non-firing`. **`explanationService` is `Object`-erased and cast internally** (the M7.5a `installRunQueryEndpoints` precedent) — keeps `com.homesynapse.automation` off rest-api's exported API so the existing `requires` stays plain. Do NOT type the param as `ExplanationService` on the public signature.
- **`GetNonFiringEndpoint`** (mirror `GetRunCausalChainEndpoint`): path param `id` (`AutomationId.of(Ulid.parse(...))`; `IllegalArgumentException`→400 `invalid-parameters`); query `expectedSince` (optional opaque cursor via the existing `PaginationCodec`/`CursorToken` → a global position; malformed→400; absent→default window, position `<=0`). Calls `explanationService.explainNonFiring(id, expectedSincePosition)`; `Optional.empty()`→**404 `not-found`** (unknown automation). On success maps `NonFiringExplanation`→the wire object (**`verdict` enum name() → the wire string; the nested `lastEvaluation` nullable**), returns `200 {data:{...}, meta:{viewPosition,timestamp}}` + the `X-HomeSynapse-View-Position` header. A non-firing verdict can change as new events arrive → a **weak ETag from `viewPosition`** (not a strong one — unlike a terminal run's immutable causal-chain).
- **`ListAutomationsEndpoint`** (mirror `ListRunsEndpoint`): optional `limit` (default 50, clamp [1,100]), `cursor`/`sort` tolerated. Calls `explanationService.listAutomations()`, maps each `AutomationSummary`→the wire object, returns `200 {data:[...], pagination:{nextCursor,hasMore,limit}, meta:{...}}`. The registry is small; an in-memory page off the full list is fine. **[Design point — pushback welcome]:** the registry isn't log-positioned, so cursor semantics differ from `/runs` — return-all-with-meta vs an in-memory offset cursor is your call; keep the `{data,pagination,meta}` envelope shape exactly. Empty `data` (not 404) when none loaded.
- **Cross-cutting (inherited — do not re-implement):** AB-1 bearer auth + the `state-store-replaying`(503) readiness gate are already installed on `/api/*`; registering these routes under `/api/*` inherits both. `meta:{viewPosition,timestamp}` from `viewPositionSupplier` + `clock`. RFC-9457 via `EndpointResponses`/`ProblemType`.

## What to Watch Out For

- **DP-B1 is load-bearing — do NOT build `SuppressionReason`.** The Doc-16 §4 7-value `SuppressionReason` + per-(automation,triggering-event) keying is POST-V1. V1 ships the 4-value `NonFiringVerdict` from run records + config + absence. If you feel pulled toward aggregating `automation_run_skipped`/`cascade_*`/`config_error` diagnostics into a suppression reason, STOP — that is the deferred depth.
- **`DISABLED` short-circuits.** A disabled automation reports `DISABLED` even if it has a recent `CONDITION_NOT_MET`/unconfirmed run — the verdict is "it's off," not the stale historical reason. Derive `enabled` first.
- **Window semantics.** `expectedSince` scopes "since when did you expect it?" — a run *before* the window must not satisfy "it fired." Default-window when the cursor is absent (recommend a 24h-equivalent lookback, matching Doc 16 §4's `NonFiringExplanation` default window — render it as a position bound, not a wall-clock filter at read time, to stay replay-deterministic).
- **Reuse, don't duplicate.** `automationOf`, `parseStatus`, `deriveOutcome`, the newest-first `readByType` scan, `firstNonBlank` already exist in `StandardExplanationService`. Factor shared private helpers; a second divergent copy of the outcome derivation is a latent inconsistency bug.
- **`lastRunId`/`lastRelevantRunId` cost.** Do not scan the whole log per automation. Bound it (single newest-first pass, or a budget then null). Both fields are explicitly nullable/best-effort.
- **Event payloads flatten types (the M7.1 residency gotcha).** `runId` is a bare `Ulid`, statuses are `String`; wrap/parse at the projection (`RunId.of`, `RunStatus.valueOf`) and handle an unknown status string defensively (don't throw on a future value — log + skip, as M7.5a does).
- **The `Object`-erasure gateway is deliberate** — it keeps `com.homesynapse.automation` off rest-api's exported API so no module-info change is needed. Do NOT type the gateway param as `ExplanationService` (that forces `requires transitive` + `api(...)` — a needless blast radius).
- **Record component / static-factory collision STOP-check** — for EVERY new record (`NonFiringExplanation`, `LastEvaluationView`, `AutomationSummary`, `ComponentView`): confirm no static factory / instance method / nested helper shares a component-accessor name (the M7.2a-2 trap that shipped twice past LLM self-review; only `javac` caught it).
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in automation or rest-api test code — the `NO_DIRECT_TIME_ACCESS` ArchUnit rule scans non-whitelisted test classes and fails `./gradlew check`. Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructor/`@BeforeEach`. (automation + rest-api are both outside the `{app,platform,test}` whitelist.)
- **Register C voice in `explanation`/`triggerSummary`.** Plain, neutral, no "we"/"sorry"/"please"; deterministic phrasing (absolute times or none — no "5 minutes ago").

## Design Points (pushback + Nick ruling welcome)

- **DP-B2 — the clean-confirmed-success edge for `/non-firing`.** The frozen `verdict` enum has 4 values, none meaning "it fired successfully." When the most-recent in-window run is a clean confirmed success, the recommended V1 default is `NEVER_TRIGGERED` + a non-null `lastRelevantRunId` + a "last fired and confirmed at <time>" explanation (so the UI distinguishes "never ran" from "ran fine"). **This is a public-API-semantics call flagged for Nick** — the alternative is an additive post-V1 5th verdict (`FIRED_CONFIRMED`, the `name`/`inProgress` C8 additive-growth precedent).
