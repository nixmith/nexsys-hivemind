## Settled Decisions (from the M9 split charter + this instruction's authoring pass — NOT open questions)

- **D-M92-1 (`open(Object)` stays):** `CoordinatorTransport.open(Object serialPort)` keeps its Phase-2 `Object` parameter — ZERO frozen-interface change, no AMD. The jSerialComm-backed implementation validates `arg instanceof SerialPort` and throws `IllegalArgumentException` (Register C message naming the expected type) otherwise. Consequence: **no jSerialComm type appears on ANY exported API signature** → module-info takes a PLAIN (non-transitive) `requires com.fazecast.jSerialComm` and Gradle takes `implementation(libs.jserialcomm)` — the `-Xlint:exports` lockstep rule is satisfied by keeping the type interior. Any public type you add whose signature would name a jSerialComm type is a design error — restructure or flag `[REVIEW]`.
- **D-M92-2 (EZSP-first, DP-C):** Full `ZnpTransport` is NOT built. ZNP exists in this unit ONLY as the auto-detect probe encoder (one SREQ frame + FCS validation of a possible SRSP) per Doc 08 §3.3's probe sequence.
- **D-M92-3 (byte-seam testability):** All protocol logic (ASH codec, ASH session, EZSP layer, probe) is built against an INTERNAL byte-channel seam (package-private interface; name is yours — e.g. `SerialByteChannel`) with exactly two implementations: the jSerialComm-backed one and the test fake. This is what makes "zero real serial I/O in unit tests" structural rather than aspirational (`NoRealIoExtension` gives no serial protection — pre-verification A13).
- **D-M92-4 (version-variant codec seam):** The EZSP codec is parameterized by the NEGOTIATED protocol version. v13 = the physical acceptance baseline; **v14 differs in status-field widths (32-bit `sl_status_t`) and is SYNTHETIC-frame unit-tested only** (z2m #30891 re-checked 2026-07-03: CLOSED, cluster pinned to ship-8.0.2/v14, no 7.4.5-line reports — physical v14 remains the AMD-96 reflash contingency, not an M9.2 target).
- **D-M92-5 (single-in-flight pacing):** ONE EZSP command outstanding at any time (ASH window=1 stop-and-wait beneath it; the bench's own EmberZNet dump shows `CONFIG_APS_ACK_TIMEOUT=1600` — burst submission is the documented ASH-collapse mode). The pipeline queues, paces, and correlates strictly serially. Timeouts surface per-command; 5 consecutive ASH ACK timeouts drive the ASH session to FAILED (Doc 08 §3.3).
- **D-M92-6 (frozen-interface coverage):** `EzspCoordinatorProtocol implements CoordinatorProtocol` covers the M9.2 scope (init/reset, version negotiation, formation/resume, watchdog, serial command execution). Interface methods that belong to M9.3/M9.4 territory (interview/ZCL/adoption paths, if present on the frozen interface) are stubbed with `UnsupportedOperationException` (Register C, naming the completing milestone) and inventoried in an `[INFO]` list in your handoff — the AMD-63-INV-01 `RESERVED_SUBPROCESS` rejection precedent. - **D-M92-7 (formation storage seam):** Network formation/resume logic is built against an injected package-private persistence seam (e.g. `NetworkParameterStore` — load/save `NetworkParameters` + the network key handle). M9.2 ships the seam + a test fake ONLY. The REAL binding (SecretStore-backed key custody per INV-SE-03, config-backed params) lands with adapter wiring at M9.3/M9.4. The seam's Javadoc must state the INV-SE-03 contract: the network key is NEVER logged, NEVER serialized into plain config, NEVER included in `toString()`.

## Technical Specification

### §Vectors — external-standard constants (embed rule: these are EMBEDDED VERBATIM per the external-standard discipline; sources: Doc 08 §3.3 as amended by AMD-96, distilling Silicon Labs UG100 [EZSP Reference Guide] + AN706 [EZSP-UART Host Interfacing / ASH]. Standing practice: RE-DERIVE each independently from the cited source/reference implementations before pinning in a test; a derivation mismatch is a STOP+report, not a silent pick.)

**ASH byte layer (AN706 / Doc 08 §3.3):**
- Reserved bytes requiring escape: `0x7E` (Flag), `0x7D` (Escape), `0x11` (XON), `0x13` (XOFF), `0x18` (Substitute), `0x1A` (Cancel). Escaping: emit `0x7D` then the byte XOR `0x20`; destuffing inverts.
- Data randomization (DATA-frame payload only): LFSR seeded `0x42`; per byte: `out = in ^ lfsr`, then `lfsr = (lfsr >> 1)` if bit0==0 else `(lfsr >> 1) ^ 0xB8`.
- CRC-CCITT: polynomial `0x1021`, init `0xFFFF`, computed over control byte + (unstuffed, randomized) data field, appended BIG-ENDIAN before the Flag.
- Frame control bytes (AN706 layout — re-derive against bellows/zigbee-herdsman before pinning): DATA = `(frmNum << 4) | (reTx << 3) | ackNum`; ACK = `0b1000_0000 | ackNum` (+ nRdy bit `0x08`); NAK = `0b1010_0000 | ackNum` (+ nRdy); RST = `0xC0`; RSTACK = `0xC1`; ERROR = `0xC2`. Frames terminate with Flag `0x7E`; a Cancel byte (`0x1A`) is sent before RST to flush the NCP's partial-frame state.
- Sequence numbers: 3-bit (mod 8); window size **1** (stop-and-wait, bellows-consistent); acknowledgments piggyback on DATA.
- Adaptive ACK timeout: start **1.6 s**, min **0.4 s**, max **3.2 s**; **5 consecutive timeouts → ASH FAILED** (propagates as transport failure).
- Reset: send Cancel + RST; await RSTACK with a **≥ 2 s** timeout; on ERROR frame instead (probe context), one retry per the §3.3 sequence. The RSTACK **reset-code byte is surfaced in diagnostics** (flow-control-mismatch vs firmware-crash discrimination).

**EZSP layer (UG100 / Doc 08 §3.3):**
- Version negotiation: command **`0x0000` (version) is ALWAYS sent first and ALWAYS uses the LEGACY single-byte frame-ID format**: `[seq][frameControl=0x00][frameId=0x00][desiredProtocolVersion]`; response carries `[protocolVersion][stackType][stackVersion(2 bytes)]`. Only after a version match is the v8+ EXTENDED format used: `[seq][fcLow][fcHigh][frameIdLow][frameIdHigh][params…]`.
- Acceptance band **v13–v14** (EmberZNet 7.4–8.x, AMD-96). Tiers: negotiated **< 13** → structured WARN `zigbee.ezsp_legacy_version` (names the negotiated version + recommends firmware update) then best-effort proceed; **< 8** → `PermanentIntegrationException` (frame format incompatible). Version is read from the NEGOTIATION AT STACK-INIT, never from any external registry (AMD-96/E6 — the bench measured `sw_version=null` externally).
- v14 variance (D-M92-4): 32-bit `sl_status_t` status widths — decode seam keyed on negotiated version; synthetic-frame tests only.
- Watchdog: `nop()` keepalive on **30 s** idle (clock-injected; missed-response feeds ASH-liveness, not a direct throw).
- Pacing (D-M92-5): single in-flight command; the bench EmberZNet dump pins `CONFIG_APS_ACK_TIMEOUT=1600` (ms) — the per-command timeout floor derives from it.

**ZNP probe (Doc 08 §3.3; probe-encode ONLY — D-M92-2):**
- Frame: `SOF(0xFE) | Length(1B) | CMD0 | CMD1 | Data(0–250B) | FCS(1B XOR over Length..Data)`.
- SYS_PING SREQ: CMD0 = `0x21` (SREQ `0x20` | subsystem SYS `0x01`), CMD1 = `0x01`, zero-length data → wire bytes `FE 00 21 01 20` (FCS = `0x00^0x21^0x01` = `0x20`). Re-derive before pinning.
- SRSP recognition: CMD0 bits 7–5 = `0x60`; FCS validated; contents otherwise ignored (probe answers "is this ZNP?", nothing more).

**Auto-detect sequence (Doc 08 §3.3 VERBATIM — implement exactly):** (1) flush RX buffer; (2) ZNP SYS_PING SREQ, wait ≤ 4 s for SRSP; (3) no response → flush again, settle 200 ms; (4) ASH RST (Cancel-prefixed), wait ≤ 4 s for RSTACK; (5) ERROR frame instead → second RST, wait ≤ 4 s. Total budget 10 s. Failure → structured ERROR `zigbee.auto_detect_failed` (port path + bytes received) + `PermanentIntegrationException` whose message tells the user: (a) verify the coordinator is connected/powered, (b) check serial port permissions, (c) set `integrations.zigbee.adapter_type` to `znp`/`ezsp` to bypass. An explicit override parameter bypasses detection entirely (the config KEY is bound by the adapter at M9.3/M9.4 — M9.2 takes it as a method parameter).

**Port identity (AMD-96/E2 + bench corpus):** key on **VID:PID** (`10c4:ea60` for the CP210x-bridged SONOFF MG21/MG24 class) + `/dev/serial/by-id/` stable path + the probe's negotiated stack version as fingerprint disambiguator (MG21 vs MG24 present identical bridges). **NEVER match on USB descriptor strings** — the bench unit reports SONOFF-branded strings, not `Silicon_Labs_CP2102N`. Serial params: 115200 baud, 8N1, flow control None.

### Behavioral contracts (the testable spine)

- **AshCodec:** pure and total — `parse(stuffedBytes)` and `emit(frame)` round-trip for every frame class; corrupt CRC → a typed rejection (structured log at the session layer, frame discarded, NEVER an exception across the transport boundary); randomization is an involution under the LFSR sequence.
- **AshSession:** after `open`, a reset handshake (Cancel+RST → RSTACK) must complete before DATA flows; retransmit on NAK or ACK-timeout with the adaptive band; the 5th consecutive timeout transitions CONNECTED→FAILED exactly once and surfaces one transport-failure signal (no failure storms); post-FAILED sends are rejected fast. Thread ownership: the session runs on the single dedicated platform thread the supervisor will allocate (IoType.SERIAL, Doc 08 §3.2) — the session itself is NOT thread-safe and must document it (mirrors `CoordinatorTransport`'s contract).
- **EzspCoordinatorProtocol:** thread-safe facade (callers are virtual threads; `ReentrantLock` only); commands execute strictly serially (D-M92-5); each command gets a `CompletableFuture`-style async result with per-command timeout; version negotiation runs exactly once per session before any other command and pins the codec variant; the tiered band behavior per §Vectors; `nop()` watchdog only when idle (never interleaves an in-flight command).
- **TransportProbe:** the 5-step sequence with the exact budgets; deterministic under a fake channel + fixed clock (every timeout path testable without real waits).
- **NetworkFormation:** form = two-tier channel preference per §3.13 + `SecureRandom`-sourced key handed DIRECTLY to the store seam (the key never appears in a log, a message, an exception, or `toString` — INV-SE-03); resume = params re-read + §3.13 mismatch handling; round-trip: form → store → resume yields operationally identical `NetworkParameters`.
- **PortWatchdog/reopen:** unplug is detected by disconnect-listener OR read-error OR ASH-liveness — `isOpen()` alone never gates; reopen targets the STABLE identity (by-id/VID:PID), not the possibly-renumbered device node; backoff is capped and clock-injected.

### Error handling

| Condition | Type | Pattern (Register C) | Recovery |
|---|---|---|---|
| Negotiated EZSP < 8 | `PermanentIntegrationException` | names negotiated version + minimum 8 | none — terminal |
| Auto-detect exhausted | `PermanentIntegrationException` | port path + received bytes + guidance (a)/(b)/(c) | manual `adapter_type` override |
| `open(Object)` wrong type | `IllegalArgumentException` | names expected `SerialPort` | caller defect |
| ASH FAILED (5 timeouts) | transport-failure signal (design: your seam — exception or callback, consistent with `CoordinatorTransport`'s contract) | last frame + counters | reset/reopen path |
| CRC-corrupt frame | discarded + structured WARN | counters in the log entry | continue (retransmit protocol handles) |
| Command timeout | per-command async failure | frame id + elapsed | caller policy (M9.4 maps to command_result) |

## What to Watch Out For

- **W1 — Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this module's code or tests. Production code in any module IS caught by the arch gate; treat Clock-injection in this module's test code as the self-enforced convention (enforcement-reach note, 2026-06-13). Use `Clock.fixed(...)`/a mutable test clock injected via constructors. The ASH adaptive-timeout and watchdog logic make this the unit where a sneaky `nanoTime()` is most tempting — don't.
- **W2 — Record component / static-factory collision (STOP-check before handoff):** scan EVERY new or modified record — no static factory, instance method, or nested helper may share a component's exact name (`valid()`/`valid` class of compile error; shipped twice past LLM self-review in M7.2a-2).
- **W3 — VT pinning:** the serial path runs on a dedicated PLATFORM thread (Doc 08 §3.2 — JNI pins carriers); everything above `EzspCoordinatorProtocol`'s lock is VT-callable. Never `synchronized` — including in test helpers.
- **W4 — Blocking reads + interruption:** jSerialComm blocking reads don't respond to `Thread.interrupt()` cleanly; design the read loop around timeout-bounded reads (`setComPortTimeouts` semi-blocking) so close/abandon can't hang a platform thread forever. The M9.1 supervisor bugs (zombie loops, double-close) are the cautionary neighbors — your close path must be idempotent and single-owner.
- **W5 — `isOpen()` lies on unplug** (charter row): a yanked USB device can keep reporting open while reads return 0/-1. Health = listener OR read-error OR ASH-liveness; never `isOpen()` alone.
- **W6 — Automatic module caveat:** jSerialComm resolves by `Automatic-Module-Name: com.fazecast.jSerialComm`. If `:integration:integration-zigbee:compileJava` reports a module-not-found under `--module-path` hygiene, verify the Gradle module-path handling for automatic modules in the convention plugin BEFORE improvising module-info changes — flag `[REVIEW]` if the build-logic needs a touch (build-logic edits are OUTSIDE the sanctioned change set except via that flag).
- **W7 — Randomizer scope:** derandomization applies to DATA-frame payloads only — control/CRC/flag bytes are never randomized; stuffing applies AFTER randomization+CRC on emit (invert exactly on parse). Order-of-operations bugs here produce plausible-looking garbage that passes short-vector tests — include one ≥16-byte vector.
- **W8 — The version handshake is the codec pivot:** nothing decodes with the extended format until negotiation completes; the negotiation frame itself is always legacy (§Vectors). Don't parameterize the version-frame codec by version — it's format-invariant by spec.
- **W9 — Register C voice everywhere** (exceptions, WARN messages): direct, neutral, no self-reference. The §3.3 guidance strings are user-facing.
- **W10 — Zero-config validity (INV-CE-02 neighborhood):** M9.2 takes overrides as PARAMETERS; do not read config, do not register schemas (M9.3/M9.4 bind those). If you find yourself importing config types beyond what signatures already carry, scope has crept.
