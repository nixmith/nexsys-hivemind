# M9.4-ADP — Config-Accepted Adoption (the Tier-1 user-acceptance surface)

**North star:** adoption is CONSENT — a device becomes part of the home only by user acceptance (Doc 02 §3.12). The conservative default is law: an absent/empty accept list adopts NOTHING (proposals sit, honestly logged). And never-false-ALIVE's spirit: adoption failure must surface loudly, never render a half-adopted device.

## §1 — The accept list

New schema key `integrations.zigbee.adopt_devices`: array of IEEE address strings, the log's own rendering (`"0x00178801101A09BB"` — `0x` + 16 hex chars; parse case-insensitively, normalize before compare). **Schema diff: exactly ONE new key** (this is the declared exception to the usual zero-schema pin — flag any second schema change as STOP). Absent or empty ⇒ this WU does nothing at runtime.

Read the key in the adapter at initialize/run (the PJ pattern — `context.configAccess()`; find the list-read idiom the config module provides; if only scalar getters exist, STOP and flag — do not improvise a parser over a raw string without flagging it as a [REVIEW]). Malformed entries: ONE WARN `zigbee.adopt_list_entry_invalid: value={}` per bad entry, skip it, never fail the boot (the list is user input).

## §2 — Adopt at proposal time (the single seam)

At the adapter's discovery listener (`onDeviceDiscovered` call site, adapter `:565` region): when the slice mints a **FRESH proposal** for IEEE `X` AND `X` ∈ the accept list AND **the proposal's interview status is COMPLETE (Nick's pin-1 — never a blind adopt of a PARTIAL interview)** ⇒ log INFO `zigbee.proposal_accepted: device={} source=config` and call `adoption.adopt(X)` immediately (same thread — the ingestion cycle; `adopt()` is internally locked and publishes `device_adopted` + registers device/entities via the existing path — do NOT modify `adopt()` itself).

**The re-link guard (load-bearing):** `adopt()` throws ISE when no proposal exists — and the Stage-2 RE-LINK path (already-adopted device re-pairs) creates NO proposal. The listener must therefore know whether THIS discovery minted a fresh proposal. Sanctioned shape: widen `onDeviceDiscovered` to return a package-private indicator (boolean or a small enum FRESH_PROPOSAL/RE_LINKED/DUPLICATE), or add a `hasProposal(ieee)` query — coder's choice, flagged in the report. **Exception-driven flow is banned** (never call adopt and catch the ISE as the re-link signal).

**Acceptance is evaluated at proposal time only** (fresh proposals; a device proposed before the key was added adopts on its next re-propose — restart + device power-cycle re-announces; this is the documented operational recipe, not a gap). **The COMPLETE gate (pin-1):** a LISTED device whose fresh proposal carries a PARTIAL/incomplete interview is NOT adopted — log ONE WARN `zigbee.proposal_incomplete_not_adopted: device={} status={}` (the operator listed it expecting adoption; the stall must be loud) and let the proposal sit (a later re-announce re-interviews and re-proposes; the accept list catches it then). `adopt()`'s own validation remains a second gate regardless (a classification/validation failure inside adopt must surface as a WARN naming the device — verify it does; if adopt can throw beyond the no-proposal ISE, catch NOTHING, let the cycle's existing error handling surface it, and confirm that handling logs rather than kills the ingestion loop — report what you find).

## §3 — Nothing else changes

`adopt()` internals, the proposal model, Stage-2 re-link, dedup, the TCJ/NCFG paths: untouched. No REST changes (Doc 09 boundary). No new events (`device_adopted` exists — zero mint, 71/41/53 hold).

## Tests (before implementation; home `integration/integration-zigbee/src/test`, default-gated)

1. Listed device, fresh COMPLETE proposal ⇒ exactly ONE `adopt` (device + entities in the registry; ONE `device_adopted` published; the `proposal_accepted` INFO logged).
2. Unlisted device ⇒ proposal sits; zero adoptions; zero new logs beyond the existing `device_proposed`.
3. Key absent ⇒ nothing adopts (conservative default — the PJ §4.2 sibling).
4. RE-LINK of an already-adopted listed device ⇒ `adopt()` NOT called (the guard), no ISE, availability path untouched.
5. Malformed list entry ⇒ ONE WARN, entry skipped, valid siblings still work.
6. Repeat announce of an adopted device ⇒ no double-adopt (existing dedup + the guard; assert ONE `device_adopted` total).
7. **(pin-1)** Listed device, fresh proposal with a PARTIAL interview ⇒ NOT adopted; ONE WARN `zigbee.proposal_incomplete_not_adopted`; the proposal sits; a subsequent COMPLETE re-proposal of the same IEEE adopts normally.
