# M9.4-NCFG — NCP Session Configuration (the join-completion blocker)

**The evidence (bench iteration 1, core `2a5fba2`):** `protocolVersion=13` → `channel_pinned: channel=20` → `network_formed: channel=20 panId=0x3fe6` → `network_up` → **`zigbee.tc_joins_enabled`** → **`zigbee.permit_join_opened: duration=254s`** — then total silence: a factory-reset Hue LCA017 (mains router) and a factory-reset SNZB-03P, both ZHA-proven on ch20, both reset in-range, produced **zero** coordinator-side activity: no `device_join` (0x0024), no `child_join` (0x0023), no `device_announce`. The read pipeline is sound (`pumpInbound(50 ms)` is the production park; the drain routes 0x0024/0x0023 — M9.4-TCJ, hub-audited).

**Causality — stated honestly (this shapes iteration 3 if needed).** ZERO 0x0024 is the load-bearing observation: `trustCenterJoinHandler` is a STACK callback that fires on association events (including DENIED joins) and is NOT gated by ZDO flags — so the failure sits at or below the ASSOCIATION layer, most plausibly the devices never attempting to join. Candidate mechanisms, all closed by the same fix, in explanatory order: (1) **`STACK_PROFILE` default 0** — the beacon advertises the stack profile; Z3.0 devices scanning for a PRO (profile 2) network ignore a profile-0 beacon outright → no association attempt → exactly the observed total silence [BENCH-VERIFY: the firmware build's actual default is unknown from here — the §1.2 read-back measures it]; (2) **`APPLICATION_ZDO_FLAGS` default 0** — the announce-delivery candidate (whether the stack forwards `Device_annce` to the host at default flags is field-ambiguous; it would explain a LATER stall, not the missing 0x0024); (3) `SECURITY_LEVEL` — 5 is believed to be the firmware default already; pinned explicitly as hygiene. Do not record any single one as "the" cause until the read-back + a join proves it.

**North star:** never-false-ALIVE extends here — a coordinator that opens a permit-join window it cannot actually surface joins through is a half-open-door lie. If a REQUIRED config value is rejected, fail honestly; the window must not open.

## §1 — `configureNcp()` (private), called from `startSession()` after version negotiation, before it returns

Placement: `run()` order is `bindTransport → startSession → resumeOrForm` (adapter `:227` calls `startSession()`), so config lands before any `networkInit`/`formNetwork` every launch — required, since memory/stack config must precede stack-up and the NCP was just reset to defaults. Add `FRAME_SET_CONFIGURATION_VALUE (0x0053)` + `FRAME_GET_CONFIGURATION_VALUE (0x0052)` + the config-id/value constants to the existing BENCH-VERIFY block (`:154-206` region) — isolated one-constant edits, the 0x0019/0x90 model. Encoding (bellows/UG100-derived, BENCH-VERIFY): `setConfigurationValue(configId u8, value u16 LE)` → 3-byte payload, response = status byte; `getConfigurationValue(configId u8)` → response = status byte + value u16 LE.

**§1.1 — the write batch** (§2 table), REQUIRED-first so an honest failure precedes the best-effort tail.

**§1.2 — the read-back (hub-added; the decisive instrument):** after the REQUIRED writes, `getConfigurationValue` each of the three REQUIRED ids and log exactly ONE INFO: `zigbee.ncp_configured: zdo_flags=0x{} stack_profile={} security_level={}` with the values THE NCP REPORTS (not the values we sent). This closes the accepted-but-not-applied hole and turns the next bench log into measurement — if a read-back mismatches what was written, log WARN `zigbee.ncp_config_readback_mismatch: id=0x{} wrote={} read={}` and (REQUIRED ids only) fail honestly like a write NAK. No key material is involved (INV-SE-03 trivially holds).

## §2 — The config batch (every id + value BENCH-VERIFY against the frozen v13 stack)

**REQUIRED — a NAK or read-back mismatch is honest failure** (throw per the checked-seam rule → the supervisor classifies; the window never opens; log `zigbee.ncp_config_rejected: id=0x{} status=0x{}`):

| Config | id (BENCH-VERIFY) | value | why it's load-bearing |
|---|---|---|---|
| `STACK_PROFILE` | 0x0C | 2 | ZigBee PRO — the beacon-compatibility candidate for the observed total silence; devices ignore a non-PRO beacon |
| `SECURITY_LEVEL` | 0x0D | 5 | standard Z3.0 APS security (believed firmware-default; pinned explicitly) |
| `APPLICATION_ZDO_FLAGS` | 0x2A | 0x0003 | RECEIVE_SUPPORTED \| HANDLE_UNSUPPORTED — the Device_annce/host-ZDO delivery candidate |

**SUPPORTING — best-effort; a NAK logs ONE WARN `zigbee.ncp_config_skipped: id=0x{} status=0x{}` and continues** (7.4.x firmware self-manages some of these and legitimately rejects — e.g. packet buffers):
`KEY_TABLE_SIZE` (0x1E)=12 · `ADDRESS_TABLE_SIZE` (0x02)=16 · `TRUST_CENTER_ADDRESS_CACHE_SIZE` (0x38)=2 · `MAX_END_DEVICE_CHILDREN` (0x11)=32 · `INDIRECT_TRANSMISSION_TIMEOUT` (0x12)=7680 (the sleepy-child buffer — the SNZB-03P leg) · `PACKET_BUFFER_COUNT` (0x01)=64 · `MULTICAST_TABLE_SIZE` (0x06)=16.
