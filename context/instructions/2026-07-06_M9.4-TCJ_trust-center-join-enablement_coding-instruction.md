# M9.4-TCJ — Trust Center Join Enablement (+ §B channel-pin)

**North star (the escalation's own words):** *a device that fails key exchange must never render as joined.* Adoption stays Device_annce-gated (structurally honest today — a failed join never announces); the new join-handler is OBSERVABILITY, never a device-synthesis path.

**The gap (hub-verified at e363320, four targeted greps returned the absences):** the coordinator forms with hashed-TCLK 0x1B84 + holds `TC_LINK_KEY` (ZigBeeAlliance09, `EzspCoordinatorProtocol:184`) as its OWN preconfigured key (consumed only at `encodeInitialSecurityState:1271`), and `permitJoin(0x0022)` opens the MAC association window (M9.4-PJ, `openPermitJoinWindow` adapter:404-420). But there is NO `setPolicy`/`EZSP_SET_POLICY(0x0055)` (TC join policy), NO `addTransientLinkKey`/`importTransientKey` (the per-join transient well-known key the device needs for the APS key exchange), and NO `trustCenterJoinHandler(0x0024)`/`childJoinHandler(0x0023)` parse. Consequence on EmberZNet 7.x: a Z3.0 device associates at MAC but the key exchange never completes → it never receives the network key → never sends Device_annce → `handleAnnounce` (ZclIngestionUnit:202) never fires → silence. ZHA/bellows pairs these same two devices because it DOES set the policy + install the transient key — the clean A/B. The M9.4a hero test used a PRE-ADOPTED FakeNcp device, so this path never ran until silicon.

## §A — Trust Center join enablement (REQUIRED; the unblocker; bench-iterative)

**§A.1 — Enable joins at window-open (the fix).** In `EzspCoordinatorProtocol`, add the two command exchanges, and CALL them from the adapter's `openPermitJoinWindow()` (adapter:404-420) BEFORE `protocol.permitJoin(duration)` — order: **policy → transient key → permitJoin**. Prefer a single protocol seam `enablePreconfiguredKeyJoins()` (widen `CoordinatorProtocol` with one method) that does both exchanges under the protocol's own lock (the `execute(...)` pattern, `:606-608` — NOT `executeLocked`, since the adapter calls it outside a held lock, same as `permitJoin`). Keep the adapter change minimal: one call before the existing `protocol.permitJoin(duration)` at :417.

1. **TC policy** — `execute(FRAME_SET_POLICY=0x0055, {policyId, decision})`. `EZSP_TRUST_CENTER_POLICY policyId = 0x00`; decision = allow-preconfigured-key-joins (bellows-derived bitmask — **BENCH-VERIFY the exact decision byte(s)**; the shape is policyId u8 + decision u8/u16). Also set `EZSP_TC_KEY_REQUEST_POLICY (0x09)` = allow-TC-key-requests if the v13 stack requires it for the device to fetch the TC link key (**BENCH-VERIFY** — a second setPolicy exchange; include it behind the same seam, easy to drop if silicon shows it unneeded).
2. **Transient link key** — install the well-known `TC_LINK_KEY` (reuse the existing constant at :184 — do NOT duplicate it) as a TRANSIENT key with a WILDCARD partner EUI (all-`0xFF`), so any joining device can use it for the initial APS key exchange. Frame: `addTransientLinkKey` (older EZSP `0x00AF`: EUI64[8] + KeyData[16]) OR `importTransientKey` (7.x security-manager `0x0111`: EUI64[8] + KeyData[16] + flags) — **BENCH-VERIFY which the frozen v13 stack answers; isolate the frame id + struct as a single BENCH-VERIFY block (the 0x0019/0x90 precedent)**. `requireSuccess(...)` on both, so a stack that rejects the enablement surfaces HONESTLY (TRANSIENT → supervisor backoff), never a silent half-open window.

**§A.2 — Observe joins honestly (never-false-ALIVE + observability).** Route + parse `trustCenterJoinHandler (0x0024)` and `childJoinHandler (0x0023)` as unsolicited callbacks (they arrive via the same `pendingCallbacks`/`drainPendingCallbacks` path the ingestion drains; today `ZclIngestionUnit.processCycle` filters to `FRAME_INCOMING_MESSAGE_HANDLER (0x0045)` only at :167-170 and DROPS 0x0024/0x0023 — extend the drain to hand these to a parse). Contract:
- **0x0024** payload (bellows-derived, **BENCH-VERIFY layout**): `newNodeId u16, newNodeEui64 u64, status u8, policyDecision u8, parentOfNewNodeId u16`. Log INFO `zigbee.device_join: device=<eui> nwk=0x<id> status=<name> decision=<name>` for a secured/started join; a DENIED or key-exchange-FAILED status logs WARN `zigbee.device_join_failed: device=<eui> status=<name>` and **synthesizes NOTHING** — no device, no state, no availability edge. A `device_left` status logs INFO and may feed availability (do NOT invent a leave path if none exists — log-only is acceptable this WU).
- **0x0023** (end-device children, e.g. the sleepy SNZB): log INFO `zigbee.child_join: child=<eui> nwk=0x<id> type=<name>`; same never-synthesize rule.
- **THE PIN:** adoption/interview remain triggered ONLY by Device_annce (the existing chain, ZclIngestionUnit:180-182,202). The join-handler NEVER starts an interview and NEVER creates a device. It is honest observability + failed-join surfacing only. Test this explicitly (a scripted failed-status 0x0024 ⇒ zero devices, one WARN).

**§A.3 — Security hygiene (bound the transient key to the window).** The transient well-known key must NOT persist as a standing credential after pairing. Verify on silicon whether the stack's transient key self-expires (EmberZNet transient keys carry a timeout, typically aligned with the permit-join window ~300 s); if it does NOT auto-expire within the window, clear it when the window closes. This WU may log the expected lifetime and leave a MODULE_CONTEXT note if silicon confirms auto-expiry; if it does not, add a clear-on-close (the window close is derivable from `permitJoinDeadline`). Never leave an unbounded wildcard well-known transient key installed. State the chosen disposition in the completion report.

**§A.4 — Tests (before implementation, per §A; home = `integration/integration-zigbee/src/test`, default-gated).** FakeNcp scripts the exchanges + callbacks; assert the LOGIC (frame ids are BENCH-VERIFY, so assert against the SAME named constants the production code uses — a wrong-but-consistent id passes the unit test and is corrected on silicon, exactly the 0x0019/0x90 model):
1. Window-open emits, IN ORDER, the policy frame(s) → the transient-key frame → the `0x0022` permitJoin, all before any join callback (extend/sibling `ZigbeePermitJoinTest` or a new `ZigbeeTrustCenterJoinTest`).
2. Key ABSENT (no permit_join_duration) ⇒ ZERO policy/transient-key frames (enablement rides the window; conservative-default-is-law extends to the join surface).
3. A scripted secured `0x0024` ⇒ one INFO `zigbee.device_join`, ZERO devices synthesized, ZERO interviews started.
4. A scripted failed/denied `0x0024` ⇒ one WARN `zigbee.device_join_failed`, ZERO devices, ZERO interviews.
5. The full happy path unchanged: a scripted `0x0024`-secured THEN a Device_annce (0x0045/ZDO 0x0013) ⇒ the EXISTING announce→interview→adopt chain fires exactly once (proves the handler didn't double-drive adoption).
6. Enablement failure honesty: a stack that NAKs setPolicy/addTransientLinkKey ⇒ the checked exception propagates (TRANSIENT), the window is NOT reported open, `isPermitJoinActive()` stays false (never-false-ALIVE — a half-open door is a lie).

**§A bench-iteration protocol (Nick, on the Pi — this is why FakeNcp can't finish it):** after the unit-green lands, Nick runs the runbook Phase 3 with a device reset in-range; if a frame id/layout is wrong the symptom is a NAK on enablement (honest PERMANENT/TRANSIENT, logged) or a join that still stalls.
