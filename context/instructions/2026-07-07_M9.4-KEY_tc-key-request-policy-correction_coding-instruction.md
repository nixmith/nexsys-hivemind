# Coding Task: M9.4-KEY — TC Key-Request Policy-ID Correction + §B Vocabulary Fold

**Subsystem:** Zigbee Adapter (Doc 08, Locked) — `integration/integration-zigbee` ONLY
**Design Doc:** Doc 08 (Locked); realizes the M9.4-TCJ enablement's own recorded intent (its BENCH-VERIFY clause resolves on silicon evidence)

## What This Implements

Iteration 4 measured the SNZB-03P's leave: `zigbee.key_establishment_failed: status=0x11` ×3 at ~5 s cadence, then `child_left`+`device_left` at +18.8 s post-join, hands off — the Z3.0 BDB TCLK-update failure class. The decode is now **verified against three independent references** and the root cause is a **policy-ID mis-map, one constant wide**:

| Fact | Verified value | Source |
|---|---|---|
| `0x11` in EmberKeyStatus | **`TC_REJECTED_APP_KEY_REQUEST`** | bellows `types/named.py` (verbatim); classic `ezsp-enum.h` (EM35x) |
| EZSP policy 0x05 (v13) | **`EZSP_TC_KEY_REQUEST_POLICY`** — governs TC responses to device TCLK key requests | GSDK 4.4.5 `ezsp-enum.h` (= EmberZNet 7.4.5, the frozen stack) |
| EZSP policy 0x09 (v13) | **`EZSP_TC_REJOINS_USING_WELL_KNOWN_KEY_POLICY`** — a DIFFERENT policy | GSDK 4.4.5 `ezsp-enum.h` |
| Decision 0x51 | `EZSP_ALLOW_TC_KEY_REQUESTS_AND_SEND_CURRENT_KEY` | GSDK 4.4.5 `ezsp-enum.h` |
| What the reference stack sets | `TC_KEY_REQUEST_POLICY` (0x05) → `ALLOW_TC_KEY_REQUESTS_AND_SEND_CURRENT_KEY` (0x51); it does NOT write policy 0x09 | bellows `ezsp/v4/config.py` `EZSP_POLICIES_SHARED` |

Our `EzspCoordinatorProtocol` declares `POLICY_TC_KEY_REQUEST = 0x09` (`:165`) and writes decision `0x51` to it at the enablement (`enablePreconfiguredKeyJoins()`, `:930-937`). **The decision VALUE was right all along; the POLICY ID is wrong.** On the frozen v13 stack that write lands on `TC_REJOINS_USING_WELL_KNOWN_KEY_POLICY`, and the REAL `TC_KEY_REQUEST_POLICY` (0x05) stays at its firmware default (deny-class) — so the TC rejects the SNZB's post-join TCLK-update key request three times and the device leaves per BDB, exactly as measured. The Hue stays because Signify firmware tolerates TCLK-update irregularities (playbook §6); ZHA keeps both because bellows writes 0x05=0x51. This is escalation-ladder rung 3 (policy-family, grounded against the reference stack) — never a guess.

Two deliverables: **§1** the one-constant correction (0x09 → 0x05, with the verified-source javadoc), and **§2** the EmberKeyStatus vocabulary fold (the unmapped 0x00/0x02/0x06–0x13/0x1E names into the §B `statusName()` decode) so every future 0x009B log line self-decodes — the instrument-first doctrine's finishing move.

## Technical Specification

### §1 — The policy-ID correction (ONE constant + javadocs)

At `:164-165` (baseline):

```java

```

Change the value to `0x05`. Keep the constant NAME (it was always the intent; the value was the defect). Rewrite the javadoc to carry the verified decode, in the module's BENCH-VERIFY idiom — e.g.:

```java
/**
 * EzspPolicyId TC_KEY_REQUEST_POLICY (0x05) — governs how the TC answers a
 * joiner's TCLK key request (the Z3.0 BDB update leg). VERIFIED against
 * GSDK 4.4.5 ezsp-enum.h (= EmberZNet 7.4.5, the frozen bench stack) and
 * bellows EZSP_POLICIES_SHARED. The original 0x09 was
 * TC_REJOINS_USING_WELL_KNOWN_KEY_POLICY — writing the allow decision there
 * left this policy at its firmware default, and the TC rejected every
 * TCLK-update request (iteration-4: key_establishment_failed status=0x11 ×3
 * → BDB leave). BENCH-VERIFY: silicon confirmation = iteration 5a
 * (key_established + the device stays).
 */
static final int POLICY_TC_KEY_REQUEST = 0x05;
```

Also update the `DECISION_ALLOW_TC_KEY_REQUESTS` javadoc (`:173-179`): its "BENCH-VERIFY whether the frozen v13 stack needs it; … drops as a one-line edit" clause is now RESOLVED — silicon proved it load-bearing; the defect was the policy id it was written to, not the decision. State that; the decision value 0x51 is the bellows default for the real policy. Javadoc-only — the constant and the enablement call site (`:937`) are otherwise untouched.

**No other production change.** `enablePreconfiguredKeyJoins()` keeps exactly its three exchanges (setPolicy 0x0055 ×2 + importTransientKey 0x0111); `POLICY_TRUST_CENTER`/`DECISION_ALLOW_PRECONFIGURED_KEY_JOINS` untouched (joins are silicon-proven); the transient-key path untouched.

## Settled Decisions (NOT open questions — implement as stated)

- **The fix is the policy ID, not the decision value.** 0x51 is verified correct (the bellows default for policy 0x05). Do not "also try" 0x52 (`ALLOW_TC_KEY_REQUEST_AND_GENERATE_NEW_KEY`) — with the hashed-TCLK formation mode the send-current path is the reference behavior. One variable per bench iteration (playbook §6).
- **Do NOT add a policy write to 0x09** under its real name. The reference stack does not write `TC_REJOINS_USING_WELL_KNOWN_KEY_POLICY`; matching the reference is the point of the A/B. Removing our mis-aimed write (which §1 does by re-aiming it) restores 0x09 to firmware default — the bellows-equivalent posture.
- **No `getPolicy` (0x0056) read-back this WU.** If iteration 5a shows 0x11 again, a read-back instrument is the NEXT rung — ruled with Nick, not improvised here (instrument-first, one instrument at a time).
- **The `zigbee.key_established` / `zigbee.key_establishment_failed` log tokens are FROZEN** (the runbook/playbook greps bind them). The fold changes only what `status=` renders for previously-unmapped bytes (hex → name).
