# M9.4-RPT — Reporting Binding, Adoption-Path Drive, Posture Routing (+ §B observability)

**North star:** never-false-CONFIRMED is enforced at INSTALL time — a device may only be confirmation-tracked through reports its reporting posture actually VERIFIED. And the anti-vacuous doctrine: every arm that can succeed silently gets ONE positive-evidence INFO, so the next bench log is measurement, not absence-of-failure.

## §1 — The EZSP `ReportingOps` binding (the missing production implementation)

ONE new package-private class (suggested `EzspReportingOps`) implementing the FROZEN `ReportingOps` interface verbatim — do NOT modify the interface — over the existing M9.4b write seams (`sendZclFrame(:1054)` / the package-private nwk overload `(:1079)` / `lookupNetworkAddress`). All four methods: `bind` (ZDO Bind_req 0x0021), `configureReporting` (ZCL global 0x06), `readReportingConfiguration` (ZCL global 0x08 → `ReportingConfigRecord`), `writeCieAddress` (IAS_CIE_Address write, §3.12). Every frame id, ZDO cluster id, ZCL command id, status code (0x86 UNSUPPORTED_ATTRIBUTE / 0x8C UNREPORTABLE_ATTRIBUTE), and the sleepy-timeout constant sit in an isolated **BENCH-VERIFY constants block** (the 0x0019/0x90 model — a silicon correction is a one-constant edit fixing code + tests together). Typed results only per the interface javadoc — no checked exception crosses the seam; a TIMEOUT is a RESULT, never a throw. All exchanges ride the ingestion thread synchronously (the single-in-flight EZSP queue discipline — no new threads, no executors).

## §2 — The adoption-path drive (both arms, the single seam again)

At the adapter's `adoptIfAccepted` region (the M9.4-ADP seam): **fresh adoption** (`PROPOSED` + adopted) ⇒ `reportingConfigurator.configureDevice(...)` with the interview's endpoints + the matched profile; **re-link** (`LINKED` outcome) ⇒ `reportingConfigurator.onRejoin(...)` — this is how iteration 4's already-adopted Hue and SNZB get configured (power-cycle → re-announce → re-link → drive). Synchronous, same thread, AFTER the adoption/re-link bookkeeping completes. **Reporting outcomes NEVER gate or fail the adopt/re-link** (Doc 02 staging) — a fully-degraded posture list is a recorded outcome, the device stays adopted. Then ONE per-device success INFO — **`zigbee.reporting_configured: device={} clusters={} verified={} degraded={}`** (counts, not identities beyond the device) — emitted on EVERY drive completion including all-degraded (the counts tell the truth; the anti-vacuous mandate: runbook step-11 pairs WARN-absence with THIS line's presence). The configurator's existing WARN vocabulary (`bind_failed`/`reporting_ack_lies`/`ias_enroll_incomplete`) must reach the log exactly as the runbook greps expect — verify the log lines exist in the M9.3 code and fire on the paths the runbook names; if any is missing at source, that is a [REVIEW], not an improvisation.

## §3 — Posture-fact routing (the AMD-97 confirmability consumption — contract pinned, mechanism coder's choice)

The returned `List<ReportingPostureFact>` must reach the installed per-device confirmation surface (`ConfirmationOverrideInstaller`'s output — the Doc 02 §3.8 adoption-installed overrides): **a capability whose confirmation rides authoritative reports stays report-confirmable ONLY where the covering cluster/attribute posture is VERIFIED; any non-VERIFIED posture (UNSUPPORTED/UNREPORTABLE/TIMEOUT/read-back-failed/ack-lies) downgrades that capability's confirmation HONESTLY per the AMD-95/AMD-97 vocabulary (DISABLED-with-recorded-reason or BEST_EFFORT as the ratified mapping dictates), with ONE WARN naming device + cluster + posture.** Never-false-CONFIRMED is the invariant being enforced; never-tracked and honestly-verdicted remain different promises (the AMD-97-literal semantics — an issuable command on a downgraded capability still renders its immediate honest verdict). Mechanism (installer hook, a second install pass, or a posture-aware decoration) is your choice — FLAG the chosen shape in the report; if it cannot be expressed without public-surface or cross-module changes, STOP and flag [REVIEW] rather than widening surface.

## §B (SEVERABLE — Nick's veto window; delete this section to strike) — key-establishment observability

Widen the ingestion drain's callback dispatch (the 0x0024/0x0023 precedent, same single-drain rule) to consume **EZSP `zigbeeKeyEstablishmentHandler` 0x009B**: parse partner EUI64 + status (BENCH-VERIFY layout), LOG-ONLY — INFO `zigbee.key_established: device={} status={}` on success-class statuses, WARN `zigbee.key_establishment_failed: device={} status={}` otherwise. **THE PIN (test-enforced): the handler never creates a device, never schedules an interview, never publishes an event, never alters adoption/availability** — pure observability. This instruments the OBS-2 discriminator: if the SNZB's post-join leave is the TCLK-update class, iteration 4's log SHOWS it.

## Tests (before implementation; home `integration/integration-zigbee/src/test`, default-gated)

1. Fresh-adopt drive: listed device adopts ⇒ bind + configureReporting + read-back frames issued per the profile's reporting spec (assert against the production BENCH-VERIFY constants); facts recorded; the `zigbee.reporting_configured` INFO verbatim with correct counts.
2. Re-link drive: an already-adopted device re-announces ⇒ `onRejoin` runs the same configuration (the iteration-4 Hue shape); no adopt, no proposal, no double-publish.
3. Ack-lies: configure SUCCESS but read-back mismatch ⇒ the honest posture + the existing `reporting_ack_lies` WARN; no VERIFIED fact minted.
4. Degrade ladder: 0x86 / 0x8C / timeout ⇒ the matching posture facts, WARNs per the existing vocabulary, adoption/re-link UNAFFECTED, the success INFO still emitted with `degraded=` counting them.
5. Posture routing (§3): a non-VERIFIED posture on a report-confirmable capability ⇒ the installed confirmation is downgraded per the ratified vocabulary + ONE WARN; a VERIFIED posture leaves it untouched. (The install-time never-false-CONFIRMED fence, test-asserted.)
6. Reporting never gates: every ops call failing ⇒ device still adopted/re-linked, entities intact, loop survives.
7. (§B) 0x009B: success and failure payloads parse and log verbatim; THE PIN — zero devices/interviews/events/adoption-state changes from the handler.

FakeNcp: a MINIMAL shared-fixture widening for the new frames is sanctioned (the NCFG built-in-model precedent — scripted responses win; per-class responders and the rig byte-untouched; declare exactly what widened).
