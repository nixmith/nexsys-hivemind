## What This Implements

The ¬ßB classifier is binary: `established()` ‚Üí INFO, everything else ‚Üí WARN `key_establishment_failed`. Silicon showed a healthy Z3.0 TCLK exchange emits in-flight PROGRESS callbacks first ‚Äî `TC_RESPONDED_TO_KEY_REQUEST` (0x06, the TC sent the key) ‚Äî so every healthy join currently logs a false failure moments before the real success. This WU adds the missing middle class: **progress ‚Üí DEBUG**, established ‚Üí INFO (unchanged), everything else ‚Üí WARN (unchanged). The progress set is RULED and exact: **{0x06 TC_RESPONDED_TO_KEY_REQUEST, 0x07 TC_APP_KEY_SENT_TO_REQUESTER, 0x0C TC_RECEIVED_FIRST_APP_KEY_REQUEST}** ‚Äî do not widen or shrink it.

## Technical Specification

### ¬ß1 ‚Äî `KeyEstablishment.progress()`

Add beside `established()` (same style, javadoc explains WHY with the iteration-5a citation):

```java
/**
 * True for the benign in-flight progress statuses ‚Äî the TC mid-exchange
 * (responded / key sent / first-of-two received). Iteration 5a measured a
 * healthy TCLK exchange emitting TC_RESPONDED_TO_KEY_REQUEST 0.26 s before
 * the genuine success; logging progress as failure is a false verdict the
 * ¬ß51 honesty greps would inherit. Ruled set (2026-07-08): exactly these three.
 */
boolean progress() {
    return status == KEY_STATUS_TC_RESPONDED_TO_KEY_REQUEST
            || status == KEY_STATUS_TC_APP_KEY_SENT_TO_REQUESTER
            || status == KEY_STATUS_TC_RECEIVED_FIRST_APP_KEY_REQUEST;
}
```

`established()` is **BYTE-UNTOUCHED** (success set stays exactly `{0x01,0x03,0x34,0x65}`). `statusName()` untouched. No new constants (all three names exist since M9.4-KEY). Record STOP-check: no member name collision (`partner`, `status`).

### ¬ß2 ‚Äî the middle arm in `handleKeyEstablishment`

Between the established-INFO arm and the failure-WARN fallthrough:

```java
if (key.progress()) {
    log.debug("zigbee.key_establishment_progress: device={} status={}",
            key.partner(), key.statusName());
    return;
}
```

**Log-token contract:** `zigbee.key_established` (INFO) and `zigbee.key_establishment_failed` (WARN) are FROZEN ‚Äî the runbook/¬ß51 greps bind them. The ONE new token is `zigbee.key_establishment_progress` at DEBUG (invisible at the bench's INFO root ‚Äî a healthy exchange's INFO stream shows exactly one `key_established`). THE PIN is unchanged: this handler still never creates/schedules/publishes/alters anything.
