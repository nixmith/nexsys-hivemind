# Coding Task: M9.5-DUR — Device/Entity Registries as Projections of the Event Log

**Subsystem:** Event Model + Device Model + Persistence + Lifecycle + Zigbee Adapter (cross-cutting, governance-class)
**Design Doc:** AMD-99 (RATIFIED 2026-07-08) amending Doc 01 §4.3 · Doc 02 §3.8/§3.12 · Doc 12 Phase 3 — all pointer-folded; the AMD text is binding
**Phase:** 3-Implementation
**Task Brief Reference:** The DUR ruling (pm-handoff v24 beat-4, Nick, verbatim: *"DUR — (i). Event-source the registries"*) → AMD-99 ratification + carry-list (pm-handoff v25 beat-1)

## What This Implements

The registries are today imperative in-memory state: `HomeSynapseCore:347` constructs a fresh `InMemoryDeviceRegistry` every boot (`entityRegistry` fresh at `:510`), nothing rehydrates either, and `DeviceAdoptedEvent` carries only an `entityId` — so adopted devices re-mint identity on every restart, the state view accumulates orphan rows, and the re-link arm is structurally unreachable across restarts (measured, bench iterations 3–5a). This WU closes the one hole in "everything is provably derived from the log": `adopt()` will publish **`device_registered`** and **`entity_registered`** events carrying the COMPLETE device/entity records (capabilities and installed DP-a confirmation tuning included), ALL registry mutation will flow through a single projection-apply path (**REG-INV-1**, register §53), and boot will reconstruct both registries by replaying those events on the existing event-bus machinery before automation definitions bind entity refs. `device_removed` becomes a first-class tombstone the projection honors from day one. Enforcement (`REGISTRY_MUTATION_ONLY_VIA_PROJECTION`) ships IN this WU — no deferral.

## Settled Decisions (from ratified AMD-99 + the v25 beat-1 carry-list — NOT open questions)

- **DP-1 (payload shape — R-A):** (A) full-fidelity records. The mirrors are **event-model-local nested records** in `com.homesynapse.event`. Enums flatten to `String` (enum `name()`) — EXCEPT `AttributeType` and `AttributeValue`, which live in `com.homesynapse.value` (a leaf module event-model already `requires transitive`) and stay TYPED. `Number`/`Duration`/`Instant`/`Ulid` are `java.base`/platform types — allowed. **Zero references to `com.homesynapse.device` types anywhere in event-model** (the AMD-52 cycle class — JPMS hard rule).
- **DP-2 (apply-path home):** `RegistryProjection` lives in `com.homesynapse.device` (device-model `requires com.homesynapse.event` — verbatim embed below). It is THE ONLY production type that calls registry mutating methods (the ArchUnit rule enforces this). The bus-facing wrapper (`Subscriber` impl) lives in `com.homesynapse.lifecycle` (lifecycle already requires `event.bus` + `device` transitively) and delegates. **The apply path consumes ONLY the event payload — never an in-scope domain object — so live-apply ≡ replay-apply by construction.**
- **DP-3 (emission contract — AMD §3):** `adopt()` publishes `device_registered` (durable at return, G-DUR3) → `RegistryProjection.apply(...)` inline → per created entity: publish `entity_registered` → apply inline → adapter-local maps + listener (unchanged) → publish the existing `device_adopted` LAST, **RETAINED byte-unchanged** (no existing consumer moves). Causal order device-before-entities. Subject refs: `SubjectRef.device(deviceId)` / `SubjectRef.entity(entityId)`.
- **DP-4 (updates — fold F1):** registration updates ride an **idempotent re-emit of the same two types — NO third event type.** Concretely: `applyPostureRouting` (`ZigbeeIntegrationAdapter` ~:704) stops calling `updateEntity` directly; when the routed capabilities differ it publishes an `entity_registered` re-emit (same entityId, refreshed capability mirrors) and applies via the projection. `reinstallOverrides` (`ZigbeeAdoptionSlice` ~:421) is **REMOVED outright** along with its call from `relink` — post-DUR the adoption-time tuning persists via replay, so the re-install is redundant; and a profile-file change silently mutating the registries is exactly what REG-INV-1 bans (the sanctioned future path for a genuine capability change is a re-emit from a real re-interview — out of scope here). `relink`'s adapter-map rebuild STAYS.
- **DP-5 (boot rebuild — R-C):** a registry-projection subscriber on the existing bus machinery, registered in Phase 3 immediately after the registries construct (Step 3.1 region). Because the registries are in-memory, the subscriber must replay from position 0 EVERY boot: **write `checkpointStore.writeCheckpoint(REGISTRY_PROJECTION_SUBSCRIBER_ID, 0L)` before subscribing** (the `CheckpointStore` contract: *"A return value of 0 means the subscriber should begin processing from the start of the event log"* — read at registration). The ONE sanctioned composition-root addition: an `awaitRegistryProjectionLive()` sibling of `awaitProjectionLive()` (same poll shape, keyed on the new subscriber id), called alongside the existing await BEFORE Step 3.4 (automation definitions bind entity refs — carry-list C3). **Anything wider than this one await = STOP-and-flag.** Sanctioned fallback if the subscriber path proves invasive (AMD §5 coder latitude — flag it, don't silently switch): a direct ordered store read of the three event types during Phase 3, then a LIVE subscription for ongoing deliveries.
- **DP-6 (adapter map rehydration — AMD §3 boundary note):** the adapter's IEEE→id / entity / binding maps rebuild FROM the rebuilt registries, not from private events. Realize at adapter startup (post-Phase-3, so the registries are caught up): enumerate `deviceRegistry` devices carrying this adapter's `integrationId` and a `HARDWARE_NAMESPACE` identifier → invoke the EXISTING `relink(ieee, device, profilesByIeee-cached-id-or-null)` for each. This is what makes ingestion and pin-2 work immediately post-restart; the announce-time `findByHardwareIdentifier` fork (~:159) then takes the LINKED arm naturally.
- **DP-7 (tombstone — R-D):** `device_removed` (existing type, zero emitters) applied by the projection on replay AND live delivery: remove the device and all its entities (deviceId from the envelope's `SubjectRef`, G-DUR7). Idempotent: removing an absent device is a no-op. NO emission surface is built (future milestone); the mechanism is proven by test (carry-list C4).
- **DP-8 (apply semantics — "idempotent by identity"):** apply is an **upsert in log order**: identity absent ⇒ create; identity present with equal state ⇒ no-op (this is what makes the live self-delivery of an adoption-time event a no-op); identity present with different state ⇒ replace (this is what makes the F1 re-emit an update). Log order is truth — last write wins. `device_removed` then re-`device_registered` (a future re-adopt) resurrects cleanly by construction.

## Locked Decisions That Apply

- **LTD-04 (ULID):** payload ids are raw `Ulid` (platform type) in event records — the typed wrappers (`DeviceId`, `EntityId`) stay device-model-side and are reconstructed at apply (`DeviceAdoptedEvent.entityId` precedent). Crockford strings only at boundaries/logs.
- **LTD-11 / virtual threads:** `ReentrantLock` only — the projection's lock, the slice's existing lock discipline. Never `synchronized`.
- **Jackson isolation (hard rule + `NO_JACKSON_IN_DOMAIN_MODEL` / `NO_JSON_TYPE_INFO_IN_EVENTS`):** zero Jackson annotations/types in event-model or device-model; all serialization lives in persistence codecs. The sealed `ExpectationRef` therefore needs its persistence-side (de)serialization handled the way AMD-87 handled `Expectation` — follow that codec's pattern, no `@JsonTypeInfo`.

## Invariants That Must Hold

- **REG-INV-1 (§53, verbatim):** *"The device and entity registries are projections of the event log. Every registry mutation flows through a single projection-apply function whose only inputs are the registration/removal event types; at boot the registries are reconstructed by replaying those events; no other code path mutates registry state. Adoption-time writes use the SAME apply function (write-ahead: the event is durable before the in-memory apply; apply is idempotent by identity, so live bus delivery of a self-published event is a no-op)."* — enforced by the new rule + the identity IT + the idempotency tests.
- **INV-ES-02/04/05/09:** state derivable from events; write-ahead (publish-before-apply, G-DUR3); subscriber idempotency (DP-8); pure-function replay (the projection derives nothing and publishes nothing during replay — it only applies; the subscriber NEVER publishes).
- **AMD-97-INV-01 (never-false-CONFIRMED):** untouched — but the tuning captured in `entity_registered` is what makes it provable across restarts; the identity IT asserts the rebuilt `ConfirmationPolicy` equals the adopted one.
- **INV-CE-04 / LTD-17:** core never sees zigbee — the mirrors are protocol-agnostic; nothing zigbee-typed enters event-model.

## What to Watch Out For

- **Record component / static-factory collision STOP-check** — scan EVERY new record (mirrors especially: `value()`, `type()`, `mode()` are collision-prone component names) for factory/method name clashes before handoff.
- **Do NOT `@EventType` the mirrors** and do not implement `DomainEvent` on them — only the two top-level events. The 41→43 pin is the tripwire; get it right by construction.
- **`Number` round-trip:** Jackson narrows/widens numerics by default. The AMD-52 float discipline + AMD-87 codec are the precedent — pin actual types in the round-trip test (an `Integer` minimum must not come back `Double`… or if the codec canonicalizes, the DOMAIN equality after `toDomain` is what must hold; assert at that layer and document the choice).
- **Deterministic collection encoding:** `Set`→sorted-`List` at the mirror boundary (hardwareIdentifiers, permissions, validValues) — replay equality and codec-fixture stability depend on it.
- **The projection must not publish** — a subscriber that publishes during REPLAY breaks INV-ES-09; the projection-apply is pure consumption. The boot INFO rides `onLive`, not `onEvent`.
- **`entityRole` null-coercion:** `Entity`'s canonical ctor coerces null→PRIMARY; the mirror carries the RESOLVED role name (never null) — map from the constructed domain record, not from raw inputs.
- **Checkpoint reset placement:** the reset-to-0 write must happen BEFORE `subscribeRuntime` (the bus reads the checkpoint at registration). Do not reset any OTHER subscriber's checkpoint.
- **The slice lock:** publishes stay OUTSIDE the slice lock (the existing F-8 discipline); the projection has its own lock — mind ordering, never nest the slice lock inside the projection lock or vice versa (document the ordering you choose).
- **Tests must inject `Clock`.** Do NOT use `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, or `System.currentTimeMillis()` in this WU's test code — Clock-injection is the standing convention (production code in any module IS caught by `NO_DIRECT_TIME_ACCESS`; treat test code the same). Use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` injected via constructors/`@BeforeEach`.
