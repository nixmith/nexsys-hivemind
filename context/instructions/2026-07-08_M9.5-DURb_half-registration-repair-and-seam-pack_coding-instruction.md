## What This Implements

M9.5-DUR made the registries projections of the log, but its emission contract is non-atomic: a process death between the durable `device_registered` and its entities' `entity_registered` publishes leaves a device that is registered but entity-less FOREVER — the announce path's LINKED arm finds the device by hardware id and re-links instead of re-proposing, no removal emitter exists, and replay faithfully rebuilds the hole. §1 closes it: the LINKED arm requires at least one registered entity, and adoption REUSES the existing device identity on a hardware-id match, so the repair is a log-clean idempotent re-emit (AMD-99 F1). §§2–6 clear the accumulated zigbee-seam debt in the same pass so the acceptance run and the 72 h soak certify tidied code.

## Settled Decisions

- **DP-B1 (repair criterion):** LINKED requires an existing device AND ≥1 registered entity: `existing.isPresent() && !entityRegistry.listEntitiesByDevice(id).isEmpty()`. The entity-less case logs ONE WARN — `zigbee.half_registration_detected: device={} deviceId={} — re-proposing for repair` — and falls THROUGH to the propose path (no early return; the normal proposal/adoption flow is the repair).
- **DP-B2 (identity continuity):** `adopt()` checks `findByHardwareIdentifier` BEFORE minting; present ⇒ REUSE that `deviceId` (the `device_registered` re-emit is then idempotent-by-identity per REG-INV-1 — same id, apply upserts). Entities mint fresh ids (none were ever durable — nothing to preserve). Scope note, recorded honestly: a PARTIAL multi-entity registration (some entities durable) still takes the LINKED arm — out of scope; Wave-1 devices are single-entity and the emission loop is milliseconds; the projection-side count makes it detectable later if it ever matters.
- **DP-B3 (§2):** `log.debug` → `log.info`, UNCONDITIONAL (count 0 prints too) — the token `zigbee.adoption_maps_rehydrated: devices={}` is FROZEN (the 5b/acceptance glance-point).
- **DP-B4 (§3):** javadoc-ONLY flips — every BENCH-VERIFY clause in the two files becomes a silicon-verified statement citing the bench record iteration that proved it (it-1 formation/resume + tc_joins · it-2 NCFG read-back + joins · it-4 RPT binding · it-5a KEY/TCLK + 0x0019/0x90 both-arms held across the arc · it-5b KEYb one-established-zero-failed). Constants BYTE-UNTOUCHED; zero behavior change; the isolated BENCH-VERIFY constant-block structure STAYS (future silicon corrections remain one-constant edits).
- **DP-B5 (§4):** `attemptReopen()` clears `permitJoinDeadline` (null) on a successful reopen — the reset NCP holds no window, so `isPermitJoinActive()` can no longer read stale-true (the M9.4-TCJ R2 recorded limitation, closed).
- **DP-B6 (§5):** the command fence consults the adopted entity's INSTALLED policy first — resolve the entity's `CapabilityInstance.confirmation()` from the registry (the tuning that now durably survives restarts); profile characterization remains the pre-adoption/no-entity fallback; `INHERENTLY_UNCONFIRMABLE` untouched. Behavioral contract (AMD-97-INV-01 semantics, the M9.4-RPT R2 row): an installed-DISABLED command still dispatches, is never ledger-tracked, and renders the immediate honest UNCONFIRMED verdict with recorded reason — never a silent bypass, never CONFIRMED.
- **DP-B7 (§6):** a census test in the app test tree (beside `RegistryMutationRuleTest`, same idiom) pinning production callers of `RegistryProjection.applyDeviceRegistered/applyEntityRegistered/applyDeviceRemoved` to EXACTLY {`ZigbeeAdoptionSlice`, `ZigbeeIntegrationAdapter`, `RegistryProjectionSubscriber`} — a new caller must arrive with a deliberate census edit (an unpublished apply() would be registry state not derived from the log).
