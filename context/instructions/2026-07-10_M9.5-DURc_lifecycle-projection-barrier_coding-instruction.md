# Coding Task: M9.5-DURc — Lifecycle IT Registry-Projection Checkpoint Barrier (test-only)

**Subsystem:** lifecycle ITs ONLY (`HeroLoopHardwareFreeIT.java`, `ZigbeeReplaySafetyIT.java`). **ZERO production diffs. ZERO other test diffs.**
**Phase:** 3-Implementation (test-infrastructure repair).

## The defect (mechanism, verified at source)

Since M9.5-DUR (`ef1d2c5`), every registry mutation flows through `RegistryProjection` (upsert-in-log-order; live self-delivery of an adoption-time `entity_registered` is a no-op ONLY when state is equal — "different state ⇒ replace" is the documented semantic). `ZigbeeAdoptionSlice.adopt()` applies synchronously on the ingestion thread AND publishes the registration events, which the `registry_projection` subscriber re-applies asynchronously on its own virtual thread. Both ITs then mutate the registry DIRECTLY from the test (`label()` → `core.entityRegistry().updateEntity(...)` — log-invisible; the ArchUnit fence covers production only) immediately after `adopt()` returns, with NO barrier awaiting the subscriber's consumption of the adoption events. On a slow CI runner the subscriber's self-delivery lands AFTER `label()` and lawfully replaces the labeled entity with its as-adopted (label-less) state. The label-targeted automation then resolves ZERO entities (LabelSelector resolves 0..n; RunInitiator returns silently on empty) → the awaited `command_issued`/frame never publishes → the count-bounded await helpers (500 polls × 20 ms) exhaust → `AssertionError` at the shared throw lines (HeroLoop :562 `awaitEnvelope`; ReplaySafety :257 `awaitTrue`). The frozen `TestClock` rules out every window-lapse alternative: a timeout means the event was never published, not late. M9.6-SEED (`a1f0f77`) is exonerated — no semantic interaction (no log assertions, fresh tempdirs, ZCL-only frame counts); the prior green CI on `04f5f70` was probabilistic.

## The fix (the repo's own pattern)

Insert a registry-projection checkpoint barrier BETWEEN the last `adopt()` and the FIRST `label()` in each IT — the `ReplayTransitionIT` await-checkpoint pattern (await a subscriber's durable consumption position before proceeding):

- **DP-1:** in each IT, add a private helper `awaitRegistryProjectionCaughtUp()`: capture the event store's current head position AFTER the last `adopt()` returns, then `awaitTrue(() -> <the registry_projection subscriber's checkpoint> >= head, "the registry projection consuming the adoption events")`. Resolve the exact seam from what the ITs already use: `core.eventBus().subscribers()` (the `awaitRuntimeSubscribersLive()` idiom) exposes `SubscriberSnapshot` with `checkpoint()` = the last delivered global position; the head position comes from the same event-store read the ITs already do (`core.eventStore().readFrom(...)` — use the max position/global ordinal the envelope surface exposes; if the store surface offers a cheaper head accessor, prefer it, state the choice).
- **DP-2:** call sites: `HeroLoopHardwareFreeIT.bootAndAdopt()` — after the `hueEntity` adopt, before `label(snzbEntity, "motion")`. `ZigbeeReplaySafetyIT` — after its `rig.adopt(...)`, before its `label(...)`.
- **DP-3:** reuse each IT's existing `awaitTrue` helper — no new await machinery, no new timeouts elsewhere, no changes to any assertion.
- **DP-4 (scope pins):** zero `src/main` diffs · zero log-token changes · zero changes to the other two lifecycle ITs (`ProcessRestartIdentityIT`, `RestartHonestyIT` — they don't label; confirm by grep and STATE the grep) · zero build-file changes.
