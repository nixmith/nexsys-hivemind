## What This Implements

`StandardAvailabilityTracker` is compiled, tested, and wired to nothing — no constructor call, no evidence feed, no publish listener (the #6 mechanism-without-driver, caught by chartered read). Entities therefore sit at `availability=UNKNOWN` forever, even with reports flowing. The v27 beat-5 read also solved the boot mystery: the existing relink emission publishes `availability_changed` at DEVICE grain and the projection applies ONLY ENTITY-grain availability (`subjectEntityIdOrNull` gates it) — a structural no-op. This WU: construct the tracker at adapter `initialize()`, feed it device-originated RX evidence from the documented seam (`onFrame`), evaluate absence each cycle with the mains read-ping arm, and publish per-ENTITY `availability_changed` through a new `TransitionListener` so the view actually moves. Honesty in both directions: ALIVE only on genuine device RX; NOT-ALIVE by battery-silence timeout or mains ping-timeout — never a false ALIVE from our own TX, never a permanent stale ALIVE on a dead mains device.

## Settled Decisions (NOT open questions)

- **DP-1 (construction):** `initialize()` constructs `StandardAvailabilityTracker(clock, powerSourceLookup, persistedAvailability, listener)`. **`persistedAvailability = Map.of()` — EMPTY, deliberately** (see the first watch-out trap: sidecar-init + edge-triggering + a view that boots UNKNOWN = the first post-boot event suppressed and the view starves). The tracker and the view thus agree at boot (both UNKNOWN) and the first RX frame emits the edge the view is waiting for — which is also the ruled FE doctrine ("availability UNKNOWN after a restart is an HONEST state"). **Sidecar snapshot-init + the boot-time view reconciliation is a NAMED deferred follow-up (one coherent WU, never split) — do not implement any of it here.**
- **DP-2 (powerSourceLookup):** `ieee -> cache.device(ieee).map(ZigbeeDeviceRecord::powerSource).orElse(0)`. No blanket fallback needed — uninterviewed records already carry 0, and N-5 gives every non-mains-proven device the conservative 25 h window.
- **DP-3 (RX evidence):** `AdapterIngestionListener.onFrame` adds `tracker.recordFrame(device, clock.instant())` beside the existing cache/interview-queue calls. That seam's own javadoc names it the availability feed; its pre-dedup placement is CORRECT (a duplicate frame is still the device talking). This is the ONLY ALIVE-evidence input.
- **DP-4 (absence + the ping arm — LOAD-BEARING):** `runCycleOnce()` calls `tracker.evaluateTimeouts()` each cycle (honoring its own javadoc). The battery 25 h arm works as-built. For each returned mains ping candidate: issue ONE ZCL read of Basic via the existing exchange seams (the `readBasic`/`zclGlobalExchange` idiom, on the run thread like every other run-cycle exchange) and feed `recordCommandResult(device, responded, clock.instant())`. A responding ping ALSO arrives as RX (onFrame fires — harmless double evidence); the ping's REAL job is the timeout: `responded=false` → `PING_TIMEOUT` → the honest offline. Without this arm a dead mains device stays ALIVE forever.
- **DP-5 (the publish shape — the view-moving contract, from the beat-5 grounding):** the `TransitionListener` publishes **one root event PER ADOPTED ENTITY of the transitioning device** (resolve via the adoption slice's entity map; a device with no adopted entities publishes nothing): `EventDraft(EventTypes.AVAILABILITY_CHANGED, 1, clock.instant(), SubjectRef.entity(entityId), priority, EventOrigin.INTEGRATION, new AvailabilityChangedEvent(prev, next), null, null)` with the relink precedent's publishRoot + swallow-`SequenceConflictException` idiom. **`next` uses the CANONICAL vocab `"online"`/`"offline"`** (the record's documented triple — not relink's `"available"`); `prev` = the tracker's prior state string or `"unknown"`. **Priority: `CRITICAL` for →offline, `NORMAL` for →online** (the payload record's own doctrine). The listener ALSO writes `cache.setAvailability(ieee, available)` on every transition — the sidecar goes permanently stale otherwise (nothing else writes it), and the deferred snapshot-init WU depends on it being current.
- **DP-6 (the relink emission):** UNTOUCHED. It is a structural projection no-op (device grain) — recorded as a known oddity; cleaning it belongs to the deferred reconciliation WU, not here.
- **DP-7 (recordCommandResult from DISPATCH: OMITTED — ruled by grounding):** `ZclDispatch.send`'s boolean is NCP-acceptance only ("true = the NCP accepted it"); no device-originated confirmation exists on the dispatch path. Wiring it as success evidence would fabricate liveness into the void. The ONLY `recordCommandResult` caller is DP-4's ping arm, whose response is genuinely device-originated.
- **DP-8 (tokens):** the tracker's existing transition INFO (`zigbee.availability_changed: device={} available={}`) goes LIVE with the wiring — declare it FROZEN from this WU forward (the runbook/scenario vocabulary gains it). No other new tokens.

## What to Watch Out For

- **(Nick's ruled trap #1, verbatim obligation):** *"the M-1 sidecar-init/edge-trigger suppression trap"* — the tracker's ctor inits sidecar-known devices SILENTLY (M-1) and `transition()` is edge-triggered, so a sidecar-initialized AVAILABLE + a view booting UNKNOWN means the first post-boot frame emits NOTHING and the view starves. Hence DP-1's EMPTY map. Do not "helpfully" pass the snapshot.
- **(Nick's ruled trap #2, verbatim obligation):** *"the load-bearing mains read-ping arm — never-false-ALIVE in the other direction"* — `evaluateTimeouts` only NOMINATES mains candidates; without the ping executor a dead mains device stays ALIVE forever. The arm is in-scope, not gold-plating.
- **Tests must inject `Clock`.** No `Clock.systemUTC()`/`Instant.now()`/`System.nanoTime()`/`System.currentTimeMillis()` in this module's test code — `Clock.fixed(...)`/mutable fake injected. The timeout legs live on the fake clock.
- The name-collision hazard from the grounding: `cache.recordFrame` ≠ `tracker.recordFrame` — both get called in onFrame; don't let a refactor conflate them.
- ReentrantLock only (the tracker already complies); the ping exchange rides the run thread's existing lock discipline — never a new thread.
- Publish-conflict handling: the relink precedent swallows `SequenceConflictException` with a log — match it.
- The dead-expression smell in `transition()` (`changed` computed then overwritten) may be tidied ONLY if byte-behavior is proven identical by the existing tracker tests — otherwise leave it.
