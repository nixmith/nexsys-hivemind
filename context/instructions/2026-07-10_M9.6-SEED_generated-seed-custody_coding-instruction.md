## What This Implements

Today every network formation writes the industry-default well-known Trust Center link key ("ZigBeeAlliance09") into the NCP's initial security state — under the ruled hashed-TCLK posture (0x1B84), per-device TCLKs are therefore derivable from a PUBLIC root: join semantics are exercised but no key secrecy is added (the v20 beat-2 R1 record). This WU mints a 16-byte cryptographically random TCLK seed at first formation, holds it in the existing zigbee custody (SecretStore, AES-256-GCM at rest), and passes it — instead of the well-known root — into the formation security struct. The well-known key KEEPS exactly one role: the per-window transient joiner-bootstrap key (the standard Z3.0 mechanism; bellows behaves identically). Migration is ruled: **the seed applies to NEW formations only** — the bench's existing well-known-root network is the sanctioned SD-5 exception and resumes untouched; the restore branch reproduces the AS-FORMED posture. After this lands, the SD-5 HARD GATE ("no non-bench network until seed-custody lands") is dischargeable.

## Settled Decisions (NOT open questions — implement as stated)

- **DP-1 (mint):** in `NetworkFormation.form(...)`, beside the existing network-key mint (`:202-208`), resolve the seed load-or-mint (16 bytes from the injected `SecureRandom`). Idempotent: an existing seed is reused, never re-minted. **The ordering doctrine extends verbatim: all newly-minted secrets are DURABLE before `ops.formNetwork` is called; params save only after formation succeeds.**
- **DP-2 (storage + marker):** seed material lives in the zigbee-data-dir SecretStore under a new fixed ref (constant `TCLK_SEED_REF = "zigbee.tclk_seed"`, the `NETWORK_KEY_REF` idiom), hex-encoded like the network key. `zigbee-network.json` is UNCHANGED — **the seed secret's PRESENCE is the posture marker**: present ⇒ this custody's network was formed with the generated seed; absent ⇒ formed on the well-known root (the pre-WU bench exactly). No schema/version field this WU (the FRAME-CTR row owns the future schema bump).
- **DP-3 (store seam):** widen `NetworkParameterStore` with seed-named methods (`saveTclkSeed(byte[])` / `loadTclkSeed()` — fixed-ref, so no String param needed; keep the interface package-private as-is). Do NOT overload the `saveNetworkKey` name (its javadoc contract says "the 16-byte AES-128 network key" — a semantic lie invites the next reader's bug). INV-SE-03's verbatim clauses apply to the seed identically — restate them in the new methods' javadoc.
- **DP-4 (atomic mint):** when BOTH the network key and the seed are minted in the same first-run, persist them via ONE `SecretStore.setAll` (the AMD-68 never-torn write). When only one is missing (asymmetric custody — possible after operator surgery), a single `set` for the missing one is correct. Restructure `form()`'s mint block accordingly (resolve both → batch-persist the newly-minted → then form).
- **DP-5 (the EZSP write):** `CoordinatorOps.formNetwork` widens to carry the TC link key: `formNetwork(int channel, int panId, long extendedPanId, byte[] networkKey, byte[] trustCenterLinkKey)`. The protocol layer becomes POSTURE-AGNOSTIC — it writes whatever TC key it is handed into `encodeInitialSecurityState` `struct[2..17]`; the posture POLICY lives in `NetworkFormation`. Make `TC_LINK_KEY` package-private (same file) so `NetworkFormation` can pass it on the well-known arms; `encodeInitialSecurityState` gains the parameter. The bitmask 0x1B84, the network-key bytes at `[18..33]`, and everything else in the struct: BYTE-UNTOUCHED.
- **DP-6 (formation arm):** fresh formation passes the seed (minted per DP-1). This is the only arm that MINTS.
- **DP-7 (restore arm):** `resume()`'s restore branch loads the seed; present ⇒ pass it; absent ⇒ pass the well-known constant — **AS-FORMED reproduction**, never a mint on restore. The NVRAM-resume arm is UNTOUCHED (it writes no security state — regression-pin that). **Recorded scope limit (state in javadoc + completion report, do NOT fix):** with presence-as-marker, a corrupted custody that lost ONLY the seed secret is indistinguishable from legitimate well-known custody — restore would honestly reproduce the wrong root and TCLK mismatches would surface at the devices' key-request legs. Bounded, named, closed later by FRAME-CTR's explicit schema.
- **DP-8 (the transient-key PIN):** `enablePreconfiguredKeyJoins()` is BYTE-UNTOUCHED — the wildcard transient key STAYS ZigBeeAlliance09 (the Z3.0 joiner bootstrap; the bellows reference posture is generated-seed + this same mechanism). **The seed NEVER rides `importTransientKey`. Pin with a test.**
- **DP-9 (policy family):** zero constant changes. `DECISION_ALLOW_TC_KEY_REQUESTS = 0x51` send-current under the generated seed is the intended reference behavior (`:200-202`) — note it in javadoc, change nothing.
- **DP-10 (observability, anti-vacuous):** ONE new INFO token, fired on the formation AND restore arms: `zigbee.tc_link_key_posture: posture={generated_seed|well_known} context={formation|restore}`. Never any key/seed material in any log (INV-SE-03). No other new tokens; all existing tokens byte-frozen.
- **DP-11 (migration):** NO live re-key path, NO NVRAM surgery, NO prompt to re-form. The bench keeps its network; operator-initiated re-formation (custody delete + re-pair) is the post-soak recipe and is OUT of this WU's code.
