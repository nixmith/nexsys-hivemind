## What This Implements

M3.7 is the final M3 milestone. It exists to prove that the seventeen-WU M3.6 composition-root cohort actually produces a system that an HTTP client can use. Through M3.6e.2 the codebase has all the pieces — `PersistenceFactory`, `InProcessEventBus`, `StateProjection` (with `MaterializedStateQueryService` query surface), Javalin server on port 7070, five package-private endpoint handlers, two operational `/internal/*` admin endpoints, and a 16-step `HomeSynapseCore.start()` bootstrap — but no existing test exercises them end-to-end with a real HTTP client. M3.4a/M3.4b cover persistence + bus + projection but stop short of HTTP. M3.6e.2 added HTTP endpoint handlers but tested them only through the package-private `EndpointContext` SPI with a `RecordingEndpointContext` stub.

This WU delivers the **M3.7 capstone E2E harness** (`HomeSynapseE2eHarness` per Research 3 REC-16) — wires the full production composition root against a `@TempDir` SQLite file and a Javalin server bound to an ephemeral port — and ~5–10 end-to-end test scenarios that publish events through `EventPublisher`, await `StateProjection` reaching `LIVE` via `LiveModeAwaiter` (REC-13), then hit the five HTTP endpoints with `java.net.http.HttpClient` and assert response shapes with `json-unit-assertj` (REC-17). The WU also closes the two OR-M3-17/OR-M3-18 placeholders in `HomeSynapseCore` step 5 by introducing `MinimalProjectionAdvancer` (Research 3 REC-20), absorbs the M3.6e.2 DLQ response-shape deviation (D-2) by plumbing `oldestParkedAt` through `SubscriberDlq` → `SubscriberSnapshot` → `DlqStatusEndpoint`, and adds `awaitility` (REC-14) and `json-unit-assertj` (REC-17) as pinned dependencies in `gradle/libs.versions.toml`.

After M3.7 commits, M3 is complete and M4 planning begins. The `HomeSynapseE2eHarness` becomes the canonical place to add future E2E coverage as M4+ subsystems land.

## What to Watch Out For

This section is where the M3.6 lessons accumulate. Read it carefully before writing the first line of code.

**Arch-rule reminder (D-04 — `NO_DIRECT_TIME_ACCESS`).** This rule applies to test code in non-whitelisted packages — `com.homesynapse.it` is NOT on the whitelist (only `com.homesynapse.{app,platform,test}..` are whitelisted). Tests must inject `Clock`; use `Clock.fixed(Instant.parse("2026-01-01T00:00:00Z"), ZoneOffset.UTC)` for deterministic tests. The existing `BurstLoadIT`/`HeapBudgetIT` in M3.4a use `Clock.systemUTC()` because `IntegrationTestHarness.start(Path, Clock)` accepts whatever you pass — but the rule still applies to any local time access (`Instant.now()`, `System.nanoTime()`, `System.currentTimeMillis()`, `Clock.systemUTC()`). Use the injected clock everywhere. Pattern reference: `coder-lessons.md` 2026-04-10 entry on `NO_DIRECT_TIME_ACCESS`.

**Xlint:exports gateway pattern (DEC-M3-16, M3.6e.1 lesson).** Any new **public** type in `rest-api` or `lifecycle` that references `io.javalin` types in its signature will fail with `-Xlint:exports -Werror`. The M3.6e.1 fix pattern: keep the implementation package-private and create a `RestFilters`-style public utility with `Object`-typed parameters. M3.7 does NOT add new public types to `rest-api` — the `DlqStatusEndpoint` change is internal — so this should not trip. But the new `HomeSynapseCore.boundHttpPort()` method is fine because `int` is a JDK type, not Javalin. Pattern reference: `coder-lessons.md` 2026-05-22 entry on M3.6e.1 XLINT_EXPORTS_FIX.

**Gradle/JPMS scope alignment (OR-M3-16, M3.6e.1 lesson).** `requires transitive` in module-info ↔ `api(project(...))` in build.gradle.kts; plain `requires` ↔ `implementation(project(...))`. M3.7 doesn't add any new module-info edges to production code (lifecycle already has all the transitive dependencies it needs for `MinimalProjectionAdvancer`), so this should not trip. The new test-only deps (awaitility, json-unit-assertj) are `testImplementation` scope and never appear in any `module-info.java`.

**Port 0 vs port 7070 (Research 3 REC-15, M3.7-critical).** Production code currently hardcodes port 7070 in `HomeSynapseCore`. M3.7 requires ephemeral ports so parallel test execution does not conflict and CI does not have to manage a fixed port. The change: `app.start(config.httpPort())` accepts the configured value; `port 0` tells Jetty to bind to any free port; after `start()` returns, `app.port()` reveals the bound port. `boundHttpPort()` MUST throw `IllegalStateException` before `start()` is called (the field `javalinApp` is null until step 12). Real Jetty's `port()` method returns the actual bound port on a started server. Verify by reading Javalin 6.7.0's API doc if unsure.

**`MinimalProjectionAdvancer` vs `DispatchingProjectionAdvancer` (Decision 1).** M3.7 uses `MinimalProjectionAdvancer` (Research 3 REC-20). The M4.0 `DispatchingProjectionAdvancer` (Research 8 REC-28) is the eventual replacement that dispatches by event-type via the `@EventType` registry. **Do NOT preemptively implement the dispatching variant** — it requires the M4-amendment-deliberation window to close, which is decoupled from M3.7 (Decision 3).

**Idempotency of test setup.** `HomeSynapseE2eHarness` must be safely re-startable across tests — each test gets a fresh `@TempDir` SQLite file, fresh port (because port 0), fresh `HomeSynapseCore` instance. Do NOT use static state in the harness. Do NOT share a `HomeSynapseCore` instance across tests (the bootstrap is expensive, but tests must be isolated).

**`@TempDir(cleanup = CleanupMode.NEVER)` on Windows (M3.4b lesson).** Crash-recovery tests that abandon a `HomeSynapseCore` instance leave SQLite file handles held on Windows. Use `CleanupMode.NEVER` to avoid test-cleanup failures; the OS reclaims temp dirs eventually. This applies to `CrashRecoveryHttpIT` specifically; other ITs can use the default `ON_SUCCESS` since they call `stop()` properly.

**No `synchronized` (LTD-11).** Use `ReentrantLock` if any test-side synchronization is needed. Virtual threads pin on `synchronized` blocks. `Awaitility`'s implementation may use `synchronized` internally — that's a library concern outside the rule's scope, but DO NOT add `synchronized` blocks to your own code.

**Awaitility default 30s timeout is too long for unit-level tests (REC-13).** Use explicit `.atMost(Duration.ofSeconds(5))`. The default `LiveModeAwaiter` overload should use 5 seconds. Override only for long-running scenarios (none expected in M3.7).

**HTTP client choice.** Use `java.net.http.HttpClient` from `java.net.http` (standard library since Java 11). Do NOT pull in Apache HttpClient, OkHttp, or any other HTTP client library — adds a dependency for zero benefit. `java.net.http.HttpClient` is in `java.net.http` which is already on the test classpath via the JDK.

**JSON assertion library.** Use `json-unit-assertj` (REC-17). Pattern: `assertThatJson(httpResponse.body()).isEqualTo("{\"data\":[]}")` or `assertThatJson(body).inPath("$.data[0].entityId").asString().matches(ULID_PATTERN)`. Do NOT use raw `String.equals` for JSON — whitespace and key order differences will cause spurious failures. Do NOT use Jackson for assertion path-traversal — `json-unit-assertj` is purpose-built for this.

**Sequence-key collision in DLQ park (D-2 absorption).** `SubscriberDlq.park(DeadLetter)` uses `DeadLetter.sequenceKey` as the dedup key in `PersistentDlqWriter` (UNIQUE constraint on `(subscriber_id, event_position)`). The `parkedAt` extension does NOT change this — it only adds a timestamp field. Make sure the UNIQUE constraint still holds after the change (no schema change required; `parkedAt` is in-memory-only on the ring side, and the persistent side already has `first_seen_at`/`last_attempt_at` columns from V002).

**`oldestParkedAt` semantics on a ring with eviction.** `SubscriberDlq.CAPACITY = 1024`. When full, the oldest entry is evicted via `ring.pollFirst()`. After eviction, the NEW oldest is the next-oldest entry. `oldestParkedAt()` correctly returns the head's `parkedAt` because the head IS the oldest. No special handling needed for eviction. The DLQ does NOT preserve the all-time-oldest parked timestamp — only the oldest still-in-ring entry.

**Test class location — `com.homesynapse.it` package.** Per the existing M3.4a/M3.4b convention, all test files live in `testing/integration-tests/src/test/java/com/homesynapse/it/`. This package is NOT whitelisted by the arch rules — D-04 applies. The package matches `IntegrationTestHarness` and the existing 5 IT classes.

**No `module-info.java` in `testing/integration-tests`.** This is a classpath-only test module. All access to package-private types in production modules goes through testFixtures (the `PersistenceTestHarness` / `InProcessEventBusFactory` pattern from M3.4a). For M3.7 the new test code only consumes public types from lifecycle (`HomeSynapseCore`, `HomeSynapseConfig`) and event-bus (`EventBus`, `SubscriberSnapshot`) and state-store (`SubscriberMode`) — no testFixture access needed.

**The existing `IntegrationTestHarness` STAYS.** M3.7's `HomeSynapseE2eHarness` is a parallel harness for HTTP-aware E2E scenarios. The M3.4a/M3.4b harness remains the right tool for performance/load/heap-budget tests (which do not exercise HTTP). Do NOT delete or merge them — they have different scopes.

**`StateProjection.create(...)` is the load-bearing 11-param call.** When you replace `NO_OP_DERIVATION` and `NO_OP_ADVANCER` at lines 264 and 266 of `HomeSynapseCore.java`, the new arguments must match the param positions exactly. Verify the call signature has not drifted from the brief's assumption. The `derivationRule` parameter is at the position the brief documents; the `advancer` parameter is at the position the brief documents. If StateProjection's API has changed since `76288af` (e.g., parameter reordering), STOP and report.
