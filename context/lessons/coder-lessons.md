## 2026-06-05 | Category: other | Source: M4.B-S2 (AMD-44 Stage 2) — adding a record component + a back-compat convenience ctor of the OLD arity is the zero-breakage way to evolve a widely-constructed record
**Discovery:** When a record gains a component (`Entity` 11→12, `ProposedEntity` 3→4) but is constructed at many sites you must not (or cannot) touch in this WU, add a **convenience constructor whose parameter list is the pre-change canonical signature**, delegating to the new canonical ctor with the new component defaulted. Every existing `new Entity(... 11 args ...)` call silently re-binds to the convenience ctor — `EntityTest`, `TestEntityFactory`, and the cross-module `StubIntegrationContext` (integration-api testFixtures) all compiled unchanged, no edit needed. There is **no overload ambiguity** because the two ctors differ in arity (11 vs 12 / 3 vs 4); ambiguity would only arise if the new component's type matched a trailing param's type AND the arities collided, which they don't. The one shape test that pins component count (`EntityTest.exactlyElevenFields` → `exactlyTwelveFields`) is the only existing test that MUST change — grep `getRecordComponents()` repo-wide first (here only `EntityTest:82` hit for these two records; `ProposedEntity` had no shape test, so a new `ProposedEntityTest` was created).
**Detail:** Guard order in the compact ctor is **coerce → validate → copy**: `if (role == null) role = DEFAULT;` BEFORE `if (!type.allows(role)) throw IAE;` BEFORE `field = List.copyOf(field);`. This ordering is load-bearing — a `null` role on a type that permits only the default (PLUG → {PRIMARY}) must coerce and pass, not NPE inside `allows(...)`. And because the default (PRIMARY) is permitted by EVERY enum constant, the back-compat ctor (which passes the default explicitly) can never construct an illegal pair, so the new construction-time guard adds ZERO breakage to the existing corpus. Deliberately NOT adding a guard for a field that was never guarded before (here `entityType`) keeps the diff minimal (it NPEs via the `allows(...)` call — documented in a code comment, not "fixed"). The compile-time legality matrix is encoded on the partner enum: each `EntityType` constant declares `EnumSet.of(...)` through an `EntityType(Set<EntityRole>)` ctor that stores `Set.copyOf(...)` — `EnumSet.of` is MUTABLE, so storing it raw would make `legalRoles()` mutable and fail the immutability test; `Set.copyOf` is mandatory. Adding a new enum constant then forces the author to declare its legal-role set (the compiler requires the ctor arg) — the constraint can't be forgotten. Note the AMD's illustrative enum-constant ORDER differed from the source order; the matrix CONTENT (the role sets) is authoritative, the ORDER stays as-is in source (pinned by a `containsExactly` test) — take data from a spec, not incidental ordering.
**Impact:** (1) To add a component to a record with many construction sites, ship a convenience ctor of the OLD arity defaulting the new field — existing callers (even cross-module testFixtures you're told not to edit) keep compiling; only the component-count shape test changes. (2) Grep `getRecordComponents()` repo-wide before editing — it's the one assertion that breaks the instant a component lands (Interface Evolution Checklist). (3) Compact-ctor order is coerce → validate → copy; a null that coerces to a universally-legal default must pass the validate step, which is why "every type permits the default" is the property that makes the change zero-breakage. (4) Encode a compile-time legality matrix on the enum via a ctor arg (forces declaration on every new constant) and store it `Set.copyOf(EnumSet.of(...))` — never retain the mutable `EnumSet`. (5) When a spec's code block lists constants in a different order than the source, the content governs, not the ordering — reordering source constants is a contract change.

## 2026-06-05 | Category: build | Source: M4.C (AMD-54..64) — the actual Spotless config enforces only header + removeUnusedImports + trailing-ws + eof, NOT import-order or indentation
**Discovery:** `build-logic/src/main/kotlin/homesynapse.java-conventions.gradle.kts` declares `spotless { java { licenseHeader(...); removeUnusedImports(); trimTrailingWhitespace(); endWithNewline() } }` — and nothing else. There is NO `googleJavaFormat()`/`palantirJavaFormat()`/`importOrder()`. So Spotless does NOT reformat indentation and does NOT sort imports — but `removeUnusedImports()` fails `check` on ANY unused import.
**Detail:** Two files in the touched set (`EventTypes.java`, `IntegrationEvents.java`) are TAB-indented while the rest of the codebase is 4-space; `TestEventSamples.java` has `com.homesynapse.value.StringValue` interleaved mid-`com.homesynapse.event.*` (not alphabetical). Both pass `check` — direct proof Spotless isn't enforcing format/import-order. Consequences for an Edit-tool WU: (a) when editing a tab-indented file you must match tabs byte-for-byte — a one-shot `^\t` grep (count tab-led lines vs space-led) settles tabs-vs-spaces cheaply before the first Edit; (b) added imports can go in any position, but every one must be USED, and Javadoc `{@link Type}` counts as usage (google-java-format's `RemoveUnusedImports` respects Javadoc refs — that's why `IntegrationEvents` keeps `import …EventType` whose only use is a `{@link}`); (c) the copyright header must be byte-exact (`/*\n * HomeSynapse Core\n * Copyright (c) $YEAR NexSys. All rights reserved.\n */`).
**Impact:** Don't spend effort hand-sorting imports for Spotless in this repo — it doesn't sort. DO audit every added import for actual use (incl. Javadoc-only `{@link}`) before declaring done, and DO preserve the file's existing indentation char (tabs in `EventTypes`/`IntegrationEvents`, spaces elsewhere). Read the spotless block in the convention plugin once for an unfamiliar repo rather than assuming google-java-format — the enforcement set changes what "format-clean" means.

## 2026-06-05 | Category: serialization | Source: M4.C (AMD-59) — a persisted record that embeds an unregistered sealed-type field degrades SILENTLY; prove it from source, escape via a command-less instance
**Discovery:** A persisted event payload round-trips through `EventPayloadCodec` only if every sealed/abstract-typed field in its subtree has a (de)serializer in `PersistenceJacksonModule`. Jackson cannot construct a sealed-interface field with no codec — `decode()` catches the failure and returns a `DegradedEvent` (a silent lossy round-trip, NOT a thrown error). `CapabilityAdded` carries a full `CapabilityInstance` whose `commands → CommandDefinition → ExpectedOutcome.expectation` field is the sealed `Expectation` (device-model), which has no codec (only the 10 ULID wrappers + `AttributeValue` are registered).
**Detail:** Verified the defect WITHOUT running the build (Coder produces files only): (1) read `PersistenceJacksonModule` — `addSerializer/addDeserializer` cover ULIDs + `AttributeValue` only; (2) confirmed `ExpectedOutcome.expectation` is typed as the sealed `Expectation` (`permits ExactMatch, WithinTolerance, EnumTransition, AnyChange`); (3) ruled out a false alarm by grepping event-model for any existing event embedding `Capability`/`Expectation`/`CommandDefinition` — none (`DeviceAdoptedEvent` carries a single `Ulid`), so the codec has never been exercised on this subtree. The instruction flagged this exact case as STOP-and-report: do NOT add a Jackson annotation to the domain type (fails `NO_JACKSON_IN_DOMAIN_MODEL`) and do NOT silently write a custom serializer (a PM design call). The escape that keeps a milestone contract-only AND green: a **command-LESS** capability (`StandardCapabilities.occupancy()` — empty commands map) round-trips losslessly (its `AttributeSchema`/`ConfirmationPolicy` subtree is plain-Jackson; the abstract `Number` min/max fields are null and omitted by `NON_NULL`), so it backs the green codec test; the command-bearing case (`onOff()`) becomes a `@Disabled("AMD-65 pending …")` acceptance test naming the follow-up amendment.
**Impact:** (1) Before asserting a new persisted event round-trips, trace its payload for ANY sealed/abstract-typed field and check `PersistenceJacksonModule` registers it — an unregistered one degrades silently to `DegradedEvent`, so a naive `isEqualTo` assertion fails with a confusing "expected X but was DegradedEvent". (2) A serde defect is provable from source alone: registered-codec list + field types + "has any existing event ever serialized this subtree?" (a clean prior-usage grep rules out a false alarm). (3) When STOP-and-report fires on serde, the contract-only escape is a minimal/command-less instance for the green test + a `@Disabled` acceptance test naming the follow-up — never an annotation, never a silent custom serializer.

## 2026-06-05 | Category: build | Source: M4.C gate fix — adding event-type string constants OR event-record classes ripples to SEVEN downstream artifacts that the producer-only Files-to-Modify list omitted
**Discovery:** Adding 7 constants to `EventTypes` (46→53) and 7 records to the `IntegrationEvents` manifests (lifecycle 5→10 + a new `CAPABILITY_EVENT_CLASSES`, so `ALL_EVENTS` 27→34) compiled cleanly but failed `./gradlew check` at `core:event-model` — and because event-model is upstream of everything, the build aborted there and the integration/persistence suites never ran (their failures were hidden behind the first one). 2. `AllEventClasses.ALL_EVENTS` (persistence test) — must concat the new manifest(s).
3. `EventCategoryMapping.TABLE` (persistence MAIN, not test) — a per-type category map sized to `ALL_EVENTS`; every new registered event needs an entry, or `explicitMappingCount()` mismatches. This is a real MAIN-source gap, not a count: the AMDs ratify event types but NOT their `EventCategory`, so the Coder picks sibling-consistent categories and flags `[REVIEW]` (categories are a derived runtime lookup, freely amendable).
4. `EventCategoryMappingTest` count (27→34).
5. `EventTypeRegistryTest` count (27→34).
6. `JacksonWarmupTest` count (27→34).
7. The two production AGGREGATION sites that build the real `EventTypeRegistry` from the per-module manifests: `HomeSynapseCore` (lifecycle, the composition root) and `IntegrationTestHarness` (integration-tests) — both 2-arg `Stream.concat(CORE, LIFECYCLE)`; a new manifest list (`CAPABILITY_EVENT_CLASSES`) must be added or the new events are unregistered in PRODUCTION (encode throws "unregistered event class"). AMD-59 §3 mandated this; nothing pinned its size, so it was a silent functional gap, not a test failure.
Plus the annotation tests (`*EventTypeAnnotationTest.EXPECTED_SUBTYPES`) which were in the instruction.
**Detail:** The discovery survey must, in the same pass that greps `getDeclaredMethods()` for interface-shape pins, ALSO grep the event-type set: `grep -rnE "isEqualTo\((N|M)\)|hasSize\((N|M)\)"` for the old counts, AND `grep -rn "LIFECYCLE_EVENT_CLASSES|CORE_PRODUCTION_EVENT_CLASSES|<new manifest>"` to find every aggregation site (test harnesses AND the production composition root). The category map is the easy-miss MAIN-source one — it's keyed by event-type string and sized to the registered set, so it's coupled even though it lives in a different module than the constants. An upstream-module failure ABORTS the whole gate (no `--continue`), so a single missed pin in `core:event-model` masks every downstream failure — fix the upstream-most one first, then re-run with `--continue`.

(3) The production composition root (`HomeSynapseCore`) and any test harness that builds a registry from the manifests are part of the lockstep — a new manifest list (not just more entries in an existing one) must be wired into every aggregation site or the events are silently unregistered at runtime. (4) `EventCategoryMapping.TABLE` (persistence main) is the easy-miss coupling — it's a per-registered-type map in a different module from the constants.

## 2026-06-06 | Category: module-boundaries | Source: M5-A SystemdHealthReporter — pure JDK 21 has no AF_UNIX datagram socket; isolate the unsupported transport behind a seam to keep the gate GREEN
**Discovery:** sd_notify uses an `AF_UNIX` `SOCK_DGRAM` socket, but OpenJDK ≤ 21 supports Unix-domain *stream* sockets only (JEP 380 delivered `SocketChannel`/`ServerSocketChannel`, never `DatagramChannel`). `DatagramChannel.open(StandardProtocolFamily.UNIX)` COMPILES but throws `UnsupportedOperationException` at runtime — and there is no pure-JDK way to bind such a socket for *receiving* either, so neither a working production transport nor a faithful real-socket unit test is possible without a native binding (JNR/JNA, not in the version catalog) or a `systemd-notify` subprocess.
**Detail:** The reporter's message protocol (`READY=1`/`WATCHDOG=1`/`STOPPING=1`/`STATUS=…`), READY-once (`AtomicBoolean` CAS), `ReentrantLock` thread-safety, and send-and-forget error handling are all fully testable; only the datagram send is not. Pattern: extract a package-private `NotifyTransport` seam (`void send(byte[])`), put the `DatagramChannel`-based `UnixDatagramTransport` behind it (correct API shape, runtime-UOE on stock JDK), and inject a *capturing* transport in tests. `./gradlew check` (compile + unit tests) stays GREEN because the real transport is never constructed in tests — the runtime gap is isolated to the composition root (M13), which is the only place that actually instantiates the reporter (with a real `$NOTIFY_SOCKET`). This is the correct STOP-and-report move: do NOT add a non-catalog dependency or a subprocess silently — flag it as a TECHNICAL PUSHBACK and let the PM choose the transport.
**Impact:** Any pure-JVM sd_notify / AF_UNIX-datagram work needs an explicit transport decision (native binding vs `systemd-notify` subprocess) BEFORE it can run on a real host — treat it as a design question, not an inline IO choice. The seam-isolation pattern (correct-shaped-but-runtime-unsupported prod impl behind a package-private interface + a capturing test double) keeps a build-deferred WU GREEN while honestly surfacing a runtime gap; reuse it whenever a JDK API compiles but isn't supported at runtime on the target platform. Because the failure is runtime-only, a "looks-done" impl can hide a non-functional deployment path — say so loudly in the handoff + a cross-agent note.

## 2026-06-06 | Category: build | Source: M5-A path-impl tests — `(Type) null` cast is redundant (and -Werror-fatal) unless it disambiguates overloads
**Discovery:** Under `-Xlint:all -Werror`, casting `null` to select a constructor/method only avoids a `[cast]` warning when there is a genuine overload ambiguity. `new LinuxSystemPaths((Path) null)` has just one single-arg ctor → the cast is redundant → `-Xlint:cast` flags it → `-Werror` fails the build. But `new SystemdHealthReporter((String) null)` has two reference-type ctors (`String` and the package-private `NotifyTransport`) → bare `null` is ambiguous → the cast is REQUIRED and is NOT flagged.
**Detail:** Decide the cast per overload-set: drop it for a single applicable overload (`new X(null)`); keep it only where two reference-typed overloads make a bare `null` ambiguous. The AssertJ `assertThatThrownBy(() -> new X(null))` form still type-checks even when the ctor declares `throws` (the checked exception is absorbed by `ThrowingCallable.call() throws Throwable`).
**Impact:** When writing null-rejection tests in this `-Werror` codebase, choose the cast by overload-set, not by habit — an unnecessary `(Type) null` is a build failure, not a style nit. This sits alongside the M2.9 `-Xlint:exports` and M4.C spotless lessons as a "the gate enforces more than it looks" build note.

## 2026-06-06 | Category: module-boundaries | Source: M5-A gate fix — `-Xlint:exports` -Werror-fatal when an exported public class exposes a supertype from a non-transitively-required module (3rd occurrence: M2.9, M3.6e.1, M5-A)
**Discovery:** `platform-systemd`'s new `module-info` did `requires com.homesynapse.platform;` (plain) while its four exported public classes (`LinuxSystemPaths`/`LocalPaths`/`SystemdHealthReporter`/`NoOpHealthReporter`) `implement` `PlatformPaths`/`HealthReporter` from that module. javac `-Xlint:exports` emits `interface … is not indirectly exported using 'requires transitive'` once per class; `-Werror` makes it fatal. Green unit tests never surface it — it is a pure main-source `compileJava` warning.
**Detail:** Rule — if a public type in an EXPORTED package exposes (as a supertype, return type, parameter, or field) a type from another module, that module must be `requires transitive`, and the Gradle dependency must be `api(...)` in lockstep (api ↔ requires transitive; implementation ↔ plain requires). Fix here: `requires com.homesynapse.platform` → `requires transitive`, and `implementation(project(":platform:platform-api"))` → `api(...)`. `org.slf4j` stayed plain/`implementation` — it appears only in method bodies, never on the public API, so it does NOT trip `-Xlint:exports`. This is the THIRD occurrence (M2.9 `SqlitePersistenceLifecycle`; M3.6e.1 the package-private-impl + public-gateway variant; now M5-A). It recurs specifically when a coding instruction hands a *verbatim* `module-info` for a NEW module that will export impls of a dependency's types and that embed is non-transitive — honoring it blindly reproduces the defect.
**Impact:** (1) When an instruction supplies a verbatim `module-info` for a new module whose exported public classes implement/return another module's types, treat a plain `requires` as suspect and promote to `requires transitive` + `api` (do NOT honor the non-transitive embed blindly — flag it as a `[REVIEW]` correction). (2) Pre-handoff self-check for every `-Werror`-sensitive module touched: for each public exported class, does its signature name a foreign-module type? If yes, that module needs `requires transitive`. A targeted `compileJava` on the touched module is the cheapest way to surface `-Xlint:exports` before handing back. (3) `-Xlint:exports` is main-source-only — passing tests do not clear it.

## 2026-06-07 | Category: module-boundaries | Source: M5-A Part 2 (AMD-87) — a non-transitive `requires X` compiles GREEN with NO direct Gradle dep when X is already on the compile classpath via another dependency's `api` (the lockstep's transitive exception)
**Discovery:** persistence's `module-info` gained `requires com.homesynapse.device` (non-transitive) with **no** matching `implementation(project(":core:device-model"))` in `build.gradle.kts` — and it compiles GREEN. device-model is already on persistence's compile classpath transitively: persistence `api(":core:state-store")` and state-store `api(":core:device-model")`, so Gradle's `api` propagation puts device-model's module on persistence's compile module path. The §13 lockstep ("plain `requires X` ⇔ `implementation(project)`") is really about ensuring X is *reachable*; when a transitive `api` chain already makes it reachable, the direct dep is redundant for a GREEN build (every OTHER `requires` in this module-info happens to be directly paired — this is the lone exception).
**Detail:** Verified from source before relying on it (Source Trust Hierarchy — actual `build.gradle.kts`/`module-info`, not summaries): state-store `module-info` `requires transitive com.homesynapse.device` (JPMS readability) + state-store `build.gradle.kts` `api(project(":core:device-model"))` (Gradle compile classpath). The instruction explicitly scoped `build.gradle.kts` out ("only the module-info changes"), and the PM's own note said "persistence already reads device transitively via `requires transitive state → transitive device`" — leaning on transitive availability was intended, not an oversight, so I did NOT add the direct dep; flagged `[INFO]` with the fragility: if a future refactor narrows state-store's `api(":core:device-model")` to `implementation`, persistence's compile breaks ("module com.homesynapse.device not found") and the direct dep must be added. SEPARATELY the persistence→device **moduleGraphAssert** STOP-gate is clean: root `build.gradle.kts` `allowed` lists `:core:.* -> :core:.*` (any core→core edge) and `restricted` has no `persistence -X> device`; the edge is acyclic (device requires value/event/platform only) and a *shortcut* over the existing persistence→state→device path so `maxHeight=7` is unchanged.
**Impact:** (1) A non-transitive `requires X` does NOT always need a direct `implementation(project)` — check whether X is already on the compile classpath via a transitive `api` chain before adding one (or before flagging the instruction as incomplete). (2) When you DO lean on transitive availability, record it (handoff `[INFO]` + a MODULE_CONTEXT gotcha) so a future narrowing of the intermediary's re-export prompts adding the direct dep. (3) Before claiming a new cross-layer JPMS edge is gate-safe, read the root `moduleGraphAssert` `allowed`/`restricted` arrays — core→core is explicitly allowed here, so the edge passes; a cycle is not the only failure mode (an edge-allowlist would be the other).

## 2026-06-07 | Category: serialization | Source: M5-A Part 2 (AMD-87) — a hand-rolled sealed-type codec delegates nested registered codecs via `provider.defaultSerializeValue` / `ctxt.readTreeAsValue`; closes the M4.C DegradedEvent gap
**Discovery:** The AMD-87 `Expectation` codec is the exact twin of the AMD-52 `AttributeValue` codec: keyed on the sealed **interface** in `PersistenceJacksonModule` (Jackson's `SimpleSerializers` walks a concrete permit's superclasses/interfaces, so all permits resolve through the one pair — never register per-permit), the serializer is an exhaustive `switch` with NO `default` (a 5th permit breaks compilation), the deserializer dispatches on a `String` tag with a rejecting `default` (open discriminator, unlike the serializer's total sealed switch). The two permits that wrap an `AttributeValue` (`ExactMatch`/`AnyChange`) REUSE the already-registered AttributeValue codec instead of re-encoding: serialize with `provider.defaultSerializeValue(value, gen)`, decode with `ctxt.readTreeAsValue(valueNode, AttributeValue.class)` (Jackson ≥2.13). The numeric permit (`WithinTolerance`) reuses the package-private `AttributeValueSerializer.writeDouble`/`AttributeValueDeserializer.readDouble` statics (same package) for AMD-52 bit-anchored-float + sentinel determinism — never re-implement `doubleToLongBits`/sentinels.
**Detail:** This resolves the M4.C `@Disabled("AMD-65 pending")` DegradedEvent gap (see the 2026-06-05 AMD-59 lesson): a command-bearing `CapabilityAdded`'s `CommandDefinition → ExpectedOutcome → Expectation(ExactMatch) → AttributeValue(BooleanValue)` subtree now round-trips because the one missing leaf (`Expectation`) got a codec that delegates its wrapped value back to the existing AttributeValue codec. Wire form: `{"t":"ExactMatch","v":{"t":"BOOLEAN","v":true}}`; `WithinTolerance` → `{"t":"WithinTolerance","target":<num-or-sentinel>,"tolerance":<num-or-sentinel>}` (the AMD-87 §2 `<bits>` notation is the bit-identity *guarantee*, realized by Schubfach round-trippable text + `"NaN"`/`"+Inf"`/`"-Inf"` sentinels — NOT a literal long; the instruction says reuse the helper, "do not re-invent the encoding"). Test edge: a record's auto-`equals` for `double` uses Double semantics (NaN==NaN true; -0.0≠+0.0), and the codec canonicalizes -0.0→+0.0, so assert `WithinTolerance` -0.0 round-trip by `Double.doubleToLongBits(decoded.target()) == doubleToLongBits(0.0)`, NOT `isEqualTo`. **Un-disable gotcha:** deleting the only `@Disabled` in a test file leaves `import org.junit.jupiter.api.Disabled;` unused → Spotless `removeUnusedImports` fails `check`; remove the import in the same edit (see the M4.C spotless lesson).
**Impact:** (1) To add a hand-rolled codec for a sealed type that mirrors an existing one, copy the precedent exactly — interface-keying, no-`default` serializer switch, `String`-tag deserializer WITH a `default`. (2) Delegate nested registered codecs, don't re-encode: `provider.defaultSerializeValue(value, gen)` (write) and `ctxt.readTreeAsValue(node, Target.class)` (read) route through whatever is registered for the runtime/target type in the SAME mapper. (3) Reuse package-private `static` float helpers across same-package codecs for determinism; never duplicate `doubleToLongBits`/sentinel logic. (4) When a `double`-bearing record round-trips through a -0.0→+0.0 canonicalizing codec, assert bit-identity via `doubleToLongBits`, not record `equals`. (5) Un-disabling a test = delete `@Disabled` AND its now-unused import.

## 2026-06-09 | Category: JPMS / contracts | Source: AMD-66–71 DOCS review E70-1 (PM-side design miss; Coder guard for M6.1/M6.4)
**Discovery:** A type that lives in a **base/shared module** (`event-model`, `value-model`, `platform-api`) may reference, in its fields/components, **only** types from that module, a leaf module, or `java.base`. Referencing a **higher-layer domain module's** type (`config`/`device`/`state`/`automation`/`integration`) inverts the JPMS edge and creates a cycle — `config requires transitive event`, so a config type inside a `com.homesynapse.event` record forces `event → config` (the AMD-52 `event↔device` class).
**Detail:** This bit AMD-70's config observability events at the design stage — caught by independent review and flattened before any code, so M6.1's `config.validation_completed` is `(int configSchemaMajor, int configSchemaMinor, int issueCount, Map<String,Integer> severityCounts)` and M6.4's `config.section_reloaded` is `(String sectionPath, int changeCount, int issueCount, String appliedClassification)` — all `java.base`/`String`, mirroring the existing all-`String` `ConfigChangedEvent`. The no-`default` interface-keyed codec pattern (AMD-52/87) works precisely because those payload types live in `value`/`event`, never above.
**Impact:** (1) Before writing **any record in a base module**, read its `module-info` and confirm every component type's module is that module, a leaf, or `java.base`. (2) If a coding instruction's event record names a higher-layer domain type in its payload, **STOP and flag it `[BLOCKING]`** — do not implement around it; the fix is flatten-to-`java.base`/`String` or route typed data through a leaf (`com.homesynapse.value`, e.g. `QuantityValue`). (3) Nick's standing directive (2026-06-09): always be mindful about contracts and JPMS.

## 2026-06-09 | Category: module-boundaries / build | Source: M6.1 — third-party libraries on the JPMS module path require `requires` directives regardless of Gradle scope
**Discovery:** Gradle `implementation(...)` scope does NOT exempt a library from JPMS: in this build every dependency that ships a `module-info.class` or an `Automatic-Module-Name` resolves on the **module path**, so a named HomeSynapse module cannot `import` it without a matching `requires` directive — "it's only an implementation dependency" is the `requires transitive`↔`api` lockstep's blind spot in reverse.
**Detail:** The M6.1 instruction mandated `config/configuration/module-info.java` unchanged ("the pipeline is package-private; the libraries are already on the `implementation` classpath") while requiring YAML/JSON-Schema implementation classes inside `com.homesynapse.config`. Jar inspection from the Gradle cache (`jar --file <jar> --describe-module --release N`) showed snakeyaml-engine 2.9 is the explicit module `org.snakeyaml.engine.v2` and json-schema-validator 1.5.6 is the explicit module `com.networknt.schema` (whose own `module-info` requires jackson-databind NON-transitively, so Jackson types don't flow through it). ## 2026-06-11 | Category: testing / filesystem | Source: M6.4 — portable atomic-write failure injection via a directory squatting on the temp-file name; multi-release module-info location
**Discovery:** The §6.8 "disk full / write failure leaves the prior file intact" contract and the REC-135 migration crash-window both need a failure injected INSIDE the atomic-write window without seams or mocks. **Detail:** Companion findings from the same WU: (1) `Files.move(tmp, target, ATOMIC_MOVE)` replaces an existing target on BOTH platforms (POSIX `rename(2)`; Windows `MoveFileEx(MOVEFILE_REPLACE_EXISTING)`) — no `REPLACE_EXISTING` needed, but if an exotic filesystem ever surfaces `AtomicMoveNotSupportedException` at the gate, adding the flag is the one-line fallback (documented in the handoff).
