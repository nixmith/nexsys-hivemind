## 2026-04-10 | Category: testing | Source: M2.5 SqliteEventStore — `./gradlew check` failure on `NO_DIRECT_TIME_ACCESS`
**Discovery:** The `NO_DIRECT_TIME_ACCESS` ArchUnit rule fires on test classes in non-whitelisted packages — not just production code. Test code in `com.homesynapse.persistence.**` that calls `Clock.systemUTC()` or `System.nanoTime()` will fail `./gradlew :app:homesynapse-app:test` even though it compiles and runs locally.
**Detail:** `SqliteEventStoreTest` initially used `Clock.systemUTC()` in `resetStore()` and `System.nanoTime()` as a filename suffix for per-test database uniqueness. Both compiled fine and the test suite ran green in isolation. But `./gradlew check` failed on `HomeSynapseArchRulesTest > noDirectTimeAccess` because `HomeSynapseArchRules.noDirectTimeAccess` scans the ENTIRE classpath including test classes, and whitelists only `com.homesynapse.app..`, `com.homesynapse.platform..`, and `com.homesynapse.test..`. Test classes in `com.homesynapse.persistence` are not exempt. The fix: follow the `InMemoryEventStoreTest` convention — a static `FIXED_INSTANT = Instant.parse("2026-01-01T00:00:00Z")` + `Clock.fixed(FIXED_INSTANT, ZoneOffset.UTC)`. The production `UlidFactory` handles sub-millisecond monotonicity internally, so every `publish()` under a fixed clock still produces a distinct, strictly-increasing `eventId`. For per-test database uniqueness, JUnit 5 `@TempDir` already gives each test a fresh directory, so `tempDir.resolve("events.db")` with a stable filename is sufficient — no nanoTime suffix needed.
**Impact:** (1) ANY new test class in a non-whitelisted package (persistence, event-model, event-bus, device-model, state-store, etc.) must avoid `Clock.systemUTC()`, `Instant.now()`, `System.nanoTime()`, and `System.currentTimeMillis()`. Use an injected `Clock.fixed(...)` and delegate uniqueness to `@TempDir` or a deterministic counter. (2) Running `./gradlew :{module}:test` is NOT sufficient to validate compliance — the arch rules run in the app module's test task, so `./gradlew check` is the authoritative gate. (3) Whenever writing tests in a new package, grep `HomeSynapseArchRules` for the whitelist first — it's a fixed list and catching this at design time avoids a commit-blocking rework.

## 2026-04-10 | Category: testing | Source: M2.5 — JUnit 5 @BeforeEach parent-first ordering exploited for dependency wiring
**Discovery:** When a parent test class declares an abstract `resetStore()` hook called from its own `@BeforeEach`, subclasses should put ALL per-test initialization inside `resetStore()` itself — not in a subclass `@BeforeEach` — because JUnit 5 runs superclass `@BeforeEach` methods BEFORE subclass `@BeforeEach` methods.
**Detail:** `EventStoreContractTest` (the abstract parent) has a `@BeforeEach` that calls `resetStore()`. If `SqliteEventStoreTest` put its `DatabaseExecutor` + `SqliteEventStore` setup in a subclass `@BeforeEach`, the parent's `@BeforeEach` would run first, call `resetStore()`, which would see null fields and NPE. Putting everything in `resetStore()` — including opening the executor, building the codec, and constructing the store — runs the wiring at exactly the right time. JUnit 5's `@TempDir` is populated before ANY `@BeforeEach` runs, so `tempDir` is safe to reference from `resetStore()`. The `@AfterEach tearDown()` runs in reverse order (subclass first), so subclass cleanup works normally.
**Impact:** Any future contract-test subclass whose setup has a nontrivial dependency on a parent-supplied hook should use this pattern. Document the ordering exploit in a class-level Javadoc block so the next reader understands why setup lives in `resetStore()` rather than `@BeforeEach`.

## 2026-04-15 | Category: persistence | Source: M2.9 SqlitePersistenceLifecycle — synchronous start() vs CompletableFuture.runAsync()
**Discovery:** `PersistenceLifecycle.start()` returns `CompletableFuture<Void>` but the implementation must be synchronous — `CompletableFuture.runAsync()` would move `JacksonWarmup.warmup()` off the calling platform thread and re-expose the carrier-pinning vulnerability (LTD-19).
**Detail:** The PM's pseudocode suggested `CompletableFuture.runAsync(...)`, but `JacksonWarmup.warmup()` MUST run on a platform thread to populate Jackson's `synchronized` caches before any virtual thread touches the codec. The startup system (Doc 12 §3.4) calls `start()` from a platform thread, so running synchronously and returning `completedFuture(null)` or `failedFuture(e)` satisfies both the interface contract and the platform-thread requirement. This also simplifies the concurrency model: the `volatile started` flag needs only single-threaded write (in `start()` and `stop()`, which the startup system calls sequentially) with multi-threaded read (in store accessors called from virtual threads).
**Impact:** Any future refactor that makes `start()` async must ensure `JacksonWarmup` still runs on a platform thread — e.g., by using a platform-thread executor or splitting the warmup into a pre-async phase. The current synchronous approach is the simplest correct solution.

## 2026-04-15 | Category: persistence | Source: M2.9 — DatabaseExecutor.start() takes 4 parameters, not just a Path
**Discovery:** `DatabaseExecutor.start()` requires `(Path dbPath, String migrationPath, List<String> migrationFiles, MigrationConfig config)` — the migration infrastructure parameters are mandatory even though the lifecycle could theoretically derive them.
**Detail:** The PM's brief speculated `DatabaseExecutor.start(databasePath)` as a 1-parameter call. The actual signature has 4 parameters because `DatabaseExecutor` delegates to `MigrationRunner` internally and needs the migration manifest (per LTD-07's "explicit ordered list" rule). The lifecycle stores the migration path and file list as `private static final` constants and passes `MigrationConfig.freshInstall()` — the only valid config for M2's single-database, no-upgrade-path scenario.
**Impact:** Future milestones adding a second database (telemetry) will construct a second `DatabaseExecutor` with its own migration path/files. The lifecycle will need two executor instances with separate migration manifests. The 4-parameter design is correct — do not attempt to "simplify" it.

## 2026-04-15 | Category: module-boundaries | Source: M2.9 build gate — JPMS `-Xlint:exports` on public class in exported package
**Discovery:** A `public` class in an exported JPMS package must not expose package-private types or types from non-transitive `requires` modules in any public method signature — including constructor parameters and return types. Under `-Werror`, each violation becomes a fatal error.
**Detail:** `SqlitePersistenceLifecycle` was declared `public final class` in the exported `com.homesynapse.persistence` package. This triggered 5 `-Xlint:exports` warnings: (1) the constructor parameter `List<Class<? extends DomainEvent>>` leaked `DomainEvent` from the non-transitive `requires com.homesynapse.event` — clients of persistence don't automatically get access to event-model types; (2–5) four store accessors (`eventStore()`, `checkpointStore()`, `viewCheckpointStore()`, `atomicCheckpointWriter()`) returned package-private implementation classes that clients of the module cannot reference. The fix was to make the class package-private (`final class` instead of `public final class`), which is consistent with every other Phase 3 implementation class in the module. External modules interact through the `PersistenceLifecycle` interface.
**Impact:** Every new Phase 3 implementation class in an exported package should default to package-private visibility. Only promote to `public` if the class must be referenced by name from another module AND all its public method signatures use only (a) types from the same module's public API, (b) types from `requires transitive` modules, or (c) JDK types. When in doubt, grep for the existing pattern: `final class` (not `public final class`) for `SqliteEventStore`, `SqliteCheckpointStore`, `DatabaseExecutor`, etc.

## 2026-04-15 | Category: anti-pattern | Source: M2.9 build gate — speculative API calls without source verification
**Discovery:** Writing method calls from memory without reading the actual source file caused 3 compilation errors and 1 Spotless violation in a single build gate run.
**Detail:** Three wrong calls in `SqlitePersistenceLifecycleTest`: (1) `EntityId.generate()` — does not exist; `EntityId` has `of(Ulid)` and `parse(String)` factories, and the correct pattern from `EventStoreContractTest` line 146 is `new EntityId(UlidFactory.generate(clock))`; (2) `SubjectRef.forEntity(entityId)` — does not exist; `SubjectRef` uses bare noun factories: `entity()`, `device()`, `integration()`, etc.; (3) missing `import java.util.concurrent.CompletableFuture` for the `CompletableFuture<Void>` local variable in test 6. Additionally, the speculative imports for `Ulid` and `UlidFactory` were present but unused (because the code called `EntityId.generate()` instead of `new EntityId(UlidFactory.generate(...))`), so Spotless flagged them as unused imports.
**Impact:** Before writing ANY method call on a type from another class/module, read the actual source file or at minimum grep for the method signature. Especially: (1) static factory methods — project convention varies (`of()`, `parse()`, `entity()`, `create()`, constructor); (2) checked exceptions — always check the `throws` clause of any interface method before calling it; (3) test helpers that call interface methods inherit the throws obligation. The 30 seconds spent reading the source file saves 5+ minutes of build-fix-rebuild cycles.

## 2026-04-15 | Category: testing | Source: M2.9 build gate — uncaught `SequenceConflictException` on `publishRoot()`
**Discovery:** `EventPublisher.publishRoot(EventDraft)` declares `throws SequenceConflictException` (checked). Any test method or helper that calls `publishRoot()` must declare this exception or the compiler rejects the code.
**Detail:** The test helper `publishTestEvent(SqlitePersistenceLifecycle)` called `lc.eventStore().publishRoot(draft)` without declaring `throws SequenceConflictException`. All 5 test methods calling the helper also lacked `throws` clauses. The existing `EventStoreContractTest` already establishes the convention — every test method that calls `publishRoot()` declares `throws SequenceConflictException`. A quick grep for `throws SequenceConflictException` in the existing test fixtures would have revealed this at design time.
**Impact:** When writing new test code that calls any `EventPublisher` method (`publish()` or `publishRoot()`), always add `throws SequenceConflictException` to the method signature. This is a checked exception — the compiler enforces it, but discovering it at build-gate time wastes a round-trip. The same pattern applies to any future interface method that declares checked exceptions: read the interface `throws` clause before writing the call.

## 2026-05-02 | Category: persistence | Source: M2-bridge Tasks 1–3 — schema reservation column pattern for forward-compatible INSERT_SQL
**Discovery:** When a V001 schema amendment adds columns that the current implementation doesn't populate (reservation columns), the INSERT_SQL must include ALL columns and bind reservations to `setNull` with the correct JDBC type matching the DDL type (BLOB → `Types.BLOB`, TEXT → `Types.TEXT`, INTEGER → `Types.INTEGER`). This prevents NOT NULL constraint violations on columns like `home_id` and `payload_size` that DO have constraints.
**Detail:** The M2-bridge expansion took INSERT_SQL from 16 to 24 bind positions. The key insight is that column order in INSERT_SQL must match the DDL column order (minus AUTOINCREMENT columns), and every column that has a NOT NULL constraint must receive a real value, not `setNull`. `chain_hash` (NOT NULL DEFAULT zeroblob(32)) required `setBytes(ZERO_HASH)` — `setNull` would throw a constraint violation even though the DDL has a DEFAULT, because an explicit INSERT that names the column overrides the DEFAULT. `payload_size` was computed as `payloadBytes.length`. The remaining 5 reservation columns are nullable so `setNull` works.
**Impact:** When future amendments add NOT NULL columns to the events table, always verify: (1) is the column in INSERT_SQL? (2) does it have a NOT NULL constraint? (3) if yes, what value should be bound? Never assume `setNull` works on a NOT NULL column, even with a DDL DEFAULT, when the INSERT statement explicitly names that column.

## 2026-05-15 | Category: phase-2 | Source: M2→M3 Bridge work unit — sealed-interface package placement under JPMS
**Discovery:** Sealed interfaces and their permits in JPMS-compiled modules MUST live in the same package. A sealed interface `CheckpointPolicy` in `com.homesynapse.state` permitting `FixedCheckpointPolicy` and `AdaptiveCheckpointPolicy` requires both records to live in `com.homesynapse.state` — splitting them into a sub-package or sibling package causes a compile-time error ("sealed type and permit must be in same package") even when both packages are exported.
**Detail:** This work unit placed all 3 sealed-related files under `core/state-store/src/main/java/com/homesynapse/state/`. The `module-info.java` already `exports com.homesynapse.state` (no new export needed). The lesson generalizes: any future sealed hierarchy in HomeSynapse must group all permits with the sealed root in one package. If the hierarchy gets large enough to feel cluttered, the answer is to extract the entire hierarchy to a dedicated module rather than splitting permits across packages.
**Impact:** When designing a new sealed type in Phase 2, decide the package up front and verify every permit you anticipate fits the same package. If you find yourself wanting to split, the type is probably the wrong shape — either it should not be sealed (use `non-sealed` or an open interface) or it should be moved to its own module.

## 2026-05-15 | Category: phase-2 | Source: M2→M3 Bridge work unit — Phase 2 records may contain trivial implementation logic when logic IS the contract
**Discovery:** Phase 2 normally prohibits implementation code, but records whose entire purpose IS the trivial logic (e.g., `FixedCheckpointPolicy.shouldCheckpoint` is one OR comparison; `AdaptiveCheckpointPolicy.shouldCheckpoint` is one mode-switch delegation) are an exception. The decision logic IS the contract — there is no separate "implementation" to extract. PM approval of this exception was implicit in the brief, but it's worth documenting because the line between "contract" and "implementation" is fuzzy for policy-style records.
**Detail:** Rule of thumb: if extracting the logic to a separate implementation file would require either (a) renaming the record to a marker / config-only data carrier and exposing the logic through a separate concrete class that takes the record as a parameter, or (b) duplicating the logic in every consumer of the policy — then the logic belongs on the record. `FixedCheckpointPolicy.shouldCheckpoint` is in the first case: extracting it would mean creating `FixedCheckpointPolicyEvaluator` that takes the record. The record-and-evaluator split adds two types and a wiring step with zero benefit because the logic is trivial and the record is immutable.
**Impact:** When defining a Phase 2 sealed-interface implementor record where the implementor's body IS one trivial comparison or delegation, put the logic on the record and document it in the deviation report as `[INFO]`. Don't introduce a Strategy-pattern split that doubles the type count for no behavioral benefit.

## 2026-05-15 | Category: verification | Source: M2→M3 Bridge work unit — Python sqlite3 as cheap SQL syntax verifier when sqlite3 CLI is unavailable
**Discovery:** When the sandbox doesn't have the `sqlite3` command-line tool, Python's stdlib `sqlite3` module is a reliable syntactic and semantic check for SQL migrations. A 10-line Python script can apply V001 then V003 against an in-memory database, query `PRAGMA table_info` and `sqlite_master`, and assert the expected post-migration state (column count, NOT NULL flags, index presence, autoindex retention).
**Detail:** The script used in this work unit applied V001 to `sqlite3.connect(':memory:')`, then applied V003, then queried `PRAGMA table_info(snapshots)` (verified 9 NOT NULL columns), `sqlite_master WHERE type='index' AND tbl_name='events'` (verified `idx_events_subject` is dropped and `sqlite_autoindex_events_1` survives), and the snapshots-table indexes (verified `idx_snapshots_subject` is present alongside the implicit PRIMARY KEY autoindex). Total time: under 5 seconds, no JDBC, no Gradle.
**Impact:** Add this to the Phase 2 verification toolkit: when changing migration SQL and sqlite3 CLI is unavailable, run a Python sqlite3 simulation as part of the self-review checklist. It does not replace the full `./gradlew check` gate but catches obvious syntax errors and index-name mistakes before they reach the build gate.

## 2026-05-17 | Category: module-boundaries | Source: M3.1 InProcessEventBus — extending EventBus interface
**Discovery:** When extending an interface that has existing implementations, new methods must be `default`.
**Detail:** The EventBus interface gained 4 methods in M3.1 but `InMemoryEventBus implements EventBus` in testFixtures. Abstract methods would break `InMemoryEventBus` — it would need to implement all 4 new methods despite being a Phase 2 fixture that only supports the original 4-method contract. Solution: declare the new methods as `default` with `throw new UnsupportedOperationException("method requires active runtime")`. The production `InProcessEventBus` overrides all defaults with real implementations. This preserves backward compatibility while extending the contract.
**Impact:** Any future interface extension in HomeSynapse where the interface has existing implementations in testFixtures (which cannot be modified in the current milestone scope) must use `default` methods. The pattern generalizes: `default` methods are the interface evolution mechanism under JPMS when backward compatibility is required.

## 2026-05-17 | Category: testing | Source: M3.1 InProcessEventBus — contract test capability hooks
**Discovery:** When a contract test base class has existing concrete subclasses, new harness methods must have defaults.
**Detail:** `EventBusContractTest` is extended by both `InMemoryEventBusTest` (Phase 2, passive bus only) and `InProcessEventBusTest` (M3.1, full active runtime). Adding new abstract methods to `EventBusContractTest` (e.g., `createActiveRuntime()`) would break `InMemoryEventBusTest`. Solution: add a `boolean supportsActiveRuntime()` capability hook (default returns `false`) and guard all Tier 5-10 tests with `assumeTrue(supportsActiveRuntime())`. `InProcessEventBusTest` overrides to return `true`. `InMemoryEventBusTest` inherits the default and skips the new tiers automatically.
**Impact:** When adding tiers to any contract test base class that has multiple subclasses with different capability levels, use a `boolean` capability method with `assumeTrue()` guards. This prevents new abstract methods from breaking existing subclasses and explicitly documents which implementations support which test tiers.

## 2026-05-17 | Category: testing | Source: M3.1 InProcessEventBus — interface shape test maintenance
**Discovery:** Interface shape tests that assert method counts break when interfaces are extended.
**Detail:** `EventBusTest` (the interface shape test from M1.f) asserted "exactly 4 declared methods" on the EventBus interface. ## 2026-05-17 | Category: concurrency | Source: M3.1 InProcessEventBus — virtual thread existence assertions
**Discovery:** `Thread.getAllStackTraces()` does not include virtual threads in Java 21.
**Detail:** An initial approach to verifying per-subscriber virtual thread creation used `Thread.getAllStackTraces().keySet().stream().filter(t -> t.getName().startsWith("hs-sub-"))` to assert VT existence. This returns an empty set because `Thread.getAllStackTraces()` only returns platform threads. Virtual threads are not enumerable through JDK APIs in Java 21. Solution: use bus introspection (`bus.subscribers()` returning `List<SubscriberSnapshot>`) to verify subscriber registration, and verify VT creation indirectly through behavioral assertions (subscriber receives events, mode transitions occur).
**Impact:** Never use `Thread.getAllStackTraces()` or `Thread.enumerate()` to assert virtual thread existence. Use application-level introspection APIs instead. If a test needs to verify VT creation, assert on the observable behavior that the VT enables (event delivery, mode transitions, etc.), not on thread enumeration.

## 2026-05-17 | Category: other | Source: M3.1 InProcessEventBus — supervisor backoff simplification
**Discovery:** Supervisor backoff was simplified in M3.1; full exponential backoff deferred to M3.2.
**Detail:** PLAN-M3-CONSOLIDATED-02 §4.4 specifies MIN=3s, MAX=30s, jitter=0.2 exponential backoff. M3.1 implements immediate retry (no backoff delay) because the full backoff sequence depends on the ReplayDriver integration (M3.2). The key M3.1 supervisor behaviors are: exception taxonomy (Error/IOException/checked → SUSPENDED; RuntimeException → retry), DLQ recording of failed events, rolling crash window (10 min, 5 crashes → SUSPENDED), and circuit breaker transition to SUSPENDED mode. The backoff timing is a separate concern layered in M3.2.
**Impact:** When implementing M3.2, the supervisor's retry path gains the exponential backoff with `Thread.sleep(backoffMs)` (safe on virtual threads ��� unmounts carrier). Tests that exercise retries should use `MutableClock` to control time rather than waiting for real sleep delays.

## 2026-05-17 | Category: module-boundaries | Source: M3.1 InProcessEventBus — JPMS enforces JDBC-free bus constraint
**Discovery:** JPMS enforces the JDBC-free bus constraint — no ArchUnit rule needed.
**Detail:** The bus module's `module-info.java` does not declare `requires java.sql`. This means any attempt to import `java.sql.Connection`, `java.sql.SQLException`, or any JDBC type in any file under `core/event-bus/src/` will fail at compile time with a module boundary error. The PLAN-M3-CONSOLIDATED-02 §4.5 references an ArchUnit rule `EVENT_BUS_DOES_NOT_IMPORT_SQLITE_DRIVER` in `HomeSynapseArchRules.java` — this rule does not exist and is not needed. The JPMS module system provides a stronger guarantee: compile-time enforcement rather than test-time enforcement.
**Impact:** Do not attempt to create an ArchUnit rule for JDBC-free enforcement in the bus module. JPMS already handles it. The `SubscriberReadConnectionFactory` and `SubscriberReadExecutor` interfaces are the mechanism for bus code to access database reads without importing `java.sql` — they abstract over the JDBC connection. When writing M3.5a, the state-store module (which DOES have `requires java.sql`) provides the `SubscriberReadConnectionFactory` implementation that creates real JDBC connections.

## 2026-05-17 | Category: other | Source: M3.1 InProcessEventBus — infrastructure exception carve-out uses only java.base types
**Discovery:** Infrastructure exception classification uses only `java.base` types since the bus cannot reference `java.sql.SQLException`.
**Detail:** The supervisor's exception taxonomy classifies exceptions for retry vs immediate suspension. Since `com.homesynapse.event.bus` does not `requires java.sql`, it cannot reference `SQLException` directly. The taxonomy uses only `java.base` types: `Error` → infrastructure (immediate SUSPENDED), `IOException` → infrastructure (immediate SUSPENDED), other checked `Exception` subclasses → infrastructure (immediate SUSPENDED), `RuntimeException` → callback fault (standard backoff/retry). This means if a `SubscriberReadExecutor` implementation wraps a `SQLException` in a `RuntimeException`, it will be treated as a callback fault (retried) rather than infrastructure (suspended). The executor implementations should wrap infrastructure failures in `IOException` or `UncheckedIOException` to get correct classification.
**Impact:** When implementing `SubscriberReadExecutor` in the persistence module (M3.5a scope), wrap `SQLException` in `UncheckedIOException` (not plain `RuntimeException`) so the supervisor correctly classifies database failures as infrastructure exceptions and suspends rather than retrying.

### 2026-05-17 — `jdk.jfr` and JDK Platform Modules Under JPMS (M3.3)

**Severity:** PM-originated error in task instruction.

**What happened:** The M3.3 task instruction stated "Do NOT add `requires jdk.jfr` to module-info.java — it compiles without it." This was wrong. The first `./gradlew :core:event-bus:check` failed because `jdk.jfr.Event` was inaccessible without an explicit `requires jdk.jfr;` directive. The Coder corrected it; the second pass compiled clean.

**Root cause:** The PM deliberation incorrectly equated "platform module" with "auto-available under JPMS." The error chain: PM deliberation made incorrect JPMS claim → reviewer didn't catch it → task instruction contained an explicit incorrect directive → Coder hit build failure → Coder corrected it.

**The correct rule:** "Platform module" means "ships with the JDK — no Maven/Gradle dependency needed." It does NOT mean "auto-required under JPMS." Only modules in the transitive closure of your declared `requires` directives are resolved. `java.base` is the only unconditionally available module. `jdk.jfr`, `java.sql`, `java.naming`, `java.desktop`, `jdk.management` all require explicit `requires` directives.

**How to check before claiming "no requires needed":**
1. Is the module `java.base`? → No `requires` needed.
2. Is the module transitively required by an existing `requires`? Trace the actual `module-info.java` chain. Example: `com.homesynapse.event.bus` → `requires transitive com.homesynapse.event` → that module's `module-info.java` → `requires transitive com.homesynapse.platform` → that requires nothing beyond `java.base`. So `jdk.jfr` is NOT in the transitive closure — explicit `requires jdk.jfr;` is mandatory.
3. If neither 1 nor 2: you need `requires <module>;`. No Gradle dependency is needed (ships with JDK), but the JPMS directive is required.

**Lesson:** When the PM or Coder makes a directive-level claim about JPMS module availability, verify it by tracing the `module-info.java` chain. Do not infer availability from "it's a JDK module."

## 2026-05-19 | Category: testing | Source: M3.4b — ThrottledWriteCoordinator cross-module wiring
**Discovery:** When a testFixture in one module needs to be referenced from a test in a different module, the testFixture cannot be package-private — but its *factory* on a public harness can be. The natural placement of a test double is alongside the type it decorates (which may itself be package-private), so a "named convenience factory" on the public harness is the cleanest cross-module seam.
**Detail:** `ThrottledWriteCoordinator` had to live in `core/persistence/src/testFixtures/java/com/homesynapse/persistence/` so it could implement the package-private `WriteCoordinator` interface (mirroring `InMemoryWriteCoordinator`). The integration-tests module is in `com.homesynapse.it` and cannot reference `ThrottledWriteCoordinator` by name. Initial attempt: import the type into `IntegrationTestHarness` — failed (package-private). Resolution: add `PersistenceTestHarness.startThrottled(...)` as a named convenience factory that captures `ThrottledWriteCoordinator::withDefaults` internally. The IT harness calls the named factory by behavior, never mentions the implementation type.
**Impact:** The same pattern applies whenever a testFixture for an internal type needs cross-module reachability. Avoid promoting the internal type to public solely for test wiring — instead expose a named factory on a public harness. The same approach worked for `InProcessEventBusFactory.create(...)` in M3.4a.

## 2026-05-19 | Category: persistence | Source: M3.4b — WriteCoordinator decorator injection
**Discovery:** Widening a `final` field's type from a concrete class to its interface is the smallest possible change to enable test-only decorator injection without altering production behavior.
**Detail:** `DatabaseExecutor.writeCoordinator` was previously typed as `PlatformThreadWriteCoordinator` (concrete). M3.4b added a package-private 3-arg constructor accepting a `Function<WriteCoordinator, WriteCoordinator>` decorator. **Discovery:** `platform-systemd`'s new `module-info` did `requires com.homesynapse.platform;` (plain) while its four exported public classes (`LinuxSystemPaths`/`LocalPaths`/`SystemdHealthReporter`/`NoOpHealthReporter`) `implement` `PlatformPaths`/`HealthReporter` from that module. javac `-Xlint:exports` emits `interface … is not indirectly exported using 'requires transitive'` once per class; `-Werror` makes it fatal. Green unit tests never surface it — it is a pure main-source `compileJava` warning.
**Detail:** Rule — if a public type in an EXPORTED package exposes (as a supertype, return type, parameter, or field) a type from another module, that module must be `requires transitive`, and the Gradle dependency must be `api(...)` in lockstep (api ↔ requires transitive; implementation ↔ plain requires). Fix here: `requires com.homesynapse.platform` → `requires transitive`, and `implementation(project(":platform:platform-api"))` → `api(...)`. `org.slf4j` stayed plain/`implementation` — it appears only in method bodies, never on the public API, so it does NOT trip `-Xlint:exports`. This is the THIRD occurrence (M2.9 `SqlitePersistenceLifecycle`; M3.6e.1 the package-private-impl + public-gateway variant; now M5-A). It recurs specifically when a coding instruction hands a *verbatim* `module-info` for a NEW module that will export impls of a dependency's types and that embed is non-transitive — honoring it blindly reproduces the defect.
**Impact:** (1) When an instruction supplies a verbatim `module-info` for a new module whose exported public classes implement/return another module's types, treat a plain `requires` as suspect and promote to `requires transitive` + `api` (do NOT honor the non-transitive embed blindly — flag it as a `[REVIEW]` correction). (2) Pre-handoff self-check for every `-Werror`-sensitive module touched: for each public exported class, does its signature name a foreign-module type? If yes, that module needs `requires transitive`. A targeted `compileJava` on the touched module is the cheapest way to surface `-Xlint:exports` before handing back. (3) `-Xlint:exports` is main-source-only — passing tests do not clear it.
