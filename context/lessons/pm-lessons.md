## 2026-03-15 | Category: design-translation | Source: Sprint 1 Block B — CausalContext rewrite
**Discovery:** The handoff block system works well for scoped, compile-and-commit units, but block-level design decisions can conflict with sprint-plan-level scope commitments. The sprint plan is the higher authority.
**Detail:** Block B's Decision 4 deferred concrete payload records to Phase 3. This was a reasonable PM judgment — payload records depend on capability schemas (Doc 02), and Block B was already large. However, the weekly sprint plan (approved by Nick) had payload types scheduled for Monday March 16. When Nick directed "finish everything from the weekly plan," the deferral was overridden. The 18 payload records were then authored directly from Doc 01 without a formal handoff block.
**Impact:** PM block decisions are bounded by the sprint plan. If a block decision defers work that the sprint plan includes, the PM should flag the conflict to Nick rather than silently deferring. Nick will either (a) adjust the sprint plan or (b) produce an additional block to cover the gap. Block decisions should never silently narrow the sprint plan's scope.

## 2026-03-15 | Category: design-translation | Source: Sprint 1 Blocks A–F — handoff format effectiveness
**Discovery:** The handoff block format (locked decisions, exact deliverables, compile gate, file placement) is highly effective for focused execution. Blocks with clear file lists and pre-resolved design decisions execute in 1–2 hours each.
**Detail:** Blocks A through F each produced clean, compilable code on first or second attempt. The most valuable sections of the handoff format are: (1) "Locked Decisions" — eliminates design-time deliberation during coding, (2) "Exact Deliverables" with file paths — eliminates placement ambiguity, (3) "Compile Gate" — provides an unambiguous exit criterion. The least valuable section: "Imports" — the coder can resolve these from context.
**Impact:** Continue the handoff block format for all future coding blocks. Consider dropping the "Imports" section to reduce handoff size. The "Locked Decisions" section is the most impactful — invest PM time there. Each decision that's pre-resolved in the handoff saves 10–15 minutes of deliberation during coding.

## 2026-03-15 | Category: constraint-enforcement | Source: Sprint 1 — scaffold module-info.java issues
**Discovery:** Scaffold `module-info.java` files should not pre-declare `exports` for packages that have no types yet. This creates build failures when `-Werror` is enabled.
**Detail:** The initial scaffold (created pre-Sprint 1) included `exports` clauses for device-model and integration-api packages that had no production types — only `package-info.java` files. These became errors under the project's strict `-Xlint:all -Werror` configuration. The fix was to comment them out with Sprint 2 TODO markers. This is a PM concern because the PM authored the scaffold structure.
**Impact:** Future module scaffolding should follow a rule: `exports` clauses are added by the handoff block that creates the first type in the package, not during initial scaffolding. The scaffold should contain only `requires` clauses for known dependencies.

## 2026-03-20 | Category: constraint-enforcement | Source: Block N review — JPMS default rule codified
**Discovery:** After three blocks (I, K, N) all required the same `requires` → `requires transitive` correction, the JPMS default rule should be encoded as a locked decision amendment rather than treated as per-block deviations.
**Detail:** Block N's LD#10 specified `requires com.homesynapse.api.rest` (non-transitive). The Coder proactively changed it to `requires transitive` based on the Block K lesson — `ApiKeyIdentity` in record components and `ApiException` in throws clauses both leak through the public API. This is the third time a handoff specified `requires` where the compiler would have rejected it. The root cause: the expanded JPMS surface (record component types, exception types in throws clauses, exception superclasses) is consistently underestimated during handoff authoring. The correct default is `requires transitive` for any inter-module dependency where the depended-on module's types appear anywhere in the exported API surface.
**Impact:** Future handoffs should default to `requires transitive` and only use non-transitive `requires` when the PM can confirm NO types from the required module appear in any Phase 2 public API signature. This rule is now codified in LD#10 of the Block N handoff and should be carried forward to all subsequent handoffs.

## 2026-06-07 | Category: constraint-enforcement | Source: M5-A gate-fix — embedded module-info must pair `requires transitive` with Gradle `api` for exported impls
**Discovery:** When a coding instruction embeds a verbatim `module-info.java` for a module whose **exported package holds public classes that expose another module's type as a supertype/return/parameter** (here: platform-systemd's `LinuxSystemPaths`/`SystemdHealthReporter` implementing platform-api's `PlatformPaths`/`HealthReporter`), the PM MUST embed `requires transitive <that-module>` (paired with Gradle `api(...)`), not plain `requires`. Plain `requires` trips `-Xlint:exports` → `-Werror` → a hard compile failure. This is the **3rd occurrence** of this exact pattern (M2.9, M3.6e.1, M5-A) — twice prior in coder-lessons, now once as a PM instruction-authoring defect.
**Detail:** The M5-A instruction's embedded module-info gave `requires com.homesynapse.platform` (non-transitive); the Coder honored it verbatim (correct behavior) and the gate failed on 4× `[exports]` warnings. The fix is the documented house lockstep (persistence module-info's own inline comment; OR-M3-16): `requires transitive` ↔ `api`, plain `requires` ↔ `implementation`. Both the PM (authoring) and the Coder (pre-gate self-check) had the lessons on file and missed applying them — primarily an instruction defect, since the embedded module-info is authoritative for the Coder.
**Impact:** (1) **Coding-instruction-authoring checklist:** before embedding a module-info, scan the module's exported public types — any that `implements`/returns/accepts a foreign module's type on their public API need that module `requires transitive` + Gradle `api`. (2) **P-class process proposal (shift-left, extends P5):** the deferred-gate policy may defer the FULL `./gradlew check`, but the Coder should run a **targeted `./gradlew :module:compileJava`** on `-Werror`-sensitive touched modules before handoff — it catches `[exports]`/redundant-cast/unused-import in ~20s and would have caught all three occurrences in-session instead of costing a Nick round-trip. (3) The DOCS review earned its keep this window on the independent Doc 15 `encrypted_kek`→`encrypted_dek` schema catch the PM had missed — keep dispatching independent review on owner-docs.

## 2026-06-09 | Category: quality-gate (JPMS / contract direction) | Source: AMD-66–71 DOCS review — E70-1 event-payload-type-residency cycle
**Discovery:** A base/shared module's type must never reference a higher-layer domain module's types — it inverts the JPMS edge and makes a cycle. The same miss hid the HTTP-test fallout of the C1 gate (the integration-tests module went un-enumerated).

(2) Pin the fail-closed SCOPE before issuing. The instruction said "honor SD-9 fail-closed-at-load" + "decide FATAL-vs-log per Doc 12 §4" — leaving the failure DOMAIN ambiguous. The Coder defaulted to FATAL-boot (one malformed def bricks the system), contradicting the loader's ratified contract ("valid sibling definitions still load... published as config_error"). Fail-closed is per-RECORD (the bad rule fails safe = does not run), not per-PROCESS; Doc 12 §4 "Automation = FATAL" is per-SUBSYSTEM-INIT. When citing a fail-closed/FATAL directive, state the domain (per-record / per-section / per-subsystem-init) explicitly.

(3) Instruction-defect signal + the FIX-07 class recurring. AB-3 carried 5 instruction defects (2 module-graph edges, the schema-ordering, the loader-nesting, the OR-M13-aware reporter selection) — all caught by Coder build-verification + WUCP Phase 2, none shipped. And verify cross-layer wiring ORDER + input-shape against the consuming subsystem's own contract, not just type signatures — loader.load(rawMap()) compiled but dropped every automation (the loader reads top-level "automations"; config nests under the "automation" section). The deferred gate is structurally blind to "compiles-but-wrong-contract"; that is what pre-issue source-verification + human WUCP Phase 2 are for.

## 2026-06-21 | Category: coding-instruction-quality / review-finding | Source: M7.2a-1 WUCP Phase 2 — an instruction's procedural ordering must not contradict a locked behavioral contract the instruction itself cites

The M7.2a-1 instruction's `RunManager` admission sequence placed the condition gate in the run body **after** concurrency-mode admission, while the SAME instruction's "What to Watch Out For" cited the locked contract that "CONDITION_NOT_MET must NOT consume a concurrency-mode slot." Those two are contradictory: evaluating conditions after mode admission transiently consumes a slot (and can deadlock a SINGLE-mode automation whose conditions keep failing). MODULE_CONTEXT/Doc-07 §3.6 is explicit — "condition evaluation occurs at trigger time, **before** mode enforcement." The Coder caught it via the "Coder Pushback Welcome" seam and implemented conditions-before-mode (a CONDITION_NOT_MET Run never enters the active set); all FSM tests pass. **Lesson:** when an instruction specifies a step ORDER (an admission/FSM sequence), cross-check that order against every locked behavioral contract the instruction cites — a procedural sequence can silently violate a contract the prose elsewhere asserts. The PM's pre-issue check should walk the sequence against the contracts, not just list the contracts. Honor the locked contract over the instruction's literal ordering; the instruction is the defect. **Corollary (carry):** when a milestone is split and the FSM is built against a stubbed seam, behaviors unobservable against the stub (e.g. QUEUED's sequential single-flight drain vs. PARALLEL) get deferred to the seam-filling milestone — track them as explicit REQUIRED carries on that milestone's backlog row so a stub-equivalent simplification cannot silently ship as the real contract.

## 2026-06-21 | Category: quality-gate / CI-as-gate-of-record | Source: M7.2a-2 gate-fix — an in-session LLM "verification: clean" is NOT a build gate

M7.2a-2's first deferred `./gradlew check` came back RED with 8 compile errors from two Java record component/static-factory name collisions (`ValidationResult.valid()`; `ModeDecision.admit/restartCancel/enqueue`), even though the Coder's in-session "adversarial verification" had reported clean. ## 2026-06-21 | Category: anti-pattern / blast-radius | Source: M7.2a-2 — don't refactor working code without cause

One of the two M7.2a-2 collisions was self-inflicted: 2a-2 needlessly refactored M7.2a-1's already-working `ModeDecision` (a clean canonical-constructor form) into static factories named for its own components — manufacturing the `admit()/restartCancel()/enqueue()` collision out of code that had shipped GREEN one milestone earlier. **Lesson: do not refactor a prior milestone's working code without a cause tied to THIS milestone's contract.** A milestone's blast radius should be the new behavior plus its minimal integration; touching a prior GREEN type "to make it nicer" adds regression risk with no contract benefit and is exactly where defects enter. Scope each instruction tightly (name the files/types in scope), and when reviewing a return, flag any diff to a prior-milestone working type that the milestone's stated scope did not require — make the Coder justify it against the contract or revert it.

## 2026-06-22 | Category: coding-instruction-quality | Source: M7.2b — an illustrative code block must not contradict a binding constraint the same instruction states

The M7.2b instruction carried an illustrative signature `public ComputedValue …` while ALSO pinning "module-info unchanged." ## 2026-06-22 | Category: coding-instruction-quality / sequencing | Source: M7.3 — defer a subscriber's LIVE composition-root wiring until its producers exist; the breadth-vs-live-path pattern

M7.3's ledger shipped GREEN as a unit, but the M7.3 instruction ALSO told the Coder to wire the live `pending_command_ledger` subscriber into the running composition root — even though nothing emits its inputs (`command_issued`), the dispatch service isn't a live subscriber, and there is no `pollExpirations` tick. ## 2026-06-22 | Category: anti-pattern / build-discipline | Source: M7.3 gate-fix — a runtime subscriber needs a paired stop() teardown, and the compile self-step is necessary-not-sufficient

The M7.3 live-wiring attempt registered `pending_command_ledger` via `eventBus.subscribeRuntime(...)` with NO matching `stop()` teardown. Its dedicated SQLite read connection stayed open after `stop()`, so on Windows the temp DB (`.db`/`-wal`/`-shm`) could not be deleted and JUnit's `@TempDir` cleanup threw `IOException` — failing all 15 composition-root tests AT TEARDOWN (the test bodies passed; the shared cleanup is what blew up). Two rules: **(1)** any `subscribeRuntime` that allocates a read connection MUST pair with a close in `stop()`, mirroring the existing post-LIVE subscriber (`automation_engine`) — a structural composition-root rule, not a per-WU nicety. **(2)** The targeted `:module:compileJava` + `spotlessApply` self-step PASSED and missed this entirely, as did the in-session LLM verification; ONLY the full `./gradlew check` (which boots the real composition root against a real temp DB) caught it. The compile self-step is **necessary-not-sufficient** — it catches signature/`[exports]`/format defects, NOT runtime/teardown/integration defects. This is the concrete argument for adding an E2E composition-root integration test to the gate (M7.4), and for never treating the compile self-step — or an LLM read-pass — as the gate of record.
