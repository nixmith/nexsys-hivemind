## 2026-03-17 | Category: anti-pattern | Source: Architecture validation review
**Discovery:** Never assert a design gap exists without verifying against the specific design doc section first. Claiming "I haven't seen a detailed upcasting strategy documented" when Doc 01 §3.10 contains a full specification is a factual error that undermines credibility.
**Detail:** During the architecture validation synthesis, the agent claimed schema evolution lacked a detailed upcasting strategy. Nick corrected this — Doc 01 §3.10 specifies pure-function upcasters on JsonNode at read time, chained v1→v2→v3, strict/lenient modes, and AMD-10 adds projection versioning. The real concern is operational experience gap (what happens when 50 upcasters chain across 2 years of production), not a design gap. Similarly, the SQLite WAL concern conflated SD cards with the actual NVMe deployment target (LTD-02 explicitly prohibits SD cards), and the event storm resilience concern didn't engage with Doc 01 §3.6's backpressure/coalescing mechanisms.
**Impact:** Before asserting any gap in the architecture: (1) cite the specific doc section you checked, (2) if you can't find a section, say "I may have missed this" rather than "this doesn't exist," (3) engage with existing mitigations before framing something as unaddressed. The strongest observations from this review — temporal automation complexity ceiling and safety-critical actuator command model — were the ones that identified genuine extension points rather than mischaracterizing existing coverage.

## 2026-03-17 | Category: scope-management | Source: AMD-25 review — multi-entity trigger tracking key
**Discovery:** Design amendments must explicitly address edge cases for multi-entity selectors even when the common case is single-entity triggers. The `(automation_id, trigger_index)` tracking key for `for_duration` timers creates a semantic limitation that is acceptable for MVP but should be documented rather than discovered by users.
**Detail:** AMD-25's duration timer tracks by `(automation_id, trigger_index)` — one timer per trigger. For triggers with area/label/type selectors resolving to multiple entities, only the first matching entity starts the timer, and only that entity's state changes can cancel it. This is correct for the overwhelmingly common case (single-entity triggers like "motion sensor in living room"), but the limitation must be visible in the documentation so users don't create group-based duration triggers expecting per-entity tracking.
**Impact:** When reviewing amendments that add behavior to existing trigger/condition/action types, always check: does this behave correctly when selectors resolve to multiple entities? The multi-entity case is easy to overlook because design doc examples almost always use single-entity scenarios.

## 2026-03-17 | Category: decomposition | Source: Block I handoff — integration-api vs integration-runtime boundary
**Discovery:** The integration-api / integration-runtime split is the cleanest example of the "contracts vs implementation" module boundary pattern. The decision criteria: "does an adapter import this type?" If yes → integration-api. If no → integration-runtime.
**Detail:** Types like IntegrationFactory, IntegrationAdapter, IntegrationContext, HealthReporter, CommandEnvelope belong in integration-api because adapters depend on them. IntegrationSupervisor, IntegrationHealthRecord, SlidingWindow belong in integration-runtime because they're supervisor-internal — adapters never see them. This same pattern applies to other module splits: the automation module will have automation-api (types that the REST API consumes) vs automation-engine (the subscriber, evaluator, RunManager internals).
**Impact:** When producing handoffs for modules with API/implementation splits, lead with the boundary question: "which types do external consumers import?" This prevents scope confusion about what belongs in which module.

## 2026-03-17 | Category: decomposition | Source: Block I handoff — forward reference handling
**Discovery:** When a module depends on types from a not-yet-specified module, use `@Nullable Object` placeholders rather than forward-declaring interfaces that might not match the actual specification.
**Detail:** IntegrationContext needs TelemetryWriter (from persistence module, Block J) and ConfigurationAccess (from configuration module, Block K). Rather than creating forward-declared interfaces in integration-api that might not match what Blocks J and K produce, the handoff uses `@Nullable Object` for TelemetryWriter and omits ConfigurationAccess entirely. When those modules are specified, IntegrationContext will be updated with the proper types. This creates a known future edit but avoids creating types that need to be reconciled later.
**Impact:** For Phase 2 blocks with cross-module forward references, prefer placeholders over forward declarations. The placeholder approach makes the dependency explicit (it's documented in the handoff), avoids type duplication, and concentrates the future edit in one place. Note: this only works because we're still in Phase 2 (interface spec) — in Phase 3 (implementation), forward references would need proper interfaces for compilation.

## 2026-03-18 | Category: planning-calibration | Source: Block I execution — sandbox disk space
**Discovery:** The Cowork sandbox VM can run out of disk space, preventing bash/Gradle execution while file Write operations still succeed (they write to the mounted user folder, not the VM filesystem).
**Detail:** During Block I execution, the VM's internal filesystem filled completely. Every bash command failed with ENOSPC because the shell needs a temp directory. However, the Write tool continued working because it writes directly to the mounted ClaudeFolder path (user's actual disk). This meant all 19 Java source files were created successfully, but the compile gate (`./gradlew compileJava`) could not be run from within the session. The compile gate must be run manually on the host machine.
**Impact:** For sessions that may need Gradle/bash execution, check disk space early. If the compile gate cannot be run in-session, document it clearly in the handoff and defer to manual execution. The alternative — smaller session scope — may be worth considering if compile verification is critical to subsequent tasks.

## 2026-04-11 | Category: anti-pattern | Source: M2.5 arch-debt retrospective
**Discovery:** Policies that DEFER enforcement steps (build gates, manual reviews, integration tests) without also TRACKING those deferrals silently create latent failures. "Deferred" without tracking is "omitted" within one or two work units. This is strictly a process failure — the policy itself (deferring `./gradlew check` to Nick's sandbox-external environment) was rational given sandbox limits.
**Detail:** M2.2 (`696ac37`) introduced `Instant.now()` in MigrationRunner; M2.4 (`4b20786`) introduced `System.nanoTime()` in JacksonWarmup. Both live in `com.homesynapse.persistence`, which is not on the `NO_DIRECT_TIME_ACCESS` ArchUnit whitelist. Neither milestone's coder session ran `./gradlew check`; the policy was to defer to Nick's environment. Neither deferral was tracked on pm-handoff.md Open Risks — in fact, pm-handoff.md hadn't been updated since 2026-03-20 due to the separate WUCP Phase 2 non-execution failure (see next entry). The violations persisted through two commits and ~24 hours until M2.5's test run caught them. Commit `d6a6065` resolved the debt via Clock injection across MigrationRunner and DatabaseExecutor and simplification of JacksonWarmup. The full forensics are in `context/audits/2026-04-11_m2.5-arch-debt-retrospective.md`.
**Impact:** Structural fix landed 2026-04-11 as part of the governance overhaul: (a) WUCP Phase 2 checklist now includes a deferred-build-gate audit that copies every deferral into pm-handoff.md Open Risks with a closure condition; (b) `project-manager/SKILL.md` §4b mandates "never issue M{x}.{y+1} while M{x}.{y}'s deferred build gate is unresolved." Every future Coder handoff with a deferred gate MUST flag it explicitly, and every PM session MUST audit Open Risks. The lesson generalizes beyond build gates — any policy that defers enforcement must be paired with a tracking mechanism in the same file that executed the deferral.

## 2026-04-11 | Category: information-routing | Source: BCP → WUCP generalization + freshness preflight
**Discovery:** A protocol that requires a human trigger to fire will eventually stop firing silently. Between 2026-03-20 and 2026-04-11 — three weeks and five milestones — BCP Phase 2 (PM-side closeout) did not execute, because nothing in the agent workflow triggered a PM session after each milestone landed. BCP Phase 1 (Coder-side) fired correctly the entire time. The asymmetry is the tell: the side that had a clear "do this at session close" checklist kept running; the side that needed "start a new session to do this" never ran.
**Detail:** The Block Completion Protocol (BCP) was authored during Phase 2 using Block-only vocabulary. When Phase 3 began, the Coder continued Phase 1 closeout (MODULE_CONTEXT, coder-handoff, coder-lessons, cross-agent-notes) because those responsibilities transfer straightforwardly from "Block X" to "Milestone M2.y." But Phase 2 (PM-side) stopped for three reasons compounding: (1) no automatic trigger — after each Coder session, Nick moved to the next coding task without starting a PM session; (2) no session-start freshness check — when a PM session eventually did fire, nothing said "verify the hivemind isn't lagging the codebase"; (3) vocabulary drift — Phase 3 used "Milestone" language while the BCP used "Block" language, creating ambiguity about whether the BCP even applied. Every PM-side artifact (PROJECT_SNAPSHOT.md, pm-handoff.md, strategic-context-map.md §6, weekly plans, master-release-plan.md) stayed frozen at its March-20/24 state while the codebase advanced through M1.a–M1.i, AMD-33, M2.1, M2.i, M2.2, M2.3, M2.4, M2.5, and the arch-debt fix. Visible drift triggered the 2026-04-11 audit.
**Impact:** Structural fixes landed in the 2026-04-11 governance overhaul: (a) BCP → WUCP (Work Unit Completion Protocol), vocabulary-neutral, with the prime rule "no work unit is done until both WUCP phases have executed, and completion is a prerequisite for starting the next"; (b) a 10-check freshness preflight at `project-manager/references/freshness-preflight.md` that MUST run at every PM session start and at every WUCP Phase 2 review, with STALE restricting the session to retroactive reconciliation only; (c) PM SKILL.md Mode 3 (Director) now asserts the preflight as the session-start gate; (d) a dual-skill-location `diff -rq` check as the final WUCP Phase 2 gate. The generalizable lesson: any protocol whose execution depends on a human remembering to trigger it is not a protocol — it's a suggestion. Convert every such protocol into a session-start checklist that fires automatically when the agent loads its role skill. If the agent doesn't run it, the protocol isn't real.
