## §2 Invariant candidates (minted at Lock if they pass the register test)

- **EXT-INV-1 — Core Never Depends on Third-Party Extensions (candidate, recommend MINT).** HomeSynapse Core boots, runs, and passes CI with the third-party extension set empty; no core feature may take a load-bearing dependency on any extension-supplied capability, type, or event. *Parents:* INV-RF-01 (crash isolation — binds failure propagation), INV-LF-02 (no outbound dependence), DECIDE-04/LTD-16 (static composition). *Novel:* the parents bind runtime failure and network direction; none binds the **dependency direction for the third-party actor class**. *CI-testable now:* composition root builds and boots with zero non-first-party integrations (the M9.1 fake-adapter path already approximates the harness). *Evidence:* L-1, L-13, L-27 — platforms where community code became load-bearing (SmartThings/webCoRE) could not evolve without breaking users; HA's split ("optional addition… at the cost of stability") is the surviving counter-shape (L-17, L-18).
- **EXT-INV-2 — Extension Capability Is Declared, Not Discovered (candidate, recommend MINT).** Every third-party extension declares its full capability surface — event classes published, operations exposed, services required, config schema version, isolation level — in static manifest data validated before load; no capability is granted at runtime that was not declared. *Parents:* the frozen `IntegrationDescriptor` (AMD-54..64: requiredServices, dataPaths, configSchemaMajor/Minor, isolationLevel), the M3.6c per-module event-class manifests, AMD-56 error-code registry discipline. *Novel:* the descriptor exists, but nothing binds the third-party actor class to declared-capability-only — today a full `EventPublisher` is injected and the permitted-publish-types check is unbuilt (gap: trust boundary paper-only, assessment §2). *CI-testable when the gate lands:* manifest-vs-behavior test + permitted-publish-types enforcement. *Evidence:* L-5, L-7; Chrome's manifest-as-trust-contract and remote-code ban; VS Code's declarative contribution points.
- **Kept as principles, NOT minted:** (a) *"The frozen SPI is the plugin contract, additive-only"* — fully carried by AMD-54..64 + AMD-55-INV-01; minting would duplicate (the INV-GA-02 near-duplicate warning). (b) *"Trust is tiered, never binary"* — testable content lands only when ladder rungs are built; premature (the AIOT-INV-2/3 precedent). (c) *"Data-first extensibility"* — owned by D5's ruling; Doc 18 restates.

## §3 The reservation list, finalized (seven seams; Doc 18 §4 rows)

1. **Dynamic loading.** Reserve JPMS child-ModuleLayer loading at the composition root, gated on (a) the LTD-16/17 amendment via the recorded process — **proposed amendment shape:** permit child-ModuleLayer plugin loading behind a named security evaluation + the trust-ladder rung the plugin declares; static composition remains the default and the V1 truth — and (b) the security evaluation the `IntegrationFactory` javadoc already anticipates (gap: dynamic loading triple-banned, assessment §2). Non-preclusion requirement: dependency resolution + engine-version gating must be designed WITH the loader, not after — openHAB's raw-JAR/marketplace era shipped dynamic modules without dependency resolution and paid for years (L-16; dossier OH-1/3/4). Evidence: L-6, L-9.
2. **Integration operation registry.** Reserve the registry that fills the reserved-empty `InvokeIntegrationAction` (gap: sealed automation hierarchies, assessment §2): operations are **declared data** (name, typed params, target integration type) registered at composition, never runtime-discovered code — the declarative-over-imperative trust win (L-5) and the analyzable target AI-authoring needs (L-32). Quotas/limits on operation invocation route to INV-RF-02's reserved path, and any cap is user-visible by design (L-23).
3. **Event-manifest layering.** Reserve per-module event-class manifests joining at ONE governed aggregation gate (gap: hardcoded 3-manifest composition-root aggregation, assessment §2). Third-party event classes enter only via their declared manifest (EXT-INV-2), namespaced per §5 below. Evidence: L-5; the M3.6c pattern already layers.
4. **Trust-boundary ladder.** Reserve the four named rungs in order: **scoped context wrappers** (LTD-17 enforcement — unscoped `EntityRegistry`/`StateQueryService` today) → **publish permissions** (permitted-publish-types check) → **INV-RF-02 quotas** → **RESERVED_SUBPROCESS** (the `IsolationLevel` enum already carries it). Doc 18 states the honesty rule: rungs are claimed only as built; no sandbox language in any external copy before the rung exists (L-2, L-4). The Homebridge child-bridge precedent is the reservation's feasibility proof: per-plugin subprocess isolation retrofitted as pure configuration because the contract stayed process-agnostic — the SPI's narrow 2-type contract preserves exactly that property (L-3). Evidence: L-1, L-2, L-3, L-6; gap: trust boundary paper-only (assessment §2).
5. **Namespace & identity governance.** Reserve: (a) **two-level publisher-scoped extension identity** (`publisher.extension`, immutable, transfer-on-abandonment governance — L-26, L-28); (b) the **type-string namespace convention** for integration types and profile identities, landing WITH M9.2 identity binding and BEFORE the M9.3 profile-registry freeze (assessment §3 items 1/3; gap: one-instance-per-type with no namespace governance); (c) an explicit **collision ruling** — recommend HA's shape: third-party may shadow only with explicit user consent + UI flagging, core wins brand preference (L-29); (d) profile/device identity disambiguation follows the fingerprint-priority + rebrand-mapping pattern (L-30) — fold into the M9.3 instruction (assessment R-2).
6. **Packaging & distribution.** Reserve what a plugin artifact IS: a **versioned, immutable release artifact + static manifest** carrying at minimum `version` (the forensic/blocking primitive — L-8) and a declared core-compatibility range that the registry/loader enforces by serving only compatible versions (L-9). Distribution may ride existing infrastructure initially (npm/GitHub-class — L-16) with the supply-chain inheritance named honestly; version-pinned channels (SmartThings' one good idea) reserved as the enterprise/installer distribution shape (L-15). Config rides `SchemaRegistry.registerIntegrationSchema` — built, never wired (gap, assessment §2) — the `config.schema.json` lesson (declarative config contract enabling UI + verification; dossier HB-4/16).
7. **Marketplace minimum trust.** Reserve the trust floor the marketplace milestone must meet on day one (the dossier's Q7 synthesis): registry of record with unique IDs; per-version automated checks + a public quality/safety signal; revocation that reaches clients (kill-switch); disclosure-based policies with a small banned list (no obfuscation, no self-update outside the pipeline, no undisclosed network/telemetry); default-off unreviewed tier; two-tier labeled core/community split with a machine-tracked, demotable quality ladder — the HA quality-scale shape, which also carries the Trust-Mark-style certification coupling (L-7, L-17, L-18, L-19, L-20, L-21, L-24, L-25, L-31). Explicitly NOT in the floor: payments, first-party hosting, continuous audit, sandbox promises.

## §4 Decision points for Nick

- **DP-18-A — AX-7, the component versioning/deprecation ruling (R-B).** *Mark:* NON-BLOCKING for Doc-18 Lock (Doc 18 frames it; the ruling is Doc 16 OQ2's own gate) — **BLOCKING before M7.2 ships user-authored components and before any AI-authoring milestone** (Doc 17 §7-4 already sequences it). *Market evidence:* the surviving ecosystems all converge on semver + declared compat range + registry-serves-compatible (L-9), a stable-contract pledge with a formally gated unstable tier (L-10), deprecation floors measured in release cycles with automated repair/migration surfaces (L-11), and immutability at the released-artifact level only — strict whole-definition immutability appears nowhere at ecosystem scale (L-14); pinning belongs to distribution channels (L-15). *Recommendation:* rule AX-7 as **semver on `(componentId, version)` + declared compat range + forward-only non-destructive migration (AMD-93-aligned) + a named deprecation floor (≥N release cycles with a Repairs-class surface)**, i.e. Doc 16's option (a) hardened with the L-9/L-11 mechanics; reject the stricter immutability model.
- **DP-18-B — the isolation line.** *Mark:* NON-BLOCKING (reservation only). *Evidence:* L-1/2/3/4/6 — in-process is a full-compromise grant everyone eventually pays for (HA 2021); subprocess buys crash/perf containment, not privilege separation; it retrofits cheaply if the contract stays process-agnostic (Homebridge). *Recommendation:* wave-1 curated adapters remain `IN_JVM` behind the quality gate (the HA-core analog); Doc 18 pins `RESERVED_SUBPROCESS` as the **intended default rung for non-curated third-party code at marketplace time**; the honesty rule (no sandbox claims before the rung exists) becomes doc text.
- **DP-18-C — R-C, the Connect "community integration repository access" paywall line.** *Mark:* NON-BLOCKING for Doc-18 Lock — **BLOCKING before any public pricing page** (assessment R-C). *Evidence, not a ruling (Nick's call):* no surveyed platform vendor-paywalls community-contributed content (L-24); HA — the direct competitor-precedent — is explicit that community content is free while paid cloud funds development, and monetizes trust via certification instead (L-25); openHAB bans paid marketplace content outright; Obsidian hosts no payments and takes no cut. The tension is with NexSys's own non-negotiables ("free tier fully functional," "no feature-gating of core functionality"). *Options to present:* (a) reword the Connect line to convenience framing (e.g., curated/one-click update channel — the value is the service, not exclusive access); (b) keep as-is and accept the brand risk. The evidence points at (a); Nick rules.

## §5 What Doc 18 should NOT attempt

It mints at most EXT-INV-1/2 and keeps every other principle unminted.
