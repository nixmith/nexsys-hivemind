# April 2026

**Master Plan Phases This Month:** P2 Foundation Implementation — persistence subsystem (maps to internal Phase 3)
**Days Remaining at Month Start:** ~228 (of the 258-day plan)

## Strategic Focus

April is the first month of real production implementation. Phase 2 (Interface Specification) closed out on 2026-03-20 with all 16 subsystem modules sealed at the interface level (~376 files). The gate into Phase 3 is the persistence subsystem: the event-sourced SQLite storage foundation that every downstream subsystem depends on. Until SqliteEventStore works, no other subsystem's production implementation can begin.

The month's work decomposes into two halves. The first half is test-first preparation: contract tests, in-memory reference implementations, and fixtures for event-model, event-bus, state-store, configuration, device-model, integration-api, and persistence. These run ahead of any production code so the tests define "correct" before "correct" is written. The second half is the persistence implementation proper: migration framework, database executors, serialization infrastructure, and SqliteEventStore itself.

April is also the month the governance layer caught up with the code. The hivemind layer had been frozen at the 2026-03-20 Phase 2 closeout while code advanced through five persistence milestones (M2.1–M2.5) plus an arch-debt fix. A full retrospective and governance overhaul is landing mid-April to prevent recurrence — the Block Completion Protocol is generalizing to a Work Unit Completion Protocol with session-start freshness checks.

## Key Milestones

1. **Test-first preparation wave** (Mar 27 – Apr 9) — M1.a through M1.i. Contract tests and in-memory reference implementations for event-model, event-bus, state-store, configuration, device-model, integration-api, persistence.
2. **Persistence subsystem landing** (Apr 10 – mid-Apr) — M2.1 (`@EventType` annotation on 22 event records), M2.2 (migration framework + V001 schema), M2.3 (DatabaseExecutor, 31 tests), M2.4 (serialization infrastructure, 62 tests), M2.5 (SqliteEventStore + TimeConversion + EventCategoryMapping, 46 tests).
3. **Arch-debt cleanup** (Apr 11) — Clock injection retrofitted into MigrationRunner and DatabaseExecutor; JacksonWarmup simplified; test-side `Clock.fixed` propagation. Resolves latent `NO_DIRECT_TIME_ACCESS` violations shipped silently in M2.2 and M2.4.
4. **Hivemind governance overhaul** (Apr 11) — BCP → WUCP generalization, freshness preflight, skill-location mirroring, dual-backlog split, retrospective audit. Restores the PM-side closeout gate.
5. **M2.6 + M2.7** (upcoming) — Combined milestone, scope TBD, likely boot-order wiring of SqliteEventStore into PersistenceLifecycle.start() plus remaining persistence gaps.

## Risks

- **Deferred build gates.** `./gradlew check` runs in Nick's sandbox-external environment, not inside the coder session. Two latent `NO_DIRECT_TIME_ACCESS` violations slipped through M2.2 and M2.4 and were only caught by the M2.5 test run. The WUCP freshness preflight and Mode 3 Director pre-flight assertion are now the enforcement mechanisms.
- **M2.5 deviations.** Six documented deviations (D-01..D-06) in the M2.5 coder-handoff — `subject_type` column added to V001, `readConnections()` accessor added to DatabaseExecutor, `TestPayload` retrofitted with `@EventType`, etc. Each deviation needs confirming that downstream milestones assume the deviated state, not the original spec.
- **Boot-order wiring deferred.** SqliteEventStore is not yet wired into PersistenceLifecycle.start() — marked as M2.8+ future work per M2.5 coder note. Must not be forgotten when the automation/event-bus subsystems come online and start expecting a real EventStore on startup.
