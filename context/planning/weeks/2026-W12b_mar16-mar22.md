## This Week's Objective

Define all public Java interfaces, records, enums, and sealed interfaces for six modules — device-model, state-store, integration-api, persistence, configuration, and automation — with full Javadoc contracts. All six modules must compile cleanly against existing Sprint 1 modules by end of week.

## How This Advances HomeSynapse

These six modules complete Waves 2 and 3 of the interface specification. Device Model defines the entity/capability type system that every higher-level subsystem queries. State Store defines the materialized-view read model. Integration API defines the adapter contract. Persistence, Configuration, and Automation round out the internal subsystem contracts. After this week, only Wave 4 (external APIs + observability + lifecycle) remains before Phase 2 is complete.

- **Device Model complexity:** Sealed Capability hierarchy is the most complex type structure in the project. May take longer than one session. Block G has the most types of any single block.
- **Automation Engine complexity:** Four sealed hierarchies (trigger, condition, action, selector) plus the run management interfaces. Block L is the second-largest block.
- **Cross-module dependencies tighten:** Starting with Block I (integration-api), modules depend on 3+ other modules. module-info.java management becomes critical.
- **Note:** DeviceId, EntityId, IntegrationId, AutomationId already exist in platform-api from Sprint 1. No need to recreate them.
