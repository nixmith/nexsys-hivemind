# Lane A1 Return — Matter Stack Survey (evidence dossier; options, no winner)

## §0 — Executive digest (one turn)

**The landscape moved decisively in the twelve months behind us, and it moved toward matter.js.** The Open Home Foundation's **matterjs-server** (TypeScript, Apache-2.0, WebSocket API) replaced the deprecated python-matter-server as Home Assistant's production Matter stack in HA 2026.6 — the add-on is `stage: stable` at v9.1.0 with no rollback path, while the repo README still carries a "Beta / not yet officially re-certified by the CSA" banner (§3.1). openHAB independently converged on the same core: its shipped Matter binding runs matter.js in an auto-downloaded Node 22+ sidecar, after its authors judged the CHIP Java SDK path "a maintenance nightmare over time" (verbatim record re-verified, §1.5 — with a **citation correction**: the real decision record is community thread t/127907, not the t/135926 the banked base cited). python-matter-server's repo was **archived read-only on 2026-06-23** (§4). No shipped headless-JVM product using the CHIP Java bindings was found (search trail, §1.4); no pure-Java Matter controller exists anywhere, and the single missing crypto primitive is named: **no maintained Java library implements SPAKE2+** — Bouncy Castle has zero SPAKE code (§5.1).

**Per option, one line each (cards in §9; no ranking):** (a) **matterjs-server sidecar** — production-proven at HA scale, 16 stable releases in the five weeks since 1.0.0, full WS API enumerated in §3.3, Matter 1.6.0 support landed 26 days after the spec; costs a supervised Node 22.13+/24.x runtime (64-bit only, ~2× the old server's RAM), an unauthenticated WS port, and sensitive fabric-credential storage files (§6). (b) **CHIP-JNI in-process** — official but Android-oriented JNI bindings buildable only inside the full GN/ninja SDK tree ("20G+" working dir per the openHAB record), zero Maven artifacts, zero known headless adopters, and C++ crash blast-radius = whole-JVM death (§1–§2). (c) **CHIP-FFM in-process** — FFM is **preview in Java 21, final only in 22+** (per-release binary incompatibility under `--enable-preview`), and CHIP exposes no C-ABI controller surface, so FFM adds a hand-written C shim on top of everything JNI needs (§2). (d) **pure Java** — priced honestly at **multiple M9-class arcs (band: ~5–10× the 14-WU Zigbee arc for a commissioning-capable controller before device breadth)**, derived from measured reference-stack sizes (matter.js: 391k LOC TS, 3,803 commits, 56 contributors, ~3.5 years; CHIP: >1.1M LOC C/C++) and the missing SPAKE2+ primitive (§5).

**Licenses are clean on every route** — CHIP, matter.js, and matterjs-server all Apache-2.0 at the LICENSE file; matterjs-server's full 199-package production npm tree contains zero GPL/LGPL/AGPL/SSPL; Node.js itself is MIT-style with no runtime copyleft (§7). **Spec tracking** favors the actively-shipped stacks: matter.js reached Matter 1.6.0 in under 4 weeks (historical lag was 4–6 months); the CHIP SDK has no v1.6 tagged release a month after the spec; big-ecosystem controllers still run 1.2/1.3-era feature sets (§8). Attestation trust arrives via a bundled daily DCL snapshot plus live DCL fetch in matterjs-server — an outbound-network dependency the design phase must walk against INV-LF (§10, BLOCKING).

**Notable single findings for the audit:** the stable matterjs-server 1.2.7 pins a *nightly alpha* of matter.js core (§3.6); the WS API has **no authentication** and binds all interfaces by default (§6.3); HA production currently runs a client two months older than its server across the schema-11/12 seam — by design, and it works (§6.4); issue #164 (BLE-commissioning crash) is **still open**, #694 is closed, and a new class — silent server↔device subscription death, #843 — is open (§3.4). RECs absent by design; this lane recommends nothing.

## §2 — In-process embedding from Java 21: the honest JNI-vs-FFM picture (evidence, not design)

### 2.1 The JDK-version interaction, stated explicitly

- **Java 21 (our core): FFM is a preview API** (JEP 442, third preview). Primary-doc proof: the Java 21 `java.lang.foreign` javadoc marks the API `[PREVIEW]` throughout [VERIFIED-current: https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/foreign/package-summary.html, fetched 2026-07-19]. (The openjdk.org JEP pages themselves returned HTTP 403 to this lane all session — honest gap §11; JEP numbers corroborated via the openjdk/jextract README which links FFM to JEP 454 [VERIFIED-current: https://raw.githubusercontent.com/openjdk/jextract/master/README.md, fetched 2026-07-19].)
- **Java 22+: FFM is final** (JEP 454) — the Java 22 javadoc carries no preview markers [VERIFIED-current: https://docs.oracle.com/en/java/javase/22/docs/api/java.base/java/lang/foreign/package-summary.html, fetched 2026-07-19].
- Using FFM on 21 requires `--enable-preview` at compile AND run, and — Oracle, verbatim: "Code that uses preview features from an older release of the Java SE Platform will not necessarily compile or run on a newer release." [VERIFIED-current: https://docs.oracle.com/en/java/javase/21/language/preview-language-and-vm-features.html, fetched 2026-07-19]. **Consequence, stated plainly: an FFM route on our Java 21 core means preview-pinned, per-release-incompatible binaries; a stable FFM route implies the core moving to 22+ first** [inference — direct from the quoted preview semantics].
- Native-access gating arrives regardless of route: JEP 472, verbatim — "The load and loadLibrary methods are restricted in JDK 24, just as the SymbolLookup::libraryLookup methods are restricted in the FFM API"; enable via `--enable-native-access=M1,M2,...`; "Library developers who rely on JNI or the FFM API should inform their users that they will need to enable native access." [VERIFIED-current: https://openjdk.org/jeps/472, fetched 2026-07-19]. Already in 21/22, FFM restricted methods warn on stderr unless `--enable-native-access` is set [VERIFIED-current: Java 21/22 java.lang.foreign package-summary, fetched 2026-07-19].

### 2.2 What each route demands against CHIP specifically

- **Per-architecture native packaging is unavoidable on both routes:** the controller is one GN-built `shared_library` (`libCHIPController`) produced only inside the tree; linux-arm64 + x86_64 means two `.so`s rebuilt from the full tree at every SDK release we track [VERIFIED-current: BUILD.gn + targets.py cites in §1.2].
- **FFM's extra tax — CHIP has no C-ABI controller surface.** jextract "leverages the clang C API in order to parse the headers" and binds C, not C++ [VERIFIED-current: openjdk/jextract README, fetched 2026-07-19]. The controller API is C++ classes — `src/controller/CHIPDeviceController.h` declares `class DLL_EXPORT DeviceController : public AbstractDnssdDiscoveryController` and `class DLL_EXPORT DeviceCommissioner : public DeviceController` inside `namespace chip::Controller` [VERIFIED-current: https://raw.githubusercontent.com/project-chip/connectedhomeip/master/src/controller/CHIPDeviceController.h, fetched 2026-07-19]. The only in-tree `extern "C"` controller surface is the Python shim (`ChipDeviceController-ScriptBinding.cpp`, functions like `pychip_DeviceController_StackInit(...)`), welded to Python-specific adapter types [VERIFIED-current: raw fetch, 2026-07-19]. **An FFM route therefore requires a hand-written, hand-maintained C shim over the C++ controller — everything the JNI route needs, plus the shim** [inference — FFM/jextract cannot consume mangled C++ classes; the existing C surface is Python-bound].
- **JNI specifics:** the JNI spec constrains loading — "The same JNI native library cannot be loaded into more than one class loader" (verbatim; violation throws `UnsatisfiedLinkError`) [VERIFIED-current: https://docs.oracle.com/en/java/javase/21/docs/specs/jni/invocation.html, fetched 2026-07-19] — which interacts with any module-layer/reload design at the composition root [inference — stated as an interaction, not designed here].

### 2.3 Crash blast-radius: native crash = JVM-process death, on either route

Four primary sources, converging: JNI spec — "Doing so could result in arbitrary consequences, including a corrupted system state or VM crash." [VERIFIED-current: https://docs.oracle.com/en/java/javase/21/docs/specs/jni/design.html, fetched 2026-07-19]. Oracle Troubleshooting Guide — "A crash, or fatal error, causes a process to terminate abnormally", with the canonical `# SIGSEGV … # Problematic frame: # C [libApplication.so+0x9d7]` example and "A bug in a native library can manifest itself as a crash in Java VM code" [VERIFIED-current: https://docs.oracle.com/en/java/javase/21/troubleshoot/troubleshoot-system-crashes.html, fetched 2026-07-19]. JEP 472 — "Calling native code can lead to arbitrary undefined behavior, including JVM crashes. Such problems cannot be prevented by the Java runtime." [VERIFIED-current: https://openjdk.org/jeps/472, fetched 2026-07-19]. FFM javadoc — binding "can result in VM crashes, or memory corruption" [VERIFIED-current: Java 22 java.lang.foreign package-summary, fetched 2026-07-19]. **Net: an in-process C++ Matter stack (JNI or FFM) shares fate with the core JVM — a SIGSEGV anywhere in ~1.1M lines of C++ terminates the whole process** [inference — direct composition of the four quotes with the §1.3 codebase measurement].

## §5 — Pure Java, scope honesty

What a from-scratch Java 21 Matter controller actually entails, priced from measured reference implementations (all sizes measured in fresh clones @ 2026-07-18 HEADs) [VERIFIED-current: git clones of project-chip/connectedhomeip + matter-js/matter.js, measured 2026-07-19].

### 5.1 Session-establishment crypto — and the named missing primitive

PASE runs SPAKE2+ (RFC 9383; Matter core spec §4.13.1, PAKE computation §3.10); CASE is the certificate-authenticated SIGMA-style exchange (§4.13.2, Sigma1/2/3) [VERIFIED-current: https://csa-iot.org/wp-content/uploads/2023/10/Matter-1.2-Core-Specification.pdf ToC + https://www.rfc-editor.org/rfc/rfc9383.html, fetched 2026-07-19]. **No maintained Java library implements SPAKE2+ as of today:** a case-insensitive grep across a fresh clone of bc-java (HEAD 2026-07-16, version 1.86-SNAPSHOT) yields **zero SPAKE source hits** (BC ships J-PAKE/EC-J-PAKE/SRP/Owl, not SPAKE2+); Maven Central search for "spake2" returns exactly 2 artifacts, both `io.yellowstonesoftware.spake2plus4s` — a **Scala** v0.1 (2023-12-27, 0 stars); the only other JVM neighbor is `MuntashirAkon/spake2-java` — plain SPAKE2 (not 2+), **LGPL-3.0** [VERIFIED-current: bc-java clone grep + https://search.maven.org/solrsearch/select?q=spake2 + https://github.com/MuntashirAkon/spake2-java, fetched 2026-07-19]. Matter's remaining primitives (P-256 ECDSA/ECDH, SHA-256, HKDF/PBKDF2, AES-128-CCM, X25519) are covered by BC/JDK [VERIFIED-current: bc-java clone — CCMBlockCipher.java, X25519.java present, fetched 2026-07-19]. Reference costs: matter.js implements SPAKE2+ in-house in **149 LOC** atop `@noble/curves` (its M/N constants pinned to draft-bar-cfrg-spake2plus-01); CHIP's `src/crypto` is 56 files / 25,383 LOC incl. eight SPAKE2P test-vector headers; PASE+CASE protocol logic: CHIP `src/protocols/secure_channel` 42 files / 15,627 LOC (CASESession.cpp/h alone 3,302 LOC); matter.js pase/ 649 + case/ 941 LOC [VERIFIED-current: clone measurements, 2026-07-19]. The LOC is small; the cost is crypto-review-grade correctness against spec test vectors (Matter spec Appendix F.5 per CHIP's TestChipCryptoPAL.cpp) [VERIFIED-current: clone, 2026-07-19].

### 5.2 TLV + Interaction Model

TLV: Matter's own binary format (spec Appendix A) — CHIP: 21 files / 13,669 LOC; matter.js: 14 files / 2,653 LOC. Interaction Model (Read/Subscribe/Write/Invoke, DataVersion filtering, event paths — spec ch. 8): matter.js `@matter/protocol` interaction+action dirs ≈ 10,598 LOC; the whole `@matter/protocol` package (action/advertisement/bdx/ble/certificate/codec/dcl/fabric/groups/icd/interaction/mdns/ota/peer/securechannel/session/thread/transport) is **239 files / 47,238 LOC**; CHIP's `src/app` is 1,300 files / 310,674 LOC (includes cluster servers) [VERIFIED-current: clone measurements, 2026-07-19].

### 5.3 mDNS/DNS-SD

### 5.8 The honest arc-scale, in M9-arc units

Anchors: the Zigbee arc = **14 WUs** for a far smaller protocol [banked: roadmap return §6.3]; matter.js took **~3.5+ years, 3,803 commits, 56 contributors** to reach an HA-production-grade controller (first commit 2022-11-04 → server GA June 2026), landing at 391k LOC TS of which ~47k is the protocol package proper; CHIP is 23,187 commits / 726 committers / >1.1M LOC [VERIFIED-current: clone measurements, 2026-07-19]. This option is priced, not dismissed: it is the only route with zero process/runtime dependencies and total stack custody — the EZSP-precedent play at ~10× the EZSP scale [inference — comparison to banked DP-C posture].

## §6 — The sidecar shape (evidence for A3; no recommendation)

### 6.1 Process lifecycle as HA runs it

The add-on supervises via **s6-overlay inside the container + HA Supervisor outside**: the server IS the s6 service (`exec node --enable-source-maps …`); the s6 `finish` script on non-zero exit writes the exitcode and `exec …/halt` — **a server crash halts the whole container**, and restart is the Supervisor's (user-toggleable watchdog) [VERIFIED-current: https://raw.githubusercontent.com/home-assistant/addons/master/matter_server/rootfs/etc/s6-overlay/s6-rc.d/matter-server/{run,finish}, fetched 2026-07-19]. The watchdog is real enough to warn about: "please disable the watchdog for the initial migration to prevent issues or mid-migration restarts" (MIGRATION_FAQ verbatim) [VERIFIED-current: MIGRATION_FAQ.md, fetched 2026-07-19]. Fixed launch args HA passes: `--storage-path /data --port … --log-level … --primary-interface <detected> --fabricid 2 --vendorid 4939` (+ loopback-only listen addresses when no host port is mapped) [VERIFIED-current: s6 run script, fetched 2026-07-19]. (4939 = 0x134B, matching the `server-2-134b` storage-dir name in the FAQ [inference — arithmetic].) Standalone Docker ships a built-in healthcheck; docs model is `--restart=unless-stopped` [VERIFIED-current: docs/docker.md, fetched 2026-07-19].

### 6.2 Storage/state it owns, and its sensitivity

Default `~/.matter_server` (`/data` in containers). Layout, verbatim (ALPHABETATESTS.md FAQ): "**server**: All node data is stored in files in this directory – one file per storage key… **ota**: The OTA update files… **certificates**: Downloaded Matter certificate chains used to validate devices when commissioning… **vendors**: Downloaded list of Matter vendors". Legacy-compat files are kept in sync: "`chip.json` - Chip configuration (fabric credentials)… Legacy files are kept updated for rollback compatibility"; chip.json "contains the global configuration of your Matter fabric, including IDs, keys, and certificates for secure communication" [VERIFIED-current: ALPHABETATESTS.md + docs/cli.md, fetched 2026-07-19]. **Sensitivity: this is the controller's fabric identity.** The documented remediation for corrupted storage is deleting the server storage dir — discarding fabric state; that losing fabric credentials forces re-commissioning every device follows from Matter's trust design (devices trust the fabric CA; new fabric = new commissioning) [VERIFIED-current: MIGRATION_FAQ remediation, fetched 2026-07-19] [inference — the re-commission consequence is Matter-architectural; no doc sentence states it literally, and no specific user-lost-storage issue was located, §11]. The docs repeat backup warnings at every migration step.

### 6.3 What the WS seam exposes vs what a controller needs

Everything device-facing rides the WS API (§3.3). **Process-start-only (CLI/env, not WS):** storage path, ports/listen addresses, primary interface, **`--fabricid`/`--vendorid` (fabric identity)**, BLE enablement, OTA enable/disable + custom-OTA import dir, test-net DCL, DCL-seed disable, time sync, dashboard, log file [VERIFIED-current: docs/cli.md argument table, fetched 2026-07-19]. Deliberate WS restrictions exist (pinned fabric label wins over `set_default_fabric_label`; label-setting "owned per connection") [VERIFIED-current: schema changelog, fetched 2026-07-19]. **Security posture: "By default the WebSocket server binds to all network interfaces… You are responsible for securing access" — there is no auth on the WS API** (verbatim warning) [VERIFIED-current: docs/cli.md, fetched 2026-07-19].

### 6.4 Version coupling server↔client

Mechanism: `SCHEMA_VERSION = 12 / MIN_SUPPORTED_SCHEMA_VERSION = 11`; schema-12 events are opt-in-gated because "older (schema-11) clients … would disconnect on an unknown event type" [VERIFIED-current: ws-controller source + schema changelog, fetched 2026-07-19]. In production HA: the add-on image pins server 1.2.6 while HA core 2026.7.1 pins `matter-python-client==0.7.1` (published 2026-05-21) — **a two-month client-server skew running in production by design, held together by the schema floor** [VERIFIED-current: add-on CHANGELOG + HA core 2026.7.1 manifest.json + PyPI, fetched 2026-07-19]. Skew has bitten before: 1.2.0 changelog "Ensures that the Python-Client does not crash anymore on unknown events from newer Server versions"; the WebRTC surface "first shipped under schema 11 … without being documented or gated" [VERIFIED-current: CHANGELOG + schema changelog, fetched 2026-07-19].

### 6.5 Packaging reality on Pi-class hardware (outside HA)

Install methods: **Docker** `ghcr.io/matter-js/matterjs-server` (tags stable/latest/dev/semver; base `node:24-trixie-slim`; unprivileged UID 1000 by default; bundles bluez, capability-granted ping; compose file in-repo; caveat verbatim: "The Docker image is provided as-is and without official support") or **npm** (`npm install matter-server && npx matter-server` on Node 22.13+/24/26). **No standalone binary exists.** Host obligations: host networking with working mDNS + IPv6/Thread routing, optional D-Bus for unprivileged BLE, firewall/conntrack tuning (battery devices "drop subscriptions every 15–30 min" behind stateful firewalls), persistent `/data`, restart policy, health probing (bare-npm supervisors must poll the port themselves), and securing the unauthenticated WS port [VERIFIED-current: docs/docker.md + README, fetched 2026-07-19] [inference — the obligations list is assembled from the cited operational docs]. 64-bit only; RAM per §3.5.

### 6.6 What breaks when the sidecar dies mid-command (issue/doc evidence, not speculation)

- **Mid-commissioning crash:** damage is bounded by the Matter failsafe — an unfinished commission rolls back when the device's failsafe expires; matter.js WIP changelog is actively tightening this ("Ensure the commissioning failsafe timer stays within the device's maximum cumulative failsafe") [VERIFIED-current: matter.js CHANGELOG WIP, fetched 2026-07-19] [inference — the rollback consequence is Matter-spec behavior, marked as such]. Recovery machinery exists and implies the prior failure: 1.0.0 "recovers automatically when the stored Node-ID counter drifts out of sync with the fabric"; 0.17.6 fixed a shutdown hang "when a BLE commission was in flight" [VERIFIED-current: both CHANGELOGs, fetched 2026-07-19]. The live counter-example: #164 (open) is precisely a crash during BLE commissioning (§3.4).
- **WS disconnect ≠ controller stops:** device subscriptions are owned server-side, independent of WS clients; the server emits `server_shutdown` on orderly exit [VERIFIED-current: README + ws-client model.ts, fetched 2026-07-19]. **In-flight WS commands are correlated only by `message_id`; nothing persists or re-delivers responses across a connection drop** [inference — from the envelope; no queue/replay mechanism appears anywhere in the API surface]. HA's client behavior on disconnect: full integration reload (`async_reload`), fresh `start_listening` dump rebuilds state; in-flight commands surface as errors [VERIFIED-current: HA core 2026.7.1 matter/__init__.py, fetched 2026-07-19].
- **Server restart → device re-subscription:** reconnects using "the last known addresses" with partial re-interview; 0.17.6 re-subscribes rebooted devices "within ~30s instead of waiting out the previous subscription's timeout"; 1.2.0 ensures changed node structures are pushed "directly after re-subscribe and not delayed" [VERIFIED-current: ALPHABETATESTS + both CHANGELOGs, fetched 2026-07-19]. The open counter-class: #843 silent subscription death (§3.4).
- **Slow/stalled WS clients:** updates coalesce and **stale events are dropped by design** rather than buffered unboundedly (1.2.0) [VERIFIED-current: CHANGELOG, fetched 2026-07-19] — an at-most-once event seam, not a log [inference — consequence of the quoted design].

### Card (a) — matterjs-server sidecar (Node process, WS API)

- **Maturity:** production at HA scale since 2026-06 (add-on `stage: stable` 9.1.0, one-way migration); 16 stable releases in 5 weeks; repo README still "Beta … not yet officially re-certified by the CSA" (§3.1). OHF-stewarded, SLSA-provenance npm publishes (§3.6).
- **Controller coverage:** commissioning (code/QR, on-network, BLE local + BLE-proxy, Thread — name-gap closed 2026-07-13), server-side subscriptions, bridges, integrated OTA-provider, fabric management, ICD; Matter 1.6.0 (feature-guarded); NFC payload-bit only; no Joint Fabric (§3.2).
- **Maintenance load:** track a fast-moving server (weekly-ish stable releases) across a schema-versioned WS seam that provably tolerates months of client skew (§6.4); watch items: stable-pins-nightly-alpha of core (§3.6), README/doc staleness (§3.2, §8.4).
- **Runtime cost:** Node 22.13+/24.x sidecar, 64-bit only, ~2× python-server RAM (no absolute MB published), `--max-old-space-size` tuning on constrained hosts; no standalone binary — Docker or npm+system-Node; supervisor must provide restart/health/storage/WS-port-security (§3.5, §6.5).
- **License:** Apache-2.0 at LICENSE; 199-package production tree zero-copyleft; Node MIT-style (§7).
- **Spec-tracking:** fastest observed route — 1.6.0 in <4 weeks; DCL trust via bundled daily seed + live fetch, reject-on-error attestation (§8.4–8.5).
- **Unknowns:** CSA re-certification timing; absolute RAM/disk on Pi; #164/#806 BLE-commissioning stability; #843 silent-subscription class; Joint Fabric roadmap (§11).

### Card (b) — CHIP-JNI in-process (libCHIPController inside our JVM)

- **Maturity:** official CSA reference SDK (the certification substrate itself); the Java binding layer is Android-oriented ("JNI bridge … for Android apps"), desktop-Linux example-grade, with open example-hygiene issues; **zero known headless-JVM adopters** (§1.2–1.4).
- **Controller coverage:** the full C++ `DeviceCommissioner` surface — commissioning incl. NFC API (v1.5.0.0 added `pairDeviceThroughNfc()` to the JNI), subscriptions, OTA, fabric table; released code at Matter 1.5.1, 1.6 unreleased (§1.1, §1.3).
- **Maintenance load:** full-tree GN/ninja rebuild per SDK release per architecture ("20G+" working dir, "builds are SLOW"); "Controller Changes" churn every release; no Maven artifacts, no out-of-tree build path; the openHAB team abandoned exactly this route as "a maintenance nightmare over time" (§1.2–1.5).
- **Runtime cost:** zero extra processes; two native `.so`s (x86_64 + arm64) shipped in our artifact; JPMS/native interactions (one-classloader JNI rule; `--enable-native-access` from JDK 24); **crash blast-radius = whole-JVM death** — a SIGSEGV in ~1.1M lines of C++ takes the core with it (§2.2–2.3).
- **License:** Apache-2.0 at LICENSE; `third_party/` tree unaudited this pass (§7.4).
- **Spec-tracking:** spec-named branches/tags; observed lag spec→release 0–5 weeks historically, but 1.6 still untagged after a month; PAA store is an operator-supplied directory defaulting to *test* certs (§8.3, §8.5).
- **Unknowns:** linux-arm64 binding viability (target constructible, nobody documented shipping it); binary sizes; the real cost of keeping a fork-free tree-build green across releases (§11).

### Card (c) — CHIP-FFM in-process (Panama bindings to a C shim)

- **Maturity:** hypothetical — no FFM bindings to CHIP exist anywhere; everything in card (b) plus a from-scratch, hand-maintained C shim over the C++ controller (jextract binds C, not C++; the only in-tree `extern "C"` surface is welded to Python) (§2.2).
- **Controller coverage:** identical to (b) in principle — whatever the shim exposes (§2.2).
- **Maintenance load:** card (b)'s full-tree build burden + shim upkeep against the per-release "Controller Changes" churn (§1.3, §2.2).

- **Spec-tracking:** as (b) (§8.3).
- **Unknowns:** shim scope/size (no precedent to measure); JDK-upgrade interaction with the frozen core toolchain; all of (b)'s unknowns (§11).

### Card (d) — Pure Java from scratch

- **Maturity:** does not exist. The one experiment (digitaldan/matter.js-java) has 2 commits, both 2023-11-11; its author shipped a Node sidecar instead; no other JVM-native attempt found (§5.7).

- **Unknowns:** jmdns Matter-fitness (unproven either way); SPAKE2+ implementation-review cost; real-device interop breadth without a quirk corpus; certification posture for a novel stack (A5's lane) (§11).

## §10 — Open questions for the design phase (each marked; answering them is A3/B-phase work, not this lane's)

1. **[BLOCKING] The DCL outbound dependency vs the local-first invariants.** matterjs-server's attestation channel is bundled-daily-seed + live `on.dcl.csa-iot.org` fetch (§8.5); CHIP's is an operator-supplied snapshot defaulting to test certs. Any option must be walked against INV-LF (no outbound dependence) and the conservative-default law: is a DCL fetch during commissioning acceptable, seed-only mode (`--disable-dcl-seed` inverse) the default, or an operator-updated snapshot channel? This shapes commissioning UX and honesty claims either way.
2. **[BLOCKING] Fabric-identity custody under any sidecar option.** `--fabricid`/`--vendorid` are process-start-only; the storage dir IS the controller identity (fabric CA keys, chip.json class); loss ⇒ re-commission the fleet (§6.2–6.3). Who owns backup/custody (SecretStore INV-SE-03 interaction), and what VID do we run? (HA runs 0x134B/4939 — theirs, not ours.)
3. **[BLOCKING] The unauthenticated WS seam.** No auth, default-binds all interfaces (§6.3). A design must pin loopback-only binding + supervisor-enforced port hygiene before any adapter speaks to it.
4. **[BLOCKING for B2 scope] BLE commissioning in or out of V1-Matter scope.** The BLE path carries the live crash/timeout issue classes (#164, #806); IP-only commissioning sidesteps them at a UX cost (§3.4). A2's hardware evidence co-decides.
5. **[NON-BLOCKING] Node runtime lifecycle policy.** Bundle vs auto-download (openHAB precedent) vs require-system-Node; Node 22 floor hits security-EOL 2027-04-30, 24.x runs to 2028 (§3.5, §7.2). A packaging-time decision with license implications already cleared (§7).
6. **[NON-BLOCKING] Version-pinning policy across the schema seam.** HA demonstrates months of tolerated skew on the schema floor (§6.4); our pin cadence (server version, client schema, and the stable-pins-nightly-alpha wrinkle §3.6) is a supply-chain-policy row, not an architecture row.
7. **[NON-BLOCKING] Subscription-liveness honesty (→ lane A4).** The #843 class (silent server↔device subscription death while `available` stays true) is exactly the never-false-ALIVE stress case; the WS seam exposes `available`, `ping_node`, and `attribute_updated` freshness as raw material (§3.4, §6.6). Mapping it is A4's charter; named here so the seam evidence lands with them.
8. **[NON-BLOCKING] CSA re-certification watch.** matterjs-server's "not yet officially re-certified … but will be in the future" (§3.1) — timing affects marketing claims only (A5's certification-economics lane), not architecture.
9. **[NON-BLOCKING] If in-process is ever seriously walked: the JDK question.** FFM-stable requires Java 22+ against our Java-21 core; JNI works on 21 but inherits JDK-24 native-access gating flags (§2.1). Evidence stands ready; no design implied.
