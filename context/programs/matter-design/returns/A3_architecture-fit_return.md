## §0 Executive digest (≤1 page)

**The four shapes, one line each against OUR architecture:**

- **(a) matterjs-server sidecar** — the SPI can front it honestly as a NETWORK adapter (the Shelly seam-map is the template `[banked: roadmap §6.2]`), and INV-RF-01's own text *pre-authorizes* out-of-process isolation as "an implementation detail, not an API contract" (`Architecture_Invariants_v1.md` §3/line 270). But it is the **first non-JVM production process** in a repo whose lifecycle doc, systemd unit, installer, and image build are single-process to the letter (§3 census below), and it lands in a genuine vocabulary hole: a first-party sidecar is neither IN_JVM-truthful nor the DP-18-B RESERVED_SUBPROCESS marketplace rung (§1.4, §9-B1). Nothing in the supervisor supervises a process (§2). The gaps are **enumerable and designable**; none is a violation.
- **(b) CHIP-JNI in-process** — SPI-frontable, but a C++ crash in-JVM is whole-JVM death: it does not degrade INV-RF-01, it **defeats** it (the supervisor's isolation is exception-catching; a SIGSEGV is not an exception — §2.2). AMD-63's javadoc names *exactly this case* as what RESERVED_SUBPROCESS was reserved for (`IsolationLevel.java` lines 26–31). Per-arch full-tree GN builds with zero Maven artifacts land on LTD-10/version-catalog doctrine with no precedent (§4.2) `[A1 §1–2]`.
- **(c) CHIP-FFM in-process** — everything in (b) plus a from-scratch C shim that exists nowhere `[A1 §2.2]`, plus preview-class FFM flags on the LTD-01-locked Java 21 launcher (§4.3). Same blast radius as (b).
- **(d) pure Java** — fits the as-built architecture without tension (it is exactly what the SPI models; the Shelly seam-map walks it), and its entire cost concentrates in one row: we become the stack's sole maintainer against a spec that revs twice a year, with no community absorbing device quirks `[A1 §9 card (d)]`.

**Three findings the hub should audit first:**
1. **The supervisor rejects RESERVED_SUBPROCESS at startup, at source** — `StandardIntegrationSupervisor.java` lines 166–170 throw `UnsupportedOperationException` (AMD-63-INV-01). Any sidecar/subprocess posture change is an amendment-class decision, not a config choice (§1.4).
2. **The NETWORK/virtual-thread supervise path is built and still unexercised by any real adapter** — re-verified at source (`StandardIntegrationSupervisor.java` lines 540–542); the only production adapter is SERIAL (`ZigbeeIntegrationFactory.java` line 133). Options (a) and (d) would be its first customer (§1.1).
3. **Doc-vs-source drift (flagged per the truth hierarchy):** Doc 12 §3.8 Step 6.1 says integration discovery "uses ServiceLoader (LTD-17)"; the as-built mechanism is the explicit factory list with ServiceLoader **banned** (DECIDE-04; ArchUnit Rule 3 `noServiceLoader` — `integration-api/MODULE_CONTEXT.md` §Utility/line 134). Source wins; Doc 12 carries stale text (§10).

**Matrix shape (full matrix in §8, verdict words per cell only — no totals, per the dispatch):** the options fail in *different dimensions* — (a)'s gaps cluster in process-model/supervision machinery (all buildable, all inventoried), (b)/(c)'s in blast radius and build doctrine (one structural), (d)'s entirely in maintenance trajectory. **No winner is stated anywhere in this file.**

## §1 The SPI walk, per option

**The frozen surface being walked.** `IntegrationFactory` (`descriptor()` pure/no-I/O, `create(IntegrationContext)`) → `IntegrationAdapter`, 8 declared methods: `initialize()`/`run()`/`close()`/`commandHandler()` + 4 AMD-55 default hooks (`onConfigUpdated`, `onOptionsUpdated`, `onReauthRequired`, `migrate`) — `integration-api/MODULE_CONTEXT.md` §M4.C/line 30. The governing lifecycle text at source, `IntegrationAdapter.java` lines 52–59: initialize "must not block on network or serial connections (INV-RF-03 — startup independence). Connection to external devices is handled by the adapter's internal reconnection logic during `run()`." `close()` "must be idempotent" (line 98). `commandHandler()` "Returning `null` indicates the adapter does not handle commands" (lines 115–117). Descriptor: 14 components; the 8-arg convenience ctor defaults `isolationLevel=IN_JVM`, `backoffParameters=defaults()` (`integration-api/MODULE_CONTEXT.md` §M4.C/line 28) — the as-built Zigbee factory uses exactly that 8-arg form (`ZigbeeIntegrationFactory.java` lines 130–138).

### 1.1 (a) matterjs-server sidecar

- **`initialize()`:** the adapter-side work (config read, schema-backed accept lists, data structures) is the Zigbee precedent verbatim. The tension is the SPAWN: launching the Node child is not "block[ing] on network or serial connections" by the letter (`IntegrationAdapter.java` line 56), but it is I/O against the external environment, and a slow/failed spawn at initialize would recreate exactly the boot-blocking class INV-RF-03 exists to kill ("A failing integration must not block system startup" — `Architecture_Invariants_v1.md` §3/line 288). **The letter permits spawn-at-initialize; the purpose argues spawn-then-connect belongs in `run()` with the reconnect loop. This is a design-phase decision, not resolvable here** (→ §9 BLOCKING-2). Note also: whether core spawns at all is prior to this (child-of-core vs systemd sibling, §3.2 / §9 BLOCKING-3) — if the sidecar is a sibling unit, `initialize()` spawns nothing and the tension dissolves into pure connect-in-`run()`.
- **`run()`:** blocks on the WebSocket read loop on a virtual thread — the built NETWORK shape. Re-verified at source: `StandardIntegrationSupervisor.java` lines 540–542, `ioType() == IoType.SERIAL ? Thread.ofPlatform() : Thread.ofVirtual()`. The banked caveat re-verified as still true: the only real adapter descriptor in the tree declares `IoType.SERIAL` (`ZigbeeIntegrationFactory.java` line 133), so "the NETWORK path exists and is untested by any real adapter" `[banked: roadmap §6.1/line 184]` — a Matter sidecar adapter is its first customer (→ §9 NB-2).
- **`close()`:** idempotent close of the WS client is trivial; whether `close()` also kills the child process is the same custody question as spawn (§9 BLOCKING-3).
- **`commandHandler()`:** honest — commands map to WS RPCs; the adapter owns `command_result` on failure classes exactly as Doc 08 §3.10 step 7 assigns it ("The adapter publishes a `command_result` event…" — `08-zigbee-adapter.md` §3.10/line 390).
- **Descriptor walk:** `ioType=NETWORK` honest. `requiredServices` — the Zigbee precedent is to declare nothing the supervisor cannot yet honor ("declaring them would be a lie the supervisor cannot honor" — `ZigbeeIntegrationFactory.java` lines 127–129 comment); same discipline applies. `dependsOn` — empty (no integration depends on another; the sidecar is not an integration and `dependsOn` takes integration-type strings only, `integration-api/MODULE_CONTEXT.md` §Gotchas/line 221 — the descriptor has **no vocabulary for a non-integration process dependency**, which is itself a finding: EXT-INV-2's manifest doctrine says capability surface is declared statically, and a required external runtime is undeclarable today). `isolationLevel` — see §1.4.

### 1.2 (b) CHIP-JNI in-process

- **`initialize()`:** `System.loadLibrary` of `libCHIPController` is local-disk I/O, not network/serial — the letter of INV-RF-03 is satisfied. A load failure surfaces as `LinkageError` → PERMANENT under the classifier (`ExceptionClassifier`: "OOM/LinkageError→PERMANENT" — `integration-runtime/MODULE_CONTEXT.md` §M9.1/line 21) — honest FAILED-no-retry, the right shape. The tension: CHIP's controller init historically brings up network stack/BLE state; everything past library-load must be deferred to `run()` by design discipline the SDK was not built around `[A1 §1.2–1.3]` [inference].
- **`run()`:** the SDK is callback-driven; a blocking `run()` requires an adapter-owned event-pump/queue shim. The SERIAL-vs-NETWORK choice is itself awkward: `IoType` exists to protect the carrier pool from pinning ("SERIAL (platform thread, JNI pinning)" — `integration-api/MODULE_CONTEXT.md` §Enums/line 84); JNI downcalls from virtual threads pin carriers, so a JNI Matter adapter honestly wants the platform-thread treatment despite being network I/O — `IoType.SERIAL` would be semantically false, `IoType.NETWORK` operationally hazardous on the Pi's 4 cores (LTD-11's stated concern, `integration-api/MODULE_CONTEXT.md` §Constraints/line 187). **Named as a descriptor-vocabulary tension for the design phase; not resolved here.**
- **`close()`/`commandHandler()`:** frontable; PIE-bare discipline binds ("permanent-failure exceptions must arrive at the supervisor unwrapped… This rule generalizes to EVERY future adapter (Matter included)" — `integration-zigbee/MODULE_CONTEXT.md` §M9.2-Gotchas/line 283).

### 1.3 (c) CHIP-FFM in-process

Identical SPI walk to (b) — FFM downcalls pin virtual threads the same way JNI does [inference], the callback-pump shim is the same, and the shim itself must first exist ("no FFM bindings to CHIP exist anywhere" `[A1 §9 card (c)/line 322]`).

### 1.4 The `IsolationLevel` truth question (all options; the known tension, stated as ruled)

The enum at source (`integration-api/src/main/java/com/homesynapse/integration/IsolationLevel.java`, full file read): IN_JVM is "the MVP behaviour and the only supported level" (lines 21–23); RESERVED_SUBPROCESS is "Reserved for post-MVP subprocess isolation of misbehaving native/JNI adapters. The M9 supervisor rejects this value with UnsupportedOperationException at startup until a future amendment activates it; no code path may treat it as runnable (AMD-63-INV-01)" (lines 26–31). Enforced at source: `StandardIntegrationSupervisor.java` lines 166–170. The ruling: "wave-1 curated adapters remain `IN_JVM` behind the quality gate…; `RESERVED_SUBPROCESS` is the intended default rung for non-curated third-party code at marketplace time" (DP-18-B, `18-extension-and-plugin-architecture.md` §3.3/line 66; ratified §7/line 123).

- **(b)/(c):** `IN_JVM` is *literally* truthful — and the enum's own javadoc names "misbehaving native/JNI adapters" as the case the subprocess rung was reserved for. Declaring a JNI adapter IN_JVM is honest about topology while running exactly the risk class the reservation anticipates. TENSION, not violation.
- **(d):** `IN_JVM` truthful, DP-18-B wave-1 posture exactly. CLEAN.
- **(a):** the gap the dispatch predicted, confirmed: the Java adapter half runs IN_JVM, but the integration's *substance* runs in a process the descriptor cannot describe. `IN_JVM` mis-states the failure topology; `RESERVED_SUBPROCESS` is (i) rejected at startup at source and (ii) ruled as the *marketplace non-curated* rung — a first-party sidecar is neither. **Where an AMD would be needed (naming only, proposing nothing):** activating/extending AMD-63's slot, or a new descriptor vocabulary for first-party external-process topology, is amendment-class work under the recorded process; DP-18-B's honesty rule ("no sandbox or isolation language in any external copy before the enforcing rung exists" — Doc 18 §2/line 43) binds any external description of it. → §9 BLOCKING-1.

## §2 Crash isolation + supervision, per option

**INV-RF-01 exact text (the wall every option hits):** "Each integration… runs in a supervised, isolated execution context. A crash, hang, or resource exhaustion in one integration must not affect the core runtime, other integrations, or the event bus" (`Architecture_Invariants_v1.md` §3/line 268). And its under-read second paragraph: "The specific isolation mechanism (in-process with virtual thread supervision, out-of-process with IPC, container-level sandboxing) is an implementation decision… The Integration API must not expose or depend on the isolation mechanism" (line 270); "designed so that it can be implemented as in-process method dispatch today and replaced with IPC, gRPC, or Unix domain sockets tomorrow without breaking a single integration" (line 272). Its test: "Deliberately crash an integration (OOM, infinite loop, unhandled exception). Verify that other integrations continue operating" (line 278).

**What the supervisor IS today (M9.1 slice, source-confirmed shape):** registration in factory-list order (Kahn deferred), thread-per-adapter allocation, HEALTHY ↔ transient-restart-cycle → FAILED with descriptor-driven exponential backoff, grace-bounded close (10 s default), exception classification, the command router. The deferred-breadth list is explicit: "DEGRADED/SUSPENDED, the probe ladder, suspension cycles, heartbeat-timeout sweeps, window-rate evaluation, JFR/resource quotas, planned-restart suppression behaviors…, dependency-graph (Kahn) ordering, integration-scoped registry/query wrappers, AUTH_FAILED routing, and the health-score formula… are ALL deferred to the post-hero supervisor-breadth unit" (`integration-runtime/MODULE_CONTEXT.md` §M9.1-Gotchas/line 231). **Nothing anywhere in Doc 05 or the runtime supervises a process** — the reader-verified absence: "No ProcessBuilder, Runtime.exec, fork/exec, child-process, or external-process supervision text exists in the document" (Doc 05 sweep, §0/line 19: "fault isolation within a single JVM process"; §10.1/line 861 names process isolation as not-implemented).

### 2.1 (a) sidecar — blast radius CLEAN, a NEW failure-mode pair, and an inventoried machinery gap

A Node process dies without taking the JVM: INV-RF-01's blast-radius clause is *better* served than in-JVM options, and line 270 pre-authorizes the topology. The honest new cost is the **liveness split**: (i) *seam-alive-process-dead* — the WS connection can look healthy (buffered, half-open) while matterjs-server is dead or wedged; the bench already codified the general lesson as W5, "`isOpen()` lies… Health = disconnect listener OR read-error OR ASH-liveness" (`integration-zigbee/MODULE_CONTEXT.md` §M9.2-Gotchas/line 272) — the sidecar analog is ping/pong + protocol-level liveness, never socket state; and (ii) *process-alive-seam-dead* — Node runs but the WS layer is deadlocked; supervisor-side health sees only what the adapter reports (`SupervisorHealthReporter` is a write-through of adapter calls, `integration-runtime/MODULE_CONTEXT.md` §M9.1/line 22). Never-false-ALIVE (§51 register class) demands the adapter report health from *observed sidecar behavior*, not connection existence.

**Supervision machinery that does NOT exist (inventory, not design):** child-process spawn + custody; process-death detection (waitpid/exit-code observation) distinct from WS-drop detection; health-probe of an external process; restart choreography that composes with the ledger's in-flight commands — Doc 05 §3.14's planned-restart machinery (queue commands, suppress availability, 60 s timeout) is *itself* deferred breadth ("only the `plannedRestart` flag is carried" — `integration-runtime/MODULE_CONTEXT.md` §M9.1-Gotchas/line 231), and a sidecar restart is a second restart domain layered on an unbuilt first; crash-loop bounding for the child (the unit's `StartLimitBurst=5/300s` protects the JVM only — `homesynapse.service` lines 18–20); INV-RF-02 resource quotas for a process JFR cannot see (Doc 05 mechanism 3 is JFR-based, §3.5/line 248 — blind to Node; the cgroup alternative is §3.2's).

### 2.2 (b)/(c) CHIP in-process — say it plainly

A C++ segfault, abort, heap corruption, or std::terminate inside `libCHIPController` is not an exception the supervisor can classify — it is JVM death. The entire isolation stack (classification table Doc 05 §3.7/line 265, restart intensity, FSM) sits *above* the failure. The INV-RF-01 test at line 278 fails by construction for every co-resident integration: crash the Matter adapter's native half and the Zigbee adapter dies with the process. The actual supervisor becomes systemd (`Restart=on-failure`, `RestartSec=10` — `homesynapse.service` lines 61–62), i.e. the recovery grain becomes the whole hub including the event bus and the live fleet's ingestion. **VERDICT: VIOLATES INV-RF-01 as written** — the only reading that saves it is accepting whole-system restart as "isolation," which the invariant's addressed-failure-mode line forecloses ("Home Assistant's single-process architecture where one misbehaving integration can degrade the entire system" — line 276). Also unaccounted: native heap sits outside `-Xmx1536m` (LTD-01 flags, `boot-contract-map.md` line 14) but inside `MemoryMax=2G` (`homesynapse.service` line 66) — native allocation pressure manifests as cgroup OOM-kill of the JVM, another whole-process death mode [inference from the two limits].

**Machinery that does not exist for (b)/(c):** none can be built *in-process* that contains a native crash — the containment machinery for this class IS the subprocess rung (AMD-63's stated purpose). What could be built short of it: crash-only mitigations (fast restart, WAL integrity on reboot — already the Doc 12 §6.8 unclean-shutdown path), which shrink MTTR, not blast radius.

### 2.3 (d) pure Java — CLEAN

Java exceptions are exactly what the supervisor classifies; the TRANSIENT-default guard (Doc 05 §3.7/line 283) and restart intensity apply unmodified. The same deferred-breadth list applies to it as to everything (probes, quotas) — a shared, not option-specific, gap.

## §3 The process model (sidecar options): what a Node sidecar means, concretely

### 3.1 The single-process census (every load-bearing place the repo assumes one JVM process today)

1. **Doc 12 is a single-process lifecycle by construction.** "The HomeSynapse process moves through a fixed sequence of lifecycle phases" (§3.1/line 81); 7 phases with integrations at Phase 6 after `READY=1` (§3.8/line 226, C12-09/line 467); shutdown is one 12-step sequence ending "Process exits with code 0" (§3.9/lines 253–292); one JVM shutdown hook (line 244); one `main()` (§8.4/line 663). The reader's sweep verdict: "The document is uniformly single-process / single-JVM… No fork, exec, spawn, worker process, sidecar, or child process appears anywhere." The only multiplicity contemplated is multi-*instance* across hubs, explicitly deferred (§14/line 827).
2. **The systemd unit is one service, one ExecStart, hardened around one process.** `homesynapse.service`: `Type=exec` + `ExecStart=/opt/homesynapse/bin/homesynapse` (lines 23, 44); readiness via authed loopback probe because real sd_notify is deferred to M13 (lines 8–13; `boot-contract-map.md` §Health/lines 70–82 — `NoOpHealthReporter` is the live path); restart policy keyed to a single process's `ExitCode` contract with `RestartPreventExitStatus=10` (lines 58–63; `boot-contract-map.md` §Exit-codes/lines 86–92); `MemoryMax=2G` scopes the cgroup (line 66); the least-privilege block (`NoNewPrivileges`, `ProtectSystem=strict`, `SystemCallFilter=@system-service`, `PrivateDevices=yes` with the staged serial-DeviceAllow ramp, lines 69–93) was tuned for a JVM — a Node child inherits all of it if child-of-core, and needs its own copy if sibling. `RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6` (line 85) happens to admit a loopback WS, but that is luck, not design.
3. **The installer/image are one-artifact, fully-offline, JVM-only.** `install.sh` line 10: "Fully offline: the tarball carries the JRE + every jar; nothing is downloaded here"; the image is a bundled jlinked JRE + classpath jars (`boot-contract-map.md` §Entry/lines 17–18); the no-systemd fallback launches exactly one launcher (`install.sh` lines 83–90); `update.sh` "only swaps /opt/homesynapse; it never touches /var/lib/homesynapse" (`boot-contract-map.md` §Update/lines 104–107). A Node runtime (>=22.13, 64-bit only `[A1 §3.5/lines 151,154]`) breaks "the tarball carries the JRE + every jar" — the image build must additionally carry a pinned Node + the matterjs-server tree, per-arch (aarch64/amd64), or the deploy story imports a package-manager dependency the "fully offline" doctrine bans.
4. **Smoke/health assume one readiness surface.** `ExecStartPost=…health-probe.sh --wait --timeout 90` probes the one HTTP API (`homesynapse.service` line 50); install-smoke (`distribution/ci/install-smoke.yml`, `smoke/run-smoke.sh`) gates on that probe. A sidecar adds a second process whose readiness/liveness the smoke harness does not observe — the honest install-smoke delta is a sidecar-liveness leg, or smoke passes while Matter is dead (the vacuous-VERIFY class the bench doctrine repeatedly killed, e.g. DP-B3's vacuous-silence fix, `integration-zigbee/MODULE_CONTEXT.md` §M9.5-DURb/line 556).
5. **Observability is in-process.** JFR is the resource-monitoring mechanism (Doc 05 §3.5/line 248); logs are one journald stream under one `SyslogIdentifier=homesynapse` (`homesynapse.service` lines 96–98). A Node child's stdout lands in the same journal only if child-of-core; a sibling unit needs its own identifier + the operator now correlates two log streams.
6. **Deploy hygiene:** the bench deploy rules (exec-bit/LF-only) `[banked: dispatch §3; bench-repo rules not re-read this session — flagged §10]` extend to any sidecar launch scripts; `install.sh` already models the discipline (`install -m 0755`, line 61).

### 3.2 The topology decision (named, not made)

Two shapes exist, with different invariant surfaces: **child-of-core** (adapter or lifecycle spawns Node; inherits the sandbox and `MemoryMax` envelope; dies with the JVM — Doc 12's phase model must then say *where* in Phases 0–6 it starts and where in the 12 shutdown steps it stops; supervision is the §2.1 machinery) vs **systemd sibling unit** (its own unit + `After=`/`BindsTo=` topology, its own restart policy, its own hardening; core's adapter merely connects — supervision cost moves to packaging, and the adapter's honest posture on sidecar-down is FAILED-with-backoff exactly like an unplugged coordinator). Each resolves the §1.1 initialize-spawn tension differently. → §9 BLOCKING-3.

### 3.3 The honest statement

A Node sidecar is a **new class of runtime dependency — the first non-JVM production process** in the deployment. Every artifact in §3.1 items 1–5 assumes its absence today. None of these is a violation; all are enumerable work, listed per-option in §8.

## §5 Event-model mint discipline: a scope-of-mint census (not a design)

**How minting works here (as-built):** a new core event type = `@EventType(EventTypes.CONSTANT)` on the record + an `EventTypes` constant + the per-module manifest entry (M3.6c) + codec + `EventCategoryMapping` + count-pin tests, all in one change ("Mints ride the DUR coding WU in lockstep: EventTypes 71→73 · @EventType 41→43 · codecs 53→55 · EventCategoryMapping +2 · every count-pin test in the SAME change" — `01-event-model-and-event-bus.md` §AMD-99-pointer/line 15). Aggregation is the composition root's explicit `Stream.concat` of `EventTypes.CORE_PRODUCTION_EVENT_CLASSES` + `IntegrationEvents.LIFECYCLE_EVENT_CLASSES`; "classpath scanning and ServiceLoader are banned (enforced by ArchUnit Rule 3, noServiceLoader)" (`event-model/MODULE_CONTEXT.md` line 55). Integration-defined types ride the dotted namespace: "Integration-defined types follow a dotted namespace convention: `{integration_name}.{event_type}`" (Doc 01 §4.3/line 475), categories declared or defaulting per §4.4/line 621. Current census: **EventTypes 73 / core production classes 43 / codec+category pins 55** (`event-model/MODULE_CONTEXT.md` masthead/line 3).
