# Lane A4 Return — The Moat Translation: honest CONFIRMED / honest ALIVE on Matter

## §0 — Executive digest (≤1 page)

**The easier/subtler verdict, proven not inherited: for native, always-on Matter devices, honest confirmation is structurally EASIER than Zigbee — the banked "arguably stronger" inference [banked: roadmap return §3.2] is PROVEN for that class on three primary-cited legs — and it is SUBTLER in exactly three places, each of which would mint a false-CONFIRMED or false-ALIVE if the Zigbee posture were inherited naively.**

Easier, with citations: (1) **reports are versioned** — a per-cluster-instance `DataVersion` "SHALL be incremented if any attribute data changes" (Matter 1.4 Core §7.10.3 [VERIFIED-mirror]), so missed intermediate changes are *detectable* (DataVersion jumps >1 between reports), which no Zigbee attribute report offers; (2) **liveness is spec-bounded, not best-effort** — "If the subscriber does not receive a Report transaction within the maximum interval … the subscriber SHALL terminate the Subscribe interaction" (1.0+1.4 Core §8.5 [VERIFIED-mirror]), with empty keep-alive reports a defined transaction — versus Zigbee's Configure-Reporting max-interval, which whole device families (Xiaomi) silently reject [banked: Doc 08 §3.7]; (3) **a second evidence channel Zigbee lacks** — Events are node-scoped, "monotonically increasing for the life of the node … preserved across restarts," timestamped at creation, buffered, and gap-detectable by EventNumber (1.4 Core §7.14 [VERIFIED-mirror]); locks emit *Mandatory* `LockOperation`/`LockOperationError` events carrying attribution (1.0 Cluster §5.2.5 [VERIFIED-mirror]).

Subtler, each a named hazard class: (1) **Invoke SUCCESS is acceptance, not outcome** — the spec delegates SUCCESS to the cluster's Effect-on-Receipt; ColorControl mandates SUCCESS *even when the target is unachievable and gets clipped* (1.0 Cluster §3.2.11.1 [VERIFIED-mirror]); the SDK's DoorLock handler sends SUCCESS before actuation ("it may take a while before the lock actually locks/unlocks" — door-lock-server.cpp [VERIFIED-current]). The Shelly `was_on` lesson has an exact Matter analogue, per device type (§1.2). (2) **Report coalescing is real and now spec-sanctioned** — subscription reports are a *delta-set of paths carrying report-time values*, not a change queue (§8.5 wording + the SDK dirty-set reads values at encode time); and since cluster revision 6 (the 1.4 data model), `CurrentLevel` and `ColorTemperatureMireds` carry the Q ("Quieter Reporting") quality: intermediate values during transitions "SHOULD NOT be reported" (1.4 Core §7.7.8 [VERIFIED-mirror]; LevelControl.xml `quieterReporting="true"` [VERIFIED-current, lane-spot-checked]). The saving grace: the **end-of-transition settled value IS reported** — Q aligns with AMD-97's settled-value caveat rather than fighting it (§1.3). (3) **No cross-interaction ordering** — nothing in §8.x orders a subscription report against the invoke response that caused it; the confirming evidence can arrive *before* the ack (§1.4, a Phase-C ledger ruling, flagged BLOCKING in §7).

**ALIVE:** honest per-class staleness ceilings are *computable from negotiated protocol parameters* — a genuine upgrade over Zigbee's fixed heuristics — and are tabulated per class in §2.5: mains-native ≈ negotiated MaxInterval + MRP grace; SIT ICD reachable-on-demand ≤15 s; LIT ICD lawful silence up to IdleModeDuration ≤18 h (subscribed) or MaximumCheckInBackoff ≤64800 s (check-in); **bridged endpoints have NO spec-backed freshness bound at all** — the spec explicitly licenses `Reachable` to be wrong in both directions and delayed (1.6 Core §9.13.5.2 via @matter/types [VERIFIED-current]). The field corroboration is already banked cross-lane: matterjs-server #843, a subscription dying silently while the node "remains `available: true` … 14+ hours" [VERIFIED-current] — the ALIVE predicate must be *positive evidence with a timestamp*, never *absence of a timeout* (§2.2).

**The strongest candidate claim** (full set + obligations, §6): *"Every command renders an evidence-graded verdict — confirmed, unconfirmed, or failed — and every state tells you whether it is first-hand or relayed through a bridge."* The competitive verify (§3.3) returns **PARTIALLY SUPPORTED, safe in its careful wording**: no surveyed platform renders per-command verified-vs-assumed verdicts or evidence-graded state; every platform surfaces bridge *topology*; and one narrow counterexample exists (openHAB's Thing-status string "Bridge reports device as not reachable" [VERIFIED-current, lane-spot-checked]) — so the honest wording is "first-hand vs bridge-relayed *as a trust property of state*," never "nobody distinguishes bridges."

**Doctrine (§5):** the wire is gone as an evidence seam (CASE encryption; no key export; decryption requires test-mode firmware that destroys production fidelity) — the honest capture point moves to the controller's decoded interaction layer, and **evidence becomes testimony**: the stack now mediates what we record. Mitigations that keep the corpus honest: dual-layer capture (semantic stream + message-level debug log), stack+version-conditioned fixture provenance (the AMD-97 "reference-stack-conditioned" lesson transfers verbatim), ciphertext pcap for timing/MRP forensics, and a second-implementation virtual DUT as the anti-overfitting instrument. The corpus fidelity requirement is a **named input to the B1 stack choice**: chip-tool's RemoteDataModelLogger emits machine-readable records *with* DataVersion; matterjs-server's WS stream drops DataVersion and coalesces by design (§5.2).

**Nick's 2026-07-19 rider (cloud-forward/trendsetter) is absorbed structurally, not bolted on:** because every verdict is an immutable event on the Doc-17 substrate ("local-first, cloud, and AI are the same log read differently" [banked: Doc 17 §2.1]), evidence-graded confirmation and provenance-graded ALIVE stream unchanged into any future cloud/AI layer — cloud renders the log, it never invents state. The trendsetter opportunity is §6-C5.

## §1 — The CONFIRMED map, precisely

### 1.1 The command's three legs, and what each proves

Trace `on_off.turn_on` (or any actuating command) through a native Matter device:

**Leg 1 — Invoke over CASE.** The command rides an authenticated, encrypted CASE session on the operational fabric. What it proves when the exchange completes: *this specific node* (session-authenticated NOC identity) received a well-formed request. What it cannot prove: anything about execution. Transport reliability is MRP (retransmission with spec-pinned backoff constants — "4.11.8. Parameters and Constants" / "4.12.2.1. Retransmissions" via ReliableMessageMgr.cpp [VERIFIED-current: https://raw.githubusercontent.com/project-chip/connectedhomeip/master/src/messaging/ReliableMessageMgr.cpp, fetched 2026-07-19]). **Lifecycle mapping: this leg is `command_dispatched` territory** — dispatch evidence, DIAGNOSTIC, exactly as Doc 08 §3.10 leg 5 [banked: Doc 08 §3.10].

**Leg 2 — the per-command status response.** The InvokeResponse carries either a CommandDataIB (cluster-specific data response) or a CommandStatusIB (plain status) per command path (1.0 Core §8.8.3.1 [VERIFIED-mirror: https://raw.githubusercontent.com/kafeibujiekun/matter-app/master/ Matter 1.0 core PDF, fetched 2026-07-19]). The load-bearing spec fact: **the core spec never defines SUCCESS as effect-achieved** — "The command SHALL be executed as defined in the cluster specification" (1.0 §8.8.2.3 / 1.4 identical [VERIFIED-mirror: https://raw.githubusercontent.com/KULeuven-COSIC/whats-the-matter/main/Matter-1.4-Core-Specification.pdf, fetched 2026-07-19]), and "SUCCESS 0x00: Operation was successful" (1.0 §8.10.1) — the meaning of "operation" is whatever the cluster's Effect-on-Receipt says, which for actuated and transitional effects is *initiation*, not completion (per-type wording, §1.2). **Lifecycle mapping: Leg 2 is the dispatch-ack — `command_result` SUCCESS/FAILURE**, the exact semantic slot the ZCL Default Response occupies in Doc 08 §3.10 legs 6–7. The rejection vocabulary is *richer* than ZCL's and maps cleanly onto `NACK_TO_FAILED`: UNSUPPORTED_COMMAND 0x81, INVALID_COMMAND 0x85 ("Command not carried out"), CONSTRAINT_ERROR 0x87, BUSY 0x9C, INVALID_IN_STATE 0xCB, NEEDS_TIMED_INTERACTION 0xC6, FAILSAFE_REQUIRED 0xCA, TIMEOUT 0x94 (1.0 §8.10.1 + 1.4 additions [VERIFIED-mirror]; machine-readable: src/protocols/interaction_model/StatusCodeList.h [VERIFIED-current]). A definite rejection status is honest `FAILED` — evidence, not assumption.

**Leg 3 — the subscription report carrying post-state (+ DataVersion increment).** The device-originated report on the standing subscription delivers the authoritative attribute's new value; the cluster instance's DataVersion "SHALL be incremented if any attribute data changes" and reports "SHALL include the latest … DataVersion associated with each cluster where there are data changes" (1.4 Core §7.10.3, 1.0 §8.5.3.4 [VERIFIED-mirror]). **Lifecycle mapping: each report datum is `state_reported` evidence; the Pending Command Ledger's expectation match earns `state_confirmed`** — Doc 01 §3.2/§3.8 absorb this without a single vocabulary change [banked: Doc 01 §3.2, §3.8]. The Matter-specific enrichment worth recording in the `state_reported` payload (a mint-discipline question for Phase C, §7): the DataVersion observed with the value — it is the per-cluster missed-change detector (a DataVersion delta >1 across consecutive reports proves suppressed intermediates; noisy — cluster-scoped, randomly initialized, wraps — but real; [inference — arithmetic on the §7.10.3 SHALLs, stated as such]).

**What earns what — the AMD-97 sentence for Matter:** Invoke-SUCCESS is *never* confirmation evidence (AMD-97-INV-01 composes unchanged: a `DISABLED`/`UNCONFIRMABLE` path never renders CONFIRMED [banked: AMD-97 §Register]); only Leg-3 report evidence matching a registered expectation earns `state_confirmed` → `CONFIRMED`. This is byte-for-byte the Zigbee posture (`CONFIRMED only ever renders on genuine report evidence` [banked: integration-zigbee MODULE_CONTEXT §ConfirmationOverrideInstaller]) — Matter changes the *quality* of Leg 3's evidence (versioned, liveness-bounded, session-authenticated), not the rule.

### 1.2 Per-device-type: where the `was_on` lesson lands

The Shelly `was_on` lesson — an acceptance ack is not an outcome — has a *spec-citable analogue per device type*, and the spec's own wording sorts the V1 classes by hazard:

- **OnOff (plug/switch/bulb-on-off): the benign class.** "On receipt of the On command, a server SHALL set the OnOff attribute to TRUE" (1.0 Cluster §1.5.7.2.1 [VERIFIED-mirror: kafeibujiekun mirror, 1.0 App Cluster PDF, fetched 2026-07-19]) — attribute change is on-receipt and unconditional, so Leg-2 SUCCESS *coincides* with the attribute-level effect. Two residual honesty gaps: (a) attribute-level ≠ physical-load-level (same as Zigbee; the report confirms the device's belief, not the relay's contacts — tolerated identically today); (b) `OnWithTimedOff` can be *discarded* silently when `AcceptOnlyWhenOn` applies ("the command SHALL be discarded," §1.5.7.6.4 [VERIFIED-mirror]) — a discard's status rendering is not spec-explicit in our extract → hazard note for the bench (measure it, §4).
- **LevelControl (bulb dimming): the transition class.** "The movement SHALL be as continuous as technically practical … the time taken to move to the new level SHALL be equal to the value of the TransitionTime field" (1.0 Cluster §1.6.6.1.1 [VERIFIED-mirror]); SUCCESS returns immediately; `RemainingTime` counts down; and `CurrentLevel` is a Q attribute since cluster rev 6 (§1.3). Confirmation must key on the settled end-of-transition value — Doc 08 §3.10's transition rule (`transition_time + 2 s` timeout adjustment [banked: Doc 08 §3.10]) transfers with a Q-cadence margin (§4 seeds).
- **ColorControl (CT): the clip-with-SUCCESS class — the sharpest analogue.** "As long as the color data fields … are within the permitted range … the resulting status code SHALL be SUCCESS. For example the MoveToColorTemperature command: if the target color temperature is not achievable by the hardware then the color temperature SHALL be clipped at the physical minimum or maximum achievable" (1.0 Cluster §3.2.11.1 [VERIFIED-mirror]). **A SUCCESS ack with a lawfully different outcome is spec-mandated behavior.** A naive `WithinTolerance(target)` expectation false-FAILs on clip; a naive trust-the-ack false-CONFIRMs. The expectation shape needs clamp-awareness (confirm against the device's own reported settled value when clipping is in play, with the honest verdict recording target-vs-achieved) — an expectation-vocabulary question for Phase C (§7, NON-BLOCKING; the ±50 K TOLERANCE band already absorbs sub-clip drift [banked: AMD-97 §Doc 02 edits]).
- **DoorLock (later wave): the actuation class, and the showcase.** Lock/Unlock/Toggle *require Timed Invoke* (Access "TO", 1.0 Cluster §5.2.4; current: DoorLock.xml `timed="true"` [VERIFIED-current: https://raw.githubusercontent.com/project-chip/connectedhomeip/master/data_model/1.4/clusters/DoorLock.xml, fetched 2026-07-19]) — which narrows the *acceptance window* and adds nothing about outcome ("prevent … 'intercept, interfere, and replay' … can be thought of as a 2-phase commit," 1.0 Core §7.6.10 [VERIFIED-mirror]). The SDK is explicit that status precedes actuation (door-lock-server.cpp ~3717: "The app should trigger the lock state change as it may take a while before the lock actually locks/unlocks" [VERIFIED-current: https://raw.githubusercontent.com/project-chip/connectedhomeip/master/src/app/clusters/door-lock-server/door-lock-server.cpp, fetched 2026-07-19]). The outcome channel is dual: `LockState` (nullable — "a lock may not know the LockState status after a power cycle until the first lock actuation is completed"; includes `NotFullyLocked`, 1.0 Cluster §5.2.3.1 [VERIFIED-mirror]) plus the *Mandatory* `LockOperation`/`LockOperationError` events with CRITICAL priority on unlock (§5.2.5 [VERIFIED-mirror]). Hazards found at source: a transition to `NotFullyLocked` emits NO event (SDK comment: "kNotFullyLocked has no appropriate event to send" [VERIFIED-current]) — attribute-report evidence is mandatory, events alone insufficient; and `AutoRelockTime` re-locks with no new command (§5.2.4.2 [VERIFIED-mirror]) — a lawful post-CONFIRMED state change the supersession/expiry vocabulary already models [banked: Doc 08 §3.6 caveat 3, AMD-98].

### 1.3 Where Matter is structurally STRONGER than Zigbee (each verdict cited)

1. **Versioned evidence.** DataVersion per cluster instance, incremented per change, reported latest (§1.1 Leg 3 citations). Zigbee attribute reports carry no version; a suppressed intermediate is *invisible* on Zigbee and *arithmetically detectable* on Matter. [VERIFIED-mirror: 1.4 Core §7.10.3 + 1.0 §8.5.3.4]
2. **Spec-bounded push liveness.** "Attribute changes SHALL be delivered as soon as possible, taking into account the minimum interval," keep-alive at max interval, subscriber-SHALL-terminate on miss (1.0+1.4 Core §8.5 [VERIFIED-mirror]). The Zigbee comparator is Configure-Reporting best-effort with device families that reject it outright and report on firmware schedule [banked: Doc 08 §3.7 exclusions]. On Matter the reporting *posture* is negotiated per subscription, not begged per device.
3. **A second, queue-semantics evidence channel.** "Events SHALL always be queued and buffered. Each Report containing events SHALL deliver queued events without reordering the queue" (1.4 Core §8.5 [VERIFIED-mirror]); node-scoped monotonic EventNumber surviving restarts, creation timestamps (1.4 Core §7.14.x [VERIFIED-mirror]). Zigbee has no attributable, numbered, gap-detectable event stream; Matter's maps naturally onto our `event_time` vs `ingest_time` distinction and the origin model (LockOperation's OperationSource/UserIndex is protocol-level PHYSICAL/USER evidence of exactly the kind Doc 01 §3.9 demands [banked: Doc 01 §3.9]). Caveat kept honest: the *counter* survives reboot by SHALL; buffered *records* are RAM-circular in the reference SDK [VERIFIED-current: EventManagement.h] — events are an evidence enhancer, not a durable log we can lean on.
4. **Strict report serialization.** "Each Report transaction initiated by the publisher SHALL complete successfully before another Report transaction is initiated" (1.0+1.4 §8.5 [VERIFIED-mirror]) — per-subscription ordering is guaranteed, so per-subject `state_reported` ordering derives cleanly.
5. **Richer, honest rejection vocabulary** (§1.1 Leg 2) — more `FAILED`-with-reason, less ambiguous timeout.

### 1.4 Where Matter is WEAKER/SUBTLER (each verdict cited) — the false-CONFIRMED hunting ground

1. **Report coalescing: the delta-set model (the lane's highest-value finding, confirmed in both spec and SDK).** The spec's reporting unit is *the changed path carrying current data*: "Each Report transaction … SHALL report the path for each delta change in the subscription data, including the attribute data that has changed … since the last Report transaction" (1.4 §8.5 [VERIFIED-mirror]) — and events get explicit queue language while attributes get none (the spec's own asymmetry signal). The SDK realizes it exactly: the dirty set stores paths+generations, never values (reporting/Engine.h: "mGlobalDirtySet is used to track the set of attribute/event paths marked dirty" [VERIFIED-current]); values are read from the data model *at encode time* (Engine.cpp RetrieveClusterData issues a fresh read [VERIFIED-current]); under pool pressure paths merge *coarser* (MergeOverlappedAttributePath [VERIFIED-current]). **Consequences:** N changes in a min-interval window render as ONE report with the latest value; an A→B→A flap renders as a dirty-path report carrying A. For never-false-CONFIRMED the direct risk is bounded — confirmation keys on the settled value, and AMD-97 caveat 2 (settled-value tolerance) transfers verbatim [banked: Doc 08 §3.6] — but two derived hazards are real: (a) **min-interval suppression delays the confirming report** — timeout seeds must add the negotiated MinInterval (§4); (b) **automation-trigger fidelity ≠ confirmation fidelity** — transient states can lawfully never surface; nothing downstream may infer "didn't happen" from "wasn't reported" (a §-worthy line for Doc-19's event-model section).
2. **Q-quality: spec-sanctioned intermediate suppression (new since the banked base).** The Q quality ("Quieter Reporting") entered the core spec at 1.3 (§7.7.8: changes "SHOULD NOT be reported" outside the attribute's declared conditions [VERIFIED-mirror: 1.3/1.4 core]), and was *applied* to `CurrentLevel`, `RemainingTime`, `CurrentFrequency` at cluster rev 6 and to `ColorTemperatureMireds` + hue/sat/XY at ColorControl rev 7 — both in the **1.4 data model** (LevelControl.xml/ColorControl.xml revision notes + `quieterReporting="true"` [VERIFIED-current, lane-spot-checked 2026-07-19]). The SDK's rendering of the rule: report "At most once per second, or at the end of the movement/transition, or when it changes from null to any other value and vice versa" (v1.4-branch level-control.cpp SetCurrentLevelQuietReport comment [VERIFIED-current] — IMPLEMENTATION-OBSERVED; the per-attribute SHOULD/SHALL strength is triangulated, not PDF-verbatim: §8). **Verdict: Q is a friend wearing a hazard's clothing** — the end-of-transition settled value is reported, which is precisely what the ledger confirms against; the hazard is only in seeds and in trigger-fidelity expectations. `OnOff` and `LockState` are NOT Q ([VERIFIED-current: OnOff.xml, DoorLock.xml `reportable="true"`]) — the discrete hero paths keep per-delta reporting.
3. **Status-vs-effect races: no cross-interaction ordering.** No clause in 1.0/1.4 §8.x orders a subscription ReportData against an InvokeResponse on a different exchange; the SDK marks the attribute dirty *during* command handling while the response encodes separately [inference — absence after targeted grep of both core specs, + SDK structure; stated as absence-of-evidence, §8]. **The confirming `state_reported` can precede `command_result`.** Doc 01 §3.8's lifecycle (ack, then match) must rule this path in Phase C: hold early-arriving matching reports against the pending expectation rather than requiring ack-first (BLOCKING question, §7).
4. **The idempotent no-change trap is spec-derivable on Matter.** Reports carry *deltas only* — a command whose target equals current state changes nothing, so NO report comes, by construction (§8.5 delta language [VERIFIED-mirror]). AMD-97 caveat 1 (idempotent commands confirm from cache/readback, never report-waiting [banked: Doc 08 §3.6]) transfers from measured-Zigbee-lesson to spec-fact — it must ship in the Matter adapter's V1, not arrive as a field surprise.
5. **The evidence path is stack-mediated** — on Zigbee we owned the wire; on Matter every report reaches us through the chosen stack's decode+subscription machinery, whose failure modes (silent subscription wedge, #843-class) masquerade as device behavior. This is §2/§5's territory; it is listed here because it bounds Leg-3 evidence quality per stack option.

## §2 — The ALIVE map

### 2.1 Subscription liveness semantics: what the max interval actually is

The max interval is a **negotiated staleness ceiling with spec teeth on both sides**: the publisher keeps the subscription alive by reporting at least every max interval ("To keep the subscription alive, a Report transaction is sent from the publisher every maximum interval, or possibly more frequently"; the empty report is a defined transaction — 1.0 Core §8.6.2 [VERIFIED-mirror]), and the subscriber SHALL terminate on a miss (§8.5 [VERIFIED-mirror]). The publisher may raise MaxInterval up to `MAX(SUBSCRIPTION_MAX_INTERVAL_PUBLISHER_LIMIT, MaxIntervalCeiling)` where the SDK pins the spec constant: `kSubscriptionMaxIntervalPublisherLimit = 3600; // seconds (60 minutes)` and "For an ICD publisher, this SHALL be set to the idle mode duration" (ReadHandler.h:57, :309–315 [VERIFIED-current: https://raw.githubusercontent.com/project-chip/connectedhomeip/master/src/app/ReadHandler.h, fetched 2026-07-19 — lane-spot-checked; this is the banked base's own file, re-verified ↻]). The controller-side timeout is MaxInterval **plus an MRP-derived round-trip grace**, not a fixed margin (`ComputeLivenessCheckTimerTimeout` … `Seconds16(mMaxInterval) + roundTripTimeout` — ReadClient.cpp [VERIFIED-current]); on expiry `OnLivenessTimeoutCallback` fires, CASE sessions are marked defunct, and resubscription follows Fibonacci backoff [VERIFIED-current: ReadClient.cpp]. HA's production policy (IMPLEMENTATION-OBSERVED comparator): ceilings of 60 s WiFi/Thread, 600 s battery; unavailable after 2 failed resubscribe attempts; presumed powered-off after 30 min ("typical TTL of mDNS") [VERIFIED-current: python-matter-server device_controller.py].

**What a subscription drop proves:** no ReportData reached *this subscriber* over *this session* within the bound — a property of path + session + publisher scheduler. **What it does not prove:** device death. Live causes include IPv6/TBR path failure, CASE session eviction, device reboot (volatile subscription state), mDNS staleness, and *subscriber-side* processing stalls [inference — enumerated from the verified mechanics; each mechanism cited in the agent dossier]. Conversely a *received* report — including the empty keep-alive — is positive, session-authenticated, timestamped liveness evidence. **The mapping to our model: ALIVE is earned by received evidence; a drop earns UNKNOWN-then-probe, never DEAD** — the exact posture of the Zigbee ping ladder (mains: active read if silent 10 min; battery: 25 h passive window [banked: Doc 08 §3.11]) with better raw material: the ceiling is per-device negotiated instead of fleet-fixed, and the keep-alive gives mains-class devices a *push* heartbeat Zigbee never had.

**Reboot and resumption:** device-side subscription persistence/resumption across reboot is an **optional, build-flag-gated CHIP feature** (`CHIP_CONFIG_PERSIST_SUBSCRIPTIONS`, SubscriptionResumptionStorage.h [VERIFIED-current]) — not verified as spec-mandated (§8) — and matter.js implements no device-side equivalent (grep-verified absence in @matter/protocol 0.17.6 [VERIFIED-current]). The `StartUp` event ("SHALL be generated by a Node as soon as reasonable after completing a boot or reboot process," 1.6 Core §11.1.6.1 via @matter/types [VERIFIED-current]) is the reboot-evidence line our availability projection can consume directly.

### 2.2 The #843 class: the instrument can lie ALIVE

matterjs-server #843 (open, 2026-07-10): "A node's active subscription can stop delivering reports while the server's client-side subscription timeout never fires. … The node remains `available: true` with frozen attribute state indefinitely (observed: 14+ hours)" [VERIFIED-current: https://github.com/matter-js/matterjs-server/issues/843, fetched 2026-07-19]; the reporter's mechanism hypothesis (`isReading` exempting a wedged subscription from its own timer) is corroborated at source in @matter/protocol 0.17.6 ClientSubscriptions.ts [VERIFIED-current], no maintainer confirmation at fetch time. HA field reports show the same *symptom family* on the CHIP stack (addons#4286 liveness timeouts with missed events; core#159629/#161886 devices stuck unavailable/unavailable-after-outage [VERIFIED-current]). **The doctrine consequence: never derive ALIVE from the stack's own availability flag.** Our availability projection must compute from *timestamped evidence records* (last report/keep-alive/check-in received at T) with the ceiling arithmetic in our code — "timer armed and last report at T" as the predicate, the stack's `available` field demoted to a diagnostic. This is the composed-behavior-under-state-history defect class the doctrine §4 names [banked: doctrine §4], arriving via a third-party instrument.

### 2.3 Reachable: testimony, not evidence

Basic Information `Reachable` (0x0028/0x0011) is optional and "For a native Node this is implicitly True" (1.6 Core §11.1.5.18 via @matter/types [VERIFIED-current]) — informationless for natives. The bridged variant is §3's frontier: bridge-computed, error-licensed both directions, delay-licensed, no freshness bound (1.6 Core §9.13.5.2 [VERIFIED-current]). Neither feeds ALIVE directly in our model; both are recordable testimony (`state_reported` on a diagnostic attribute) that can *corroborate* but never *earn* availability.

### 2.4 ICD/sleepy devices, version-pinned ↻

Cluster history re-verified against the CSA-generated data-model XMLs shipped in the SDK [VERIFIED-current: data_model/{1.2,1.3,1.4,1.5}/clusters/ICDManagement.xml, fetched 2026-07-19]: **1.2** introduces ICD Management (0x0046, rev 1, CheckInProtocolSupport only) · **1.3** adds LIT/UAT/check-in — all three features `provisionalConform` · **1.4** de-provisionalizes (rev 3), adds `MaximumCheckInBackoff` + DSLS (dynamic SIT↔LIT); CSA's 1.4 announcement names LIT + Check-In as the headline ("a new Check-In Protocol to ensure reliable communication" [VERIFIED-current: csa-iot.org 1.4 newsroom]) · **1.5/1.6** leave the cluster at rev 3 (announcements name no ICD changes [VERIFIED-current]). ## §3 — Bridged endpoints: the honesty frontier

### 3.1 Mechanics of bridge attribution

A bridge is an Aggregator device type (0x000E) whose Descriptor PartsList enumerates bridged endpoints; each bridged device is a Bridged Node endpoint ("represents a device on a foreign network, but is not the root endpoint of the bridge itself" — Device Library §2.5 via @matter/node [VERIFIED-current]) carrying the Bridged Device Basic Information cluster (0x0039) with `Reachable` **mandatory** and the `ReachableChanged` event ("shall be generated when there is a change in the Reachable attribute … After (re)start of a bridge this event may be generated" — 1.6 Core §9.13.7.2 [VERIFIED-current: @matter/types]). The spec's own honesty about the attribute is the frontier's charter: "Determination of reachability might not be perfect … the Matter Node SHOULD be aware of the risk of false positives and negatives … detection … may be delayed due to the technique employed (e.g. detecting that a number of expected messages from the bridged device did not arrive)" (1.6 Core §9.13.5.2 [VERIFIED-current]). 1.4 adds `KeepActive`/`ActiveChanged` for bridged ICDs [VERIFIED-current: ZAP XML]. **Attestation attaches to the bridge**: commissioning, NOC, and the DAC exchange exist only at node scope (the bridge's endpoint 0); the data model defines no per-bridged-endpoint attestation or operational-credentials cluster [inference — structural, from the verified device-type/cluster inventory; the spec's own prose sentence on bridged non-attestation was NOT directly fetched — §8]. The SDK's reference bridge *simulates* Reachable ("the `reachable` attribute is simulated" — examples/bridge-app README [VERIFIED-current]). Bridged share of installed base remains unquantified [banked: roadmap §3.2, unchanged].

### 3.2 The honest per-endpoint posture in AMD-97 vocabulary — OPTIONS (the design rules; evidence here)

The vocabulary absorbs bridges without a new verdict value — the question is where the bridge-attribution fact lives. Three options, each with the evidence that prices it:

- **O1 — a provenance axis (recorded fact, not a verdict cap).** Add an evidence-provenance field per entity/characterization entry (`NATIVE | BRIDGE_ATTESTED`, riding the corpus `provenance` block that already exists [banked: bench IR schema §2]) and render it in the read-API/UI. Confirmability stays *measured per capability*: a Hue-behind-bridge that reliably reports its authoritative attribute IS `CONFIRMABLE` — the report evidence is real; it is one attested hop weaker in provenance. Evidence for: the report path through a bridge is still device-originated data relayed under the bridge's CASE session; punishing it to BEST_EFFORT would discard measurable, corpus-provable confirmation behavior (the Zigbee corpus already proves the same physical devices confirm reliably). Evidence against: provenance becomes a second axis the UI/read-API must carry (frozen v1.1 surface question → contract conversation, not a raw workaround [banked: bench charter §5 rider]).
- **O2 — ceiling cap: bridged endpoints max out at BEST_EFFORT.** Simplest honesty guarantee (never CONFIRMED on relayed evidence). Evidence against: it conflates *evidence quality* with *evidence provenance* — the corpus would record measured VERIFIED_REPORTS behavior the verdict then refuses to honor; and it makes the marketing claim weaker (we'd be hiding a distinction, not surfacing it). The S31 precedent cuts *against* O2: the downgrade there was for genuinely unverifiable seeds, with a sanctioned re-upgrade on evidence [banked: integration-zigbee MODULE_CONTEXT DP-WL-6] — bridged reports are verifiable evidence.
- **O3 — split axes: confirmation per measured behavior (O1's rule), ALIVE strictly provenance-gated.** Bridged `Reachable=true` NEVER feeds ALIVE (it is testimony with no freshness bound — §2.3/§2.5); leaf availability renders only as "bridge-attested reachable as of T" + relayed-report recency; `ReachableChanged=false` maps to an honest degrade (availability UNKNOWN-by-bridge-testimony, distinct from our-evidence offline; commands to an endpoint the bridge attests unreachable → immediate honest UNCONFIRMED with reason recorded, INV-SA-03 style [banked: AMD-97 §Doc 02 edits]). Bridge-node unreachable → all its endpoints degrade together (one cause, recorded once, fanned honestly).

[inference — option construction is the lane's; every load-bearing input above is cited. The lane notes O1+O3 compose and O2 conflicts with the measured-evidence thesis, but the design rules in Phase C.]

## §5 — The corpus/bench doctrine for Matter

### 5.1 The wire is gone; name it and move the seam

Wire pcap of production Matter traffic is not a practical evidence seam: payloads are CASE-encrypted; the official matter-dissector decrypts only with manually entered keys or `CHIP_CONFIG_SECURITY_TEST_MODE` well-known keys, is "experimental status … may have bugs that may cause Wireshark to crash," Linux-only [VERIFIED-current: github.com/project-chip/matter-dissector README]; mainline Wireshark parses headers only ("[Encrypted Payload]" otherwise) [VERIFIED-current: gitlab.com wireshark MR !9865]; there is **no session-key export facility** in chip-tool/SDK — the documented community workflow is firmware patching, and "Decryption for anything incorporating a nonce (i.e. production devices) it is just not implemented" [community-reported: connectedhomeip#33094]; vendor guidance says rebuild with test mode, "MUST NEVER BE ENABLED IN PRODUCTION BUILDS" [VERIFIED-current: docs.silabs.com matter-wireshark]. Test-mode firmware destroys the production-faithful-DUT property the capture exists to record — so for retail DUTs the wire yields only ciphertext. **Keep the ciphertext pcap anyway**: message headers, sizes, timing, and MRP retransmission counters are cleartext — the timing/forensics side-channel that partially restores wire-class custody [inference — from the header-parsing evidence above].

### 5.2 The honest capture point per A1 stack option (evidence; B1 consumes)

The capture seam moves to the controller's decoded interaction layer, and each stack option exposes a different-quality record — **corpus fidelity is hereby a named B1 input**:

- **(a) matterjs-server sidecar:** the WS push stream is semantically useful but *lossy as evidence*: `attribute_updated` is `[node_id, "endpoint/cluster/attr", value]` — **no DataVersion, no timestamps** [VERIFIED-current: ws-controller WebSocketControllerHandler.ts:366]; `node_updated` is coalesced by design ("can be triggered several times for one logical change … share the per-node coalescing key" [VERIFIED-current: source comments]); invoke request/response pairs never appear on the push stream (responses go only to the calling client) [VERIFIED-current: event-enum absence]; `node_event` DOES carry event_number + device timestamps [VERIFIED-current: model.ts]. The decode layer beneath sees more: matter.js's `DecodedAttributeReportValue` carries `version` (DataVersion) [VERIFIED-current: AttributeDataDecoder.ts:46], and `MATTER_LOG_LEVEL=debug` logs every message with exchange metadata + payload bytes [VERIFIED-current: MessageExchange.ts/MessageCodec.ts; cli.md `--log-file` rotation]. **Fixture posture for (a): dual-layer capture (WS stream + debug message log), with the WS layer treated as the semantic record and the message log as the audit layer** — and a standing option to upstream a DataVersion-bearing WS event (the schema-version seam exists: server schema 12 with `require_schema` gating [banked: A1 §3.3]).
- **(b) CHIP (chip-tool / JNI):** the strongest machine-readable record today — `RemoteDataModelLogger` emits JSON with `clusterId/endpointId/attributeId/dataVersion/value` for reports, command responses, events, and errors [VERIFIED-current: examples/chip-tool RemoteDataModelLogger.cpp]; human DMG logs pretty-print full IM payloads incl. `DataVersion = 0x…` [VERIFIED-current: AttributeDataIB.cpp]. Caveat: the interactive/WebSocket server around it is community-reported unstable ("more or less not thought to be asynchronous"; segfault reports) [community-reported: connectedhomeip#32478].
- **(c/d) pure Java (if ever):** the only option that restores wire-class custody — we would own the decode, so the fixture IS our observation again. Priced elsewhere as multi-arc [banked: A1 §0/§5]; recorded here only as the custody consequence.

**The trust consequence, stated per the dispatch: evidence becomes testimony.** Whatever stack we choose, a decode/subscription/liveness bug in it is *baked into fixtures and indistinguishable from device behavior* without a second witness (§2.2's #843 is the live exhibit). Mitigations the doctrine adopts: (1) dual-layer capture per option above; (2) **fixture provenance grows a stack conditioning field** — stack name + version + schema pinned per capture, because the corpus's Zigbee lesson ("latency bands are reference-stack-conditioned" [banked: AMD-97 §Consumption]) transfers verbatim and worsens: server versions change event ordering across releases [VERIFIED-current: matterjs-server CHANGELOG 0.5.x ordering fixes]; (3) a **second-implementation cross-check instrument** — replay the same DUT stimulus under CHIP and matter.js and diff the decoded records; divergence = instrument finding, not device finding [inference — the discriminator pattern applied to stacks].

### 5.3 What a "Matter fixture" is (the seeded-log architecture, served)

The seeded-log substrate consumes `state_reported`-level events and is protocol-agnostic by construction [banked: Doc 01 §3.2; bench IR schema §4 "the captured stream is a seeded event log"] — nothing changes downstream. The Matter fixture is the *upstream* artifact: a captured, decoded interaction-record stream — attribute reports (path, value, DataVersion where the layer exposes it, receipt timestamp), invoke request/status pairs (command path, status code, timed-invoke flag), node events (EventNumber, priority, device timestamp, receipt timestamp), and subscription lifecycle records (establish/parameters/keep-alives/drops) — normalized into the corpus IR exactly as Path B does for Zigbee [banked: bench IR schema §4], with `provenance.source=BENCH_CAPTURE` plus the new stack-conditioning fields (§5.2). Two Matter-specific fixture rules: record the **negotiated Min/MaxInterval** with every capture (it conditions every latency number); segment **session-establishment time from command time** (CASE setup 2–30 s must never contaminate a confirm envelope — §4 cross-cutting).

`import_test_node` — present in BOTH servers, verified at source ("Import test node(s) from a HA or Matter server diagnostics dump," node ids ≥ reserved test range [VERIFIED-current: python-matter-server device_controller.py; matterjs-server TestNodeCommandHandler.ts:123]) — is a real controller-layer fixture-INJECTION seam: it seeds *state snapshots* (attribute maps), not behavior streams [inference — from the model shape], so its bench role is interview/registry/projection tests (adopt-against-a-known-node, bridge-topology rehearsal), not confirmation-latency replay. Confirmation replay stays at our seeded-log layer, where it already lives.

### 5.4 The virtual-DUT option ↻

`chip-all-clusters-app` is upstream's own certified-testing instrument: "Certification tests are all integration tests"; the app "serves as a reference example application that can function as a device under test (DUT)"; the CSA Test Harness itself runs it when the DUT is a controller [VERIFIED-current: project-chip.github.io integration_tests.html + certification-tool TH User Guide]. Fidelity: cluster *logic* fidelity is high (same SDK cluster code real CHIP-based products ship [inference — same repo, reference role]); timing fidelity is low (desktop process — no radio, no sleep, no flash stalls [inference]); and upstream itself treats its composition as load-bearing and *changes it* (endpoint-layout change breaking downstream tests [community-reported: connectedhomeip#34374]); session-layer CI flake exists [community-reported: #11440, #39360]. matter.js ships virtual device examples pairable with any ecosystem [VERIFIED-current: @matter/examples README] — the second-implementation DUT for §5.2's cross-check. **Bench role verdict: virtual DUTs are instrument-validation and protocol-behavior scaffolding (cheap, scriptable, pre-hardware), never a substitute for retail-silicon capture — the corpus's ground-truth rung stays physical** [inference — doctrine §1's evidence-capital thesis applied]. The upstream python_testing/Mobly framework and YAML suites are reusable against physical products ("you'll be prompted to manually reboot or factory reset the device" [VERIFIED-current: python.html]) — a free stimulus library for bench operator scripts, with the caveat that many CI tests need test-event-triggers retail devices won't honor [inference — flagged in the harness docs].

### 5.5 The ratchet, day one

The field-incident→scenario pipeline needs, on day one of any Matter bench: (1) the dual-layer capture running *continuously* with rotation (the S31/power-cut lesson: evidence must predate the incident [banked: doctrine §1–2]) — matterjs-server's `--log-file` + 24 h rotation ×7 [VERIFIED-current: cli.md] or the chip JSON logger equivalent; (2) the ciphertext pcap ring buffer on the bench VLAN (A2's isolation design); (3) the scenario format extended with Matter stimulus vocabulary (commission, invoke, subscription-drop, bridge-reachable-flap, ICD check-in) — a SCENARIO_FORMAT delta the B1+ runner inherits [banked: bench charter B0/B1]; (4) the #843 class as the FIRST authored Matter scenario family: wedge-the-instrument, assert our availability projection renders honest staleness while the stack still says `available: true` — the fixture-paired assert proving our ALIVE predicate's false-verdict boundary [banked: doctrine §3]. What gets HARDER than Zigbee, said honestly and completely: we lose single-artifact ground truth (wire → testimony), we gain stack-version conditioning on every number, and instrument-vs-device arbitration becomes a designed capability (second witness) instead of a free property of owning the coordinator.

**C2 — the provenance claim (the differentiator §3 proves available).**
